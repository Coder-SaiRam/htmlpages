GROUP BY & Aggregations
COUNT, SUM, AVG, MIN, MAX — computing stats per group
💡
Execution order: WHERE filters rows → GROUP BY groups them → HAVING filters groups. You cannot use COUNT() or SUM() inside WHERE. Use HAVING for that.
📊
Aggregate stats per department
headcount, average salary, min/max, total payroll — all in one query.
group_by.sqlSQL
SELECT
    d.dept_name,
    COUNT(e.emp_id)           headcount,
    ROUND(AVG(e.salary), 0)  avg_salary,
    MIN(e.salary)             min_sal,
    MAX(e.salary)             max_sal,
    SUM(e.salary)             total_payroll
FROM  employees  e
JOIN  departments d ON e.dept_id = d.dept_id
GROUP BY d.dept_name
HAVING   COUNT(e.emp_id) >= 2   -- filter groups, not rows
ORDER BY total_payroll DESC;
OUTPUT HR (0 employees) excluded by HAVING  ·  3 rows

CTE (WITH) — name your subqueries for readability
Compute dept averages first, then compare each employee against their own dept avg.
cte.sqlSQL
WITH dept_avg AS (
    SELECT dept_id, ROUND(AVG(salary), 0) avg_sal
    FROM   employees
    WHERE  dept_id IS NOT NULL
    GROUP BY dept_id
)
SELECT
    e.name, e.salary,
    da.avg_sal                             dept_avg,
    ROUND(e.salary - da.avg_sal, 0)      vs_dept_avg
FROM      employees e
JOIN      dept_avg  da ON e.dept_id = da.dept_id
ORDER BY vs_dept_avg DESC;

Recursive CTE — walk the org hierarchy tree
Anchor at top-level managers, recursively find their reports.
recursive_cte.sqlSQL
WITH RECURSIVE org AS (
    -- Anchor: start from employees with no manager
    SELECT emp_id, name, manager_id, 0 depth
    FROM   employees WHERE manager_id IS NULL
    AND    dept_id IS NOT NULL

    UNION ALL

    -- Recursive: add direct reports of current level
    SELECT e.emp_id, e.name, e.manager_id, o.depth+1
    FROM   employees e
    JOIN   org       o ON e.manager_id = o.emp_id
)
SELECT REPEAT('  ', depth) || name org_chart, depth
FROM  org ORDER BY emp_id;

Window Functions
ROW_NUMBER · RANK · DENSE_RANK · LAG · LEAD · Running Totals
💡
Key difference from GROUP BY: GROUP BY collapses 9 rows → 3 groups. Window functions keep all 9 rows AND add a computed column using a window of related rows. You cannot filter on window results in WHERE — wrap in a CTE first.
🏆
RANK vs DENSE_RANK vs ROW_NUMBER — side by side
All three rank salary per department. Watch how ties are handled.
ranking.sqlSQL
SELECT
    name, dept_id, salary,
    ROW_NUMBER() OVER (PARTITION BY dept_id ORDER BY salary DESC) row_num,
    RANK()       OVER (PARTITION BY dept_id ORDER BY salary DESC) rnk,
    DENSE_RANK() OVER (PARTITION BY dept_id ORDER BY salary DESC) dense_rnk
FROM   employees
WHERE  dept_id IS NOT NULL
ORDER BY dept_id, salary DESC;

Top-N per group — highest paid per department
Cannot filter window results in WHERE. Use CTE wrapper.
top_n_per_group.sqlSQL
WITH ranked AS (
    SELECT
        e.name, d.dept_name, e.salary,
        DENSE_RANK() OVER (
            PARTITION BY e.dept_id
            ORDER BY e.salary DESC
        ) rnk
    FROM  employees  e
    JOIN  departments d ON e.dept_id = d.dept_id
)
SELECT name, dept_name, salary, rnk
FROM   ranked
WHERE  rnk <= 2   -- top 2 earners per department
ORDER BY dept_name, rnk;

Modifying data — always use transactions for multi-step changes
✏️
INSERT, UPDATE, DELETE with RETURNING
RETURNING shows you exactly what changed — no separate SELECT needed.
dml.sqlSQL
-- INSERT a new employee
INSERT INTO employees (emp_id, name, dept_id, salary, joined)
VALUES (10, 'Jack', 10, 71000, '2024-01-10')
RETURNING *;

-- UPDATE: 10% raise for Engineering
UPDATE employees
SET    salary = ROUND(salary * 1.10)
WHERE  dept_id = 10
RETURNING name, salary AS new_salary;

-- UPSERT: insert or update on conflict
INSERT INTO employees (emp_id, name, dept_id, salary, joined)
VALUES (2, 'Bob', 10, 80000, '2020-07-01')
ON CONFLICT (emp_id) DO UPDATE SET salary = EXCLUDED.salary
RETURNING emp_id, name, salary;

Indexes
Sequential scan vs Index scan — visualised
⚡
How indexes speed up lookups
Without index: read every row. With index: jump directly. O(n) vs O(log n).
❌ No Index — Sequential Scan (reads ALL rows)
Row 1: emp_id=1 → not 7, continue...
Row 2: emp_id=2 → not 7, continue...
Row 3: emp_id=3 → not 7, continue...
Row 4: emp_id=4 → not 7, continue...
Row 5: emp_id=5 → not 7, continue...
Row 6: emp_id=6 → not 7, continue...
Row 7: emp_id=7 → ✓ FOUND!
Row 8: emp_id=8 → still must check
Row 9: emp_id=9 → done (read all 9)
9 reads. With 1M rows → 1,000,000 reads
✅ With B-Tree Index — Binary Search (log n)
Index tree: [1 2 3 4 5 6 7 8 9]
Check mid=5 → need higher half
Check mid=7 → exact match! Get row pointer
Jump directly to Row 7 ✓
Rows 1,2,3,4,5,6,8,9 never read
Rows 1,2,3,4,5,6,8,9 never read
Rows 1,2,3,4,5,6,8,9 never read
Rows 1,2,3,4,5,6,8,9 never read
∼3 reads. With 1M rows → only ∼20 reads!
indexes.sqlSQL
-- Basic index on frequently queried column
CREATE INDEX idx_dept    ON employees(dept_id);
CREATE INDEX idx_salary  ON employees(salary);

-- Composite: order matters! Supports WHERE dept_id=? AND salary>?
-- Also supports WHERE dept_id=? alone (leftmost prefix rule)
-- Does NOT help: WHERE salary>? alone (skips first column)
CREATE INDEX idx_dept_sal ON employees(dept_id, salary);

-- Partial: only index high earners — smaller and faster
CREATE INDEX idx_high_sal ON employees(salary)
    WHERE salary > 80000;

-- Covering: includes extra cols → no table row fetch needed
CREATE INDEX idx_covering ON employees(dept_id)
    INCLUDE (name, salary);

-- Check which plan is chosen
EXPLAIN ANALYZE
SELECT name, salary FROM employees WHERE dept_id = 10;

Transactions & ACID
Atomicity · Consistency · Isolation · Durability
🔒
Transfer budget between departments — atomically
If either UPDATE fails, both roll back. No partial state ever persists.
transaction.sqlSQL
BEGIN;  -- start transaction

    -- Step 1: Reduce Engineering budget by 50000
    UPDATE departments
    SET    budget = budget - 50000
    WHERE  dept_id = 10;

    -- Step 2: Add 50000 to Finance budget
    UPDATE departments
    SET    budget = budget + 50000
    WHERE  dept_id = 30;

    -- If Step 2 throws an error → ROLLBACK undoes Step 1 too

COMMIT;  -- persist both changes atomically

-- Verify
SELECT dept_name, budget FROM departments
WHERE  dept_id IN (10, 30);
OUTPUT After COMMIT — both changed or neither  ·  2 rows
dept_namebudget
Engineering450000
Finance500000
🚨
Rollback scenario: If the Finance UPDATE fails, the ENTIRE transaction rolls back. Engineering stays at 500000. No half-transfers, no inconsistent state.

Seq Scan vs Index Scan — what to look for
Run EXPLAIN ANALYZE before adding random indexes. Let the plan guide you.
❌ No index — Seq Scan
EXPLAIN ANALYZE

WHERE  salary > 85000;

CREATE INDEX idx_sal

⚡ SQL Execution Order

SQL runs in this order — NOT the order you write it
1
FROM / JOIN
Identify tables & build combined row set
↓
2
WHERE
Filter rows. No aliases or aggregates allowed here
↓
3
GROUP BY
Group filtered rows into buckets
↓
4
HAVING
Filter groups. Aggregates like COUNT() allowed here
↓
5
SELECT
Compute columns, aliases, window functions
↓
6
ORDER BY → LIMIT
Sort & trim. SELECT aliases usable here (evaluated after SELECT)

📌 Key Concepts

WHERE vs HAVING
WHERE filters rows before grouping. HAVING filters groups after. Never use COUNT() in WHERE.
IS NULL vs = NULL
= NULL always returns UNKNOWN (no rows). Always use IS NULL or IS NOT NULL.
INNER vs LEFT JOIN
INNER drops unmatched rows. LEFT keeps all left rows, NULLs on unmatched right.
GROUP BY collapses rows
9 rows → 3 groups. Window functions keep all 9 rows and add a computed column.
RANK gaps, DENSE_RANK doesn't
Ties: RANK gives 1,1,3. DENSE_RANK gives 1,1,2. Use DENSE_RANK for top-N filtering.
COALESCE(a, fallback)
Returns first non-NULL. COALESCE(dept_name, 'Unassigned') for safe NULL handling.
Covering Index
Holds all columns a query needs. DB never reads the table. Fastest possible read (index-only scan).
Recursive CTE
Anchor query + recursive part via UNION ALL. Classic use: org hierarchies, category trees.
ON CONFLICT (UPSERT)
INSERT ... ON CONFLICT (col) DO UPDATE SET ... Atomic insert-or-update. No race conditions.

❓ Interview Q&A

What happens when you use = NULL in a WHERE clause?
It always returns zero rows. NULL comparisons return UNKNOWN, not TRUE or FALSE. Use IS NULL or IS NOT NULL.
Why can't you use a SELECT alias in WHERE?
Because WHERE is evaluated before SELECT in execution order. The alias doesn't exist yet. Wrap in a CTE or subquery.
INNER JOIN vs LEFT JOIN — when to use which?
INNER when you only want matched rows. LEFT when you need all rows from the left table regardless of match — use COALESCE to handle NULLs on the right.
ROW_NUMBER vs RANK vs DENSE_RANK?
ROW_NUMBER: always unique (1,2,3,4). RANK: ties same rank, skips next (1,1,3). DENSE_RANK: ties same rank, no gap (1,1,2). Use DENSE_RANK for top-N per group.
When should you NOT add an index?
Small tables (<10K rows), low-cardinality columns (boolean, status with 2-3 values), heavily written columns. Every write pays an index update cost.
What does EXPLAIN ANALYZE tell you?
The plan the DB chose, how many rows were actually processed, and real timings. Look for Seq Scan on large tables. Compare estimated vs actual rows — big gap means stale stats, run ANALYZE.
