Advanced Window Functions
NTILE, PERCENT_RANK, CUME_DIST, FIRST_VALUE, NTH_VALUE, frame clauses
💡
Window function anatomy: fn() OVER (PARTITION BY col ORDER BY col ROWS/RANGE BETWEEN ... AND ...). All three parts are optional. The frame clause controls which rows are included in the calculation.
📊
NTILE, PERCENT_RANK, CUME_DIST — distribution functions
Divide customers into buckets, find relative position within a group.
distribution_fns.sqlPostgreSQL
SELECT
    c.name, c.segment,
    SUM(o.total_amt)                                         total_spend,
    NTILE(4) OVER (ORDER BY SUM(o.total_amt) DESC)          quartile,
    ROUND(PERCENT_RANK() OVER (ORDER BY SUM(o.total_amt) DESC)*100,1) pct_rank,
    ROUND(CUME_DIST()    OVER (ORDER BY SUM(o.total_amt)      )*100,1) cume_dist_pct
FROM      customers c
JOIN      orders    o ON c.cust_id = o.cust_id
WHERE     o.status = 'delivered'
GROUP BY c.cust_id, c.name, c.segment
ORDER BY total_spend DESC;

FIRST_VALUE, LAST_VALUE, NTH_VALUE — peek at specific rows in window
Show each order alongside the customer's first-ever order and best order.
first_last_nth.sqlPostgreSQL
SELECT
    c.name,
    o.order_id,
    o.order_date,
    o.total_amt,
    FIRST_VALUE(o.total_amt) OVER w                           first_order_amt,
    LAST_VALUE(o.total_amt)  OVER w                           last_order_amt,
    MAX(o.total_amt)         OVER (PARTITION BY o.cust_id)  best_order_ever
FROM  orders    o
JOIN  customers c ON o.cust_id = c.cust_id
WHERE o.cust_id = 1   -- Aditya Kumar (3 orders)
WINDOW w AS (
    PARTITION BY o.cust_id ORDER BY o.order_date
    ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
)  -- UNBOUNDED FOLLOWING needed for LAST_VALUE to see all rows!
ORDER BY o.order_date;

⚠️
LAST_VALUE gotcha: By default, the window frame is ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW. So LAST_VALUE only sees up to the current row, not the actual last row. Always specify ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING to get the true last value.

Moving Average & 3-Month Rolling Sum — Frame Clauses
ROWS vs RANGE frame clause: ROWS is physical row count, RANGE is value-based.
moving_avg.sqlPostgreSQL
-- Monthly revenue trend with 3-month moving average
WITH monthly AS (
    SELECT
        DATE_TRUNC('month', order_date) AS month,
        SUM(total_amt)               AS revenue
    FROM   orders
    WHERE  status = 'delivered'
    GROUP BY 1
)
SELECT
    month,
    revenue,
    SUM(revenue)  OVER (ORDER BY month
        ROWS BETWEEN 2 PRECEDING AND CURRENT ROW) rolling_3m,
    ROUND(AVG(revenue) OVER (ORDER BY month
        ROWS BETWEEN 2 PRECEDING AND CURRENT ROW)) moving_avg_3m
FROM  monthly
ORDER BY month;

Chained CTEs — pipeline of transformations
Each CTE builds on the previous one. Easier to debug than nested subqueries.
chained_cte.sqlPostgreSQL
-- Step 1: total spend per customer
-- Step 2: segment-level stats
-- Step 3: identify which segments beat the overall average
WITH
cust_spend AS (
    SELECT c.cust_id, c.name, c.segment,
           SUM(o.total_amt) spend
    FROM   customers c
    JOIN   orders    o ON c.cust_id = o.cust_id
    WHERE  o.status = 'delivered'
    GROUP BY c.cust_id, c.name, c.segment
),
seg_stats AS (
    SELECT segment,
           COUNT(*) cnt, SUM(spend) total, ROUND(AVG(spend),0) avg_spend
    FROM   cust_spend
    GROUP BY segment
),
overall AS (
    SELECT ROUND(AVG(spend),0) overall_avg FROM cust_spend
)
SELECT
    s.segment, s.cnt, s.avg_spend,
    o.overall_avg,
    CASE WHEN s.avg_spend > o.overall_avg THEN '⬆ Above avg'
         ELSE '⬇ Below avg' END vs_overall
FROM  seg_stats s, overall o
ORDER BY s.avg_spend DESC;

Recursive CTE — product category hierarchy + path
Walk a tree structure. Works for org charts, folder trees, category hierarchies.
recursive_cte.sqlPostgreSQL
-- categories table: cat_id, name, parent_id
-- Electronics > Mobile > Smartphones
--             > Laptop
-- Apparel     > Shoes > Sports
WITH RECURSIVE cat_tree AS (
    -- Anchor: root categories (no parent)
    SELECT cat_id, name, parent_id,
           0                        depth,
           name::text                 path
    FROM   categories
    WHERE  parent_id IS NULL

    UNION ALL

    -- Recursive: add children
    SELECT c.cat_id, c.name, c.parent_id,
           ct.depth + 1,
           ct.path || ' > ' || c.name
    FROM   categories  c
    JOIN   cat_tree    ct ON c.parent_id = ct.cat_id
)
SELECT
    REPEAT('  ', depth) || name  display,
    path,
    depth
FROM  cat_tree
ORDER BY path;

Gap Detection — find missing order IDs or date gaps
Classic interview problem: find gaps in a sequential series using LAG.
gap_detection.sqlPostgreSQL
-- Find gaps in order sequence (missing order_ids)
WITH gaps AS (
    SELECT
        order_id,
        LAG(order_id) OVER (ORDER BY order_id) prev_id
    FROM orders
)
SELECT
    prev_id + 1  gap_start,
    order_id - 1 gap_end,
    order_id - prev_id - 1 missing_count
FROM  gaps
WHERE order_id - prev_id > 1;   -- gap > 1 means IDs were skipped
OUTPUTGaps in order_id sequence · No gaps in our dataset — illustrative
gap_startgap_endmissing_count
0 rows — our orders 1001-1012 are consecutive. If order 1005 was deleted: gap_start=1005, gap_end=1005, missing_count=1

Table Partitioning
Range, List, Hash partitioning — splitting large tables for performance
💡
Why partition? When a table has 100M+ rows, queries on recent data still scan old data. Partitioning physically splits the table so queries only touch the relevant partition — called partition pruning.
🗂️
Range Partitioning by year — orders table at scale
Partition orders by year so queries for 2024 data never touch 2021-2023 partitions.
range_partition.sqlPostgreSQL
-- Parent table (no data stored here directly)
CREATE TABLE orders_partitioned (
    order_id   BIGSERIAL,
    cust_id    INT  NOT NULL,
    order_date DATE NOT NULL,
    status     TEXT,
    total_amt  NUMERIC(12,2)
) PARTITION BY RANGE (order_date);

-- Child partitions (one per year)
CREATE TABLE orders_2022 PARTITION OF orders_partitioned
    FOR VALUES FROM ('2022-01-01') TO ('2023-01-01');

CREATE TABLE orders_2023 PARTITION OF orders_partitioned
    FOR VALUES FROM ('2023-01-01') TO ('2024-01-01');

CREATE TABLE orders_2024 PARTITION OF orders_partitioned
    FOR VALUES FROM ('2024-01-01') TO ('2025-01-01');

-- This query ONLY scans orders_2024 — partition pruning!
EXPLAIN
SELECT * FROM orders_partitioned
WHERE  order_date >= '2024-01-01';
-- Plan shows: Seq Scan on orders_2024 (orders_2022, orders_2023 pruned)
Partition TypeWhen to useExample
RANGETime-series data, date/number rangesorders by year/month
LISTKnown, finite set of valuesorders by country/status
HASHEven distribution, no natural rangeusers by user_id % N
✅
Partition pruning requires the WHERE clause to filter on the partition key column directly. WHERE order_date::text LIKE '2024%' disables pruning (function on column). Use WHERE order_date >= '2024-01-01' AND order_date < '2025-01-01'.

5 Query Anti-Patterns with Fixes
Common patterns that silently kill performance.
❌ Anti-pattern 1: Function on indexed column
-- Index on order_date is IGNORED
-- DB computes YEAR() for every row first
SELECT * FROM orders
WHERE EXTRACT(YEAR FROM order_date) = 2023;
-- Plan: Seq Scan (full table)
✅ Fix: Use range condition
-- Index on order_date IS used
-- Sargable: condition directly on column
SELECT * FROM orders
WHERE  order_date >= '2023-01-01'
  AND  order_date <  '2024-01-01';
-- Plan: Index Scan on idx_order_date
❌ Anti-pattern 2: OFFSET pagination (gets worse at page 1000)
-- OFFSET 10000 scans & discards 10000 rows
-- Gets slower every page: O(n)
SELECT * FROM orders
ORDER BY order_id
LIMIT 20 OFFSET 10000;
-- Scans first 10020 rows just to return 20
✅ Fix: Keyset / Cursor pagination (always O(1))
-- Client sends last seen order_id as cursor
-- Index seek directly to that point: O(log n)
SELECT * FROM orders
WHERE  order_id > 1010  -- last seen cursor
ORDER BY order_id
LIMIT  20;
-- Index scan on order_id: always fast
⚠️
Anti-pattern 3: SELECT * — fetches unused columns, prevents index-only scans, increases network payload. Always list only the columns you need. Anti-pattern 4: NOT IN with NULLs — WHERE id NOT IN (1, 2, NULL) returns zero rows because NULL comparisons are UNKNOWN. Use NOT EXISTS instead.

Locking & MVCC
Row-level locks, deadlocks, isolation levels, optimistic vs pessimistic
🔒
Pessimistic Locking — FOR UPDATE, FOR SHARE, SKIP LOCKED
Lock rows at read time to prevent concurrent modification. Critical for inventory, payments.
locking.sqlPostgreSQL
-- Session A: lock the stock row before decrementing
BEGIN;
SELECT prod_id, stock
FROM   products
WHERE  prod_id = 101
FOR UPDATE;  -- acquires row-level exclusive lock

-- Session B runs same query → BLOCKS until A commits/rolls back

UPDATE products
SET    stock = stock - 1
WHERE  prod_id = 101;
COMMIT;

-- Job queue pattern: workers claim tasks without blocking each other
SELECT order_id, cust_id
FROM   orders
WHERE  status = 'pending'
ORDER BY order_id
LIMIT  1
FOR UPDATE SKIP LOCKED;  -- skip rows already locked by other workers
OUTPUTFOR UPDATE SKIP LOCKED — Worker 1 claims order 1010, Worker 2 claims 1012 simultaneously · 2 rows (one per worker)

Worker 110107pending

What dirty reads, non-repeatable reads, and phantom reads mean in practice.

-- Set isolation level for a transaction
BEGIN ISOLATION LEVEL REPEATABLE READ;

-- Non-repeatable read scenario (READ COMMITTED):
-- T1 reads Aditya spend = 194998
-- T2 inserts a new order for Aditya + COMMIT
-- T1 reads again → different result! (non-repeatable)

-- With REPEATABLE READ, T1 always sees its snapshot
-- T2's insert is invisible until T1 commits
COMMIT;
🚨
Deadlock prevention rule: Always acquire locks in the same order across all transactions. If T1 locks products then orders, T2 must also lock products then orders — never reverse. PostgreSQL detects deadlocks and kills one transaction automatically, but it causes errors and retries.

Trigger — auto-update updated_at & log changes
Triggers run automatically on INSERT/UPDATE/DELETE. Use for auditing and derived columns.
trigger.sqlPostgreSQL
-- Step 1: Create the trigger function
CREATE OR REPLACE FUNCTION update_timestamp()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = NOW();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Step 2: Attach trigger to orders table
CREATE TRIGGER trg_orders_updated
BEFORE UPDATE ON orders
FOR EACH ROW
EXECUTE FUNCTION update_timestamp();

-- Step 3: Any UPDATE now auto-sets updated_at
UPDATE orders
SET    status = 'delivered'
WHERE  order_id = 1010;
-- updated_at is now automatically set to NOW()
✅
BEFORE vs AFTER triggers: BEFORE triggers can modify the NEW row before it's saved (like our updated_at). AFTER triggers run post-commit so they can't modify the row but are safe for sending notifications or writing to audit logs.

JSON Queries
JSONB operators, indexing JSON, querying nested data
🧩
JSONB operators — extract, filter, update nested data
JSONB is binary-stored JSON with indexes. Much faster than JSON type for querying.
jsonb_queries.sqlPostgreSQL
-- products table has a JSONB column: metadata
-- {"brand": "Apple", "specs": {"ram": "8GB", "storage": "256GB"}}

-- Extract top-level key (>> returns text)
SELECT name, metadata->>'brand' brand
FROM   products
WHERE  metadata->>'brand' = 'Apple';

-- Extract nested key (-> returns JSONB, ->> returns text)
SELECT
    name,
    metadata->'specs'->>'ram'     ram,
    metadata->'specs'->>'storage' storage
FROM   products
WHERE  category = 'Electronics';

-- Check key exists (@> contains operator)
SELECT name FROM products
WHERE  metadata @> '{"brand": "Apple"}'::jsonb;

-- GIN index for fast @> queries
CREATE INDEX idx_prod_meta ON products USING GIN (metadata);

-- Update a nested JSON value (returns new JSONB)
UPDATE products
SET    metadata = jsonb_set(metadata, '{specs,ram}', '"16GB"')
WHERE  prod_id = 102;

Full-text search on product names — tsvector + tsquery
PostgreSQL built-in FTS: tokenise → index → rank. No Elasticsearch needed for moderate scale.
full_text_search.sqlPostgreSQL
-- Add a search vector column (computed from name + category)
ALTER TABLE products
    ADD COLUMN search_vec tsvector
    GENERATED ALWAYS AS (
        to_tsvector('english', name || ' ' || category)
    ) STORED;

-- GIN index for fast text search
CREATE INDEX idx_prod_search ON products USING GIN(search_vec);

-- Search: find products matching "book" OR "python"
SELECT
    name, category,
    ts_rank(search_vec, query) rank
FROM   products,
       to_tsquery('english', 'book | python') query
WHERE  search_vec @@ query
ORDER BY rank DESC;

Materialized Views
Pre-computed results, refresh strategies, when to use vs regular views
💾
Materialized View vs Regular View — the difference
Regular view: re-runs query every time. Materialized view: stores result on disk, must be refreshed.
👁️ Regular VIEW
• No storage — it's just a saved SQL query
• Always fresh — runs query every SELECT
• Slow for complex aggregations on large tables
• Cannot be indexed directly
💾 MATERIALIZED VIEW
• Stores result on disk like a table
• Must call REFRESH to update
• Fast reads — even for complex aggregations
• Can be indexed like a regular table
materialized_view.sqlPostgreSQL
-- Create materialized view: monthly sales summary
CREATE MATERIALIZED VIEW mv_monthly_sales AS
SELECT
    DATE_TRUNC('month', o.order_date)  month,
    d.dept_name                          category,   -- from products category
    COUNT(DISTINCT o.order_id)           orders,
    SUM(oi.qty * oi.unit_price)          revenue
FROM   orders      o
JOIN   order_items oi ON o.order_id = oi.order_id
JOIN   products    p  ON oi.prod_id = p.prod_id
WHERE  o.status = 'delivered'
GROUP BY 1, 2;

-- Add index on the materialized view
CREATE INDEX idx_mv_month ON mv_monthly_sales(month);

-- Refresh strategies:
-- FULL: locks table during refresh
REFRESH MATERIALIZED VIEW mv_monthly_sales;

-- CONCURRENTLY: no lock, needs unique index, slower
CREATE UNIQUE INDEX idx_mv_uniq ON mv_monthly_sales(month, category);
REFRESH MATERIALIZED VIEW CONCURRENTLY mv_monthly_sales;

-- Schedule via pg_cron (runs every day at 2am):
-- SELECT cron.schedule('refresh-mv', '0 2 * * *',
--   'REFRESH MATERIALIZED VIEW CONCURRENTLY mv_monthly_sales');
✅
Use Materialized Views for: dashboards, reports, aggregations over millions of rows where slight staleness is acceptable. Avoid for: data that must always be current (inventory, account balances).

Full syntax. All 3 parts optional.

Divide rows into n equal buckets (1=best)

❓ Senior-Level Q&A

What is MVCC and why does PostgreSQL use it?
Multi-Version Concurrency Control. Each transaction sees a snapshot of data at its start time. Writers don't block readers, readers don't block writers. PostgreSQL stores old row versions in the heap (dead tuples). VACUUM cleans them up. This enables high concurrency without read-write locks.
ROWS BETWEEN vs RANGE BETWEEN — what's the difference?
ROWS BETWEEN 2 PRECEDING: exactly 2 physical rows before. RANGE BETWEEN INTERVAL '7 days' PRECEDING: all rows within 7 days by value — could be 0 or 100 rows. Use ROWS for fixed-count windows, RANGE for value-based time windows.
When should you use a Materialized View vs a regular table?
Use a Materialized View when: (1) the query is complex and expensive, (2) slight data staleness is acceptable, (3) you want to index the result. Use a regular table when you need real-time accuracy or need to write to it directly.
What is partition pruning and when does it fail?
Partition pruning = the DB skips partitions that can't match the WHERE clause. It fails when: (1) you apply a function to the partition key (YEAR(order_date)), (2) you use an implicit type cast, (3) the WHERE clause references a non-partition column. Always filter directly on the partition key column.
Explain the N+1 query problem in SQL.
N+1 happens when code fetches N rows, then runs 1 additional query per row (often in a loop). Example: fetch 100 customers, then for each customer query their orders → 101 queries. Fix: use a JOIN or a single query with GROUP BY to get all data at once.
How do you safely drop an index on a production table?
Use DROP INDEX CONCURRENTLY idx_name. Normal DROP acquires an ACCESS EXCLUSIVE lock, blocking ALL queries. CONCURRENTLY takes only a weaker lock but takes longer. Test on a replica first and monitor query performance after dropping.
