Caching

The single biggest performance lever in system design — serve data from memory, not disk.

⚡

Why, Where & How

5 cache positions · 4 strategies · 3 failure modes

Cache stores frequently-read data in a faster layer — typically RAM — so expensive reads to disk or databases happen rarely. The fundamental driver is the latency gap: Redis reads at ~0.5ms while a cold DB query takes 10–50ms. At 1M requests/sec, that gap is the difference between 5 servers and 500.

Databases Deep Dive

SQL vs NoSQL is never about which is better — it's always about access patterns. Know when to use each and why.

🗃

SQL vs NoSQL, Indexing, Sharding, Replication

ACID (Atomicity, Consistency, Isolation, Durability) — all-or-nothing transactions, constraints enforced, no dirty reads, committed = permanent. BASE (Basically Available, Soft-state, Eventually consistent) — prefers availability over consistency, accepts temporary inconsistency for scalability.

Indexing: A B-Tree index on user_id turns a full-table scan (O(n)) into a tree traversal (O(log n)). Every index slows writes (must update index) and speeds reads. Composite indexes: (user_id, created_at) supports queries filtering by both. Covering index: index contains all queried columns — no table lookup needed.

Sharding splits your dataset horizontally across multiple DB instances. Range sharding: user_ids 1–1M on shard 1. Problem: hot shards. Hash sharding: shard = hash(user_id) % N. Problem: range queries. Directory sharding: lookup table maps keys to shards. Most flexible, but lookup table is a bottleneck. Twitter uses Snowflake IDs — the ID itself encodes the shard.

-- Why composite index order matters -- Query: SELECT * FROM orders WHERE user_id=5 AND status='paid' AND created_at > '2024-01-01' -- Good: (user_id, status, created_at) — uses all 3 cols CREATE INDEX idx_orders ON orders (user_id, status, created_at); -- Bad: (created_at, user_id) — can't use index for user_id filter first -- Rule: equality conditions first, range condition last -- Snowflake ID: 64 bits — encodes datacenter + worker + sequence -- Sortable by time, globally unique, no coordination needed tweet_id = timestamp(41 bits) | datacenter(5) | worker(5) | sequence(12)

Read replicas

  • Offload reads from primary
  • Async replication (lag: ms–sec)
  • Use for reporting/analytics
  • Rule: add when reads > 80%

Replication lag trap

  • "Read your own writes" bug
  • You post → read replica misses it
  • Fix: route your own reads to primary
  • Fix: sticky sessions per user

Interview answer

  • Start: "It depends on access patterns"
  • ACID needed? → PostgreSQL/MySQL
  • Write-heavy IoT? → Cassandra
  • Global ACID? → CockroachDB/Spanner

Load Balancing

Distributing traffic across servers to maximise throughput, minimise latency, and eliminate single points of failure.

⚖

L4 vs L7 · Algorithms · Health Checks · Sticky Sessions

HAProxy, Nginx, AWS ELB, Envoy — and when each is the right tool

Layer 4 (TCP/UDP) load balancers see IP packets and distribute based on source/destination IP and port. Fast — no HTTP parsing overhead. Layer 7 (HTTP) load balancers inspect the full HTTP request — can route based on URL path, headers, cookies, or request body. More powerful but slightly higher latency. Twitter uses a 2-tier model: L4 (HAProxy) → L7 (Nginx/Envoy).

Message Queues & Event Streaming

Decouple services, absorb traffic spikes, and enable async processing — the backbone of every large system.

📨

Kafka · RabbitMQ · SQS · Pub/Sub Patterns

Ordering guarantees · consumer groups · at-least-once delivery

A message queue decouples producers (publish events) from consumers (process them independently). The producer doesn't wait. This enables three critical properties: temporal decoupling (consumer can be down), rate decoupling (consumer processes at its own pace), and fan-out (N consumers each get every message independently).

Kafka deep dive: Messages are written to topics, which are split into partitions. Each partition is an ordered, immutable log. Consumer groups each get their own offset — multiple consumer groups can independently read the same topic. Partitions enable parallelism: N partitions = N consumers processing in parallel. Replication factor 3 = message survives 2 broker failures. LinkedIn invented Kafka and runs 7 trillion messages/day on it.

Queue benefits

  • Producer returns instantly (async)
  • Absorbs 10× traffic spikes
  • Services scale independently
  • Dead letter queues for failures
  • Replay events (Kafka 7-day log)

Queue tradeoffs

  • Eventual processing — no sync response
  • At-least-once delivery → need idempotency
  • Ordering only within partition
  • Consumer lag monitoring required

Idempotency is mandatory

  • Queues deliver at-least-once
  • Consumer may see duplicate messages
  • Fix: deduplicate by message ID in DB
  • Fix: use SQS FIFO (exactly-once)

Protecting your system from abuse, preventing cascading failures, and enforcing fair usage — a favourite standalone interview question.

🚦

  • API Gateway — easiest, centralised

  • Application code — most flexibility
  • CDN (Cloudflare) — before hitting origin

  • Redis Cluster: shard by user_id

  • Fix: Redis Lua script (atomic) per shard
  • Approximate: each node limits to N/shards

CDN & Edge Computing

Serving content from the nearest point-of-presence reduces latency by 10–100× and absorbs massive traffic with zero origin load.

🌍

How CDNs Work · CloudFront · Edge Functions · Cache Invalidation

450+ PoPs globally — every static asset should be CDN-served

A CDN (Content Delivery Network) operates a global network of Points of Presence (PoPs) — servers distributed across cities worldwide. When a user requests cdn.example.com/image.jpg, DNS resolves to the nearest PoP. If that PoP has the object cached, it responds directly (cache hit). If not, it fetches from the origin server (your S3 or EC2), caches it, and serves it. All subsequent requests for that object hit the PoP cache.

Cache invalidation is the hardest CDN problem. Options: (1) TTL expiry — content expires after max-age seconds. (2) Versioned URLs — app.abc123.js — changing filename bypasses all caches. Preferred for immutable assets. (3) CloudFront invalidation API — explicitly purge a path. Costs money, takes 15–30 seconds to propagate. (4) Surrogate keys (Fastly) — tag objects with a key, purge all objects with that tag in one API call.

CDN benefits

  • Serves users from nearest PoP (~5ms vs ~150ms)
  • Absorbs 99%+ of static traffic
  • DDoS protection at the edge
  • Infinite scalability for static content
  • HTTP/2 + TLS termination at edge

CDN caveats

  • Egress costs from CDN provider
  • Cache invalidation propagation delay
  • Dynamic content bypasses CDN
  • GDPR: PoPs store EU user data abroad

What to put on CDN

  • All static assets (JS, CSS, images, fonts)
  • Public API responses with Cache-Control header
  • User profile images, media files
  • NOT: authenticated or personalised responses

Consistent Hashing

How to distribute data across N servers so that adding or removing a server only moves 1/N of the keys — not all of them.

🔄

Virtual Nodes · Ring Hashing · Hotspot Prevention

Used by Cassandra, DynamoDB, Memcached, CDNs

With naive modular hashing (key % N), adding one server reshuffles ~N-1/N keys (almost everything). This causes a cache storm — every server misses, DB gets hammered. Consistent hashing places both servers and keys on a ring (0 to 2³²). Each key is served by the first server clockwise. Adding a server only moves keys between the new server and its predecessor — roughly 1/N of keys total.

Virtual nodes solve the hot-spot problem: without them, when one server is removed its entire load shifts to one neighbour. With virtual nodes, each physical server occupies V positions on the ring (V=100–200 is common). When a server is removed, its load spreads across all other servers proportionally. Cassandra uses 256 virtual nodes per server by default.

// Consistent hashing (simplified) const ring = new SortedMap(); // Place each server at V virtual node positions servers.forEach(server => { for (let i = 0; i < VIRTUAL_NODES; i++) { ring.set(hash(`${server}:${i}`), server); } }); // Route key: find first ring position >= hash(key) getServer(key) { const h = hash(key); return ring.firstEntryGte(h)?.value ?? ring.first().value; }

Why consistent hashing wins

  • Only 1/N keys remapped when adding server
  • Virtual nodes → even load distribution
  • No central lookup table required
  • Graceful scale-up and scale-down

Where it's used

  • Cassandra: data partitioning across nodes
  • DynamoDB: partition key routing
  • Redis Cluster: 16,384 hash slots
  • Memcached: client-side server selection
  • CDN: routing requests to edge servers

Circuit breaker pattern: When Service A calls Service B and B is slow/failing, A should stop trying after N failures and open the circuit — immediately returning a fallback response for a timeout period. This prevents A from queuing up thousands of pending B calls and exhausting its thread pool, which would cascade the failure to everything calling A. Hystrix (Netflix) popularised this. Resilience4j and Envoy implement it today.

Service discovery: With 100+ microservices, IP addresses change constantly. Services register themselves with a service registry (Consul, Eureka, AWS Cloud Map) on startup. Callers query the registry for the current healthy instances. Kubernetes handles this natively via DNS — payment-service.default.svc.cluster.local resolves to the current pods.

Service mesh (Istio/Envoy)

  • mTLS between every service (zero-trust)
  • Automatic retry + circuit breaking
  • Distributed tracing injection
  • Traffic splitting (canary deploys)
  • No application code changes needed

Saga pattern (distributed txns)

  • No 2PC across microservices (too slow)
  • Saga: sequence of local transactions
  • Each step publishes event for next step
  • Compensating transactions on failure
  • Choreography (events) vs Orchestration (central)

When to use microservices

  • Multiple teams need independent deploys
  • Services have very different scale needs
  • NOT: small team, early stage startup
  • Strangler fig: migrate monolith gradually

Search Systems

Inverted indexes, distributed search, autocomplete — why SQL LIKE is never the answer at scale.

🔍

Elasticsearch · Inverted Index · Autocomplete · Faceted Search

Earlybird (Twitter) · TF-IDF relevance · real-time indexing pipeline

A search engine stores an inverted index — mapping every token (word) to every document containing it. "alice" → [doc1, doc7, doc342]. A full-text query looks up each word's posting list, intersects them, and ranks results by TF-IDF (term frequency × inverse document frequency). This is O(k) where k is matching documents — far better than O(n) full table scan of SQL LIKE.

Autocomplete / typeahead: Two approaches. (1) Prefix trie in Redis: store all completions, query with ZRANGEBYLEX. O(log n + k) per query. Fast but memory-intensive for large vocabularies. (2) Elasticsearch prefix query: works well for millions of documents. Facebook uses tries with an n-gram inverted index. Twitter uses Earlybird — a custom Lucene fork with append-only in-memory indexing that makes tweets searchable within 5–15 seconds of posting.

Distributed Consensus

How distributed systems agree on a single truth — the problem at the heart of every strongly-consistent distributed database.

🗳

Raft · Paxos · Leader Election · ZooKeeper · etcd

CAP theorem · quorum · split-brain prevention

The consensus problem: In a network of N nodes where messages can be delayed or nodes can fail, how do they agree on a single value? This underpins everything from distributed databases to leader election to Kubernetes cluster state.

Raft (2013) was designed to be understandable where Paxos was notoriously complex. Three roles: Leader (handles all writes), Follower (replicates), Candidate (wants to become leader). Leader sends heartbeats; if followers don't hear from leader within election timeout, they become candidates and start an election. Candidate wins majority vote → becomes leader. All log entries replicate to majority before committing (quorum = floor(N/2)+1). Used by: etcd (Kubernetes), CockroachDB, TiKV, Consul.

Split-brain: Network partition splits 5 nodes into 2+3. Both halves think the other is dead. Without quorum rules, both could accept writes → data divergence. Raft prevents this by requiring quorum (3/5) for any decision. The minority partition (2 nodes) can't make progress — it simply stalls until the partition heals.

etcdRaft~10K writes/secKubernetes cluster state, service discovery

CAP theorem recap

  • During network Partition: choose C or A
  • C: return error if can't confirm latest
  • A: return stale data rather than error
  • CP: etcd, ZooKeeper, HBase
  • AP: Cassandra, DynamoDB, CouchDB

Interview insight

  • PACELC extends CAP: even without partition, tradeoff between latency (L) and consistency (C)
  • Spanner chose C+L via atomic clocks
  • Most real systems are tunable: Cassandra consistency levels

The three pillars of observability: Metrics (aggregated numbers over time — CPU 73%, error rate 0.1%), Logs (discrete events — "user X purchased item Y at time Z"), Traces (request journey across services — request 123 spent 5ms in auth, 40ms in payment, 2ms in DB). SLO/SLI/SLA: SLI (Service Level Indicator) = the actual metric (e.g., 99.2% requests succeed). SLO (Service Level Objective) = the target (e.g., 99.9% uptime). If you've burned your error budget for the month, you stop new feature work and fix reliability.

Security & Authentication

OAuth 2.0, JWTs, mTLS, secrets management, DDoS protection — security is increasingly expected in system design interviews.

🔒

Auth · Zero-Trust · Secrets Management · DDoS

OAuth 2.0 flows · JWT anatomy · Vault · WAF

Authentication (who are you?) vs Authorization (what can you do?). Modern systems use OAuth 2.0 for delegated authorization — users grant apps access without sharing passwords. The Authorization Code flow is the secure flow for web apps: user → auth server → code → app → token. PKCE (Proof Key for Code Exchange) extends this to mobile/SPA where a client secret can't be stored securely.

JWT (JSON Web Token) structure: header.payload.signature. Header: algorithm (RS256). Payload: claims (user_id, roles, exp). Signature: HMAC(header+payload, secret). Verification is stateless — any server with the public key can verify without a DB lookup. Downside: can't revoke before expiry. Fix: short expiry (15min) + refresh tokens stored in Redis.

Designing WhatsApp at scale: Each user maintains a persistent WebSocket connection to a chat server. When Alice sends a message to Bob: (1) Alice's device sends via WebSocket to her chat server. (2) Chat server persists message in Cassandra (partition key: conversation_id, clustering key: message_id timestamp). (3) Chat server looks up Bob's current chat server via Redis (user_id → server_id). (4) Routes message to Bob's server. (5) Bob's server pushes to Bob's WebSocket. (6) Delivery receipt flows back. If Bob is offline: message is queued in SQS; push notification sent via FCM/APNs.

1

Connection establishment

User connects via WebSocket. Server registers user_id → server_id mapping in Redis. Heartbeat every 30s to maintain presence.

2

Message send

Client sends message over WebSocket. Server assigns message_id (Snowflake), persists to Cassandra, acknowledges to sender.

3

Message delivery

Look up recipient's server_id in Redis. Route via internal pub/sub (Kafka topic per server or Redis pub/sub). Recipient server pushes to WebSocket.

4

Offline handling

If recipient offline: store in undelivered queue. On reconnect, fetch missed messages via Cassandra range query on conversation_id + last_seen_id.

5

Read receipts & presence

Read receipt: client → server → Kafka event → sender's server → WebSocket push. Presence: Redis TTL per user (expire = offline), refresh on activity.

Group chat fan-out

  • Group message → N recipients
  • Naive: N separate deliveries
  • Better: Kafka topic per group
  • Each member's server subscribes
  • Cap group size for fan-out feasibility

Scale numbers

  • WhatsApp: 100B messages/day
  • ~1M WebSocket connections per server
  • Each connection: ~1KB RAM (Go goroutine)
  • 100 servers = 100M concurrent users

System Design Patterns & Interview Framework

Universal patterns that appear across every system, and the 5-step framework that structures every interview answer.

🏗

Universal Patterns · Interview Template · Numbers Cheatsheet

CQRS · event sourcing · saga · strangler fig · the 5-step framework
↕

CQRS

Command Query Responsibility Segregation — separate read and write models. Writes go to a normalised SQL DB. Reads from a denormalised Cassandra or Elasticsearch view. Each optimised independently.

📜

Event Sourcing

Store all state changes as an immutable event log. Current state = replay of all events. Perfect audit trail. Can rebuild any derived view. Used by: accounting systems, CQRS write side, Kafka as source of truth.

🔄

Saga pattern

Distributed transaction across microservices without 2PC. Chain of local transactions; each publishes an event. On failure: compensating transactions roll back. Choreography (events) vs Orchestration (central saga).

🌱

Strangler fig

Migrate a monolith to microservices gradually. New features built as microservices. Existing monolith features migrated one by one. Route traffic via a facade. The monolith "strangles" as microservices grow around it.

🔌

Sidecar / Ambassador

Deploy a helper container alongside every app container. Handles cross-cutting concerns: logging, mTLS, retry logic, circuit breaking. Envoy in Istio is a sidecar. Keeps app code clean of infrastructure concerns.

📋

Outbox pattern

Write to DB and message queue atomically. Store pending events in an "outbox" table in the same DB transaction. A separate poller reads outbox and publishes to Kafka. Prevents dual-write consistency bugs.

The 5-step interview framework

1

Clarify requirements (5 min)

How many users? Read or write heavy? Latency SLA? Global or regional? Mobile clients? What's in scope vs out of scope?

2

Estimate scale (5 min)

DAU × actions ÷ 86,400 = QPS. Peak = avg × 3. Storage = rows × row_size × years. Bandwidth = QPS × payload size. Numbers anchor every decision.

3

High-level design (15 min)

Draw: Client → DNS → CDN → LB → App Servers → Cache → DB → Queue. Name specific technologies. Keep it simple first.

4

Deep dive on bottlenecks (15 min)

DB schema and indexes. Sharding key choice. Cache invalidation strategy. Fan-out approach. Failure modes. Let interviewer guide which area to deepen.

5

Discuss tradeoffs (5 min)

There is no perfect answer. "I chose X because Y, which means I accepted Z." Show you understand what you optimised for and what you gave up.
