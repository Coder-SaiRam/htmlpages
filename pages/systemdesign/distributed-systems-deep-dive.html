Section 01

Why Distribution Is Hard

A single machine is simple: one CPU, one memory, one disk — everything is synchronous. The moment you add a second machine, you introduce partial failures, network latency, and the impossibility of a global clock. This is the fundamental shift.

The Eight Fallacies of Distributed Computing

1. The network is reliable

2. Latency is zero

3. Bandwidth is infinite

4. The network is secure

5. Topology doesn't change

6. There is one administrator

7. Transport cost is zero

8. The network is homogeneous

Core Distributed System Problems

💀

Partial Failure

Some nodes crash, some work fine, some respond slowly. The system can't tell the difference between a dead node and a slow node. This is the "detection" problem — solved by timeouts, heartbeats, and gossip protocols.

🕐

No Global Clock

Two nodes can't agree on "what time it is now". Network Time Protocol gives ~1ms accuracy — but 1ms is an eternity at nanosecond processor speeds. Solution: Lamport clocks (logical ordering) and Vector clocks (causality tracking).

🌐

Network Partition

Two groups of nodes that can't talk to each other must independently decide what to do. Do they keep accepting writes (AP) and risk divergence? Or reject writes (CP) and sacrifice availability? This is the CAP dilemma.

🔄

Ordering & Causality

If Node A writes X=1, then Node B reads X and writes Y=X+1, Node C must apply these in the right order. Without coordination, C might see Y=X+1 before seeing X=1 — causality violation. Solved by causal consistency models.

⚡

Failover Process

Primary crashes. Patroni/Orchestrator detects via missed heartbeats (default: 3× 2s = 6s). Elects new primary from replicas — picks the one with highest LSN (most up-to-date WAL). Updates DNS/VIP to point to new primary. Old primary, if it recovers, must be demoted (fencing). Total: ~30–60 seconds downtime.

🔱

Split-Brain Problem

Network partition: Primary and Replica both think they're the primary. Both accept writes. When partition heals: conflicting data on both nodes. Solution: STONITH (Shoot The Other Node In The Head) — fencing ensures only ONE node can write. Tools: Patroni + etcd, MHA, Orchestrator.

📖

Read-Write Splitting

Application layer routes writes to primary, reads to replicas. Problem: read-your-writes violation. User writes a post, immediately reads timeline — hits replica, post not there yet. Solutions: (1) Read from primary for N seconds after write, (2) session stickiness to primary, (3) monotonic read: always route same session to same replica.

⚠️ The Replication Lag Trap — Real Production Bug

User on e-commerce: adds item to cart (write to primary), immediately sees cart (read from replica with 20ms lag). Cart appears empty. User adds item again. Now has 2 items. Charges twice. This is not theoretical — it happens at Instagram, Twitter scale constantly. Fix: sticky sessions (route same user to same replica), or read-your-writes (route to primary for 500ms after each write).

Section 04

Consistency Models — A Full Spectrum

Not all consistency is the same. There's a precise spectrum from strongest (linearizability) to weakest (eventual). Every distributed system must explicitly choose where it sits — and the choice determines what bugs are possible.

Strongest

Linearizability (Strict Consistency)

Every operation appears instantaneous and globally ordered. Once a write completes, every subsequent read on ANY node returns that value. Implemented by: Single-leader with synchronous replication, Zookeeper, etcd. Cost: High latency — every write must wait for all replicas. Used for: Distributed locks, leader election, configuration.

Strong

Sequential Consistency

All nodes see operations in the same order, but that order may differ from real-time wall clock. "There is A ordering, and everyone agrees on it." Weaker than linearizability — allows some reordering vs real time. Used by most CPUs' memory models.

Medium

Causal Consistency

Operations related by causality are seen in order. "If A happened before B, everyone sees A before B." Concurrent operations (no causal relationship) may be seen in any order. Used by: DynamoDB (with vector clocks), CockroachDB optional mode. Balance of availability and correctness.

Weak

Eventual Consistency

If no new updates, all replicas will EVENTUALLY converge to the same value. Zero guarantees about timing. Used by: Cassandra, DynamoDB default, Redis async replication, DNS. Highest availability — never blocks. Risk: reading stale data during convergence window.

Weakest

Read-Your-Writes / Monotonic Reads

Session-level guarantees. You always see your own writes. Reads never go backwards in time (if you saw version 5, you never see version 3 after). Not a global consistency guarantee — only per-session. Pragmatic middle ground used by most user-facing systems.

💡 Cassandra Quorum Math

N=3 replicas. Write to QUORUM (2/3). Read from QUORUM (2/3). At least 1 node in common → always reads latest write. Formula: R + W > N = strong consistency.

Strong: R=QUORUM(2), W=QUORUM(2), R+W=4 > N=3 ✅
Weak: R=ONE(1), W=ONE(1), R+W=2 ≤ N=3 ❌ may miss write
Performance cost: 2× network hops for QUORUM vs 1× for ONE.

Section 05

CAP Theorem — The Real Explanation

Consistency, Availability, Partition Tolerance — you can only guarantee 2 of 3 during a network partition. But the real insight is that partitions are INEVITABLE — so the real choice is between C and A.

CA systems (no partition tolerance) only work on a single machine or LAN where partitions are "impossible". In practice, all distributed systems must handle partitions — so CA is not a real distributed option.

What "Choosing CP vs AP" Actually Means

💥 The Partition Scenario

Nodes A and B cannot communicate. User writes X=10 to Node A. Another user reads from Node B.

CP choice: Node B refuses to serve reads (returns error). Zero stale reads. Sacrifices availability.
AP choice: Node B serves X=old_value (stale data). Always available. Sacrifices consistency.

SystemChoiceUnder Partition Behavior
etcd / ZookeeperCPMinority partition refuses reads/writes. Majority continues.
HBaseCPRegion goes offline, returns errors until healed.

CouchDBAPMulti-master, accepts divergent writes. MVCC on reconciliation.
MongoDB (primary)CPPrimary goes down → election → no writes during ~30s election.
DynamoDBAPEventual consistency default. Strong reads add CP guarantee.
Redis ClusterAPServes potentially stale data. Async replication = possible loss.
💡 PACELC — Better Than CAP

CAP only covers partition scenarios. PACELC covers normal operation too:

During Partition (P): choose A or C
Else (E) normal: choose Latency (L) or Consistency (C)

DynamoDB: PA/EL — prefers availability + low latency
CockroachDB: PC/EC — always consistent, accepts latency

Section 06

Distributed Transactions

The hardest problem in distributed systems: making multiple databases agree on a single atomic outcome. Three approaches — 2PC, 3PC, and Saga — each with different failure profiles.

Two-Phase Commit (2PC)

Ph1

Prepare Phase

Coordinator asks all participants: "Can you commit?" Each participant: acquires locks, writes to WAL, responds YES or NO. All must vote YES to proceed. If any votes NO → immediately send ROLLBACK to all.

Ph2

Commit Phase

If all YES: Coordinator sends COMMIT to all participants. Each participant commits, releases locks, acknowledges. If coordinator crashes HERE (after some committed, some not) → blocking problem. Participants wait with locks held until coordinator recovers.

💥 2PC Blocking Problem — The Fatal Flaw

Coordinator crashes after sending COMMIT to Participant A but before reaching Participant B. Participant B is stuck holding locks, waiting for coordinator to recover. Other transactions that need those locks also wait. System is blocked. Recovery: coordinator log + manual intervention. This is why 2PC is used sparingly.

2PC — Message Sequence
Coordinator → All Participants: PREPARE txn_id
Participant A → Coordinator:    VOTE YES  (locks held)
Participant B → Coordinator:    VOTE YES  (locks held)
Participant C → Coordinator:    VOTE NO   (constraint fail)

-- ANY NO → abort entire transaction:
Coordinator → A,B,C:           ROLLBACK txn_id
-- All release locks

-- ALL YES → commit:
Coordinator → All:             COMMIT txn_id
Coordinator CRASHES HERE ← blocking problem if partial commit
-- Participant B waits indefinitely...

Saga Pattern — The Modern Alternative

Break a long transaction into a sequence of local transactions, each with a compensating transaction (rollback action). No locks held between steps — much better availability.

T1

Book Seat (Seat DB)

Compensating: Release seat lock. If this fails, nothing to undo.

T2

Deduct Inventory (Inventory DB)

Compensating: Add back to inventory. If T3 fails → run C2 → C1.

T3

Charge Payment (Payment DB)

Compensating: Issue refund. If T4 fails → run C3 → C2 → C1.

T4

Send Confirmation (Notification)

Compensating: Send cancellation email. If this fails → run C4 → C3 → C2 → C1.

💡 Choreography vs Orchestration Saga

Choreography: Each service publishes event → next service reacts. Decentralized. Hard to debug (no single flow view).

Orchestration: Central saga orchestrator sends commands to services. Easy to track state. Services don't need to know about each other. Used by Temporal, AWS Step Functions.

3️⃣

Three-Phase Commit (3PC)

Adds a PRE-COMMIT phase between PREPARE and COMMIT. Allows coordinator to timeout and abort if it crashes mid-commit. Solves the blocking problem of 2PC. Cost: 3× network round trips. Used rarely — Saga is more practical for microservices.

🔑

Distributed Locking (Redlock)

For coordination across services: Redlock algorithm acquires locks on N Redis nodes (N=5 typical). Lock valid if acquired on majority (⌊N/2⌋+1 = 3) in less than TTL time. Clock drift and GC pauses make this tricky — Martin Kleppmann's critique is worth reading before using in production.

📋

Outbox Pattern

Write to local DB + outbox table in ONE transaction. Separate process polls outbox table, publishes to Kafka, marks as sent. Guarantees exactly-once event emission without distributed transaction. Used by Stripe, Shopify, Debezium CDC to reliably publish events.

Consensus Algorithms — Raft Deep Dive

How distributed nodes agree on a single value despite failures. Raft is the readable alternative to Paxos — used by etcd, CockroachDB, TiKV, and Consul.

Raft: Three Roles

LEADER
Receives all writes
Sends heartbeats
←→
FOLLOWER
Receives log entries
Votes in elections
←→
CAND.
Transitional state
Requests votes
01

Leader Election

Follower times out (150–300ms random), becomes Candidate, increments term, votes for itself, requests votes from others. First to get majority (⌊N/2⌋+1) wins. Random timeout prevents split votes.

02

Log Replication

Client sends write to Leader. Leader appends to local log, sends AppendEntries RPC to all Followers. Follower appends and acknowledges. When majority ack: Leader commits, applies to state machine, returns to client.

03

Safety

Raft ensures at-most-one leader per term. Log matching property: if two entries have same index and term, all preceding entries are identical. Candidate must have log at least as up-to-date as voter's log to win election.

Raft Log Entry Flow

Raft — Write Commit Sequence
-- Client: SET key="balance" value=100
-- All requests go to LEADER

1. Leader appends to local log:
   Log[5] = {term:3, cmd:"SET balance 100"}

2. Leader sends AppendEntries to all followers:
   AppendEntries{
     term: 3,
     leaderId: "node-1",
     prevLogIndex: 4,
     prevLogTerm: 3,
     entries: [{term:3, cmd:"SET balance 100"}],
     leaderCommit: 4  -- last committed index
   }

3. Followers append, reply Success=true
4. Leader receives majority ack (3/5 nodes):
   → commits log[5], updates commitIndex=5
   → applies to state machine: balance=100
   → responds to client: OK

5. Followers learn of commit via next heartbeat:
   leaderCommit=5 → followers apply log[5] too

Section 08

Cache Consistency — Across All Instances

Your application has 50 pods. Each pod reads from Redis, but Redis has 6 nodes in a cluster. Your PostgreSQL has 3 replicas. Getting all of these to agree on "what is the current value of X" requires careful cache invalidation strategies.

Cache Invalidation Strategies

1

Cache-Aside (Lazy Loading)

App reads from cache. Miss → read from DB → write to cache → return. Write: update DB, DELETE (not update) cache key. Next read re-populates. Problem: Cache stampede on popular keys. Fix: Mutex lock on miss (one thread fetches, others wait).

2

Write-Through

Write hits BOTH cache and DB synchronously. Cache always has fresh data. Problem: Extra write latency. Hot keys may be written frequently but rarely read. Fix: Use TTL to eventually evict cold keys.

3

Write-Behind (Write-Back)

Write to cache only. Async flush to DB. Very fast writes. Problem: Data loss if cache crashes before flush. Used for: metrics counters, analytics, view counts where some loss is OK.

4

Read-Through

Cache sits in front of DB. On miss: cache fetches from DB itself (not the app). App always talks only to cache. Simplifies app code. Used by: AWS ElastiCache with DAX, Memcached front-end to RDS.

Cache Consistency Problems & Solutions

Cache Inconsistency Scenarios
-- RACE CONDITION 1: Stale write
T1: App A writes user.balance=100 → DB
T2: App B reads user.balance → cache HIT → returns 50 (stale!)
-- Gap between DB write and cache invalidation
-- Fix: Write to DB + immediately DELETE cache key (not update!)
-- Next read: cache miss → fetches fresh 100 from DB

-- RACE CONDITION 2: Double delete
T1: App A: writes balance=100, deletes cache key
T2: App B: reads (miss), queries DB gets 50 (replication lag!)
T2: App B: writes 50 to cache
-- Cache now has 50, DB has 100 → inconsistent!
-- Fix: Cache-Aside with short TTL (5-30s) as safety net

-- CACHE STAMPEDE: Popular key expires
WATCH cache_key
IF (cache miss) -- thousands of requests simultaneously
  SET mutex_key NX EX 10  -- only ONE gets this
  IF (got mutex):
    value = db.query()
    cache.set(cache_key, value, ttl=300)
    cache.delete(mutex_key)
  ELSE:
    sleep(50ms) + retry  -- wait for winner to populate
🔬 Redis Cluster Consistency

Redis Cluster shards keys across 16,384 hash slots. Each slot assigned to a master node. 3 masters, 3 replicas. Problem: Redis uses ASYNC replication. If master crashes before replica receives write → data lost. Redis WAIT command: WAIT 1 100 — wait for 1 replica to ack within 100ms. This makes specific writes synchronous but not the whole cluster.

Global Infrastructure — Where Everything Lives

A real-world picture of how application servers, databases, and caches are distributed across regions and cities — how data flows between them, where replicas live, and what "local" vs "global" actually means for your users in Bangalore, New York, or Frankfurt.

🌍 Global Region Layout — 3 Primary Regions

Zoomed In — Single Region (Mumbai)

What a single AWS region actually looks like — how a request from a user in Bangalore flows through CDN, load balancer, app servers, cache, and database across two availability zones.

How a Request Travels — Bangalore User

1

User in Bangalore → CDN Edge

DNS resolves to nearest Cloudflare/Akamai PoP (Bangalore PoP). Static assets (HTML, JS, CSS, images) served directly. ~5ms latency. 80% of traffic handled here — DB never touched.

2

Cache Miss → Route53 Geo-DNS

Dynamic request forwarded to AWS Route53. Geo-routing detects user is in India → routes to ap-south-1 (Mumbai). A user in London would route to eu-central-1 (Frankfurt) instead.

3

Mumbai ALB → App Server

Application Load Balancer uses round-robin + health checks across 4 app servers in two AZs. Request lands on app-mum-01 (AZ-A) or any available pod.

4

App Server → Redis (cache check)

App checks Redis master (AZ-A) first. Hit → returns in ~1ms. Miss → goes to PostgreSQL. Cache key format: user:{id}:profile

5

App → PgBouncer → DB

Writes → PgBouncer (AZ-A) → Postgres Primary (AZ-A, Powai DC). Reads → PgBouncer read-pool (AZ-B) → Postgres Replica (AZ-B, Navi Mumbai). Replica lags ~0ms (sync replication within region).

6

WAL stream → AZ-B Replica + DR

Every DB write streams via PostgreSQL WAL to AZ-B replica synchronously (zero lag, within region). The Singapore DR replica receives the same WAL asynchronously (~20ms cross-ocean lag). If entire Mumbai region fails → Singapore promoted to primary in ~60s.

💡 Why Two Availability Zones?

AZ-A and AZ-B are physically separate data centres in the same city — different power grids, different network backbones, different cooling. If AZ-A (Powai) loses power or floods, AZ-B (Navi Mumbai) keeps running with no data loss. The ALB automatically stops routing to unhealthy AZs. This is why "high availability" means at minimum 2 AZs — not just 2 servers in the same rack.

⚠️ What Happens When Mumbai Goes Down Completely

T+0s: Mumbai region loses power / network.
T+10s: Route53 health checks detect ap-south-1 ALB unreachable.
T+30s: Route53 DNS TTL (30s) expires. New DNS responses route to Singapore DR.
T+45s: Singapore Postgres async replica promoted to primary (DBA/automated via Patroni).
T+60s: All new traffic hits Singapore. ~20ms of writes potentially lost (async lag window).
T+2h: Mumbai restored. Becomes new replica. Syncs from Singapore primary. Eventually fails back.

This is why financial systems use synchronous cross-region replication (zero data loss, higher latency) and most web apps use async DR (small data loss window, lower normal-operation latency).

⚠️ The Hidden Danger — Replication Slot Overflow

Primary buffers missed WAL in a replication slot. If replica stays down too long, this slot fills your PRIMARY disk and crashes it too. Fix: set max_slot_wal_keep_size = 10GB in PostgreSQL. Once exceeded, slot is dropped — replica must do full base backup to rejoin.

Writes committed on the old primary that hadn't streamed to any replica are lost forever. Amount lost = replication lag at moment of crash. Intra-region (sync replica): 0ms lost. Cross-region async replica: up to 90ms of writes lost.

🎯 The Three Rules That Prevent All of These

Rule 1 — Always set max_slot_wal_keep_size. Without it, a down replica fills your primary disk and takes down the whole system. Set it to something reasonable (10–50GB) so slots get dropped before they kill the primary.

Rule 2 — Never use synchronous_commit=on without a runbook. It gives you zero data loss but requires a DBA to manually intervene when the sync replica dies. If nobody is awake at 3AM, every write hangs until morning.

Rule 3 — Monitor replication lag, not just uptime. A replica that is up but 5 minutes behind is more dangerous than a crashed replica — it serves confidently wrong data. Alert when lag exceeds 500ms. Page on-call when it exceeds 5 seconds.

Section 12

Distributed System Patterns

Battle-tested solutions to recurring distributed systems problems. Each pattern addresses a specific failure mode or coordination challenge.

💓

Heartbeat + Failure Detection

Every node sends heartbeat every N seconds to all neighbors. If no heartbeat received in 3N seconds → node marked suspect → 6N → declared dead. Phi Accrual Detector (Cassandra): continuous failure probability based on arrival time statistics. More accurate than binary dead/alive.

⏱️

Logical Clocks

Lamport Clock: Increment on each local event. Send max(own, received)+1 on message. Orders events but can't detect concurrent events.

Vector Clock: Array of counters, one per node. Detects true causality and concurrency. Used by DynamoDB, Riak for conflict detection.

📡

Gossip Protocol

Epidemic-style information spread. Each node randomly shares state with N neighbors every T seconds. State propagates in O(log N) rounds. Eventually all nodes know all states. Used by: Cassandra (cluster membership), Redis Cluster (topology), Consul (service discovery). Fault-tolerant by design.

🔢

CRDT (Conflict-free Replicated Data Type)

Data structures that can be updated concurrently on any node and merged automatically. No coordination needed. Types: G-Counter (add only), PN-Counter (add/remove), G-Set, OR-Set, LWW-Register. Used by: Redis (sets), Riak (data types), Figma (collaborative editing).

📋

Event Sourcing

Don't store current state — store all events that led to it. Replay events to reconstruct state. Benefits: complete audit log, time-travel debugging, rebuild any read model. Used by: Kafka + event-driven architectures, Axon Framework, EventStoreDB. Trade-off: eventual consistency in derived views.

🏠

Circuit Breaker

Wraps remote calls. State machine: CLOSED (normal) → OPEN (too many failures, fail fast) → HALF-OPEN (test one request). Prevents cascade failures. Used by: Netflix Hystrix, Resilience4j, Istio service mesh. Configuration: error threshold 50%, window 60s, sleep 30s.
