The Three Concepts Explained

Forward proxy, reverse proxy, and load balancer are often confused. Here is exactly what each one is, who it serves, and where it lives in the network.

🗺

Side-by-side comparison

Who is "in the middle" and whose identity is hidden?
Forward Proxy
Sits in front of the client
Client → Forward Proxy → Internet
Server sees the proxy's IP, not the client's.

Use: corporate firewalls, VPNs, anonymity, content filtering
Reverse Proxy
Sits in front of the server
Client → Reverse Proxy → Server
Client sees the proxy's IP, not the server's.

Use: SSL termination, caching, hide server topology
Load Balancer
A specialised reverse proxy
Client → Load Balancer → Server 1/2/3
Distributes requests across multiple servers.

Use: scalability, high availability, zero-downtime deploys
// the key mental model A forward proxy hides the client from the server. A reverse proxy hides the server from the client. A load balancer is a reverse proxy that distributes load across multiple servers. Every load balancer is a reverse proxy, but not every reverse proxy is a load balancer.

Forward Proxy

Acts on behalf of the client — the server never knows who the real client is. Your company's internet filter is a forward proxy.

🕵

How a Forward Proxy Works — With Real IPs

Step-by-step packet flow · DNS · IP masking · corporate use case

In a forward proxy setup, the client is configured to route all outbound traffic through the proxy. The proxy makes the request on behalf of the client and returns the response. From the origin server's perspective, the connection came from the proxy IP, not the client's real IP.

Packet-level walkthrough
1

Client sends request to proxy

The browser is configured with proxy 10.0.0.1:3128. It sends an HTTP CONNECT or GET request to the proxy, not directly to google.com.

SRC: 192.168.1.45:54821 → DST: 10.0.0.1:3128 GET http://google.com/ HTTP/1.1 Host: google.com
2

Proxy checks policy (allow/deny)

Corporate proxy checks ACL rules. google.com → ALLOWED. youtube.com → BLOCKED (entertainment). Proxy logs: user 192.168.1.45 requested google.com at 14:23:07.

3

Proxy opens NEW connection to origin

The proxy makes a fresh TCP connection to google.com. The original client TCP connection is separate. Google sees a connection from the proxy's public IP only.

SRC: 203.0.113.50:41200 → DST: 172.217.16.46:443 GET / HTTP/1.1 Host: google.com
4

Response flows back through proxy

Google responds to the proxy. The proxy receives the response, optionally caches it, and forwards it back to the original client 192.168.1.45.

Corporate use cases

  • Block social media during work hours
  • Log all employee internet activity
  • Scan downloads for malware
  • Cache frequently-visited pages
  • Single outbound IP for firewall rules

Privacy / VPN use cases

  • Hide real IP from websites
  • Bypass geo-restricted content
  • Anonymise traffic (Tor is multi-hop proxy)
  • Access region-locked APIs
  • Security research anonymity

HTTPS and CONNECT tunnels

  • For HTTPS, browser sends HTTP CONNECT
  • Proxy creates TCP tunnel (doesn't decrypt)
  • TLS handshake happens end-to-end
  • SSL inspection: proxy is a MITM (corp only)
  • Requires installing corp CA cert on devices

Real tools

  • Squid — most common, open source
  • Nginx (forward proxy mode)
  • Shadowsocks — encrypted proxy
  • SOCKS5 proxy — protocol-agnostic
  • Cloudflare WARP, VPNs
// HTTPS connect tunnel — what happens at the packet level Client: CONNECT google.com:443 HTTP/1.1
Proxy: 200 Connection Established
After this, the proxy is a blind TCP pipe. The TLS handshake and all encrypted traffic passes through without the proxy seeing the plaintext. The proxy only knows the domain name from the CONNECT request — not the URL path, headers, or body.

Reverse Proxy

Sits in front of your servers — clients think they're talking to your app but they're actually talking to the proxy. Nginx and Cloudflare are reverse proxies.

🔀

Reverse Proxy — With Real IPs & Full Request Flow

SSL termination · caching · compression · header rewriting · security

A reverse proxy intercepts all inbound requests before they reach your application servers. Clients have no idea your real servers exist — they communicate only with the proxy's IP. This gives you a single, clean entry point where you can handle SSL, authentication, caching, rate limiting, and routing in one place.

Production topology — api.myapp.com example
User Browser
98.12.44.201
→
Nginx Reverse Proxy
api.myapp.com → 52.10.8.3
SSL ends here ✓
→
App Server 1 · 10.0.1.10:8080
App Server 2 · 10.0.1.11:8080
App Server 3 · 10.0.1.12:8080
App servers have private IPs only. They are NOT reachable from the internet. Only the proxy's IP (52.10.8.3) is public.
Full request lifecycle — packet by packet
1

Client sends HTTPS request

DNS resolves api.myapp.com → 52.10.8.3. Client opens TLS connection to 52.10.8.3:443.

SRC: 98.12.44.201:52300→ DST: 52.10.8.3:443 GET /api/users/42 Authorization: Bearer eyJ...
2

Proxy performs SSL termination

The TLS session ends at the proxy. It decrypts the request using its TLS certificate. This is why you only need ONE SSL certificate on the proxy, not on every app server.

3

Proxy modifies and forwards request

Proxy adds X-Forwarded-For header so app servers know the real client IP. Sends request to one of the upstream app servers over plain HTTP on the internal network.

SRC: 10.0.0.1:45010→ DST: 10.0.1.11:8080 GET /api/users/42 HTTP/1.1 X-Forwarded-For: 98.12.44.201 X-Real-IP: 98.12.44.201
4

App server responds to proxy

App server processes the request and responds. It doesn't know anything about TLS, SSL certificates, or the real client IP (unless reading X-Forwarded-For).

SRC: 10.0.1.11:8080→ DST: 10.0.0.1:45010 200 OK {"id":42,"name":"Alice"}
5

Proxy re-encrypts and returns to client

Proxy may add response headers (CORS, security headers, cache-control). Re-encrypts with TLS and sends back to the original client over the still-open TLS session.

🔐

SSL Termination

Handle TLS at the proxy, use plain HTTP internally. Only ONE certificate needed. Reduces CPU load on app servers. Internal traffic stays encrypted (mTLS) if needed.

Most common use
⚡

Response Caching

Proxy caches frequently-returned responses. GET /api/products served from cache — app server never receives the request. Nginx proxy_cache_valid 200 10m.

Performance
🗜

Gzip Compression

Proxy compresses all responses before sending to client. A 500KB JSON response compresses to ~40KB. App servers don't need compression logic. gzip on; gzip_types application/json

Bandwidth saving
🛡

Security Layer

Block malicious IPs, rate limit at edge, apply WAF rules, strip sensitive headers. App servers are completely shielded — no public IP, no direct access possible.

Security
🔀

Path-based Routing

/api/* → API servers. /static/* → S3/CDN. /admin/* → admin service. One domain, multiple backend services, split by URL prefix.

Microservices
📊

Observability

Log every request: IP, method, URL, response code, duration. Emit metrics. One place to see all traffic. App servers don't need logging infrastructure.

Operations

Load Balancer

A specialised reverse proxy that distributes requests across multiple servers — the foundation of horizontal scalability.

⚖

Complete Load Balancer Architecture With IPs

Redundant LB pair · upstream pool · health checks · session persistence

A load balancer sits between clients and a pool of backend servers. Every incoming request is routed to one server in the pool based on the chosen algorithm. When a server dies, the LB detects this via health checks and stops sending traffic to it. When a new server starts, the LB adds it automatically. The client always connects to the same single public IP — the LB's VIP (Virtual IP).

// VIP and VRRP — how HA load balancers work The VIP (Virtual IP) 52.10.8.3 is not tied to a single physical machine. It floats between LB1 and LB2. VRRP (Virtual Router Redundancy Protocol) runs between LB1 and LB2. LB1 broadcasts heartbeats every second. If LB2 misses 3 heartbeats, it promotes itself to ACTIVE and takes over the VIP. This failover takes <1 second — clients experience a brief pause but connections recover automatically.

Why use a load balancer

  • Scale horizontally — add servers freely
  • Zero-downtime rolling deploys
  • Auto-remove failed servers
  • SSL offloading from app servers
  • Single entry point for rate limiting

LB as a SPOF?

  • Without HA: yes — LB dies, everything dies
  • Fix: active-passive pair with VRRP
  • Fix: AWS ELB (multi-AZ by default)
  • Fix: Anycast DNS routes around failures
  • The LB pair itself has no SPOF above it

Stateless is the golden rule

  • If servers hold session state → sticky sessions needed
  • Sticky sessions break horizontal scale
  • Solution: move sessions to Redis
  • Then any server can handle any request
  • This is the most important LB principle

Load Balancing Algorithms

Six algorithms — each optimised for a different traffic pattern. Choosing the wrong one causes hot servers and wasted capacity.

🎯

All 6 Algorithms — With Worked IP Examples

Round-robin · weighted · least connections · IP hash · random · consistent hash
Algorithm 1
Round Robin

Each new request goes to the next server in sequence. Simple and perfectly even — but ignores server capacity and current load. Request 1 → S1, Request 2 → S2, Request 3 → S3, Request 4 → S1, and so on.

Request routing — 6 requests, 3 servers:
Req #198.1.2.3:55001→10.0.1.10 (S1)
Req #2102.5.6.7:44200→10.0.1.11 (S2)
Req #355.22.33.44:61000→10.0.1.12 (S3)
Req #477.8.9.10:50220→10.0.1.10 (S1) ← wraps around
Req #5190.4.5.6:33410→10.0.1.11 (S2)
Req #612.3.4.5:28800→10.0.1.12 (S3)
Result: S1=2, S2=2, S3=2 requests — perfectly even ✓
Best for: Homogeneous servers with similar request durations  |  Problem: S1 has 8GB RAM, S2 has 32GB — both get equal load
Algorithm 2
Weighted Round Robin

Assign a weight to each server proportional to its capacity. A server with weight 4 gets 4× more requests than a server with weight 1. Powerful for heterogeneous fleets where new high-capacity servers should serve more traffic.

10 requests, weights: S1=1, S2=2, S3=3 (total weight=6):
R1→S1 R2→S2 R3→S2 R4→S3 R5→S3 R6→S3 R7→S1 R8→S2 R9→S2 R10→S3
S1: 16.7% · S2: 33.3% · S3: 50% — proportional to capacity ✓
Algorithm 3
Least Connections

Route each new request to the server with the fewest active connections. Ideal when requests have variable durations — long DB queries vs instant cache hits. Dynamically balances load in real time as connections open and close.

Current state — a slow DB query is holding connections on S1:
S1 · 10.0.1.10
80 conns
S2 · 10.0.1.11
30 conns
S3 · 10.0.1.12
15 conns
→ New request arrives. Round-robin would send to S1. Least-conn sends to S3 (15 connections) — the server with lightest load.
Algorithm 4
IP Hash (Sticky Sessions)

hash(client_IP) % N — the same client IP always maps to the same server. Creates "stickiness" without cookies. Required when servers hold in-memory session state. Problem: fails silently if a server dies (that client's session is lost) and causes imbalance behind NAT.

hash(98.12.44.201) % 3 = 1 → always → 10.0.1.10 (S1)
hash(142.250.80.100) % 3 = 2 → always → 10.0.1.11 (S2)
hash(203.45.67.89) % 3 = 0 → always → 10.0.1.12 (S3)
⚠ Problem: 50% of corporate clients behind NAT 10.0.0.1 → ALL hash to same server!

Layer 4 vs Layer 7 Load Balancing

L4 sees TCP packets. L7 sees the full HTTP request. Each is faster or more capable in different ways.

🧱

Deep Comparison With Packet-Level Examples

When L4 is better · when L7 is essential · two-tier production setup
L4 — Transport Layer
Sees: SRC IP, DST IP, SRC port, DST port, TCP flags
Cannot see: HTTP method, URL, headers, cookies

Decision: hash(SRC_IP) % N or round-robin on connections

Speed: Wire-speed — millions of packets/sec
Examples: HAProxy TCP mode, AWS NLB, LVS
L7 — Application Layer
Sees: Full HTTP request — method, URL, headers, body, cookies
Can do: Path routing, header-based routing, A/B testing, auth

Decision: if path.startsWith('/api') → api_pool

Speed: ~100K requests/sec (parse + route overhead)
Examples: Nginx, HAProxy HTTP, AWS ALB, Envoy
What each layer sees — same request, different views
L4 view (only TCP/IP headers):
SRC: 98.12.44.201:52300
DST: 52.10.8.3:443
TCP SYN · Seq: 0x4a2f
✗ Cannot see: URL, method, headers, cookies
L7 view (full HTTP request):
GET /api/products?cat=shoes
Host: api.myapp.com
Cookie: session=abc123; ab_group=B
Authorization: Bearer eyJhbGci...
✓ Route: /api/* → api_pool · A/B: group B → new_backend
// production pattern: two-tier load balancing Large systems use L4 at the edge (handles millions of connections at wire speed, terminates DDoS) and L7 behind it (does smart routing, auth, A/B testing). Twitter, Facebook, and Netflix all use this two-tier pattern. L4 (HAProxy) → L7 (Envoy/Nginx) → application servers.

Health Checks & Failure Handling

How the load balancer detects dead servers, drains connections gracefully, and restores traffic when a server recovers.

💓

Active Checks · Passive Checks · Circuit Breaker · Graceful Drain

The exact state machine that governs server health transitions

A load balancer only helps if it stops sending traffic to dead servers. Active health checks probe each server at regular intervals. Passive health checks watch real traffic responses — if a server returns 5xx errors, it's marked unhealthy. Most production systems use both.

# Nginx upstream health check configuration upstream app_servers { least_conn; # Algorithm: fewest active connections server 10.0.1.10:8080 weight=3 max_fails=3 fail_timeout=30s; server 10.0.1.11:8080 weight=3 max_fails=3 fail_timeout=30s; server 10.0.1.12:8080 weight=3 max_fails=3 fail_timeout=30s; server 10.0.1.13:8080 backup; # Only used if all primaries are down } # HAProxy active HTTP health check backend app_pool balance leastconn option httpchk GET /health HTTP/1.1\r\nHost:\ localhost http-check expect status 200 # Must return 200 OK default-server inter 5s fall 3 rise 2 # Every 5s · 3 fails=down · 2 pass=up server s1 10.0.1.10:8080 check server s2 10.0.1.11:8080 check server s3 10.0.1.12:8080 check
1

Active health check runs every 5 seconds

LB sends GET /health HTTP/1.1 to each server. Expects 200 OK within 2s. Your /health endpoint checks DB connection, Redis ping, disk space — returns 200 only if all dependencies are healthy.

2

Server 10.0.1.11 starts failing

Health check gets a 503 at T+0, T+5, T+10. After 3 consecutive failures (fall=3), the server is marked DOWN. The LB stops routing new requests to it. Active connections are allowed to finish.

3

Traffic redistributes to remaining servers

S1 and S3 now share all traffic. With least-conn, load distributes proportionally. The backup server 10.0.1.13 is NOT activated because 2 of 3 primary servers are still healthy.

4

Server recovers — gradual traffic restoration

Health check passes 2× in a row (rise=2). Server re-enters rotation. LB starts sending a small percentage of traffic to verify stability before ramping up to full capacity.

5

Zero-downtime deploy — graceful drain

Before deploying to S1: send DRAIN signal. LB stops sending new connections to S1 but lets active connections finish (connection draining). After all connections close (typically 30s), deploy, then re-enable.

Full Request Journey — 12 Hops

Corporate employee accessing SaaS app — forward proxy + reverse proxy + load balancer all in one scenario
1

User types app.mycompany.com in Chrome

Employee laptop IP: 192.168.10.55. Browser is configured to use corporate proxy 10.1.0.1:3128. Sends HTTP CONNECT to proxy.

SRC: 192.168.10.55→ DST: 10.1.0.1:3128 (Forward Proxy) CONNECT app.mycompany.com:443 HTTP/1.1
2

Corporate forward proxy allows and tunnels

Squid proxy: app.mycompany.com → ALLOWED (whitelist). Creates TCP tunnel. Replies 200 Connection Established. Employee's TLS handshake now flows through the proxy tunnel to the internet.

3

DNS resolves app.mycompany.com → Cloudflare Anycast IP

DNS query returns 104.26.15.200 — a Cloudflare IP. Because of Anycast routing, this routes to the nearest Cloudflare PoP (Mumbai datacenter for this user). The traffic never leaves the continent until it must.

4

Cloudflare acts as reverse proxy (CDN edge)

TLS terminates at Cloudflare. Serves from CDN cache if possible. Applies DDoS protection, WAF rules, rate limiting. For this dynamic API request: cache miss → forwards to origin.

SRC: 104.26.15.200 (Cloudflare)→ DST: 203.0.113.10:443 (Origin) CF-Connecting-IP: 203.0.113.50 CF-Ray: 7f2a3b4c5d6e
5

L4 Load Balancer (HAProxy) — AWS NLB

Origin IP 203.0.113.10 is the NLB's VIP. L4 LB distributes based on least connections — routes to the L7 LB with fewest active TCP connections.

6

L7 Load Balancer (Nginx) — path routing

Nginx reads the URL. /api/* → routes to api_pool. /static/* → routes to S3 via proxy_pass. /admin/* → routes to admin_pool + verifies admin JWT. All on private network 10.0.0.0/8.

GET /api/v2/dashboard → DST: 10.0.2.15:8080 (API Server) X-Forwarded-For: 192.168.10.55 X-Real-IP: 192.168.10.55
7

API Server processes the request

App server 10.0.2.15 receives plain HTTP. Reads JWT from Authorization header, validates against Redis token store. Queries PostgreSQL on 10.0.3.10:5432. Returns JSON response.

8

Response travels back through every layer

App → Nginx (adds security headers, gzip compresses) → HAProxy → Cloudflare (adds CF headers, caches for next request) → Forward Proxy (logs the response) → Employee browser.

200 OK Content-Encoding: gzip X-Response-Time: 42ms Strict-Transport-Security: max-age=31536000
