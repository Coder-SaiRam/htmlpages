Section 01

The Big Picture

Every system design question asks the same thing: how do you scale to millions without it collapsing?

00

Full System Architecture

How a request travels from your phone to a database and back
When a user opens an app, a request flows through several layers: DNS resolves the domain to an IP, a CDN serves cached assets from the nearest edge node, a Load Balancer distributes traffic across servers, App Servers handle business logic, Cache answers common reads instantly, and the Database persists everything. Understanding this flow is the foundation of every design interview.
Real-World Example — Instagram

When you open Instagram: your phone hits CloudFront (CDN) for your feed images, Route 53 (DNS) resolves api.instagram.com, an AWS ELB routes your request to one of thousands of app servers, Memcached serves your friend count, and Cassandra holds the post data. Your like triggers a Kafka event that fans out notifications asynchronously.

Section 02

Scalability

Growing the system without rewriting it — the first question every interviewer asks.

01

Horizontal vs Vertical Scaling

The foundation of scalable architecture
Load Balancing Stateless Design Auto-scaling
Vertical scaling (scale-up) means upgrading the machine — more CPU, more RAM. Simple but has a hard ceiling: there's only one machine, so it's a single point of failure, and the biggest cloud instances are exponentially expensive.

Horizontal scaling (scale-out) adds more machines. Requires stateless servers (no session data stored locally) and a load balancer. The key insight: stateless means any server can handle any request, so you can spin up 1 or 1000 without changing code.

Horizontal Pros

  • No single point of failure
  • Near-infinite scale potential
  • Cheaper commodity hardware
  • Rolling deployments possible
  • Auto-scaling with cloud

Horizontal Cons

  • Requires stateless design
  • Network complexity increases
  • Distributed consistency harder
  • Data sharding challenges
Real-World — Netflix

Netflix runs 100,000+ AWS EC2 instances. When a new season of Stranger Things drops, traffic spikes 10×. AWS Auto Scaling Groups spin up hundreds of new stateless streaming servers within minutes. No single server holds any user session — everything is pulled from a shared cache (EVCache, Netflix's Redis fork).

Always say "stateless app servers" before adding a load balancer. Interviewers want to know you understand WHY horizontal scaling works — because no request depends on hitting the same server twice.
Section 03

Caching

The single biggest performance win — serve data from memory, not disk.

02

Caching Strategies

Cache-aside, Write-through, Write-back — and when to use each
Redis Memcached LRU / TTL Cache Hit Rate
A cache sits in front of your database and serves hot data from memory (microseconds) instead of disk (milliseconds). The cache hit rate — percentage of reads served from cache — is the key metric. A 99% hit rate means only 1% of reads touch the database. Three main strategies differ in when and how cache gets updated.

Caching Pros

  • Reduces DB load by 99%+
  • Sub-millisecond response times
  • Absorbs read traffic spikes
  • Reduces infrastructure cost

Caching Cons

  • Cache invalidation is hard
  • Stale data risk
  • Cache stampede on cold start
  • Extra infrastructure to manage
Real-World — Twitter

Twitter uses Twemcache (a fork of Memcached) and Redis extensively. Your home timeline is pre-computed and cached for active users — when someone you follow tweets, a fan-out service pushes the tweet ID into each follower's cached timeline. Lady Gaga has 50M followers, so Twitter uses a hybrid: cache for normal users, compute-on-read for celebrity tweets.

The hardest problem in caching is invalidation — knowing when to remove stale data. The phrase "there are only two hard things in computer science: cache invalidation and naming things" is practically a meme in interviews. Always discuss your TTL strategy and how you handle stale reads.
When asked about caching, mention the cache stampede problem: when the cache expires, 10,000 requests simultaneously hit the DB. Solutions: staggered TTLs, mutex locks, or background refresh.
Section 04

Databases

SQL vs NoSQL is never about which is better — it's about which fits your access patterns.

03

SQL vs NoSQL + Sharding

Picking the right storage engine — and scaling it
PostgreSQL Cassandra MongoDB DynamoDB Sharding
Relational (SQL) databases enforce ACID transactions and are exceptional at complex JOINs and structured data. NoSQL databases sacrifice strict consistency for horizontal scalability and flexible schemas. Neither is universally better — your read/write patterns and consistency requirements dictate the choice.
FeatureSQL (Postgres)NoSQL (Cassandra)
ConsistencyACID — strongEventual (tunable)
SchemaFixed, structuredFlexible, dynamic
JoinsNative, powerfulNo — denormalize data
Write scaleVertical + read replicasLinear horizontal scale
Best forOrders, finance, authIoT, time-series, feeds
ExamplesPostgreSQL, MySQL, SQLiteCassandra, DynamoDB, MongoDB
Sharding splits your database horizontally — user IDs 0–10M on shard 1, 10M–20M on shard 2. The shard key choice is critical: a bad key (like user creation date) creates hot spots. A good key (like hashed user ID) distributes evenly. Resharding is painful — plan ahead.

Read Replicas

  • Offload read traffic from primary
  • Zero downtime for read queries
  • Async replication = tiny lag
  • Geographic distribution

Replication Lag

  • Read-after-write inconsistency
  • "I just posted but can't see it"
  • Sticky sessions as workaround
  • Monitoring replication lag critical
Real-World — Uber

Uber migrated from PostgreSQL to MySQL then to Schemaless (their custom NoSQL on MySQL) for trips data. Their driver location updates — 1 million writes/second — go into Apache Kafka → HBase (NoSQL). Financial trip data (exact fare, payment) stays in MySQL with ACID transactions. The split-brain approach: NoSQL for high-write telemetry, SQL for money.

When asked "SQL or NoSQL?", never say NoSQL is more scalable. Say: "If I need ACID transactions and complex queries, SQL. If I need massive write throughput with flexible schema, NoSQL. For most systems I'd start with PostgreSQL and add NoSQL only when needed."
Section 05

CAP Theorem

The fundamental constraint of distributed systems — you can only guarantee two of three.

04

Consistency, Availability, Partition Tolerance

The theorem that shapes every distributed database decision
CP Systems AP Systems Network Partitions
In any distributed system, network partitions (communication failures between nodes) are inevitable. When they happen, you must choose: do you return a potentially stale value (choose Availability), or do you block until you can guarantee the latest value (choose Consistency)? There is no system that can be both strongly consistent and always available during a partition.

CP (pick Consistency)

  • Every read gets latest data
  • No dirty reads possible
  • HBase, Zookeeper, Spanner
  • Banking, order management

AP (pick Availability)

  • Always responds (maybe stale)
  • Higher uptime, lower latency
  • Cassandra, DynamoDB, CouchDB
  • Social feeds, product catalog
Real-World — Amazon Shopping Cart

Amazon's Dynamo paper (2007) chose AP for the shopping cart — it's better to show a slightly stale cart than fail the request. If two updates conflict (you added on mobile, partner added on laptop), Dynamo uses vector clocks and last-write-wins to resolve conflicts. Amazon would rather lose a conflict than show a 503 error to a customer about to buy something.

The modern framing is PACELC — extending CAP to cover the latency vs consistency tradeoff even when there's no partition. Most production systems tune consistency levels per-operation: "quorum reads" for strong consistency on payment confirmations, "eventual reads" for profile views.
Section 06

Message Queues

Decouple services, absorb traffic spikes, and enable async processing at scale.

05

Async Architecture with Queues

Kafka, RabbitMQ, SQS — when and why to introduce a queue
Kafka RabbitMQ AWS SQS Fan-out
A message queue sits between a producer (publishes events) and one or more consumers (process them independently). The producer doesn't wait for consumers — it fires and forgets. This decoupling lets each service scale independently and protects downstream services from traffic bursts.

Message Queue Pros

  • Decouples producer from consumer
  • Absorbs write traffic bursts
  • Retry logic on consumer failure
  • Fan-out to multiple consumers
  • Event sourcing / audit trail

Message Queue Cons

  • Added infrastructure complexity
  • Message ordering hard at scale
  • Duplicate message handling needed
  • Eventual processing = no sync response
Real-World — LinkedIn (Kafka's birthplace)

LinkedIn invented Apache Kafka to handle their activity stream — every page view, profile view, connection request. At peak, Kafka at LinkedIn processes 7 trillion messages/day. Each event is published once and consumed by multiple services in parallel: analytics pipeline, recommendation engine, notification service, activity feed — all independently, without coupling.

Introduce a queue whenever you hear "async", "burst traffic", "decouple", or "at least once delivery". Also mention idempotency — since queues can deliver messages more than once, consumers must handle duplicates gracefully.
Section 07

Interview Framework

Structure your answer and you're 80% of the way there before drawing a single box.

06

The 5-Step Answer Template

Used at Google, Meta, Amazon, and every FAANG interview
The framework is about managing time. You have 45 minutes. Spend 5 on requirements, 5 on estimation, 15 on high-level design, 15 on deep dive, and 5 on tradeoffs. Interviewers don't want a perfect answer — they want to see how you think through ambiguity.
Example — "Design a URL Shortener" (bit.ly)

Clarify: 100M URLs/day, reads 10× writes, URLs never change. Estimate: 1000 writes/sec, 10K reads/sec. 100M × 365 × 5 years × 500 bytes = ~80TB. Design: API server → hash function (base62 7-char) → Redis cache → MySQL. Deep dive: hash collision handling, custom aliases, expiry. Tradeoff: MD5 vs counter vs random ID — counter is predictable, random needs collision check.
