Why Caching Exists

The fundamental speed gap between memory and disk is what makes caching not just useful — but essential.

⏱

The Latency Hierarchy

Why memory beats disk by 1,000× — and why that gap dictates architecture

Every computer system has a storage hierarchy. The closer data is to the CPU, the faster it can be accessed — but the more expensive and smaller the storage. Caching exploits this by keeping hot data (frequently accessed) in a faster layer so expensive reads to slower layers happen rarely.

// key insight A Redis read is 100× faster than a typical DB query and 300× faster than a cold disk read. At 1M requests/second, the difference between cache hit and DB hit is the difference between 6 servers and 600 servers.

What caching solves

  • Eliminates repeated expensive computation
  • Reduces DB read load by 90–99%
  • Absorbs traffic spikes
  • Cuts infrastructure costs
  • Enables sub-50ms API responses

What caching introduces

  • Stale data (consistency tradeoff)
  • Cache invalidation complexity
  • Cache stampede on cold start
  • Extra infrastructure to manage
  • Memory cost

The 80/20 rule of caching

  • 80% of reads hit 20% of data
  • Cache that 20% → most reads served fast
  • Known as the "hot key" principle
  • Most systems benefit immediately

Caching Strategies

How data gets into the cache, how it gets updated, and how reads and writes flow through the system.

📖

Read Strategies

How reads are served — the cache-aside pattern is the most widely used

Cache-aside pros

  • Simple to implement
  • Cache failures don't break reads
  • Only caches what's actually read
  • Most widely used pattern

Cache-aside cons

  • First read after miss is slow
  • Cache stampede risk
  • Data can be stale until TTL expires
  • Application owns cache logic

When to use

  • Read-heavy workloads (10:1+)
  • Data that changes infrequently
  • You want cache failure resilience
  • Starting a new system — simplest

Eviction Policies

When the cache is full, something must go. The policy you choose directly affects your hit rate.

🗑

LRU, LFU, TTL, and Beyond

Choosing the wrong eviction policy can cut your hit rate by 30%+

Eviction policy determines which entry gets removed when the cache is full and a new entry needs to be stored. The right choice depends entirely on your access pattern — is data accessed in bursts (LFU better), accessed recently (LRU better), or valid for a fixed time (TTL better)?

🕐

LRU — Least Recently Used

Evicts the entry that hasn't been accessed for the longest time. Assumes recent access predicts future access. Implemented as a doubly-linked list + hashmap.

Most Common
📊

LFU — Least Frequently Used

Evicts the entry accessed the fewest times overall. Better for stable "hot item" access patterns. More complex: needs frequency counter per key.

Popularity-based
⏰

TTL — Time to Live

Entry expires after a fixed duration regardless of access. Guarantees freshness. Often combined with LRU: evict expired first, then LRU among the rest.

Freshness guarantee
🎲

Random Replacement

Evicts a random entry. Surprisingly competitive with LRU in practice, and much cheaper to implement. Used in some CPU cache designs.

Simple
🔢

FIFO — First In First Out

Evicts the oldest-inserted entry. No regard for access frequency. Useful when data has inherent temporal ordering and older data is naturally less relevant.

Queue-like
🏆

W-TinyLFU (Caffeine)

Combines LRU for new items (warm them up) with LFU for popular items. The best practical policy — used in Caffeine, Guava, and Redis 4.0+. Beats LRU by 30–80% on real workloads.

State of the Art
// interview key point Redis uses approximated LRU by default (samples N random keys and evicts the least-recently used among them). This is cheaper than a true LRU doubly-linked list and performs nearly as well. Redis 4.0+ added allkeys-lfu for frequency-based eviction.

Critical Patterns & Failure Modes

The problems that kill production systems — and exactly how to defend against each one.

⚡

Cache Stampede (Thundering Herd)

When a cache key expires and 10,000 requests simultaneously hammer the database

A popular key expires. Before any server can repopulate it, thousands of simultaneous requests all miss the cache and all send queries to the database. The DB is hit with N× its normal load, potentially collapsing under the pressure — and taking down the entire service.

1

Key expires at T+0

Redis key for "trending:global" expires after its TTL. 50,000 users are currently viewing trending topics.

2

All 50,000 requests miss

Every server simultaneously checks Redis, gets a cache miss, and decides to query the database to repopulate.

3

DB collapses under 50,000 queries

The database receives 50,000 identical slow queries at once. Response time shoots from 2ms to 40 seconds. Cascading failure begins.

Fix 1: Mutex / lock

  • Only 1 server acquires DB lock
  • Others wait or serve stale data
  • Use Redis SET NX (atomic lock)
  • Release lock after repopulation

Fix 2: Probabilistic early expiry

  • Before TTL expires, random % refresh
  • No simultaneous expiry spike
  • Formula: refresh if rand() < β·log(ttl·remaining)
  • Used by Twitter, Facebook

Fix 3: TTL jitter

  • Add random ±20% to all TTLs
  • Prevents synchronized mass expiry
  • TTL = base_ttl + random(0, jitter)
  • Simplest fix — do this always

Fix 4: Background refresh

  • Dedicated worker refreshes before expiry
  • Key never actually "misses"
  • Requires knowing access patterns
  • Best for high-traffic stable keys
// Fix: Redis mutex pattern (SET NX) async function getWithStampedePrevention(key) { let val = await redis.get(key); if (val) return JSON.parse(val); // Try to acquire mutex (atomic, expires in 5s) const lock = await redis.set(`lock:${key}`, 1, 'NX', 'EX', 5); if (!lock) { // Another server has the lock — wait briefly and retry await sleep(50); return getWithStampedePrevention(key); } // We have the lock — query DB and repopulate const data = await db.query(...); const jitteredTTL = 3600 + Math.floor(Math.random() * 600); await redis.setex(key, jitteredTTL, JSON.stringify(data)); return data; }
💣

Cache Penetration & Cache Avalanche

Two more production killers you must know cold for interviews

Cache Penetration

  • Queries for keys that DON'T exist
  • Every query hits DB (cache never helps)
  • Common DDoS vector: fake IDs
  • Fix: cache null results with short TTL
  • Fix: Bloom filter at the gate

Cache Avalanche

  • Large chunk of cache expires at once
  • All expire simultaneously → mass DB hits
  • Often from same TTL on batch-loaded data
  • Fix: jitter TTLs by ±20%
  • Fix: stagger bulk cache warming

Hot Key Problem

  • 1 key gets millions of reads/sec
  • Single Redis shard overwhelmed
  • Fix: replicate hot key to N shards
  • Fix: local in-process L1 cache
  • Fix: read from replica shards
// bloom filter — cache penetration defense A Bloom filter is a probabilistic data structure that answers "does key X possibly exist?" in O(1) with zero false negatives (if it says NO, the key definitely doesn't exist — skip the DB). At 1% false positive rate, a Bloom filter for 1 billion keys needs only ~1.2 GB of RAM vs TBs for the actual data. Twitter, LinkedIn, and Cassandra all use Bloom filters at scale.
// Bloom filter + cache null pattern (combined defense) async function getUser(userId) { // Gate 1: Bloom filter (very fast, RAM-only) if (!bloomFilter.mightContain(userId)) { return null; // Definitely doesn't exist — skip cache + DB } // Gate 2: Cache lookup const cached = await redis.get(`user:${userId}`); if (cached === 'NULL') return null; // Cached null result if (cached) return JSON.parse(cached); // Gate 3: DB query const user = await db.findUser(userId); if (!user) { await redis.setex(`user:${userId}`, 300, 'NULL'); // Cache the miss return null; } await redis.setex(`user:${userId}`, 3600, JSON.stringify(user)); return user; }

When to Use Cache — Decision Guide

Cache is not always the answer. Here's the complete decision framework.

🎯

Use Cache When...

These conditions signal that caching will have high impact
🔄

High read:write ratio

If the same data is read 10× more than it's written, caching provides compounding benefit. User profiles (read millions of times, updated rarely) are perfect cache candidates.

Rule: >10:1 read:write
💰

Expensive computation

If computing the result takes significant CPU or time (ML inference, complex aggregations, report generation), cache the result. Compute once, serve many.

Rule: compute > 50ms
🌍

Geographic distribution

CDN edge caches serve content from the nearest point-of-presence. A user in Mumbai shouldn't wait for a response from a US data center for static content.

Rule: multi-region users
📈

Traffic spikes

Cache absorbs sudden traffic bursts. A news article going viral sends 100K requests/sec for the same data — a single cache entry handles all of them.

Rule: unpredictable spikes
⚡

Low latency SLA

If you need sub-100ms responses, you cannot afford DB round trips for every request. Cache is mandatory once your latency target is tighter than what your DB can provide.

Rule: SLA < 100ms
💵

Cost reduction

DB reads cost money (AWS RDS charges per I/O). If 95% of reads hit cache instead of the DB, your database bill drops by 95%. At scale, Redis pays for itself within days.

Rule: DB I/O costs high

Do NOT cache when...

  • Data changes every request (real-time prices)
  • Strong consistency is mandatory (bank balances)
  • Data is unique per user with no sharing
  • Cache size exceeds benefit (tiny traffic)
  • Compliance requires no data replication

Cache with caution when...

  • Stale data has real consequences
  • Invalidation logic is complex
  • Data is user-specific (privacy concerns)
  • Write rate approaches read rate
  • Data size exceeds available memory

Cache Invalidation — The Hard Problem

"There are only two hard things in CS: cache invalidation and naming things." — Phil Karlton

🧨

Invalidation Strategies

How to keep cache and database in sync — each approach has distinct tradeoffs

Cache invalidation is hard because you must decide: when does cached data become too stale to serve? Too aggressive invalidation → high miss rate, defeats the purpose. Too lenient → users see wrong data. The right strategy depends on your consistency requirements.

StrategyHow it worksConsistencyComplexityReal-world use
TTL expiry Cache entry auto-expires after fixed duration Eventual (up to TTL stale) Very low Most common — profiles, feeds, config
Event-driven invalidation On write → publish event → consumers delete cache keys Near-instant Medium (pub/sub infra) Twitter, Facebook, product catalogs
Write-through invalidation On write → update both cache and DB atomically Strong Medium Financial data, inventory
Version-based keys Key = "user:123:v5" — increment version on change Strong High (version tracking) Immutable URLs, CDN assets
CDC (Change Data Capture) Monitor DB binlog → auto-invalidate on any change Near-instant High (Debezium/Kafka) Large-scale systems needing strong consistency
// cache aside race condition — the classic bug Thread A reads key (miss) → Thread B writes new value → Thread B deletes cache → Thread A stores old value in cache. Now cache has stale data indefinitely until TTL expires. Fix: delete cache after DB write (not before), and use short TTLs as a safety net even with event-driven invalidation.

Choose Redis when

  • Need sorted sets (leaderboards)
  • Pub/sub messaging
  • Lua scripting / atomic ops
  • Persistence matters
  • 90% of production systems

Choose Memcached when

  • Simple string cache only
  • Multi-threaded (better CPU use)
  • Memory per item is critical
  • No persistence needed
  • Very high throughput, simple keys

Use both (L1 + L2) when

  • Caffeine (in-process) → Redis cluster
  • Sub-ms for hottest keys
  • <1ms for shared data
  • Used by Twitter, Netflix
  • Dramatically cuts Redis load
