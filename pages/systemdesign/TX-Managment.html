🎯 The Core Principle — State Is the Only Problem

Every concurrency problem in this doc — double debit, dirty read, stale replica, lost update — happens because something has state. The router has no state: hash("rahul@upi") gives the same answer on all 100 routers independently. The database has state: balance=₹1000. That's where all the coordination mechanisms live.

🐘 DB Replicas   ← copied state, repl lag

Stateless = Scale to Any Number

10 routers or 1000 — they all compute hash("rahul@upi") → Node 1. Same answer, zero coordination.

⚡ MVCC + WAL — How Reads and Replicas Work Simultaneously

MVCC — Readers Don't Block

PostgreSQL keeps multiple row versions. Writer creates a new version. Reader sees the version from their snapshot time. They never block each other. Ananya reading balance and Sai writing balance run in parallel — Ananya sees the committed value at her query start time.

WAL — How Replicas Sync

Every write to DB Primary generates a WAL (Write-Ahead Log) entry. This streams to replicas. Replica-1 (same AZ, sync) applies it in ~1ms — zero lag. Replica-2 (cross-region, async) applies it in ~90ms. During that 90ms window, reads from Replica-2 return the old value.

Optimistic Lock Retry

Optimistic locking uses a version column (v=42). Write includes WHERE version=42. If someone else already wrote (version is now 43), you get 0 rows updated. You detect this, re-read the fresh balance, re-check, and retry. No blocking — conflict detected at commit time.

🎯 DB Is the Single Truth Point

It doesn't matter if you have 2 app nodes or 1000. They all write to the same primary DB. The DB engine serializes via row-level locks. SELECT FOR UPDATE works across all app nodes simultaneously — no application-level coordination needed. This is the correct place to enforce financial consistency in a distributed system.

1
Geographic Routing

A user in Delhi makes a UPI payment. The orchestrator reads their IP and PSP identifier, determines the nearest region (ap-south-1 Mumbai vs ap-southeast-1 Singapore), and routes accordingly. Without this, a user in Delhi might hit a server in London adding 200ms of latency to every transaction.

2
Account Affinity (Consistent Hash)

This is the most important job. hash("rahul@hdfc") always maps to PSP Cluster A. All 1000 simultaneous payments to Rahul's account go to the same cluster — they are serialised before reaching the DB. The DB row lock still exists as a safety net, but it rarely needs to block anyone because the queue already did the work.

hash("rahul@hdfc") = 0x3A7F → Cluster A
All 1000 req/sec → Cluster A → queue
DB sees 1 req at a time → lock rarely fires ✅
3
Health-Aware Failover

App Node 3 crashes. Without an orchestrator, a round-robin load balancer still sends 20% of traffic to Node 3 — those requests fail. The orchestrator does active health checks (every 2s). When Node 3 fails 3 consecutive checks, it is immediately removed from the pool. Zero traffic reaches it. All requests reroute to healthy nodes — invisibly to users.

Node 3: missed 3 health checks → REMOVED
Traffic redistributed to Node 1, 2, 4 ✅
User sees zero failures. Auto, no manual step.
4
Rate Limiting & Fraud Gate

A bot sends 50,000 payment requests per second from one VPA. Without an orchestrator, all 50K reach app nodes and then the DB — which collapses under the load. The orchestrator enforces rate limits per VPA (e.g. max 10 req/sec per account). The 49,990 excess requests are dropped at the entry point — never reaching any app server or DB.

Bot: 50,000 req/sec on rahul@hdfc
NPCI: rate limit = 10 req/sec per VPA
49,990 dropped at entry. DB untouched. ✅
5
Protocol Translation & PSP Routing

UPI involves multiple PSPs (PhonePe, Google Pay, Paytm). When Sai (PhonePe) pays Rahul (HDFC), NPCI must route: PhonePe PSP validates Sai → NPCI routes to HDFC PSP → HDFC credits Rahul. NPCI understands which PSP owns which VPA and routes accordingly. No app node needs to know the full PSP topology — the orchestrator handles all of it.

sai@phonepe → NPCI → phonepe PSP (debit)
rahul@hdfc → NPCI → HDFC PSP (credit)
NPCI coordinates both sides ✅
Why It Is Stateless — The Key Property

The orchestrator stores NO account state. It only stores routing rules (which VPA → which cluster — a small lookup table). hash("rahul@hdfc") returns the same cluster on every NPCI node independently — no coordination needed. This means you can run 10 NPCI nodes or 100 — they all compute the same answer. Scale is unlimited.

NPCI Node 1: hash("rahul@hdfc") → Cluster A ✅

🎯 The Complete Mental Model

Orchestrator is an optimisation. Remove it and the system still produces correct results (the DB lock still prevents double-debit). But with 1000 concurrent requests for the same account all hitting the DB simultaneously, 999 block — system is correct but unusable at scale.

DB lock is a guarantee. Remove it and the system routes beautifully — but the moment two requests for the same account reach different nodes simultaneously (failover, routing hiccup, scaling event), you get a double debit. Data corrupted. Money gone.

They are sequential, not competing. Orchestrator acts first — optimises traffic. DB lock acts last — guarantees correctness. You need both, and neither can replace the other.

Isolation Levels — Where They Live

Where Do Isolation Levels Actually Live?

Most developers think isolation levels are a Spring/Hibernate concept. They are not. They live inside the PostgreSQL storage engine — deep in the same layer that handles MVCC row versioning, WAL writes, and lock management. Spring, Hibernate, and JDBC can only request a level. The actual enforcement — snapshots, lock decisions, anomaly prevention — all happens inside PostgreSQL.

The Full Stack — Who Does What
PostgreSQL Engine
Maintains MVCC row versions. Takes snapshots. Applies visibility rules. Acquires row locks. Detects deadlocks. Enforces CHECK constraints. This is where isolation actually happens.
ENFORCES
JDBC Driver
Sends SET TRANSACTION ISOLATION LEVEL to PostgreSQL over TCP. Sends BEGIN, COMMIT, ROLLBACK. Zero logic of its own — it is a pipe.
MESSENGER
Spring @Transactional
Calls connection.setTransactionIsolation(level) before BEGIN. That is the entire job. Spring is a remote control. PostgreSQL is the TV.
REMOTE CTRL
Hibernate / JPA
Translates Java objects to SQL. Manages first-level (session) and second-level cache. Delegates all transaction and isolation concerns to Spring + JDBC. Does not touch isolation logic.
DELEGATES
Your App Code
Declares what it needs via @Transactional(isolation = ...). That annotation triggers the whole chain above. Just a declaration.
DECLARES

⚠️ The Connection Pool Trap — Silent Wrong Isolation

HikariCP reuses physical DB connections across many requests. If one request sets ISOLATION LEVEL SERIALIZABLE and the connection returns to the pool without being reset, the next request using that connection silently runs as SERIALIZABLE even though it didn't ask for it. HikariCP handles this correctly by default — but raw connection pools may not. Always verify connection.setTransactionIsolation() is called fresh at the start of each transaction.

Snapshot Internals

What Is a Snapshot? Not a Copy of the Database

The most common misconception about MVCC: people think a snapshot is a physical copy of the data — like a database backup taken at that instant. It is not. A snapshot is just three numbers and a small list of transaction IDs. Maybe 50–100 bytes. It takes microseconds to create. No rows are copied. No tables are copied. Nothing is physically duplicated.

What a Snapshot Actually Contains

-- The entire snapshot structure:
snapshot = {
  xmin: 1050, ← oldest active transaction ID
  xmax: 1060, ← next transaction ID to assign
  in_progress: [1052, 1055, 1058]
}

-- Size: ~50-100 bytes
-- Creation time: microseconds
-- Data copied: ZERO bytes
✅ Where the snapshot lives

The snapshot is stored in PostgreSQL's shared memory in a structure called ProcArray — a small in-memory array of all currently active backend processes and their transaction IDs. Taking a snapshot means walking this array (which fits in a few kilobytes of RAM) and copying the current active transaction list. The whole operation is a memory scan — no disk I/O.

Where the Data Actually Lives — Multiple Versions on Disk

PostgreSQL keeps multiple versions of every row on disk all the time — this is the MVCC storage model. Each row version has two hidden fields stamped at write time:

-- accounts table — what PostgreSQL stores on disk:
xminxmaxidbalance 9010rahul₹1000 ← current live row

-- Sai starts TXN 1055, writes balance=₹0:
9011055rahul₹1000 ← old version (xmax stamped) 10550rahul₹0 ← new version (uncommitted)

Both rows exist on disk simultaneously. No copying. Just two row versions.

The snapshot does not create these versions — they already exist from normal write operations. The snapshot just provides the reference point: "as of which transaction IDs should I decide visibility?"

How the Snapshot Decides What You See — Pure Arithmetic

When your query runs, PostgreSQL walks through every candidate row version and applies these visibility rules. No locks on reads. Just integer comparisons on transaction IDs:

-- Visibility rule (simplified):
A row version is VISIBLE to my transaction if:
  xmin < snapshot.xmax ← was created before I started
  AND xmin NOT IN snapshot.in_progress ← was committed, not in-progress
  AND (xmax = 0 ← not deleted
      OR xmax > snapshot.xmax ← deleted after I started
      OR xmax IN snapshot.in_progress) ← deleted by in-progress txn

-- Applied to our example:
-- Ananya's snapshot: {xmin:1050, xmax:1060, in_progress:[1052,1055,1058]}

-- Row version 1: xmin=901, xmax=1055, balance=₹1000
  xmin=901 < 1060 ✅
  xmin=901 NOT IN [1052,1055,1058] ✅ (committed long ago)
  xmax=1055 IS IN in_progress → treat as not-yet-deleted ✅
  → VISIBLE — Ananya sees ₹1000 ✅

-- Row version 2: xmin=1055, xmax=0, balance=₹0
  xmin=1055 IS IN [1052,1055,1058] → created by in-progress txn ❌
  → INVISIBLE — Ananya cannot see ₹0 ❌

The Only Difference Between READ COMMITTED and REPEATABLE READ — Snapshot Timing

Both isolation levels use the exact same snapshot mechanism and visibility rules. The only difference is when the snapshot is taken:

READ COMMITTED (default)

A fresh snapshot is taken at the start of each SQL statement. So if you run two SELECTs in the same transaction and someone commits between them, your second SELECT sees the new committed data.

BEGIN
SELECT balance ← snapshot HERE → ₹1000
... Sai commits ₹0 ...
SELECT balance ← NEW snapshot → ₹0
COMMIT
Two different values in same txn!
Non-repeatable read — acceptable for most use cases
REPEATABLE READ

The snapshot is taken once at transaction BEGIN and reused for every statement. No matter what commits happen during your transaction, you always see the same view of the world.

BEGIN ← snapshot taken HERE once
SELECT balance → uses snapshot → ₹1000
... Sai commits ₹0 ...
SELECT balance → SAME snapshot → ₹1000
COMMIT
Both reads return same value ✅
Perfect for bank statements, reports

VACUUM — Cleaning Up Dead Row Versions

Every UPDATE creates a new row version and marks the old one dead (sets xmax). If you update Rahul's balance 1000 times, PostgreSQL has 1000 dead row versions on disk. These accumulate until AUTOVACUUM runs in the background to clean them. AUTOVACUUM scans tables and removes row versions whose xmax transaction is committed and older than the oldest active transaction — meaning no snapshot could ever need to see them anymore.

AUTOVACUUM fires when:
• >20% of rows are dead
• autovacuum_vacuum_scale_factor
• Runs in background
• No table lock held
What it removes:
• Dead row versions (old xmax)
• Only if no snapshot needs them
• Updates visibility map
• Updates table statistics
Without AUTOVACUUM:
• Table bloat (GBs of dead rows)
• Sequential scans slow down
• XID wraparound → crash
• Must run VACUUM FULL manually
🎯 The Complete Mental Model — Three Sentences

A snapshot is just a list of in-progress transaction IDs at a point in time — ~100 bytes, no data copied.

The multiple row versions that make MVCC work already exist on disk from normal write operations — each row has xmin (who created it) and xmax (who deleted/updated it) stamped on it. The snapshot just provides the "as of when" reference to decide which version is visible to your query.

Reads never lock anything. Visibility is pure arithmetic on transaction IDs — comparing xmin/xmax against your snapshot. This is why PostgreSQL can run thousands of concurrent reads and writes with no blocking between them.

MVCC Internals — xmin, xmax, in_progress

xmin, xmax, in_progress — How Every Row Knows Its History

These are the three fields that make MVCC work. Two of them (xmin, xmax) are hidden system columns stamped on every row in every table. One of them (in_progress) is part of the snapshot, not the row. Together they let PostgreSQL answer one question — "should this row version be visible to your transaction?" — using pure integer arithmetic, zero locks, zero data copying.

xmin — "Who created this row version"

xmin stands for "transaction minimum." It is the Transaction ID (XID) of the transaction that ran the INSERT or UPDATE that produced this row version. It is stamped permanently at write time and never changes. Every transaction gets a unique auto-incrementing number — like a bank token. TX 901, TX 1055, TX 1060 — each one is unique globally in that database instance.

-- Rahul account created inside transaction 901:
INSERT INTO accounts (id, balance) VALUES ('rahul', 1000);
-- PostgreSQL stamps the row:
xmin = 901 ← "TX 901 gave birth to this row version"
-- This stamp is permanent. Never changes for this version.

xmax — "Who killed this row version"

xmax is the Transaction ID of the transaction that ran UPDATE or DELETE on this row version. xmax = 0 means nobody has killed it yet — it is still the current live version. When Sai's TX 1055 updates the balance, PostgreSQL does NOT overwrite the old row. It stamps the old row with xmax = 1055 and creates a brand new row version with xmin = 1055.

-- Sai (TX 1055) updates balance to 0:
UPDATE accounts SET balance = 0 WHERE id = 'rahul';
-- PostgreSQL does NOT overwrite. It does two things:
Old row: xmax = 1055 ← "TX 1055 killed me"
New row: xmin = 1055, xmax = 0 ← "TX 1055 created me, alive"
-- Both rows physically on disk simultaneously.

in_progress — "Who was running when my snapshot was taken"

in_progress is NOT a column on the row. It lives inside the snapshot structure — a tiny object (~60 bytes) created when your transaction begins. PostgreSQL walks the ProcArray — a shared memory structure with one slot per active database connection — and captures which transaction IDs are currently running. That list becomes in_progress.

-- ProcArray at the moment TX2 starts:
Slot 1: PID 101, xid = 1055 (TX1 — Sai, RUNNING)
Slot 2: PID 102, xid = 1060 (TX2 — Ananya, starting)
Slot 3: PID 103, xid = 1062 (TX3 — Priya, starting)
-- TX2 snapshot captured:
in_progress = [1055] ← only TX1 was running
-- Size: ~60 bytes. No data copied. Microseconds.

Are xmin and xmax visible to you?

Yes — they are hidden system columns on every table, automatically added by PostgreSQL. SELECT * never shows them, but you can explicitly select them:

-- Normal query:
SELECT * FROM accounts;
-- id | balance (xmin, xmax hidden)

-- Explicit system column query:
SELECT xmin, xmax, * FROM accounts;
-- xmin | xmax | id | balance
-- 901 | 0 | rahul | 1000

-- Cannot use xmin/xmax as column names:
CREATE TABLE test (xmin INT); -- ERROR: reserved

MVCC Snapshot Walkthrough — 7 Steps

in_progress — The Time Capsule That Separates Isolation Levels

in_progress is a frozen photograph of which transactions were running at the exact moment your snapshot was taken. It does not update. It does not care what happens after. And it is the single field that separates REPEATABLE READ from READ COMMITTED.

💡 What in_progress is NOT

Not a wait queue. Transactions in the list do not know they are listed there. They are not signalled or paused.

Not a lock. Being in in_progress does not cause any blocking on readers. You just get an older version immediately.

Not updated mid-transaction (for REPEATABLE READ). Even if TX1055 commits while Ananya is mid-query, her in_progress list does not change.

-- How in_progress controls visibility:
IF xmin IN in_progress
  → row created by running tx → HIDE
IF xmax IN in_progress
  → row killed by running tx → SHOW (kill not final)
IF xmax NOT IN in_progress AND xmax != 0
  → row killed by committed tx → HIDE (officially dead)
The One Difference Between REPEATABLE READ and READ COMMITTED
READ COMMITTED

Fresh snapshot (new in_progress list) taken before each SQL statement. Committed transactions disappear from in_progress. You start seeing their writes.

BEGIN
SELECT ← snapshot: in_progress=[1055] → Rs.1000
... TX1055 commits ...
SELECT ← NEW snapshot: in_progress=[] → Rs.0
COMMIT
REPEATABLE READ

One snapshot taken at BEGIN. Same in_progress list reused for every statement. Committed transactions stay in in_progress because the snapshot was frozen before they committed.

BEGIN ← snapshot: in_progress=[1055] FROZEN
SELECT ← same snapshot → Rs.1000
... TX1055 commits ...
SELECT ← SAME snapshot → Rs.1000 still!
COMMIT
⚠️ It Depends on the DB Engine

MVCC with xmin/xmax is PostgreSQL's specific implementation. MySQL InnoDB also uses MVCC but implements it differently — through an undo log (keeps old versions in a separate structure rather than in the same table heap). Oracle uses a similar undo log approach. The behaviour is similar across all — readers don't block writers — but the internal mechanics differ. This is why READ UNCOMMITTED in PostgreSQL behaves the same as READ COMMITTED — PostgreSQL's MVCC simply has no code path to serve uncommitted row versions to readers. Other engines like MySQL with READ UNCOMMITTED actually skip the committed check and can serve dirty reads.

🎯 Three Fields, One System

xmin — hidden system column on every row. "Who created me." Stamped at write time, permanent.

xmax — hidden system column on every row. "Who killed me." 0 means still alive. Stamped when a row is updated or deleted.

in_progress — part of the snapshot, not the row. "Who was running when my snapshot was taken." Captured from ProcArray at BEGIN. Frozen for REPEATABLE READ. Refreshed per statement for READ COMMITTED.

These three integers, compared arithmetically on every row version your query touches, are the entire MVCC system. No locks on reads. No data copying. No blocking between readers and writers.

Deep Dive 01

Transaction Isolation Levels — The Full Spectrum

Not all database reads are equal. SQL defines 4 isolation levels — each one trades consistency for performance. PostgreSQL implements all of them using MVCC (no actual locking for reads). Choosing the wrong one means dirty reads, phantom rows, or non-repeatable reads. Here is what each one means in practice.

READ UNCOMMITTED
Weakest
Almost nobody uses this
What happens

You can read rows that another transaction has written but NOT YET COMMITTED. If that transaction rolls back, you read data that never officially existed. Called a "dirty read".

UPI Scenario

Sai starts debit, writes ₹0 but hasn't committed yet. Priya reads ₹0. Sai's transaction then fails and rolls back to ₹1000. Priya decided "Rahul has ₹0" based on data that no longer exists.

PostgreSQL

PostgreSQL treats READ UNCOMMITTED identically to READ COMMITTED — it simply cannot do dirty reads by design. MVCC always serves committed versions.

READ COMMITTED
Default in PG
Most apps use this
What happens

Each statement sees only data committed before that statement began. You never read dirty (uncommitted) data. But within a single transaction, two reads of the same row can return different values if someone committed between them.

UPI Scenario

Ananya reads balance=₹1000 (snapshot 1). Sai commits ₹0. Ananya reads balance again (snapshot 2) = ₹0. Same transaction, two different values. "Non-repeatable read" — acceptable for most use cases.

Scenario 4 in simulator

This is what the "Read While Writing" scenario demonstrates. Ananya's read gets a snapshot of committed data at statement time. Safe and fast.

REPEATABLE READ
Strong
Bank statements, reports
What happens

The entire transaction sees a single consistent snapshot taken at transaction start. All reads within this transaction see the same data, even if others commit in between. No dirty reads, no non-repeatable reads.

UPI Scenario

Rahul's bank statement generation uses REPEATABLE READ. It reads balance (₹1000) then reads all transactions, then computes totals — all from the same snapshot. Even if Priya credits ₹500 during generation, the statement is perfectly consistent.

PostgreSQL default for

Dumps, exports, audit reports, any multi-statement read that must be internally consistent. Set per transaction: SET TRANSACTION ISOLATION LEVEL REPEATABLE READ

SERIALIZABLE
Strongest
Full ACID, max safety
What happens

Transactions execute as if they ran one-by-one, even if they actually ran concurrently. DB detects any anomaly (phantom reads, write skew, serialization failures) and aborts conflicting transactions. Application must handle the abort and retry.

UPI Scenario

CockroachDB uses SERIALIZABLE for all transactions always. Every UPI transfer is fully isolated. Cost: higher abort rate on hot accounts (like a celebrity receiving thousands of payments/sec) — all must retry on conflict.

Used by

Google Spanner, CockroachDB always. PostgreSQL optionally. Core banking systems for end-of-day reconciliation. Any place where "write skew" (two transactions reading same data, each writing to different rows based on it) would cause bugs.

Write Skew — The Subtle Villain

Two doctors are on call. Each reads "are there at least 2 doctors on call?" — YES. Each decides to go off-call. Both commit. Now 0 doctors are on call. Neither transaction saw a conflict but together they violated a business constraint.

Txn A: SELECT count(*) ON_CALL → 2 ✅
Txn B: SELECT count(*) ON_CALL → 2 ✅
Txn A: UPDATE SET on_call=false WHERE id=1
Txn B: UPDATE SET on_call=false WHERE id=2
Result: 0 doctors on call 💀 — neither saw conflict
Fix: SERIALIZABLE isolation or explicit lock on count

Phantom Read — New Rows Appearing

Transaction A reads "all transactions above ₹1000" and gets 3 rows. Before A finishes, Transaction B inserts a new row above ₹1000. A reads again and gets 4 rows. The "phantom" row appeared mid-transaction. Non-repeatable read is about changed values; phantom is about new rows.

Txn A: SELECT WHERE amount>1000 → [3 rows]
Txn B: INSERT amount=5000 → committed
Txn A: SELECT WHERE amount>1000 → [4 rows] 👻
Phantom row appeared. Audit report is inconsistent.
Fix: REPEATABLE READ or SERIALIZABLE isolation
Deep Dive 02

MVCC Internals — How PostgreSQL Actually Does It

MVCC (Multi-Version Concurrency Control) is the single most important reason PostgreSQL is safe under concurrent load. Instead of blocking readers when a writer is active, PostgreSQL keeps multiple versions of each row. Every transaction sees a consistent snapshot of the database as it existed at a specific point in time.

How Row Versioning Works

Every row in PostgreSQL has two hidden system columns: xmin (transaction ID that created this version) and xmax (transaction ID that deleted/updated this version, or 0 if still current).

-- accounts table (what PostgreSQL actually stores):
xminxmaxidbalance TXN 1010rahul₹1000 ← current (xmax=0)
-- Sai starts TXN 105, writes ₹0:
TXN 101TXN 105rahul₹1000 ← old version TXN 1050rahul₹0 ← new version (uncommitted)
-- Ananya (TXN 104, started before 105) reads:
sees xmin=101, xmax=105 → 105 not committed → sees ₹1000 ✅
-- After Sai commits TXN 105:
New readers see xmin=105, xmax=0 → ₹0
-- VACUUM cleans up old versions periodically

Snapshot Isolation — The Snapshot ID

When your transaction starts, PostgreSQL takes a snapshot: "all transaction IDs currently in progress." Any row version whose xmin is in that "in progress" set is invisible to you — you see the version that existed before those transactions started.

-- Snapshot at Ananya's transaction start:
xmin=104, in_progress=[105, 106], xmax=infinity
-- Rules for visibility:
xmin < 104 AND xmax=0 → VISIBLE (old committed)
xmin < 104 AND xmax in [105,106] → VISIBLE (deleted by in-progress)
xmin in [105,106] → INVISIBLE (created by in-progress)
-- This means:
Sai's ₹0 row (xmin=105) → INVISIBLE to Ananya
₹1000 row (xmin=101) → VISIBLE ✅
-- No locks needed for reads. Pure math.
✅ Why MVCC Is Fast

Readers never lock anything. Writers create new row versions instead of overwriting. Reads and writes run in parallel at full speed. VACUUM eventually garbage-collects old versions. The only lock is the row-level write lock between competing writers — SELECT FOR UPDATE.

VACUUM — Cleaning Up Dead Row Versions

Every UPDATE creates a new row version and marks the old one dead (xmax set). If you update the same row 1000 times, you have 1000 dead versions sitting in the table taking up space. AUTOVACUUM runs in the background to clean these up. Without it, tables bloat, query plans degrade, and eventually XID wraparound crashes the whole cluster.

AUTOVACUUM triggers when:
• >20% of rows are dead
• autovacuum_vacuum_threshold
• Runs in background
• No table lock needed
VACUUM ANALYZE does:
• Removes dead tuples
• Updates table statistics
• Fixes visibility map
• Prevents XID wraparound
Production problems without it:
• Table bloat (GBs of dead rows)
• Slow sequential scans
• XID wraparound (crash!)
• Pg must shut down to fix
Deep Dive 03

WAL — How Every Write Is Made Durable and Replicated

WAL (Write-Ahead Log) is the backbone of both PostgreSQL crash recovery and streaming replication. Every change to data is first written to the WAL — an append-only log on disk — before the actual data pages are modified. "Write-ahead" means: WAL is written first, data pages later. This guarantees durability even if the server crashes mid-write.

WAL Write Sequence — What Happens on Every COMMIT

1
App sends COMMIT
UPDATE accounts SET balance=0 WHERE id='rahul'
2
WAL record written to WAL buffer
LSN 0/1A3F4B8: TXN 105 UPDATE accounts id=rahul bal=0
3
WAL flushed to disk (fsync)
pg_wal/000000010000000000000001 written. This is the durability guarantee — crash now, WAL survives.
4
Success returned to client ✅
Once WAL is on disk, the commit is durable. Data pages in memory (shared_buffers) still show old value — that is fine.
5
WAL streams to replicas (async)
WAL records stream to Replica-1 (1ms), Replica-2 (90ms). Replicas apply and send back confirmation (reply_lsn).
6
Checkpointer writes dirty pages (background)
Eventually, modified data pages are written from shared_buffers to actual data files. WAL segments before checkpoint LSN are no longer needed for recovery.

Replication Modes — Detailed Comparison

Synchronous Replication

Primary waits for at least one replica to confirm WAL received AND written to its own WAL (not just in memory) before returning success to client. Config: synchronous_commit = on. Zero data loss guarantee. Cost: every write adds one round-trip time to its latency.

Asynchronous Replication (default)

Primary returns success after writing to its own WAL. Replica receives WAL in the background. If primary crashes before replica applies: up to replication-lag worth of commits lost. Config: synchronous_commit = off. Intra-DC lag: ~1ms. Cross-region: ~90ms.

Remote Write (Compromise)

Primary waits for replica to confirm WAL received and written to OS buffer — but NOT necessarily fsynced to disk. Config: synchronous_commit = remote_write. Lower latency than full sync. Very small data loss risk (OS crash on replica before fsync).

LSN — Log Sequence Number

Every WAL record has a unique LSN (like 0/1A3F4B8). The gap between primary's current LSN and replica's applied LSN = replication lag in bytes. Convert to time: if you write 10MB/sec and lag is 900KB, lag ≈ 90ms. Monitor: SELECT * FROM pg_stat_replication.

Deep Dive 04

Deadlocks — When Two Transactions Wait for Each Other Forever

A deadlock happens when Transaction A holds a lock that Transaction B needs, AND Transaction B holds a lock that Transaction A needs. Neither can proceed. PostgreSQL detects this automatically (deadlock_timeout default: 1 second) and kills one transaction. The killed one gets an error and must retry.

Classic Deadlock — Transfer Between Two Accounts

-- Sai: transfer ₹500 from Rahul → Priya
TXN A: LOCK Rahul (FOR UPDATE) ✅
TXN A: wants to LOCK Priya next...
-- Simultaneously: Priya transfers ₹200 to Rahul
TXN B: LOCK Priya (FOR UPDATE) ✅
TXN B: wants to LOCK Rahul next...
💀 DEADLOCK: A waits for B, B waits for A
PostgreSQL detects after 1s → kills one → other proceeds
✅ Prevention: Always Lock in Same Order

If all transactions always lock accounts in the same order (e.g., lower ID first), deadlocks are impossible. TXN A and TXN B both try to lock Rahul first — one blocks, the other proceeds. No circular wait.

PostgreSQL Deadlock Detection

PostgreSQL does NOT check for deadlocks on every lock acquisition — that would be too expensive. Instead it starts a timer when a lock wait begins. If the wait exceeds deadlock_timeout (default 1s), it then does a full dependency cycle check across all waiting transactions.

-- PostgreSQL error your app receives:
ERROR: deadlock detected
DETAIL: Process 12345 waits for ShareLock
on transaction 105; blocked by 12346.
-- Your application MUST handle this:
catch (PSQLException e) {
  if (isDeadlock(e)) retry(txn);
}
-- The killed transaction is fully rolled back
-- All its locks released. Other transaction proceeds.
Deep Dive 05

Lock Types in PostgreSQL — From Row to Table

PostgreSQL has multiple levels of locking: row-level (what SELECT FOR UPDATE uses), table-level (what DDL uses), and advisory locks (application-defined). Understanding which lock your query takes is critical for avoiding unexpected blocking in production.

Lock ModeTaken ByBlocks Who?UPI Use Case
ROW SHARE SELECT FOR UPDATE / FOR SHARE Conflicts with EXCLUSIVE table locks only Reading Rahul's account before debit — holds row while checking balance
ROW EXCLUSIVE INSERT / UPDATE / DELETE Other ROW EXCLUSIVE if same row Writing new balance after debit — row locked until commit
SHARE UPDATE EXCLUSIVE VACUUM / CREATE INDEX CONCURRENTLY DDL only. Reads and writes proceed normally AUTOVACUUM cleaning dead row versions while app writes — no blocking
ACCESS EXCLUSIVE ALTER TABLE / DROP TABLE / VACUUM FULL Everything — blocks ALL reads and writes Adding a column to accounts table — ALL UPI transactions pause. Schedule at 3AM.
Advisory Lock pg_advisory_lock(key) Only other advisory locks with same key Application-level mutex: ensure only one job processes a batch at a time
⚠️ The ALTER TABLE Trap in Production

Every ALTER TABLE accounts ADD COLUMN ... takes an ACCESS EXCLUSIVE lock. This blocks ALL reads and ALL writes on that table for the duration of the ALTER. On a busy UPI system processing 10,000 transactions/second, running ALTER TABLE during business hours means every transaction hangs. Solutions: (1) Use ALTER TABLE ... ADD COLUMN ... DEFAULT NULL — in PostgreSQL 11+ this is instant (metadata-only). (2) Use CREATE INDEX CONCURRENTLY — builds index without blocking. (3) Schema migrations must always use lock_timeout = '2s' so they abort rather than hang.

Deep Dive 06

Distributed Transactions in UPI — When Two Banks Are Involved

Everything so far assumed one database. But a real UPI transfer touches TWO banks: Sai's bank (debit ₹1000 from Sai) and Rahul's bank (credit ₹1000 to Rahul). These are completely separate databases. You cannot use SELECT FOR UPDATE across two different banks. This is the distributed transaction problem.

How NPCI Actually Handles This — Saga Pattern

UPI does NOT use 2PC (Two-Phase Commit) across banks — it is too slow and the blocking problem makes it impractical across internet links between banks. Instead UPI uses a Saga pattern — a sequence of local transactions, each with a compensating transaction (rollback action).

T1
Debit Sai's account (SBI)
Local transaction on SBI DB. Debit ₹1000 from Sai. Compensating action: re-credit ₹1000 to Sai.
T2
Credit Rahul's account (HDFC)
Local transaction on HDFC DB. Credit ₹1000 to Rahul. Compensating action: debit ₹1000 back from Rahul.
C1
If T2 fails — Run compensating transaction
NPCI orchestrator triggers C1: re-credit ₹1000 to Sai (reverse T1). Both databases return to original state. Sai gets refund within 3 business days per RBI rules.

The Debit-Credit Gap — The Real Risk Window

Between T1 completing and T2 completing there is a brief window where Sai's money is debited but Rahul has not received it yet. This is called the "debit-credit gap." If NPCI crashes in this exact window, the saga is incomplete and must be recovered from the NPCI transaction log.

-- NPCI transaction state machine:
INITIATED → DEBIT_PENDING → DEBITED
         → CREDIT_PENDING → COMPLETED
-- If crash in DEBITED state:
On recovery: find all DEBITED transactions
Retry T2 (credit Rahul) or run C1 (refund Sai)
-- Idempotency key prevents double credit:
T2 includes txn_id. If T2 already applied,
second attempt returns success (no-op).
ℹ️ Why Your "Failed" UPI Sometimes Shows Debit

The app shows failure because NPCI timed out waiting for T2 confirmation. But T1 (your debit) already committed at SBI. T2 (Rahul's credit) may or may not have committed. NPCI will retry T2 or run C1 (refund). This is why UPI refunds can take up to 3 business days — it is the saga compensation running.

🎯 The Complete Mental Model

Stateless layers (routers, app nodes) — scale to any number. No state = no consistency problem = no coordination needed.

Stateful layers (DB Primary) — one source of truth. MVCC for reads (never blocking). SELECT FOR UPDATE for competing writes. CHECK constraints as the absolute last line of defense.

Replication layers (Replicas) — copies of state. Always slightly behind (async) or incur latency (sync). Reads from replicas can be stale. Writes always go to primary.

Cross-system (Saga) — for transactions spanning multiple databases. No global ACID. Use local transactions + compensating transactions + idempotency keys. Accept eventual consistency with guaranteed eventual correctness.
