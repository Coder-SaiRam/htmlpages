What is JPA? The Hierarchy Explained
JDBC → JPA → Hibernate → Spring Data JPA
🔌
JDBC
raw database API
Direct database calls. Manual SQL. Manual ResultSet mapping. No object mapping. Maximum control, maximum boilerplate. Every query = 20+ lines of code.
🔌 Driving manual car — you control every gear shift
📜
JPA Spec
Java EE standard
Java Persistence API — a specification (interfaces only). Defines how Java objects map to DB tables. NOT an implementation — just a contract. Providers implement it.
📜 Car safety standards — rules that car makers must follow
⚙️
Hibernate
JPA implementation
Most popular JPA provider. Implements JPA spec + adds extras (Criteria API, L2 cache, etc). Spring Boot uses Hibernate as default. Does the actual SQL generation.
⚙️ Toyota — implements the safety standards + adds their own features
🚀
Spring Data JPA
abstraction on top
Spring layer on top of Hibernate/JPA. Auto-generates repositories. Derive queries from method names. Pagination, sorting, auditing built-in. This is what you mostly use.
🚀 Self-driving car — you say "go to mall", it handles everything
😩

    }
}

With Spring Data JPA

// That's it. Spring generates the implementation!
public interface UserRepository extends JpaRepository<User, Long> {

    // Spring derives SQL from method name automatically
    Optional<User> findByEmail(String email);

    List<User> findByNameContainingIgnoreCase(String name);

    List<User> findByAgeGreaterThanAndActiveTrue(int age);

    long countByDepartment(String department);
}

// Usage in service — just inject and call
@Service
public class UserService {
    private final UserRepository repo;

    public Optional<User> getByEmail(String email) {
        return repo.findByEmail(email);  // done!
    }
}

Spring Data JPA Architecture
layers and components
Spring Data JPA — Full Architecture
📥 REQUEST FLOW
1
🌐 @RestController / @Service
Your business logic calls Repository methods. No SQL knowledge needed here.
2
📋 Spring Data Repository Interface
Your interface extends JpaRepository<Entity, ID>. Spring generates implementation at startup via proxy pattern.
3
🔀 SimpleJpaRepository (generated impl)
Spring's auto-generated proxy. Translates method names to JPQL. Manages Entity lifecycle.
4
🗂️ EntityManager (JPA)
JPA's core API. Manages persistence context (1st-level cache). Tracks entity state. The gateway to Hibernate.
5
⚙️ Hibernate SessionFactory
JPA Provider. Generates actual SQL. Manages 2nd-level cache. Translates JPQL → SQL dialect.
6
🔌 JDBC + DataSource (HikariCP)
Connection pool sends SQL to actual database. HikariCP is the default fast connection pool in Spring Boot.
7
🗄️ Database (MySQL / PostgreSQL / H2)
Executes SQL, returns ResultSet. Spring maps it back to your Entity object.
🧩 KEY COMPONENTS
📦 Persistence Context
First-level cache. EntityManager maintains a map of all loaded entities per transaction. Identical queries within same transaction return same object from cache — no DB hit.
🔄 Entity Lifecycle
Transient → new object, not in DB
Managed → tracked by EntityManager
Detached → was managed, no longer tracked
Removed → scheduled for DELETE
⚡ HikariCP Connection Pool
Pool of pre-opened DB connections. Default pool-size: 10. Reuses connections instead of opening new ones. Configure: spring.datasource.hikari.maximum-pool-size=20
🗺️ Dialect
Hibernate generates DB-specific SQL. MySQL dialect uses LIMIT, PostgreSQL uses LIMIT/OFFSET, Oracle uses ROWNUM. Auto-detected from driver.

@Entity                             // marks this as a JPA entity
@Table(name = "products",           // custom table name
    indexes = {
        @Index(name = "idx_sku", columnList = "sku", unique = true),
        @Index(name = "idx_category", columnList = "category_id")
    }
)
@EntityListeners(AuditingEntityListener.class)  // for auditing
@Data                               // Lombok: getters, setters, equals, hashCode
@Builder
@NoArgsConstructor
@AllArgsConstructor
public class Product {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY) // auto-increment
    private Long id;

    @Column(name = "product_name",  // custom column name
            nullable = false,
            length = 200)
    private String name;

    @Column(unique = true, nullable = false, length = 50)
    private String sku;             // Stock Keeping Unit

    @Column(precision = 10, scale = 2)  // for BigDecimal: 10 total digits, 2 decimal
    private BigDecimal price;

    @Enumerated(EnumType.STRING)    // store enum as "ACTIVE" not 0
    @Column(nullable = false)
    private ProductStatus status;   // enum: ACTIVE, INACTIVE, DISCONTINUED

    @Lob                            // Large Object — stores in separate LOB column
    @Column(columnDefinition = "TEXT")
    private String description;

    @Column(name = "image_url")
    private String imageUrl;

    @Version                        // optimistic locking — auto-incremented on update
    private Long version;

    // Auditing fields (auto-populated)
    @CreatedDate
    @Column(updatable = false)
    private LocalDateTime createdAt;

    @LastModifiedDate
    private LocalDateTime updatedAt;

    @CreatedBy
    @Column(updatable = false)
    private String createdBy;

    // Relationships (see Section 4)
    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "category_id")
    private Category category;

    @OneToMany(mappedBy = "product", cascade = CascadeType.ALL, orphanRemoval = true)
    private List<ProductImage> images = new ArrayList<>();

    @ElementCollection                  // for simple value collections
    @CollectionTable(name = "product_tags", joinColumns = @JoinColumn(name = "product_id"))
    @Column(name = "tag")
    private Set<String> tags = new HashSet<>();
}

public class Category {
    @Id @GeneratedValue(strategy = IDENTITY)

    private String name;

    // One category has many products
    @OneToMany(mappedBy = "category",     // "category" = field in Product

public class Product {
    @Id @GeneratedValue(strategy = IDENTITY)
    private Long id;
    private String name;

    // Many products belong to one category
    @ManyToOne(fetch = FetchType.LAZY)    // LAZY = don't load until accessed
    @JoinColumn(name = "category_id")     // FK column name in products table
    private Category category;
}

⚠️
@ManyToMany Best Practice: Avoid simple @ManyToMany if you might need extra fields later. Create a dedicated join entity (like Enrollment). Also use Set not List — Hibernate deletes ALL rows and re-inserts with List on update (Hibernate bug).

CascadeType & orphanRemoval
controlling child lifecycle
CascadeTypeWhat propagates to childUse case
PERSISTsave() on parent saves childNew entities together
MERGEupdate on parent updates childUpdating object graphs
REMOVEdelete parent → deletes childrenDependent entities
REFRESHreload parent → reloads childrenMulti-user scenarios
DETACHdetach parent → detaches childrenRare
ALLAll of the aboveOwned relationships
cascade + orphanRemoval
@Entity
public class Order {
    @Id @GeneratedValue(strategy = IDENTITY)
    private Long id;

    // cascade ALL: save/delete order → save/delete items
    // orphanRemoval: remove item from list → DELETE from DB
    @OneToMany(mappedBy = "order",
               cascade = CascadeType.ALL,
               orphanRemoval = true)
    private List<OrderItem> items = new ArrayList<>();

    // Helper methods to maintain both sides
    public void addItem(OrderItem item) {
        items.add(item);
        item.setOrder(this);        // set bidirectional link
    }

    public void removeItem(OrderItem item) {
        items.remove(item);         // orphanRemoval deletes from DB
        item.setOrder(null);
    }
}
🌊 cascade = when parent saves/deletes, children follow automatically. orphanRemoval = child removed from collection → deleted from DB.

Custom Repository Implementation
when auto-derivation isn't enough
Custom implementation pattern
// 1. Define custom interface
public interface ProductRepositoryCustom {
    List<Product> searchWithFilters(ProductFilter filter);
}

// 2. Implement it
@Repository
@RequiredArgsConstructor
public class ProductRepositoryCustomImpl
        implements ProductRepositoryCustom {

    private final EntityManager em;

    @Override
    public List<Product> searchWithFilters(ProductFilter filter) {
        CriteriaBuilder cb = em.getCriteriaBuilder();
        CriteriaQuery<Product> query = cb.createQuery(Product.class);
        Root<Product> root = query.from(Product.class);

        List<Predicate> predicates = new ArrayList<>();
        if (filter.getName() != null)
            predicates.add(cb.like(root.get("name"), "%" + filter.getName() + "%"));
        if (filter.getMinPrice() != null)
            predicates.add(cb.ge(root.get("price"), filter.getMinPrice()));

        query.where(predicates.toArray(new Predicate[0]));
        return em.createQuery(query).getResultList();
    }
}

// 3. Main repository extends both
public interface ProductRepository
        extends JpaRepository<Product, Long>,
                ProductRepositoryCustom {
    // both auto-derived and custom methods available!
}

JPQL & Native SQL Queries
@Query — full control over SQL
📝
JPQL — Object-Oriented Query Language
queries on entities, not tables
JPQL @Query examples
public interface OrderRepository extends JpaRepository<Order, Long> {

    // Basic JPQL — uses entity class name, not table name!
    @Query("SELECT o FROM Order o WHERE o.status = :status")
    List<Order> findByOrderStatus(@Param("status") OrderStatus status);

    // JOIN FETCH — solves N+1 problem (loads related in 1 query)
    @Query("SELECT o FROM Order o " +
           "JOIN FETCH o.customer c " +
           "JOIN FETCH o.items i " +
           "WHERE o.id = :id")
    Optional<Order> findByIdWithDetails(@Param("id") Long id);

    // Aggregation
    @Query("SELECT SUM(o.totalAmount) FROM Order o " +
           "WHERE o.customer.id = :customerId " +
           "AND o.status = 'COMPLETED'")
    BigDecimal getTotalSpentByCustomer(@Param("customerId") Long customerId);

    // Date comparison
    @Query("SELECT o FROM Order o WHERE o.createdAt BETWEEN :from AND :to")
    List<Order> findOrdersInDateRange(
        @Param("from") LocalDateTime from,
        @Param("to") LocalDateTime to);

    // Subquery
    @Query("SELECT p FROM Product p WHERE p.price = " +
           "(SELECT MAX(p2.price) FROM Product p2 WHERE p2.category = p.category)")
    List<Product> findMostExpensivePerCategory();

    // UPDATE query — must use @Modifying
    @Modifying
    @Transactional
    @Query("UPDATE Product p SET p.status = :status WHERE p.category.id = :catId")
    int updateStatusByCategoryId(
        @Param("status") ProductStatus status,
        @Param("catId") Long categoryId);

    // DELETE query
    @Modifying
    @Transactional
    @Query("DELETE FROM Product p WHERE p.status = 'DISCONTINUED' AND p.updatedAt < :date")
    int deleteOldDiscontinued(@Param("date") LocalDateTime date);
}

Native SQL Queries
when JPQL isn't enough
nativeQuery = true
public interface ProductRepository extends JpaRepository<Product, Long> {

    // Native SQL — uses actual table/column names
    @Query(value = "SELECT * FROM products WHERE sku = :sku",
           nativeQuery = true)
    Optional<Product> findBySkuNative(@Param("sku") String sku);

    // Use DB-specific features: full-text search (PostgreSQL)
    @Query(value = "SELECT * FROM products " +
                   "WHERE to_tsvector('english', name || ' ' || description) " +
                   "@@ plainto_tsquery('english', :keyword)",
           nativeQuery = true)
    List<Product> fullTextSearch(@Param("keyword") String keyword);

    // Window functions (PostgreSQL/MySQL 8+)
    @Query(value = """
        SELECT p.*, 
               RANK() OVER (PARTITION BY p.category_id ORDER BY p.price DESC) as price_rank
        FROM products p
        WHERE p.active = true
        """, nativeQuery = true)
    List<Object[]> findProductsWithRank();

    // Native with pagination (must specify countQuery!)
    @Query(value = "SELECT * FROM products WHERE status = :status",
           countQuery = "SELECT COUNT(*) FROM products WHERE status = :status",
           nativeQuery = true)
    Page<Product> findByStatusNative(
        @Param("status") String status, Pageable pageable);

    // Returning DTO projection from native query
    @Query(value = """
        SELECT p.id, p.name, p.price, c.name as categoryName
        FROM products p
        JOIN categories c ON p.category_id = c.id
        WHERE p.active = true
        """, nativeQuery = true)
    List<ProductSummaryView> findProductSummaries();  // interface projection
}

Transactions — @Transactional Deep Dive
ACID · Propagation · Isolation
🔒
@Transactional — Core Usage
TransactionalService.java
@Service
@RequiredArgsConstructor
public class OrderService {

    private final OrderRepository orderRepo;
    private final InventoryRepository inventoryRepo;
    private final PaymentService paymentService;

    // ✅ @Transactional on service method — NOT on repository
    @Transactional
    public Order createOrder(CreateOrderRequest req) {
        // All operations in SAME transaction
        // If ANY fails → ALL rolled back automatically

        // 1. Create order
        Order order = Order.builder()
            .customerId(req.getCustomerId())
            .status(OrderStatus.PENDING)
            .build();
        orderRepo.save(order);  // INSERT (not committed yet)

        // 2. Deduct inventory
        inventoryRepo.decrementStock(req.getProductId(), req.getQty());

        // 3. Process payment — if this throws, everything rolls back!
        paymentService.charge(req.getCustomerId(), req.getAmount());

        order.setStatus(OrderStatus.CONFIRMED);
        return order;
        // ← transaction commits here if no exception
    }

    // Read-only: performance optimization — no dirty checking
    @Transactional(readOnly = true)
    public List<Order> getOrderHistory(Long customerId) {
        return orderRepo.findByCustomerId(customerId);
    }

    // Custom rollback rules
    @Transactional(rollbackOn = {PaymentException.class, StockException.class},
                   noRollbackOn = {NotificationException.class})
    public void processOrderWithNotification(Long orderId) {
        // Payment/Stock failures roll back transaction
        // Notification failures do NOT roll back
    }

    // Timeout — roll back if takes more than 30 seconds
    @Transactional(timeout = 30)
    public void bulkUpdatePrices(List<PriceUpdate> updates) {
        updates.forEach(u -> productRepo.updatePrice(u.getId(), u.getPrice()));
    }
}
🌊
Propagation Types
PropagationBehaviour
REQUIRED (default)Join existing TX. Create new if none. Most common.
REQUIRES_NEWAlways create new TX. Suspend existing. Good for audit logs.
SUPPORTSJoin if exists, proceed without TX if not.
NOT_SUPPORTEDSuspend TX, run without transaction.
MANDATORYMust have existing TX. Throws if none.
NEVERMust NOT have TX. Throws if one exists.
NESTEDNested TX with savepoints. Rollback nested but not outer.

Common @Transactional Pitfalls
Pitfalls to avoid
// ❌ PITFALL 1: Self-invocation — proxy is bypassed!
@Service
public class BadService {
    @Transactional
    public void outer() {
        inner();  // calls this.inner() directly — NO proxy = NO transaction!
    }
    @Transactional(propagation = REQUIRES_NEW)
    public void inner() { ... }
}
// ✅ FIX: Inject self or move inner() to separate @Service

// ❌ PITFALL 2: @Transactional on private method — AOP can't intercept
@Transactional  // IGNORED on private!
private void saveInternal() { ... }
// ✅ FIX: Make it public or protected

// ❌ PITFALL 3: Checked exceptions DON'T rollback by default!
@Transactional
public void save() throws IOException {  // IOException is checked
    repo.save(entity);
    throw new IOException("error");  // WON'T rollback!
}
// ✅ FIX:
@Transactional(rollbackOn = IOException.class)
public void save() throws IOException { ... }

N+1 Problem & Fetch Strategies
the most common JPA performance issue
😱
The N+1 Problem — How It Happens
1 query to get N orders → N more queries to get each order's items
Performance Killer
N+1 problem code
// Innocent-looking service code
List<Order> orders = orderRepo.findAll(); // Query 1: SELECT * FROM orders (100 rows)

for (Order order : orders) {
    // Each access fires a NEW query!
    String customerName = order.getCustomer().getName();
    // Query 2: SELECT * FROM customers WHERE id=1
    // Query 3: SELECT * FROM customers WHERE id=2
    // ...
    // Query 101: SELECT * FROM customers WHERE id=100
}
// TOTAL: 101 queries for what should be 1 JOIN query!

// Even worse with collections:
for (Order order : orders) {
    int itemCount = order.getItems().size();
    // +100 more queries for items!
}
// TOTAL: 201 queries 😱
🐌
100 orders → 101 SQL queries! With 1000 orders → 1001 queries. Response time goes from milliseconds to seconds. DB connections maxed out.
✅
Goal: always fetch related data in the minimum number of queries — ideally 1 or 2 queries total regardless of result size.
How to detect N+1
Enable SQL logging (show-sql=true), then watch for many similar SELECT statements. Or use Hibernate Statistics bean. Or use p6spy or datasource-proxy library to count queries per request.
💊
Solutions — 4 Ways to Fix N+1
Fix 1: JOIN FETCH in JPQL
// Best: load everything in 1 query
@Query("SELECT o FROM Order o " +
       "JOIN FETCH o.customer " +         // include customer
       "LEFT JOIN FETCH o.items i " +     // include items (LEFT for empty orders)
       "LEFT JOIN FETCH i.product " +     // include item's product
       "WHERE o.status = :status")
List<Order> findWithDetailsAndStatus(@Param("status") OrderStatus status);

// WARNING: JOIN FETCH with multiple collections causes duplicates!
// Use DISTINCT or separate queries for multiple collections
@Query("SELECT DISTINCT o FROM Order o " +
       "JOIN FETCH o.customer " +
       "WHERE o.id IN :ids")
List<Order> findWithCustomer(@Param("ids") List<Long> ids);
Fix 2: @EntityGraph
// Define in Entity class
@Entity
@NamedEntityGraph(name = "Order.withCustomerAndItems",
    attributeNodes = {
        @NamedAttributeNode("customer"),
        @NamedAttributeNode(value = "items",
            subgraph = "items.product"),
    },
    subgraphs = @NamedSubgraph(name = "items.product",
        attributeNodes = @NamedAttributeNode("product"))
)
public class Order { ... }

// Use in Repository
@EntityGraph("Order.withCustomerAndItems")
List<Order> findByStatus(OrderStatus status);

// Or inline (no need for @NamedEntityGraph)
@EntityGraph(attributePaths = {"customer", "items", "items.product"})
List<Order> findByCustomerId(Long customerId);
Fix 3: Batch Fetching (application.yml)
// application.yml — applies globally
spring.jpa.properties.hibernate.default_batch_fetch_size: 100
// Now lazy loading fires: WHERE id IN (1,2,3,...100) — 1 query!

// Or per-relationship
@OneToMany(mappedBy = "order")
@BatchSize(size = 100)          // load 100 orders' items in 1 query
private List<OrderItem> items;
Fix 4: Use Projections / DTOs
// Don't load the full entity — load only what you need
@Query("SELECT new com.example.dto.OrderSummary(" +
       "o.id, o.status, c.name, SIZE(o.items)) " +
       "FROM Order o JOIN o.customer c " +
       "WHERE o.createdAt > :from")
List<OrderSummary> findOrderSummaries(@Param("from") LocalDateTime from);

// OrderSummary DTO
@Value                          // Lombok immutable DTO
public class OrderSummary {
    Long orderId;
    OrderStatus status;
    String customerName;
    int itemCount;
}
🎯 Rule of thumb: Use JOIN FETCH for single objects, Batch Size for collections, Projections when you don't need the full entity.
🔍
LAZY vs EAGER — When to Use Which
TypeDefault forLoads whenRiskBest Practice
LAZY@OneToMany, @ManyToManyFirst access to the fieldLazyInitializationException outside sessionUse for collections — always safest
EAGER@ManyToOne, @OneToOneAlways when parent is loadedN+1 problem, loads data you may not needChange to LAZY + use JOIN FETCH when needed
✅
Golden Rule: Make ALL relationships LAZY. Then use JOIN FETCH or @EntityGraph to load what you need, when you need it. This gives you full control with zero surprise queries.

Scenario A: E-Commerce — Order Management with Optimistic Locking
Concurrent orders → Inventory control → Optimistic locking prevents oversell
Concurrency
Inventory.java — @Version prevents oversell
@Entity
public class Inventory {
    @Id private Long productId;
    private int availableStock;

    @Version                    // Hibernate checks version on UPDATE
    private Long version;       // if changed since read → throws OptimisticLockException
}

@Service
@RequiredArgsConstructor
public class OrderService {

    private final InventoryRepository inventoryRepo;
    private final OrderRepository orderRepo;

    @Transactional
    @Retryable(value = OptimisticLockingFailureException.class, maxAttempts = 3)
    public Order placeOrder(PlaceOrderRequest req) {
        // If 2 users buy last item simultaneously:
        // User A reads: stock=1, version=5
        // User B reads: stock=1, version=5
        // User A updates: stock=0, version=6 ← succeeds
        // User B tries: WHERE id=X AND version=5 → 0 rows affected
        //               → OptimisticLockException thrown!
        //               → @Retryable retries → re-reads stock=0 → fails with StockException

        Inventory inv = inventoryRepo.findById(req.getProductId())
            .orElseThrow(() -> new ProductNotFoundException(req.getProductId()));

        if (inv.getAvailableStock() < req.getQuantity()) {
            throw new InsufficientStockException("Not enough stock");
        }

        inv.setAvailableStock(inv.getAvailableStock() - req.getQuantity());
        inventoryRepo.save(inv);  // throws OptimisticLockException if version mismatch

        Order order = Order.builder()
            .customerId(req.getCustomerId())
            .productId(req.getProductId())
            .quantity(req.getQuantity())
            .status(OrderStatus.CONFIRMED)
            .build();

        return orderRepo.save(order);
    }
}

Scenario B: Healthcare — Complex Reporting with Specifications

Specifications
PatientSpecification.java
// Repository extends JpaSpecificationExecutor for Specification support
public interface PatientRepository extends JpaRepository<Patient, Long>,
                                            JpaSpecificationExecutor<Patient> {}

// Specification — reusable, composable predicates
public class PatientSpecification {

    public static Specification<Patient> hasDoctor(Long doctorId) {
        return (root, query, cb) ->
            doctorId == null ? null
                : cb.equal(root.get("doctor").get("id"), doctorId);
    }

    public static Specification<Patient> ageGreaterThan(Integer age) {
        return (root, query, cb) ->
            age == null ? null : cb.greaterThan(root.get("age"), age);
    }

    public static Specification<Patient> diagnosedWith(String condition) {
        return (root, query, cb) ->
            condition == null ? null
                : cb.like(cb.lower(root.get("diagnosis")), "%" + condition.toLowerCase() + "%");
    }

    public static Specification<Patient> admittedBetween(
            LocalDate from, LocalDate to) {
        return (root, query, cb) ->
            (from == null || to == null) ? null
                : cb.between(root.get("admissionDate"), from, to);
    }
}

// Service — compose specifications dynamically
@Service
public class PatientService {
    private final PatientRepository repo;

    public Page<Patient> search(PatientFilter filter, Pageable pageable) {
        Specification<Patient> spec = Specification
            .where(PatientSpecification.hasDoctor(filter.getDoctorId()))
            .and(PatientSpecification.ageGreaterThan(filter.getMinAge()))
            .and(PatientSpecification.diagnosedWith(filter.getCondition()))
            .and(PatientSpecification.admittedBetween(filter.getFrom(), filter.getTo()));

        return repo.findAll(spec, pageable);  // Spring builds the WHERE clause!
    }
}
Bulk operations — efficient updates
@Service
@RequiredArgsConstructor
public class BulkService {

    private final EntityManager em;

    // Efficient bulk update — no entity loading
    @Transactional
    public int dischargeAllPatients(Long wardId) {
        return em.createQuery(
            "UPDATE Patient p SET p.status = 'DISCHARGED', " +
            "p.dischargeDate = :now " +
            "WHERE p.ward.id = :wardId AND p.status = 'ADMITTED'"
        )
        .setParameter("now", LocalDate.now())
        .setParameter("wardId", wardId)
        .executeUpdate();          // returns count updated rows
    }

    // Batch INSERT — much faster than saving one by one
    @Transactional
    public void bulkImportPatients(List<Patient> patients) {
        int batchSize = 50;
        for (int i = 0; i < patients.size(); i++) {
            em.persist(patients.get(i));
            if (i % batchSize == 0 && i > 0) {
                em.flush();   // write batch to DB
                em.clear();   // clear persistence context to free memory
            }
        }
    }
}

❓ Interview Q&A — Core Concepts
JPA vs Hibernate? JPA is the specification (interfaces). Hibernate is the implementation. Spring Data JPA is a layer on top of Hibernate that auto-generates repository code.
What is N+1 problem? Loading N entities then accessing a LAZY relationship on each → N additional queries. Fix with JOIN FETCH, @EntityGraph, or @BatchSize. Detect with show-sql=true.
LAZY vs EAGER? LAZY loads relationship when first accessed. EAGER loads always. Best practice: make all relationships LAZY, use JOIN FETCH when you need the data.
Optimistic vs Pessimistic locking? Optimistic (@Version): assume no conflict, check at save time → throws on conflict. Pessimistic (SELECT FOR UPDATE): lock row immediately. Optimistic better for high-read apps.
What is Persistence Context? First-level cache managed by EntityManager. All entities loaded in a transaction are cached here. Identical lookups hit cache not DB. Cleared at transaction end.
❓ Interview Q&A — Advanced
How does @Transactional work? Spring creates AOP proxy. Before method: open transaction. On success: commit. On RuntimeException: rollback. Self-invocation (this.method()) bypasses proxy — no transaction!
ddl-auto values? none: do nothing. validate: check schema matches. update: update schema (dev). create: create tables on start. create-drop: create then drop on close. Production: none or validate.
JPQL vs Native SQL? JPQL: DB-agnostic, uses entity names, portable. Native SQL: DB-specific, can use DB features (full-text, window functions). Use JPQL first, native only when JPQL can't do it.
Page vs Slice? Page fires 2 queries: data + COUNT(*). Returns total pages/elements. Slice fires 1 query + checks for next page. Use Slice for infinite scroll — avoids expensive count on huge tables.
How to do bulk operations? @Modifying @Query for bulk UPDATE/DELETE. EntityManager.persist() in loop with flush+clear every 50 for bulk INSERT. Set spring.jpa.properties.hibernate.jdbc.batch_size=50.
