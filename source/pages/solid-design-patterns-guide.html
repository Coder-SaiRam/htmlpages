// Section 0 · Foundations
SOLID Principles
Five object-oriented design principles by Robert C. Martin ("Uncle Bob"). They make code maintainable, testable, and extendable. Think of them as guiding forces — not rigid laws. Applying them well separates good code from great architecture.
📦
S — Single Responsibility Principle
One Class, One Reason to Change
A class should have only one job. If it handles auth, sends emails, and generates PDFs — three different teams have reasons to touch it. A bug in email logic can silently break auth.
⚡ When to Apply
  • A class mixes concerns: auth + email + DB + reporting in one place
  • Multiple teams need to modify the same class for unrelated features
  • Unit testing requires too many unrelated mocks to set up
  • A "small change" to one feature keeps breaking another
🏢 Real Scenario — E-commerce Order

An Order class calculates totals, saves to MySQL, sends emails, and generates PDF invoices. When the email team changes the template, they risk breaking order calculation. Fix: split into OrderCalculator, OrderRepository, OrderNotifier, InvoiceService.

❌ Violates SRP — 4 reasons to change
class UserService {
  // Concern 1: Auth
  public User login(String u, String p) {…}
  // Concern 2: Email
  public void sendWelcomeEmail(User u) {…}
  // Concern 3: PDF
  public PDF generateReport(User u) {…}
  // Concern 4: DB
  public void save(User u) {…}
}
✅ Follows SRP — 1 reason each
class AuthService   { User login(…) {…} }
class EmailService  { void sendWelcome(…){…} }
class ReportService { PDF  generate(…) {…} }
class UserRepository{ void save(…)      {…} }

// Each class: 1 reason to change ✓
// Teams work independently ✓
// Easy to unit test ✓
🔓
O — Open/Closed Principle
Open for Extension, Closed for Modification
Once a tested class works, don't edit its internals to add new features. Extend it via interfaces or inheritance. This protects existing code from regression bugs when new features are added.
⚡ When to Apply
  • You have a growing if/else or switch that gains a new branch every sprint
  • Adding a new payment method or notification channel forces editing core logic
  • A class is heavily tested and you want zero regression risk
  • Building a framework — let users plug in extensions without touching your source
🏢 Real Scenario — Payment Gateway

A system supports CreditCard and PayPal. Adding Crypto means opening PaymentProcessor and adding another else if — risking breakage. OCP fix: define a PaymentMethod interface. Adding Crypto = one new class. PaymentProcessor never changes again.

❌ Edit core class every time
class PaymentProcessor {
  public void pay(String type, double amt) {
    if (type.equals("credit")) { … }
    else if (type.equals("paypal")) { … }
    else if (type.equals("crypto")) {
      // Must edit this class again! ❌
    }
  }
}
✅ Add a new class, nothing changes
interface PaymentMethod {
  void pay(double amount);
}
class CreditCard implements PaymentMethod {…}
class Crypto     implements PaymentMethod {…}

// This class NEVER changes again ✓
class PaymentProcessor {
  void process(PaymentMethod m, double amt) {
    m.pay(amt);
  }
}
🔄
L — Liskov Substitution Principle
Subtypes Must Be Substitutable for Their Bases
If S is a subtype of T, replacing T with S must not break the program. Child classes must honour ALL contracts of their parent — never weaken, never break them.
⚡ Warning Signs (LSP Violation)
  • Subclass throws UnsupportedOperationException for a parent method
  • You see instanceof checks to handle specific subclasses differently
  • Overridden method does nothing, or silently ignores a parameter
  • Replacing parent with child object breaks existing unit tests
🏢 Classic Trap — Square extends Rectangle

Intuitively Square IS-A Rectangle. But Rectangle's contract says: setting width doesn't change height. A Square must break this rule. Fix: both extend a Shape base class — they're siblings, not parent/child.

❌ Violates LSP — breaks parent contract
class Rectangle {
  protected int w, h;
  void setWidth(int w)  { this.w = w; }
  void setHeight(int h) { this.h = h; }
  int  area() { return w * h; }
}
class Square extends Rectangle {
  void setWidth(int w) {
    this.w = this.h = w; // ← breaks contract!
  }
}
// setWidth(5);setHeight(4); → 25 not 20 ❌
✅ Siblings under shared abstraction
abstract class Shape {
  abstract int area();
}
class Rectangle extends Shape {
  int w, h;
  int area() { return w * h; }
}
class Square extends Shape {
  int side;
  int area() { return side * side; }
}
// Both safely substitute Shape ✓
🎯
I — Interface Segregation Principle
No Class Should Implement Methods It Doesn't Need
Fat interfaces force implementers to stub out irrelevant methods. Prefer many small, focused interfaces. Clients depend only on what they actually use.
⚡ When to Apply
  • You see empty method bodies or throw new UnsupportedOperationException()
  • An interface keeps growing, forcing all implementers to update
  • Some implementers only use 2 of 10 methods — the rest are dead weight
  • "Why does a Printer need a scan() method?"
🏢 Real Scenario — Robot Workers

A Worker interface has work(), eat(), sleep(). A Robot implementation throws exceptions for eat and sleep. Fix: split into Workable, Eatable, Sleepable. Robot only implements Workable.

❌ Fat interface forces stubs
interface Worker {
  void work();
  void eat();
  void sleep();
}
class Robot implements Worker {
  public void work()  { /* ok */ }
  public void eat()   {
    throw new UnsupportedOperationException();
  }
  public void sleep() {
    throw new UnsupportedOperationException();
  }
}
✅ Small interfaces — take what you need
interface Workable  { void work(); }
interface Eatable   { void eat(); }
interface Sleepable { void sleep(); }

class Robot implements Workable {
  public void work() { /* ok */ }
}
class Human implements Workable,
    Eatable, Sleepable { … }
// Each implements only what it needs ✓
🔌
D — Dependency Inversion Principle
Depend on Abstractions, Not Concretions
High-level business logic must not depend on low-level details (DB, HTTP, filesystem). Both depend on interfaces. This makes business logic testable, portable, and immune to infrastructure changes.
⚡ When to Apply
  • Business logic creates new MySQLRepo() or new HttpClient() directly inside itself
  • Unit tests need a running database, real HTTP server, or real filesystem
  • Switching from MySQL to PostgreSQL forces editing business logic files
  • High-level class imports low-level framework class names
🏢 Real Scenario — OrderService + Database

An OrderService creates new MySQLOrderRepository() inside itself. To write a unit test you need MySQL running. Fix: OrderService depends on an OrderRepository interface. Inject MySQLRepo in production, MockRepo in tests — zero changes to business logic.

❌ Hard-coded — untestable
class OrderService {
  // Directly creates concrete class!
  private MySQLOrderRepo repo =
      new MySQLOrderRepo();

  void placeOrder(Order o) { repo.save(o); }
}
// Need real MySQL to test this ❌
// Can't swap to PostgreSQL easily ❌
✅ Injected interface — fully testable
interface OrderRepository {
  void save(Order order);
}
class OrderService {
  private final OrderRepository repo;
  // Constructor injection — no concretion
  OrderService(OrderRepository r) {
    this.repo = r;
  }
  void placeOrder(Order o) { repo.save(o); }
}
// Test: pass MockRepo → no DB needed ✓

🏗️ CREATIONAL
Singleton Pattern
Ensures a class has exactly ONE instance with a global access point. Like a single database connection pool — you want one shared managed pool, not hundreds of separate ones competing for resources.
⚡ Use When
  • Exactly ONE instance is needed: Logger, Config manager, Thread pool, DB connection pool
  • Shared state must be globally consistent (e.g., app configuration loaded once)
  • Object creation is expensive — load config file once, reuse everywhere
⚠️ Avoid when: You need it purely as a global variable substitute. Overuse makes unit testing hard — you can't inject a mock. Prefer dependency injection in most cases.
🏢 Scenario — Application Logger

Your app needs ONE Logger writing to one log file. If every class creates its own Logger instance, you get file locking conflicts and interleaved messy logs. A Singleton Logger ensures all components write through the exact same instance.

Logger.java
Java
public class Logger {
    // volatile ensures correct visibility across threads
    private static volatile Logger instance;

    private Logger() {} // private — no external instantiation

    // Double-checked locking: thread-safe lazy init
    public static Logger getInstance() {
        if (instance == null) {
            synchronized (Logger.class) {
                if (instance == null)
                    instance = new Logger();
            }
        }
        return instance;
    }

    public void log(String level, String msg) {
        System.out.printf("[%s] %s%n", level, msg);
    }
}

// Same instance everywhere in the app ✓
Logger.getInstance().log("INFO", "Order placed");
Logger.getInstance().log("INFO", "Payment processed");

🏗️ CREATIONAL
Factory Method Pattern
Defines an interface for creating objects but lets subclasses (or a factory class) decide which concrete class to instantiate. The client only knows the interface — it never hard-codes which class it gets.
⚡ Use When
  • You have a growing if/switch block deciding which class to instantiate
  • A class can't anticipate what type of objects to create at compile time
  • You want creation logic isolated in one place (pairs beautifully with OCP)
  • Framework design — let users register custom implementations
🏢 Scenario — Multi-Channel Notifications

Your app sends notifications via Email, SMS, and Push. Each has different setup. A NotificationFactory creates the right type. When WhatsApp is added, you add ONE new class — zero edits to existing code. The factory grows, nothing breaks.

NotificationFactory.java
Java
// Product interface
interface Notification {
    void send(String recipient, String message);
}

// Concrete products
class EmailNotification implements Notification {
    public void send(String to, String msg) {
        System.out.println("📧 Email → " + to + ": " + msg);
    }
}
class SMSNotification implements Notification {
    public void send(String to, String msg) {
        System.out.println("📱 SMS → " + to + ": " + msg);
    }
}
class PushNotification implements Notification {
    public void send(String to, String msg) {
        System.out.println("🔔 Push → " + to + ": " + msg);
    }
}

// Factory — central creation point
class NotificationFactory {
    public static Notification create(String channel) {
        return switch (channel.toLowerCase()) {
            case "email" -> new EmailNotification();
            case "sms"   -> new SMSNotification();
            case "push"  -> new PushNotification();
            default -> throw new IllegalArgumentException("Unknown: " + channel);
        };
    }
}

// Client — never knows which concrete class it got
Notification n = NotificationFactory.create("email");
n.send("user@example.com", "Your order has shipped!");

🏗️ CREATIONAL
Builder Pattern
Constructs complex objects step-by-step. Eliminates telescoping constructors with 8+ parameters. Each step is optional with sensible defaults. The fluent chain is readable and self-documenting.
⚡ Use When
  • Object has many optional fields (5+) making constructors unreadable
  • You want self-documenting, fluent construction code
  • Different "configurations" of the same object are needed
  • Real-world: Lombok's @Builder, OkHttp's Request.Builder, Java's StringBuilder
🏢 Scenario — HTTP Request Builder

Building an HTTP request needs a URL (required), but headers, timeout, auth token, body, retry policy are optional. A Builder chains only what you need — making construction readable and impossible to misuse with wrong parameter order.

HttpRequest.java
Java
class HttpRequest {
    private final String  url;           // required
    private final String  method;        // default GET
    private final String  body, authToken;
    private final int     timeoutMs;
    private final boolean followRedirects;

    private HttpRequest(Builder b) {
        url=b.url; method=b.method; body=b.body;
        authToken=b.authToken; timeoutMs=b.timeoutMs;
        followRedirects=b.followRedirects;
    }

    public static class Builder {
        private final String  url;
        private String  method="GET", body, authToken;
        private int     timeoutMs=5000;
        private boolean followRedirects=true;

        public Builder(String url)       { this.url=url; }
        public Builder method(String m)  { method=m; return this; }
        public Builder body(String b)    { body=b; return this; }
        public Builder auth(String t)    { authToken=t; return this; }
        public Builder timeout(int ms)   { timeoutMs=ms; return this; }
        public Builder noRedirects()     { followRedirects=false; return this; }
        public HttpRequest build()        { return new HttpRequest(this); }
    }
}

// Usage — readable, fluent, impossible to pass args in wrong order
HttpRequest req = new HttpRequest.Builder("https://api.example.com/orders")
    .method("POST")
    .body("{\"item\":\"laptop\"}")
    .auth("Bearer eyJhbGci...")
    .timeout(10000)
    .build();

🏗️ CREATIONAL
Prototype Pattern
Creates new objects by cloning existing ones. When object creation is expensive (DB lookups, asset loading, heavy computation), copying and tweaking an existing object is orders of magnitude faster than re-initializing.
⚡ Use When
  • Object initialization is expensive and you need many similar objects
  • Game dev: clone enemy templates to spawn hundreds efficiently
  • Document templates: clone base document, fill in variable parts
  • Configuration objects: clone base config and override specific keys
🏢 Scenario — Game Enemy Spawning

An Orc enemy loads sprites, sounds, and AI config from disk — takes 200ms. Spawning 500 Orcs the normal way: 100 seconds! With Prototype: create ONE template Orc (200ms), clone it 499 times (microseconds each). Total: ~200ms.

EnemyPrototype.java
Java
abstract class Enemy implements Cloneable {
    protected String type, spriteSheet, aiConfig;
    protected int    health, damage, speed;

    public Enemy clone() {
        try { return (Enemy) super.clone(); }
        catch (CloneNotSupportedException e) { throw new RuntimeException(e); }
    }
}

class Orc extends Enemy {
    public Orc() {
        type = "Orc"; health = 100; damage = 20;
        // Expensive operations — load once
        spriteSheet = loadSprite("orc.png");  // 100ms
        aiConfig    = loadConfig("orc_ai.json"); // 100ms
    }
}

// Create template ONCE, clone MANY times
Enemy orcTemplate = new Orc();  // 200ms

for (int i = 0; i < 500; i++) {
    Enemy orc = orcTemplate.clone();      // microseconds!
    orc.health = 80 + (int)(Math.random() * 40);
    spawnAt(orc, randomPosition());
}
// Total: ~200ms instead of 100 seconds ✓

🏗️ CREATIONAL
Abstract Factory Pattern
Creates families of related objects without specifying their concrete classes. Like a furniture store — when you choose "Victorian", ALL pieces (chair, table, sofa) come in Victorian style. Consistency is enforced by design.
⚡ Use When
  • Your system works with multiple families of related products that must be used together
  • Enforce consistency — if "Dark Theme", ALL UI components must be dark
  • Cross-platform toolkits: Windows controls vs Mac controls vs Web controls
  • Database drivers: MySQL factory creates MySQLConnection + MySQLCommand together
🏢 Scenario — Cross-Platform UI Toolkit

A GUI framework renders Buttons and Checkboxes for Windows and Mac. An UIFactory interface creates matched components. App code calls factory.createButton() — it doesn't know if it's Windows or Mac. Switching OS = swapping one factory object.

UIAbstractFactory.java
Java
// Product interfaces
interface Button   { void render(); }
interface Checkbox { void render(); }

// Windows family
class WinButton   implements Button   { public void render() { System.out.println("[Win Button]");   } }
class WinCheckbox implements Checkbox { public void render() { System.out.println("[Win Checkbox]"); } }

// Mac family
class MacButton   implements Button   { public void render() { System.out.println("[Mac Button]");   } }
class MacCheckbox implements Checkbox { public void render() { System.out.println("[Mac Checkbox]"); } }

// Abstract Factory interface
interface UIFactory {
    Button   createButton();
    Checkbox createCheckbox();
}
class WinFactory implements UIFactory {
    public Button   createButton()   { return new WinButton(); }
    public Checkbox createCheckbox() { return new WinCheckbox(); }
}
class MacFactory implements UIFactory {
    public Button   createButton()   { return new MacButton(); }
    public Checkbox createCheckbox() { return new MacCheckbox(); }
}

// Client — consistent family guaranteed, never mixed!
class App {
    void buildUI(UIFactory factory) {
        factory.createButton().render();   // always matched ✓
        factory.createCheckbox().render(); // always matched ✓
    }
}
new App().buildUI(new WinFactory()); // switch to MacFactory — one line change

🔩 STRUCTURAL
Adapter Pattern
Converts one interface into another that clients expect. Like a power plug adapter letting a UK plug work in a US socket. Perfect for integrating third-party libraries or legacy code without modifying either side.
⚡ Use When
  • Integrating a third-party library with an incompatible interface (Stripe, Twilio, AWS SDKs)
  • Reusing legacy code in a new system without touching the legacy code
  • Your app expects one interface, external code provides another
🏢 Scenario — Stripe Payment Integration

Your app uses PaymentGateway.charge(customerId, amount). Stripe's SDK has createCharge(stripeId, amountInCents). You can't change either. A StripeAdapter wraps Stripe and exposes your expected interface. Tomorrow swap to PayPal? New adapter, zero client changes.

StripeAdapter.java
Java
// Your app's expected interface (Target)
interface PaymentGateway {
    boolean charge(String customerId, double dollars);
}

// Third-party Stripe SDK — cannot modify!
class StripeSDK {
    public boolean createCharge(String stripeId, long cents) {
        System.out.println("Stripe: charged " + cents + " cents");
        return true;
    }
}

// Adapter — wraps Stripe, exposes your interface
class StripeAdapter implements PaymentGateway {
    private final StripeSDK stripe;
    public StripeAdapter(StripeSDK s) { this.stripe = s; }

    @Override
    public boolean charge(String customerId, double dollars) {
        long cents = (long)(dollars * 100); // translate units
        return stripe.createCharge(customerId, cents);
    }
}

// Client — clean, consistent API. Never knows about Stripe!
PaymentGateway gw = new StripeAdapter(new StripeSDK());
gw.charge("cust_abc", 49.99);

🔩 STRUCTURAL
Decorator Pattern
Dynamically adds responsibilities to objects without subclassing. Stack decorators like pizza toppings — each wraps the previous result, adding its own behaviour. Avoids class explosion when you need many feature combinations.
⚡ Use When
  • Add behaviour to individual objects without affecting others of the same class
  • Subclassing leads to class explosion (CoffeeWithMilkAndSugarAndVanilla?)
  • Features need to be added/removed at runtime (feature flags, permission levels)
  • Real-world: Java I/O streams, HTTP middleware, Spring Security filter chains
🏢 Scenario — Coffee Shop Customization

A coffee shop has base SimpleCoffee. Customers mix: Milk, Sugar, Vanilla, Caramel, Extra Shot. With 5 toppings = 2⁵=32 subclasses! With Decorator: each topping wraps any coffee, adds cost/description. Stack in any order, any combination — 5 classes total.

CoffeeDecorator.java
Java
interface Coffee {
    String getDescription();
    double getCost();
}

class SimpleCoffee implements Coffee {
    public String getDescription() { return "Coffee"; }
    public double getCost()        { return 1.00; }
}

// Abstract decorator — wraps any Coffee
abstract class CoffeeDecorator implements Coffee {
    protected final Coffee wrapped;
    CoffeeDecorator(Coffee c) { this.wrapped = c; }
}

// Concrete decorators — each adds its own touch
class MilkDecorator extends CoffeeDecorator {
    MilkDecorator(Coffee c) { super(c); }
    public String getDescription() { return wrapped.getDescription() + ", Milk"; }
    public double getCost()        { return wrapped.getCost() + 0.25; }
}
class VanillaDecorator extends CoffeeDecorator {
    VanillaDecorator(Coffee c) { super(c); }
    public String getDescription() { return wrapped.getDescription() + ", Vanilla"; }
    public double getCost()        { return wrapped.getCost() + 0.50; }
}
class ExtraShotDecorator extends CoffeeDecorator {
    ExtraShotDecorator(Coffee c) { super(c); }
    public String getDescription() { return wrapped.getDescription() + ", Extra Shot"; }
    public double getCost()        { return wrapped.getCost() + 0.75; }
}

// Stack decorators like toppings!
Coffee order = new ExtraShotDecorator(
                 new VanillaDecorator(
                   new MilkDecorator(
                     new SimpleCoffee())));
// "Coffee, Milk, Vanilla, Extra Shot" → $2.50 ✓

🔩 STRUCTURAL
Facade Pattern
Provides a simplified, unified interface to a complex subsystem. Like a hotel concierge — one person coordinates taxi, laundry, and restaurant reservations so you don't call each department separately.
⚡ Use When
  • A subsystem has many interdependent classes; clients need a simple entry point
  • Layering your app — controllers call only the service facade, not repositories directly
  • Building a library: expose a clean API, hide messy internals
  • Real-world: Spring's JdbcTemplate, REST controllers orchestrating services
🏢 Scenario — Order Placement Flow

Placing an order requires: check inventory → charge payment → save to DB → send email. The controller shouldn't orchestrate all 4 services manually. An OrderFacade.placeOrder() handles everything behind one clean method call.

OrderFacade.java
Java
// Complex subsystems — don't expose to clients directly
class InventoryService {
    boolean checkStock(String id, int qty) { return true; }
    void    reserve(String id, int qty)    { }
}
class PaymentService {
    boolean charge(String customerId, double amt) { return true; }
}
class EmailService {
    void sendConfirmation(String email, String orderId) { }
}
class OrderRepository {
    String save(Order o) { return "ORD-001"; }
}

// FACADE — one clean method hides all the complexity
class OrderFacade {
    private final InventoryService inv   = new InventoryService();
    private final PaymentService   pay   = new PaymentService();
    private final EmailService     email = new EmailService();
    private final OrderRepository  repo  = new OrderRepository();

    public String placeOrder(OrderRequest req) {
        if (!inv.checkStock(req.productId, req.qty))   return "OUT_OF_STOCK";
        if (!pay.charge(req.customerId, req.amount))   return "PAYMENT_FAILED";
        inv.reserve(req.productId, req.qty);
        String id = repo.save(new Order(req));
        email.sendConfirmation(req.email, id);
        return id; // "ORD-001"
    }
}

// Controller — beautifully simple ✓
String orderId = new OrderFacade().placeOrder(request);

🔩 STRUCTURAL
Proxy Pattern
A surrogate that controls access to another object — transparently adding caching, access control, lazy loading, or logging. The client calls the same interface, never knowing there's a proxy in between.
⚡ Four Types of Proxy
  • Virtual proxy: Delay expensive creation until first use (lazy-loading images/objects)
  • Caching proxy: Cache results of expensive DB queries or API calls
  • Protection proxy: Check permissions before allowing access
  • Logging proxy: Log all calls without touching the original class
🏢 Scenario — Caching Database Proxy

Every call to getUserById() hits the DB — 50ms each. With a caching proxy, the first call fetches from DB, subsequent calls with the same ID return instantly from cache. Client code is identical — zero changes required.

CachingProxy.java
Java
interface UserService {
    User getUserById(int id);
}

// Real service — hits the database
class DatabaseUserService implements UserService {
    public User getUserById(int id) {
        System.out.println("🗄️  DB query for id=" + id);
        return new User(id, "Alice"); // 50ms
    }
}

// Caching Proxy — same interface, adds cache transparently
class CachingUserService implements UserService {
    private final UserService             real  = new DatabaseUserService();
    private final Map<Integer,User>       cache = new HashMap<>();

    public User getUserById(int id) {
        return cache.computeIfAbsent(id, k -> {
            System.out.println("⚡ Cache miss — fetching from DB");
            return real.getUserById(k);
        });
    }
}

// Client — identical code, dramatic performance difference!
UserService svc = new CachingUserService();
svc.getUserById(42); // 🗄️  DB hit — 50ms
svc.getUserById(42); // ⚡ Cache hit — 0.1ms!
svc.getUserById(42); // ⚡ Cache hit — 0.1ms!

🔩 STRUCTURAL
Composite Pattern
Treats individual objects and groups of objects uniformly via the same interface. A folder's getSize() recursively sums files inside it — the same method works on a single file or a folder containing 10,000 files.
⚡ Use When
  • You have hierarchical tree structures: file systems, org charts, DOM trees, menus
  • Client code should treat leaves and containers (groups) identically
  • Recursive operations: sum totals, render whole UI tree, calculate total org salary
FileSystem.java
Java
interface FileSystemItem {
    String getName();
    long   getSize();  // works on File AND Folder!
    void   print(String indent);
}

// Leaf — no children
class File implements FileSystemItem {
    private final String name; private final long size;
    File(String n, long s) { name=n; size=s; }
    public String getName() { return name; }
    public long   getSize() { return size; }
    public void   print(String i) { System.out.println(i+"📄 "+name+" ("+size+"b)"); }
}

// Composite — has children
class Folder implements FileSystemItem {
    private final String name;
    private final List<FileSystemItem> children = new ArrayList<>();
    Folder(String n) { name=n; }
    void add(FileSystemItem i) { children.add(i); }
    public String getName() { return name; }
    public long   getSize() {
        return children.stream().mapToLong(FileSystemItem::getSize).sum();
    }
    public void print(String i) {
        System.out.println(i+"📁 "+name);
        children.forEach(c -> c.print(i+"   "));
    }
}

// Build tree and use uniformly
Folder root = new Folder("project");
Folder src  = new Folder("src");
src.add(new File("Main.java",  3500));
src.add(new File("Utils.java", 1200));
root.add(src);
root.add(new File("README.md", 800));
root.print("");                 // recursive print
root.getSize();                // → 5500 bytes

🎭 BEHAVIORAL
Strategy Pattern
Defines a family of algorithms, encapsulates each, and makes them interchangeable at runtime. Eliminates massive if/else selecting different algorithm branches. This is OCP applied to algorithms.
⚡ Use When
  • Multiple variants of an algorithm exist and you need to switch between them at runtime
  • A class is bloated with if/else selecting different algorithm branches
  • Sorting strategies, discount engines, routing algorithms, compression methods
  • You want to test each algorithm in complete isolation
🏢 Scenario — E-commerce Discount Engine

Checkout applies different discount strategies: percentage off, flat amount, BOGO, seasonal. Instead of one huge conditional, each strategy is a class. Swap at runtime based on customer tier, cart value, or active promotions.

DiscountStrategy.java
Java
// Strategy interface
interface DiscountStrategy {
    double apply(double price);
    String getDescription();
}

// Concrete strategies
class PercentageDiscount implements DiscountStrategy {
    private final double pct;
    PercentageDiscount(double pct) { this.pct = pct; }
    public double apply(double p)  { return p * (1 - pct/100); }
    public String getDescription() { return pct + "% off"; }
}
class FlatDiscount implements DiscountStrategy {
    private final double amt;
    FlatDiscount(double amt) { this.amt = amt; }
    public double apply(double p)  { return Math.max(0, p - amt); }
    public String getDescription() { return "$" + amt + " off"; }
}
class NoDiscount implements DiscountStrategy {
    public double apply(double p)  { return p; }
    public String getDescription() { return "No discount"; }
}

// Context — uses whatever strategy is currently set
class ShoppingCart {
    private DiscountStrategy strategy = new NoDiscount();
    private double subtotal = 100.00;

    void setStrategy(DiscountStrategy s) { strategy = s; }

    double checkout() {
        double final_ = strategy.apply(subtotal);
        System.out.printf("%s: $%.2f → $%.2f%n",
            strategy.getDescription(), subtotal, final_);
        return final_;
    }
}

// Swap strategy at runtime ✓
ShoppingCart cart = new ShoppingCart();
cart.setStrategy(new PercentageDiscount(20)); cart.checkout(); // $80.00
cart.setStrategy(new FlatDiscount(15));      cart.checkout(); // $85.00

🎭 BEHAVIORAL
Observer Pattern
Defines a one-to-many dependency: when one object changes state, all dependents are notified automatically. The backbone of event-driven systems, reactive programming, and pub/sub architecture.
⚡ Use When
  • Change in one object requires notifying others — and you don't know how many at compile time
  • You want loosely coupled, event-driven architecture
  • Real-world: UI event listeners, stock price feeds, domain events in microservices
  • MVC pattern: Model notifies Views when data changes
🏢 Scenario — Order Status Events

When order status changes to "SHIPPED", multiple systems react: send email, update inventory, notify analytics, trigger billing. Instead of Order calling all systems directly, it fires an event. Each system subscribes independently — completely decoupled.

OrderObserver.java
Java
// Observer interface
interface OrderObserver {
    void onStatusChanged(int orderId, String status);
}

// Concrete observers — each reacts independently
class EmailNotifier implements OrderObserver {
    public void onStatusChanged(int id, String s) {
        System.out.println("📧 Email: Order " + id + " is " + s);
    }
}
class InventoryUpdater implements OrderObserver {
    public void onStatusChanged(int id, String s) {
        if("SHIPPED".equals(s)) System.out.println("📦 Deducting stock for order "+id);
    }
}
class AnalyticsTracker implements OrderObserver {
    public void onStatusChanged(int id, String s) {
        System.out.println("📊 Tracked: order "+id+" → "+s);
    }
}

// Subject — manages observers and fires events
class Order {
    private final int id;
    private String status = "PENDING";
    private final List<OrderObserver> observers = new ArrayList<>();

    Order(int id) { this.id = id; }
    void subscribe(OrderObserver o)   { observers.add(o); }
    void unsubscribe(OrderObserver o) { observers.remove(o); }

    void updateStatus(String newStatus) {
        status = newStatus;
        observers.forEach(o -> o.onStatusChanged(id, status));
    }
}

// Usage — fully decoupled ✓
Order order = new Order(1001);
order.subscribe(new EmailNotifier());
order.subscribe(new InventoryUpdater());
order.subscribe(new AnalyticsTracker());
order.updateStatus("SHIPPED"); // All three observers fire!

🎭 BEHAVIORAL
Command Pattern
Encapsulates a request as an object — enabling undo/redo, queuing, logging, and transaction rollback. Like a restaurant order ticket: the waiter doesn't cook, they pass the ticket. The kitchen processes it independently.
⚡ Use When
  • You need undo/redo functionality (text editors, Photoshop, IDE actions)
  • Operations need to be queued or scheduled for later execution
  • Macro recording: capture a sequence of commands and replay them
  • Transaction logs: persist commands and replay on system failure
🏢 Scenario — Text Editor Undo/Redo

Every user action (type, delete, bold, insert) is a Command with execute() and undo(). The editor maintains a history stack. Ctrl+Z pops and calls undo(). Ctrl+Y re-executes. Adding "paste" functionality = adding one new Command class.

TextEditorCommand.java
Java
interface Command { void execute(); void undo(); }

// Receiver — the object being operated on
class TextDocument {
    private StringBuilder content = new StringBuilder();
    void insert(int pos, String text) { content.insert(pos, text); }
    void delete(int pos, int len)   { content.delete(pos, pos+len); }
    String getText()                  { return content.toString(); }
}

// Command knows how to execute AND undo itself
class InsertCommand implements Command {
    private final TextDocument doc; private final int pos; private final String text;
    InsertCommand(TextDocument d, int p, String t) { doc=d; pos=p; text=t; }
    public void execute() { doc.insert(pos, text); }
    public void undo()    { doc.delete(pos, text.length()); }
}

// Invoker — manages history for undo/redo
class TextEditor {
    private final Deque<Command> history   = new ArrayDeque<>();
    private final Deque<Command> redoStack = new ArrayDeque<>();

    void execute(Command c) { c.execute(); history.push(c); redoStack.clear(); }
    void undo() {
        if(!history.isEmpty()) { Command c=history.pop(); c.undo(); redoStack.push(c); }
    }
    void redo() {
        if(!redoStack.isEmpty()) { Command c=redoStack.pop(); c.execute(); history.push(c); }
    }
}

TextDocument doc    = new TextDocument();
TextEditor   editor = new TextEditor();
editor.execute(new InsertCommand(doc, 0, "Hello"));
editor.execute(new InsertCommand(doc, 5, " World")); // "Hello World"
editor.undo();   // "Hello"
editor.redo();   // "Hello World"

🎭 BEHAVIORAL
Chain of Responsibility
Passes a request along a chain of handlers until one handles it. The sender doesn't know which handler will process it. Like escalating a support ticket up tiers until someone with enough authority resolves it.
⚡ Use When
  • More than one handler may process a request; the handler isn't known at compile time
  • HTTP middleware pipelines (auth → logging → rate limiting → handler)
  • Approval workflows with escalation thresholds (Team Lead → Manager → Director)
  • Adding a new handler = adding one class, zero edits to existing code
ApprovalChain.java
Java
abstract class Approver {
    protected Approver next;
    protected final String name;
    protected final double limit;

    Approver(String n, double l) { name=n; limit=l; }

    // Fluent chaining: lead.setNext(mgr).setNext(dir)
    Approver setNext(Approver a) { next=a; return a; }

    void approve(double amount) {
        if(amount <= limit)    System.out.printf("✅ %s approved $%.0f%n", name, amount);
        else if(next != null) { System.out.printf("↑  %s escalated $%.0f%n",  name, amount); next.approve(amount); }
        else                   System.out.printf("❌ $%.0f exceeds all limits%n", amount);
    }
}

class TeamLead extends Approver { TeamLead() { super("Team Lead", 1_000); } }
class Manager  extends Approver { Manager()  { super("Manager",   10_000); } }
class Director extends Approver { Director() { super("Director",  100_000); } }

// Build chain, then submit any amount
Approver lead = new TeamLead();
lead.setNext(new Manager()).setNext(new Director());
lead.approve(500);    // ✅ Team Lead approved
lead.approve(5_000);  // ↑ escalated → ✅ Manager approved
lead.approve(50_000); // ↑ → ↑ → ✅ Director approved

🎭 BEHAVIORAL
State Pattern
Allows an object to change behaviour when its internal state changes. Instead of massive if/switch checking a status field, delegate to the current State object — it handles behaviour for that exact state and manages transitions.
⚡ Use When
  • Object behaviour changes dramatically based on its current state
  • You have large if/switch on a status or state field scattered everywhere
  • State transitions need to be explicit and have rules (PENDING → SHIPPED, not SHIPPED → PENDING)
  • Vending machines, order workflows, TCP connections, traffic lights
🏢 Scenario — Order State Machine

Order states: PENDING → CONFIRMED → SHIPPED → DELIVERED (or CANCELLED). Each state allows different actions: PENDING can be cancelled, SHIPPED cannot. State classes enforce these rules without any if/else in client code.

OrderStateMachine.java
Java
interface OrderState {
    void confirm(OrderCtx ctx);
    void ship(OrderCtx ctx);
    void cancel(OrderCtx ctx);
}

class PendingState implements OrderState {
    public void confirm(OrderCtx c) { c.setState(new ConfirmedState()); }
    public void ship(OrderCtx c)    { System.out.println("❌ Must confirm first"); }
    public void cancel(OrderCtx c)  { c.setState(new CancelledState()); }
}
class ConfirmedState implements OrderState {
    public void confirm(OrderCtx c) { System.out.println("Already confirmed"); }
    public void ship(OrderCtx c)    { c.setState(new ShippedState()); }
    public void cancel(OrderCtx c)  { c.setState(new CancelledState()); }
}
class ShippedState implements OrderState {
    public void confirm(OrderCtx c) { System.out.println("Already shipped"); }
    public void ship(OrderCtx c)    { System.out.println("Already shipped"); }
    public void cancel(OrderCtx c)  { System.out.println("❌ Cannot cancel shipped order"); }
}

// Context — delegates to current state
class OrderCtx {
    private OrderState state = new PendingState();
    void setState(OrderState s) { state=s; System.out.println("→ "+s.getClass().getSimpleName()); }
    void confirm() { state.confirm(this); }
    void ship()    { state.ship(this); }
    void cancel()  { state.cancel(this); }
}

OrderCtx order = new OrderCtx();
order.ship();     // ❌ Must confirm first
order.confirm(); // → ConfirmedState
order.ship();     // → ShippedState
order.cancel();   // ❌ Cannot cancel shipped order

🎭 BEHAVIORAL
Template Method Pattern
Defines an algorithm's skeleton in a base class; subclasses override specific steps without changing the structure. The "Hollywood Principle": don't call us, we'll call you. The framework controls the flow; subclasses fill the hooks.
⚡ Use When
  • Multiple classes share the same algorithm structure but differ in specific steps
  • Avoid code duplication in similar algorithms (CSV export vs JSON export)
  • Framework design: control the flow, let subclasses provide custom steps
  • Real-world: JUnit test lifecycle, Spring's JdbcTemplate, data processing pipelines
🏢 Scenario — Data Export Pipeline

Exporting CSV and JSON follows: fetch data → format records → write to file → notify complete. Same steps, only "format records" differs. Template Method defines the fixed skeleton; subclasses provide just the format-specific logic.

DataExporter.java
Java
abstract class DataExporter {

    // Template method — the FIXED skeleton (final, no override)
    public final void export(String dest) {
        List<Record> data = fetchData();     // step 1 — shared
        String formatted  = formatData(data); // step 2 — abstract
        writeTo(dest, formatted);             // step 3 — shared
        onComplete(dest);                     // step 4 — hook (optional)
    }

    private List<Record> fetchData() {
        System.out.println("📥 Fetching data...");
        return List.of(new Record("Alice",30), new Record("Bob",25));
    }
    protected abstract String formatData(List<Record> data); // Must implement
    private void writeTo(String d, String c) { System.out.println("💾 Writing → " + d); }
    protected void onComplete(String d) { System.out.println("✅ Done: " + d); } // Hook
}

class CsvExporter extends DataExporter {
    @Override
    protected String formatData(List<Record> data) {
        StringBuilder sb = new StringBuilder("name,age\n");
        data.forEach(r -> sb.append(r.name).append(",").append(r.age).append("\n"));
        return sb.toString();
    }
}
class JsonExporter extends DataExporter {
    @Override
    protected String formatData(List<Record> data) {
        return "[{" + data.stream().map(r ->
            "\"name\":\""+r.name+"\",\"age\":"+r.age).collect(joining("},{"))+"}]";
    }
}

// Same export() call — different formats ✓
new CsvExporter().export("users.csv");
new JsonExporter().export("users.json");

💡 Pattern Selection Guide

🔨 Object Creation Problem?
  • Need ONE instance → Singleton
  • Need to choose a type → Factory Method
  • Need matching family → Abstract Factory
  • Too many parameters → Builder
  • Expensive init → Prototype
🔩 Structure Problem?
  • Incompatible interface → Adapter
  • Add features without class explosion → Decorator
  • Hide complex subsystem → Facade
  • Control access / caching → Proxy
  • Hierarchical tree → Composite
🎭 Behaviour Problem?
  • Switch algorithm at runtime → Strategy
  • React to state changes → Observer
  • Need undo/redo → Command
  • Object grows with state → State
  • Multiple handlers → Chain of Resp.
