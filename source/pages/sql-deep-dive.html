SQL Foundations & Data Types
DDL · DML · DQL · DCL — the four SQL command categories
📋
DDL
Data Definition Language

Defines database structure. CREATE, ALTER, DROP, TRUNCATE. These auto-commit — cannot be rolled back in most DBs.

🏗️ Designing the blueprint of a building
✏️
DML
Data Manipulation Language

Works with data in rows. INSERT, UPDATE, DELETE, MERGE. Wrapped in transactions — can be rolled back.

🪑 Moving, adding, removing furniture
🔍
DQL
Data Query Language

Reads data. SELECT is the only command. Most complex and most used. Never modifies data.

👀 Reading a book — doesn't change it
🔐
DCL
Data Control Language

Manages permissions. GRANT and REVOKE. Controls who can do what on which objects.

🗝️ Giving or revoking building access cards

SELECT Deep Dive
WHERE · ORDER BY · LIMIT · DISTINCT · CASE · String & Date functions
🔍 SELECT Execution Order — Critical Concept

SQL is not executed in the order you write it. Understanding the execution order explains why you can't use a WHERE clause alias defined in SELECT.

📥 SQL CLAUSE EXECUTION ORDER
1
FROM / JOIN
Identify all source tables. Build the cartesian product, then apply JOIN conditions.
↓
2
WHERE
Filter individual rows. Aliases from SELECT not yet available here.
↓
3
GROUP BY
Group filtered rows into buckets by specified columns.
↓
4
HAVING
Filter groups. Aggregate functions allowed here (COUNT, SUM, etc).
↓
5
SELECT
Compute expressions, aliases, window functions. Column list evaluated here.
↓
6
DISTINCT
Remove duplicate rows from result set.
↓
7
ORDER BY
Sort the final result. SELECT aliases CAN be used here (after SELECT is evaluated).
↓
8
LIMIT / OFFSET
Trim the sorted result to the requested page of rows.
✨ WHERE Clause — All Conditions
where_examples.sqlSQL
-- ─── COMPARISON OPERATORS ───────────────────────────────────────────────
SELECT * FROM employees WHERE salary > 50000;
SELECT * FROM employees WHERE salary BETWEEN 40000 AND 80000;   -- inclusive both ends
SELECT * FROM employees WHERE hire_date >= '2022-01-01';

-- ─── NULL CHECKS ────────────────────────────────────────────────────────
SELECT * FROM employees WHERE manager_id IS NULL;         -- top-level managers
SELECT * FROM employees WHERE manager_id IS NOT NULL;
-- ⚠️ Never use = NULL — it always returns false! Use IS NULL instead.

-- ─── LIKE / ILIKE (pattern matching) ────────────────────────────────────
SELECT * FROM employees WHERE email LIKE '%@gmail.com';      -- ends with
SELECT * FROM employees WHERE first_name LIKE 'J%';            -- starts with J
SELECT * FROM employees WHERE last_name ILIKE '%smith%';       -- case-insensitive (PostgreSQL)
SELECT * FROM employees WHERE phone LIKE '___-____';           -- _ matches single char

-- ─── IN / NOT IN ────────────────────────────────────────────────────────
SELECT * FROM employees WHERE department_id IN (1, 3, 5);
SELECT * FROM employees WHERE job_level NOT IN ('JUNIOR', 'MID');
-- ⚠️ NOT IN with NULLs: if list has a NULL, entire NOT IN returns UNKNOWN!

-- ─── CASE expression ────────────────────────────────────────────────────
SELECT
    first_name,
    salary,
    CASE
        WHEN salary < 40000  THEN 'Entry'
        WHEN salary < 80000  THEN 'Mid'
        WHEN salary < 120000 THEN 'Senior'
        ELSE                        'Executive'
    END AS salary_band,
    CASE is_active WHEN TRUE THEN 'Active' ELSE 'Inactive' END status
FROM employees;

-- ─── USEFUL STRING FUNCTIONS ────────────────────────────────────────────
SELECT
    UPPER(first_name),                          -- JOHN
    LOWER(email),
    CONCAT(first_name, ' ', last_name) full_name,
    first_name || ' ' || last_name full_name2,   -- PostgreSQL concat shorthand
    TRIM('  hello  '),                          -- 'hello'
    LENGTH(first_name),
    SUBSTRING(email, 1, POSITION('@' IN email) - 1) username,
    REPLACE(phone, '-', '') clean_phone
FROM employees;

-- ─── DATE FUNCTIONS ─────────────────────────────────────────────────────
SELECT
    hire_date,
    EXTRACT(YEAR FROM hire_date)          hire_year,
    DATE_PART('month', hire_date)          hire_month,
    AGE(NOW(), hire_date::timestamp)       tenure,
    DATE_TRUNC('month', hire_date)         first_of_month,
    hire_date + INTERVAL '90 days'        probation_end,
    TO_CHAR(hire_date, 'DD Mon YYYY')      formatted
FROM employees;

JOINs — All Types Explained
INNER · LEFT · RIGHT · FULL · CROSS · SELF — with visual explanations
🔵
INNER JOIN
Matching rows only

Returns rows where condition is TRUE in both tables. Most common join.

🔶
LEFT JOIN
All left + matching right

All rows from left table. Matching rows from right. NULL where no match on right.

🔷
RIGHT JOIN
All right + matching left

Mirror of LEFT JOIN. Rarely used — just swap table order and use LEFT JOIN instead.

🔮
FULL JOIN
All rows both sides

All rows from both tables. NULL where no match on either side. Good for reconciliation.

✖️
CROSS JOIN
Cartesian product

Every row in A × every row in B. 10 rows × 10 rows = 100 rows. Rarely needed.

🔁
SELF JOIN
Table joined to itself

Join a table to itself with aliases. Classic use: employee → manager hierarchy.

📌 All JOIN Types with Real Examples
joins.sqlSQL
-- ─── INNER JOIN — only matched rows ────────────────────────────────────
SELECT
    e.employee_id, e.first_name, e.last_name,
    d.dept_name, d.budget
FROM employees e
INNER JOIN departments d ON e.department_id = d.department_id;
-- Employees WITHOUT a department are EXCLUDED

-- ─── LEFT JOIN — all employees, even without a department ───────────────
SELECT
    e.first_name,
    COALESCE(d.dept_name, 'Unassigned') AS department
FROM employees e
LEFT JOIN departments d ON e.department_id = d.department_id;

-- ─── Find employees WITHOUT a department (anti-join pattern) ───────────
SELECT e.* FROM employees e
LEFT JOIN departments d ON e.department_id = d.department_id
WHERE d.department_id IS NULL;   -- NULL means no match found — anti-join!

-- ─── FULL OUTER JOIN — see all mismatches on BOTH sides ─────────────────
SELECT
    e.first_name       employee,
    d.dept_name        department
FROM employees e
FULL OUTER JOIN departments d ON e.department_id = d.department_id
WHERE e.employee_id IS NULL OR d.department_id IS NULL;  -- orphans on both sides

-- ─── SELF JOIN — employee with their manager ────────────────────────────
SELECT
    e.first_name                           employee,
    COALESCE(m.first_name, 'No Manager') manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.employee_id;  -- same table, different alias

-- ─── MULTIPLE JOINS chained ─────────────────────────────────────────────
SELECT
    o.order_id, o.created_at,
    c.customer_name,
    p.product_name, oi.quantity, oi.unit_price,
    d.dept_name                              -- sales rep's department
FROM orders o
JOIN customers      c  ON o.customer_id  = c.customer_id
JOIN order_items    oi ON o.order_id     = oi.order_id
JOIN products       p  ON oi.product_id  = p.product_id
LEFT JOIN employees rep ON o.rep_id      = rep.employee_id
LEFT JOIN departments d ON rep.department_id = d.department_id
WHERE o.created_at >= '2024-01-01'
ORDER BY o.created_at DESC;
💡
Anti-Join Pattern: LEFT JOIN ... WHERE right_table.id IS NULL is the fastest way to find rows in table A that have NO match in table B — faster than NOT EXISTS or NOT IN on most databases.

Aggregations & GROUP BY
COUNT · SUM · AVG · MIN · MAX · HAVING · ROLLUP · GROUPING SETS
📊 Aggregate Functions — Complete Reference
aggregations.sqlSQL
-- ─── BASIC AGGREGATION ──────────────────────────────────────────────────
SELECT
    COUNT(*)                   total_employees,      -- counts all rows
    COUNT(manager_id)          has_manager,          -- NULLs are EXCLUDED
    COUNT(DISTINCT department_id) unique_depts,      -- distinct count
    SUM(salary)                total_payroll,
    AVG(salary)                avg_salary,
    ROUND(AVG(salary), 2)     avg_salary_rounded,
    MIN(salary)                lowest_salary,
    MAX(salary)                highest_salary,
    MAX(hire_date)             most_recent_hire,
    STRING_AGG(first_name, ', ' ORDER BY first_name) all_names  -- PostgreSQL
FROM employees
WHERE is_active = TRUE;

-- ─── GROUP BY — aggregate per group ─────────────────────────────────────
SELECT
    d.dept_name,
    COUNT(e.employee_id)       headcount,
    ROUND(AVG(e.salary), 0)   avg_salary,
    SUM(e.salary)              total_salary,
    MIN(e.hire_date)           oldest_hire
FROM employees e
JOIN departments d ON e.department_id = d.department_id
WHERE e.is_active = TRUE
GROUP BY d.dept_name          -- group by non-aggregated SELECT column
HAVING COUNT(e.employee_id) > 5   -- filter GROUPS (not rows) — use HAVING not WHERE
ORDER BY total_salary DESC;

-- ─── ROLLUP — subtotals + grand total ───────────────────────────────────
SELECT
    COALESCE(dept_name, 'TOTAL') dept,
    COALESCE(job_level, 'ALL')   level,
    COUNT(*) headcount,
    SUM(salary) payroll
FROM employees e
JOIN departments d ON e.department_id = d.department_id
GROUP BY ROLLUP(dept_name, job_level);  -- adds subtotal rows automatically

-- ─── FILTER inside aggregate (PostgreSQL) ───────────────────────────────
SELECT
    department_id,
    COUNT(*) total,
    COUNT(*) FILTER (WHERE job_level = 'SENIOR')  senior_count,
    COUNT(*) FILTER (WHERE salary > 90000)        high_earners,
    AVG(salary) FILTER (WHERE is_active = TRUE)   active_avg_salary
FROM employees
GROUP BY department_id;
⚠️
WHERE vs HAVING: WHERE filters individual rows before grouping. HAVING filters groups after aggregation. You cannot use aggregate functions in WHERE — use HAVING for that.

Subqueries & CTEs
Inline · Correlated · EXISTS · WITH (CTE) · Recursive CTE
❌ Deep Nested Subquery — Hard to Read
SELECT * FROM (
    SELECT dept_id, AVG(sal) avg_sal
    FROM (
        SELECT department_id dept_id, salary sal
        FROM employees
        WHERE is_active = TRUE
    ) active_emps
    GROUP BY dept_id
) dept_avgs
WHERE avg_sal > 70000;
✅ CTE — Readable, Reusable
WITH active_emps AS (
    SELECT department_id, salary
    FROM employees
    WHERE is_active = TRUE
),
dept_avgs AS (
    SELECT department_id, AVG(salary) avg_sal
    FROM active_emps
    GROUP BY department_id
)
SELECT * FROM dept_avgs
WHERE avg_sal > 70000;
🔄 Subquery Types & EXISTS
subqueries.sqlSQL
-- ─── SCALAR SUBQUERY — returns single value ─────────────────────────────
SELECT
    first_name, salary,
    salary - (SELECT AVG(salary) FROM employees) diff_from_avg
FROM employees;

-- ─── IN / NOT IN with subquery ───────────────────────────────────────────
SELECT * FROM employees
WHERE department_id IN (
    SELECT department_id FROM departments
    WHERE budget > 1000000
);

-- ─── EXISTS — faster than IN for large datasets ──────────────────────────
-- Find managers (employees who manage at least 1 person)
SELECT e.first_name, e.last_name
FROM employees e
WHERE EXISTS (
    SELECT 1 FROM employees sub
    WHERE sub.manager_id = e.employee_id   -- correlated: references outer query
);

-- ─── CORRELATED SUBQUERY — runs once per outer row ───────────────────────
-- Employees earning more than their department average
SELECT e.first_name, e.salary, e.department_id
FROM employees e
WHERE e.salary > (
    SELECT AVG(e2.salary)
    FROM employees e2
    WHERE e2.department_id = e.department_id  -- correlated to outer e
);

-- ─── RECURSIVE CTE — org hierarchy traversal ────────────────────────────
WITH RECURSIVE org_tree AS (
    -- Anchor: start from CEO (no manager)
    SELECT employee_id, first_name, manager_id, 0 AS level
    FROM employees
    WHERE manager_id IS NULL

    UNION ALL

    -- Recursive: find direct reports of current level
    SELECT e.employee_id, e.first_name, e.manager_id, ot.level + 1
    FROM employees e
    JOIN org_tree ot ON e.manager_id = ot.employee_id
)
SELECT
    REPEAT('  ', level) || first_name AS org_chart,
    level
FROM org_tree
ORDER BY level, first_name;

Window Functions
ROW_NUMBER · RANK · DENSE_RANK · LAG · LEAD · NTILE · Running Totals
💡 Window Functions — The Big Difference

Unlike GROUP BY, window functions do not collapse rows. Each row keeps its identity while getting access to aggregate information from a defined "window" of rows.

GROUP BY collapses rows
-- 100 employees → 5 department rows
SELECT department_id, AVG(salary)
FROM employees
GROUP BY department_id;
-- Result: 5 rows
OVER() keeps all rows
-- 100 employees → still 100 rows!
SELECT first_name, salary,
    AVG(salary) OVER(
        PARTITION BY department_id
    ) dept_avg
FROM employees;
-- Result: 100 rows, each has dept_avg
🏆 All Window Functions with Examples
window_functions.sqlSQL
SELECT
    first_name,
    department_id,
    salary,
    hire_date,

    -- ─── RANKING ────────────────────────────────────────────────────────
    ROW_NUMBER() OVER(PARTITION BY department_id ORDER BY salary DESC)
        AS row_num,       -- unique: 1,2,3,4 — no ties

    RANK() OVER(PARTITION BY department_id ORDER BY salary DESC)
        AS rank_num,      -- ties get same rank, skips next: 1,1,3,4

    DENSE_RANK() OVER(PARTITION BY department_id ORDER BY salary DESC)
        AS dense_r,       -- ties same rank, NO gap: 1,1,2,3

    NTILE(4) OVER(ORDER BY salary DESC)
        AS quartile,      -- divide into 4 buckets (Q1=top 25%)

    PERCENT_RANK() OVER(ORDER BY salary)
        AS pct_rank,      -- 0.0 to 1.0 — relative position

    -- ─── OFFSET (LAG/LEAD) ─────────────────────────────────────────────
    LAG(salary, 1) OVER(PARTITION BY department_id ORDER BY hire_date)
        AS prev_salary,   -- previous row's salary (NULL for first row)

    LEAD(salary, 1) OVER(PARTITION BY department_id ORDER BY hire_date)
        AS next_salary,   -- next row's salary (NULL for last row)

    salary - LAG(salary) OVER(ORDER BY hire_date)
        AS salary_change, -- difference from previous hire's salary

    -- ─── RUNNING AGGREGATES ────────────────────────────────────────────
    SUM(salary) OVER(PARTITION BY department_id ORDER BY hire_date
                     ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)
        AS running_payroll,   -- cumulative sum within dept

    AVG(salary) OVER(ORDER BY hire_date
                     ROWS BETWEEN 2 PRECEDING AND CURRENT ROW)
        AS moving_avg_3,      -- 3-row moving average

    -- ─── FIRST / LAST VALUE ────────────────────────────────────────────
    FIRST_VALUE(salary) OVER(PARTITION BY department_id ORDER BY salary DESC)
        AS dept_top_salary    -- highest salary in dept, on every row

FROM employees
ORDER BY department_id, salary DESC;
🎯 Classic Window Function Use Case — Top N Per Group
top_n_per_group.sqlSQL
-- Top 3 highest paid employees per department
WITH ranked AS (
    SELECT
        first_name, department_id, salary,
        DENSE_RANK() OVER(PARTITION BY department_id ORDER BY salary DESC) rnk
    FROM employees
    WHERE is_active = TRUE
)
SELECT * FROM ranked WHERE rnk <= 3;
-- Cannot filter on window functions in WHERE — must use CTE or subquery!

Indexes — The Backbone of Performance
B-Tree · Hash · Composite · Partial · Covering · When NOT to index
🌳 How B-Tree Index Works

Without an index, a query does a Full Table Scan — reads every single row. With a B-Tree index, it does a Binary Tree Search — O(log n) lookups. A table with 1M rows: full scan = 1,000,000 reads. B-Tree = ~20 reads.

ℹ️
The DB optimizer decides whether to use an index. If it estimates that >5–10% of rows will match, it may prefer a full scan over the index. This is why low-cardinality columns (e.g. boolean, gender) make poor index candidates.
📋 Index Types — Complete Reference
indexes.sqlSQL
-- ─── BASIC B-TREE (default) ─────────────────────────────────────────────
CREATE INDEX idx_employees_email    ON employees(email);
CREATE UNIQUE INDEX idx_emp_email_u ON employees(email);  -- enforces uniqueness too

-- ─── COMPOSITE INDEX — column ORDER matters! ────────────────────────────
CREATE INDEX idx_emp_dept_salary ON employees(department_id, salary);
-- Supports: WHERE dept=? AND salary=?   ✅
-- Supports: WHERE dept=?               ✅ (leftmost prefix rule)
-- Does NOT: WHERE salary=?             ❌ (skips first column)

-- ─── PARTIAL INDEX — only index rows matching a condition ───────────────
CREATE INDEX idx_active_employees ON employees(last_name)
    WHERE is_active = TRUE;
-- Only active employees in the index — smaller, faster for active-only queries!

-- ─── COVERING INDEX — include extra columns (no heap fetch needed) ───────
CREATE INDEX idx_emp_dept_covering
    ON employees(department_id)
    INCLUDE (first_name, last_name, salary);  -- PostgreSQL 11+ INCLUDE
-- Query only touches the index — no table row lookup needed!
-- SELECT first_name, salary WHERE department_id = ? → index-only scan

-- ─── EXPRESSION / FUNCTIONAL INDEX ──────────────────────────────────────
CREATE INDEX idx_email_lower ON employees(LOWER(email));
-- Used by: WHERE LOWER(email) = 'john@example.com'
-- Plain email index would NOT be used for LOWER() queries!

-- ─── HASH INDEX (PostgreSQL) — equality only ─────────────────────────────
CREATE INDEX idx_emp_hash_id ON employees USING HASH (employee_id);
-- Faster than B-Tree for = lookups, but useless for BETWEEN/ORDER BY/LIKE

-- ─── GIN INDEX — for JSONB, arrays, full-text ───────────────────────────
CREATE INDEX idx_metadata_gin ON employees USING GIN (metadata);
-- Enables: WHERE metadata @> '{"role": "admin"}'

-- ─── CHECK index usage ────────────────────────────────────────────────
EXPLAIN ANALYZE
SELECT * FROM employees WHERE email = 'john@example.com';
-- Look for "Index Scan" vs "Seq Scan" in EXPLAIN output

Primary key / Foreign key columnsYES — alwaysPKs auto-indexed. FK columns need manual index for JOIN performance.

Columns used in ORDER BY / GROUP BYYESAvoids sort step if index already sorted.

Transactions & Locking
ACID · Isolation Levels · Optimistic vs Pessimistic · Deadlocks
⚛️
A — Atomicity
All or Nothing

Every statement in a transaction succeeds, or ALL are rolled back. No partial updates.

💸 Transfer $100 — debit AND credit, or neither
🔒
C — Consistency
Valid State Always

Transaction brings DB from one valid state to another. All constraints hold before and after.

⚖️ Account balance can never go negative
🚧
I — Isolation
Concurrent Safety

Concurrent transactions behave as if sequential. Controlled by isolation levels.

👤 Each user sees a consistent snapshot
💾
D — Durability
Survives Crashes

Committed data is permanently written to disk. Survives power loss, crashes.

✅ Once confirmed, your booking is permanent
🔐 Transactions & Locking — Patterns
transactions.sqlSQL
-- ─── BASIC TRANSACTION ───────────────────────────────────────────────────
BEGIN;                              -- or START TRANSACTION

    UPDATE accounts SET balance = balance - 500 WHERE id = 1;
    UPDATE accounts SET balance = balance + 500 WHERE id = 2;
    INSERT INTO audit_log(action, amount) VALUES('TRANSFER', 500);

COMMIT;                             -- persist all 3 changes atomically
-- or ROLLBACK; to undo everything

-- ─── SAVEPOINTS — partial rollback ──────────────────────────────────────
BEGIN;
    INSERT INTO orders(customer_id, total) VALUES(42, 250.00);
    SAVEPOINT order_created;

    INSERT INTO payments(order_id, amount) VALUES(LASTVAL(), 250.00);
    -- if payment fails:
    ROLLBACK TO order_created;       -- undo only the payment, keep the order
    -- then handle payment retry...
COMMIT;

-- ─── PESSIMISTIC LOCKING — lock row immediately ──────────────────────────
BEGIN;
SELECT * FROM inventory
WHERE product_id = 101
FOR UPDATE;                         -- exclusive lock, blocks other writers
-- Other transactions trying FOR UPDATE on same row will WAIT here
UPDATE inventory SET stock = stock - 1 WHERE product_id = 101;
COMMIT;

-- FOR SHARE — allow other reads but block writes
SELECT * FROM inventory WHERE product_id = 101 FOR SHARE;

-- SKIP LOCKED — non-blocking, skip locked rows (job queues!)
SELECT * FROM job_queue
WHERE status = 'PENDING'
ORDER BY created_at
LIMIT 1
FOR UPDATE SKIP LOCKED;            -- multiple workers safely dequeue jobs!
Isolation LevelDirty ReadNon-Repeatable ReadPhantom ReadUse When
READ UNCOMMITTED✅ Possible✅ Possible✅ PossibleAnalytics only — never for financial data
READ COMMITTED (default)❌ Prevented✅ Possible✅ PossibleMost OLTP apps — good default
REPEATABLE READ❌ Prevented❌ Prevented✅ PossibleReport generation, consistent reads
SERIALIZABLE❌ Prevented❌ Prevented❌ PreventedFinancial transactions, maximum safety
🚨
Deadlock Prevention: Always acquire locks in the same order across transactions. If Transaction A locks row 1 then row 2, and Transaction B locks row 2 then row 1 — deadlock! Solution: standardize lock order or use SELECT ... FOR UPDATE NOWAIT to fail fast instead of waiting.

Schema Design & Normalization
1NF · 2NF · 3NF · Denormalization · Naming Conventions
📐 Normal Forms — When to Normalize
FormRuleProblem it Solves
1NFNo repeating groups. Each cell has one atomic value. Each row is unique.Eliminates: "tags = 'sql,java,python'" in one column
2NF1NF + every non-key column depends on the WHOLE primary key (no partial dependency)Eliminates duplicate data in composite-key tables
3NF2NF + no transitive dependency (non-key column depends on another non-key column)Eliminates: storing dept_name alongside dept_id in employees
BCNFStricter 3NF — every determinant is a candidate keyEdge cases in multi-valued dependencies
💡
Denormalization is sometimes correct. 3NF is ideal for OLTP (writes). For OLAP/reporting, intentional denormalization (star schema, materialized views) trades storage for query speed. Know when to break the rules.
🏗️ Schema Best Practices
schema_best_practices.sqlSQL
-- ✅ NAMING: snake_case, plural tables, singular columns, _id suffix for FKs
CREATE TABLE order_items (            -- plural table name

CREATE OR REPLACE FUNCTION set_updated_at()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = NOW();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_order_items_updated

⚡ Common Slow Query Patterns & Fixes
❌ Function on indexed column (kills index)
SELECT * FROM employees
WHERE YEAR(hire_date) = 2023;
-- Wrapping in YEAR() prevents index use!
-- Forces full table scan every time.
✅ Range condition — index works
SELECT * FROM employees
WHERE hire_date BETWEEN
    '2023-01-01' AND '2023-12-31';
-- Direct comparison uses index scan ✅
❌ SELECT * — fetches unused data
SELECT *
FROM orders o
JOIN customers c ON o.customer_id = c.id
WHERE o.status = 'PENDING';
-- Transfers all columns over network
-- Prevents covering index scans
✅ Explicit columns + covering index
SELECT o.order_id, o.total, c.name
FROM orders o
JOIN customers c ON o.customer_id = c.id
WHERE o.status = 'PENDING';
-- Only fetches what you need
-- Enables index-only scans ✅
❌ OFFSET pagination — slow on large data
SELECT * FROM orders
ORDER BY created_at DESC
LIMIT 20 OFFSET 100000;
-- DB scans 100,020 rows to skip 100,000
-- Gets slower the deeper you paginate!
✅ Keyset / Cursor pagination — always O(1)
SELECT * FROM orders
WHERE created_at < '2024-05-01 12:00:00'   -- last seen cursor
ORDER BY created_at DESC
LIMIT 20;
-- Uses index, consistent speed at any depth ✅

CREATE VIEW v_employee_summary AS

    e.employee_id, e.first_name, e.last_name,

-- ─── STORED PROCEDURE (PostgreSQL) ──────────────────────────────────────
CREATE OR REPLACE PROCEDURE transfer_funds(
    p_from_id    BIGINT,
    p_to_id      BIGINT,
    p_amount     NUMERIC
)
LANGUAGE plpgsql
AS $$
DECLARE
    v_balance NUMERIC;
BEGIN
    SELECT balance INTO v_balance FROM accounts WHERE id = p_from_id FOR UPDATE;

    IF v_balance < p_amount THEN
        RAISE EXCEPTION 'Insufficient funds: balance=%, required=%', v_balance, p_amount;
    END IF;

    UPDATE accounts SET balance = balance - p_amount WHERE id = p_from_id;
    UPDATE accounts SET balance = balance + p_amount WHERE id = p_to_id;
    INSERT INTO transfers(from_id, to_id, amount) VALUES(p_from_id, p_to_id, p_amount);

    COMMIT;
END;
$$;

-- Call it
CALL transfer_funds(1, 2, 500.00);

-- Month-over-month revenue with % change
WITH monthly_revenue AS (
    SELECT
        DATE_TRUNC('month', created_at)   month,
        SUM(total_amount)               revenue,
        COUNT(DISTINCT customer_id)     unique_buyers,
        COUNT(*)                        order_count
    FROM orders
    WHERE status = 'COMPLETED'
    GROUP BY 1
)
SELECT
    month,
    revenue,
    unique_buyers,
    LAG(revenue) OVER(ORDER BY month)   prev_month_revenue,
    ROUND(
        (revenue - LAG(revenue) OVER(ORDER BY month))
        / NULLIF(LAG(revenue) OVER(ORDER BY month), 0) * 100, 1
    ) pct_change                          -- NULLIF prevents divide by zero
FROM monthly_revenue
ORDER BY month;

-- Top 10 products by revenue with category context
SELECT
    p.product_name,
    c.category_name,
    SUM(oi.quantity)                    units_sold,
    SUM(oi.quantity * oi.unit_price)    gross_revenue,

📊 Scenario B: SaaS — User Cohort Retention
Classic cohort analysis — what % of users from each signup month are still active N months later
cohort_retention.sqlSQL
WITH user_cohorts AS (
    -- Assign each user to their signup month cohort
    SELECT
        user_id,
        DATE_TRUNC('month', created_at) cohort_month
    FROM users
),
user_activity AS (
    -- Get each user's active months from events table
    SELECT DISTINCT
        user_id,
        DATE_TRUNC('month', event_at) activity_month
    FROM user_events
),
cohort_size AS (
    SELECT cohort_month, COUNT(user_id) cohort_users
    FROM user_cohorts GROUP BY 1
)
SELECT
    uc.cohort_month,
    cs.cohort_users                        cohort_size,
    EXTRACT(MONTH FROM
        AGE(ua.activity_month, uc.cohort_month)) months_since_signup,
    COUNT(DISTINCT ua.user_id)             retained_users,
    ROUND(COUNT(DISTINCT ua.user_id) * 100.0
          / cs.cohort_users, 1)            retention_pct
FROM user_cohorts uc
JOIN user_activity ua  ON uc.user_id = ua.user_id
JOIN cohort_size  cs  ON uc.cohort_month = cs.cohort_month
WHERE ua.activity_month >= uc.cohort_month
GROUP BY uc.cohort_month, cs.cohort_users, months_since_signup
ORDER BY uc.cohort_month, months_since_signup;
⚙️ Scenario C: Transactional Job Queue with FOR UPDATE SKIP LOCKED
Reliable multi-worker queue — no duplicate processing, no polling overhead
job_queue.sqlSQL
CREATE TABLE job_queue (
    job_id       BIGSERIAL PRIMARY KEY,
    job_type     VARCHAR(50)  NOT NULL,
    payload      JSONB        NOT NULL,
    status       VARCHAR(20)  NOT NULL DEFAULT 'PENDING',
    attempts     INT          DEFAULT 0,
    max_attempts INT          DEFAULT 3,
    run_at       TIMESTAMPTZ  DEFAULT NOW(),
    created_at   TIMESTAMPTZ  DEFAULT NOW()
);
CREATE INDEX idx_job_queue_pending ON job_queue(status, run_at)
    WHERE status = 'PENDING';   -- partial index — only pending jobs!

-- Worker: atomically claim one job (safe for N concurrent workers)
WITH claimed AS (
    SELECT job_id FROM job_queue
    WHERE  status = 'PENDING'
      AND  run_at <= NOW()
      AND  attempts < max_attempts
    ORDER BY run_at
    LIMIT 1
    FOR UPDATE SKIP LOCKED      -- skip rows claimed by other workers!
)
UPDATE job_queue SET
    status   = 'PROCESSING',
    attempts = attempts + 1
WHERE job_id = (SELECT job_id FROM claimed)
RETURNING *;   -- returns claimed job to the application

⚡ Key SQL Commands

SELECT ... FROM ... WHERE
Core read query. Add JOIN, GROUP BY, ORDER BY, LIMIT as needed.
INSERT INTO ... VALUES
Add rows. Use RETURNING * to get inserted row back (PostgreSQL).
UPDATE ... SET ... WHERE
⚠️ Always include WHERE. Missing WHERE = update ALL rows!
DELETE FROM ... WHERE
⚠️ Always include WHERE. TRUNCATE for deleting all rows — much faster.
GROUP BY + HAVING
GROUP BY groups rows. HAVING filters groups. WHERE filters rows BEFORE grouping.
JOIN ON vs USING
JOIN t ON a.id = b.a_id or JOIN t USING(id) when column names match.
COALESCE(val, fallback)
Returns first non-NULL value. Use for default values: COALESCE(name, 'Unknown').
NULLIF(a, b)
Returns NULL if a = b, else a. Classic use: NULLIF(denom, 0) to prevent division by zero.
CASE WHEN ... THEN ... END
SQL's if-else. Works in SELECT, WHERE, ORDER BY, even GROUP BY.
WITH cte AS (...)
Named subquery. Improves readability. Can be referenced multiple times in same query.
WINDOW fn() OVER (PARTITION BY ... ORDER BY ...)
Aggregate per window without collapsing rows. Cannot filter on window result in WHERE.
EXPLAIN ANALYZE
Your best friend for slow queries. Look for Seq Scan on large tables — add an index!

❓ Interview Q&A

WHERE vs HAVING — what's the difference?
WHERE filters individual rows before grouping happens. HAVING filters groups after aggregation. You cannot use aggregate functions like COUNT() in WHERE — that's what HAVING is for.
What is the N+1 query problem in SQL context?
Running 1 query to get N rows, then N additional queries for each row's related data. Fix with a single JOIN or a WHERE IN subquery. Symptoms: many near-identical queries in slow logs, response time proportional to result size.
INNER JOIN vs LEFT JOIN — when to use which?
Use INNER JOIN when you only want rows that have a match in both tables. Use LEFT JOIN when you need all rows from the left table regardless of whether there's a match — use COALESCE to handle NULLs on the right side.
ROW_NUMBER vs RANK vs DENSE_RANK?
ROW_NUMBER: always unique, no ties (1,2,3,4). RANK: same number for ties, skips next (1,1,3). DENSE_RANK: same for ties, no gap (1,1,2). Use DENSE_RANK for Top-N filtering — it's the most predictable for WHERE rnk <= 3.
What is a covering index?
An index that contains all columns needed to satisfy a query — the DB never needs to visit the actual table rows (index-only scan). Create with INCLUDE clause: CREATE INDEX ON t(lookup_col) INCLUDE (col1, col2). Dramatic performance boost for read-heavy queries.
When should you NOT add an index?
Small tables (<10K rows), low-cardinality columns (boolean, status with 2-3 values), and columns that are updated very frequently. The optimizer ignores indexes when it expects to scan >5-10% of rows — and every write pays the index update cost regardless.
What is a window function and how is it different from GROUP BY?
GROUP BY collapses many rows into one per group. Window functions compute aggregate values over a defined "window" but keep every row. Use OVER (PARTITION BY dept ORDER BY salary) to define the window. You cannot filter on a window function in WHERE — wrap in a CTE first.
What is ACID and why does it matter?
Atomicity: all-or-nothing. Consistency: constraints always hold. Isolation: concurrent transactions don't interfere. Durability: committed data survives crashes. Without ACID, concurrent transactions can corrupt data — e.g. two users both reading stock=1, both buying, leaving stock=-1.
What is the difference between DELETE, TRUNCATE, and DROP?
DELETE FROM t WHERE ...: removes rows, can be rolled back, fires triggers, slow on large tables. TRUNCATE TABLE t: removes ALL rows instantly (DDL), cannot be rolled back in most DBs, resets sequences. DROP TABLE t: destroys the entire table structure + data permanently.
