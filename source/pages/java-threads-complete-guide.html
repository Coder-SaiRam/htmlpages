Process vs Thread vs Virtual Thread
the foundation
🏢
Process
isolated program instance
An independent program in execution. Has its own memory space (heap, stack, code, data). Isolated — one process cannot access another's memory. Heavyweight: creating costs ~1ms, ~1MB+ RAM minimum. IPC needed for communication.
✅ Strong isolation
✅ Crash in one doesn't affect others
❌ Expensive to create and switch
❌ IPC (pipes, sockets) for communication
🏢 Separate office buildings — employees can't walk between them; need phone calls
🧵
OS Thread (Platform Thread)
lightweight execution unit
Exists within a process. Shares process memory (heap). Has its own stack (~1MB default), registers, and program counter. OS scheduler decides which thread runs on which CPU core. Java thread = 1:1 OS thread mapping.
✅ Shared memory = easy communication
✅ Much faster than processes
❌ ~1MB stack per thread
❌ OS context switch overhead (~1-10μs)
🧵 Employees in same building — share resources, communicate easily
🪶
Virtual Thread (Project Loom)
Java 21+ lightweight threads
JVM-managed threads. Millions can exist simultaneously. Tiny heap stack (few KB, grows dynamically). When blocked on I/O, JVM automatically parks virtual thread and frees the OS thread for other work. No OS involvement.
✅ Millions per JVM
✅ Near-zero blocking cost
✅ Same Thread API — easy migration
❌ Not for CPU-bound tasks
🪶 Extremely efficient task scheduler — idle tasks sleep on a cot, active tasks get a real desk

→run() ends

🔬
State Transitions in Code
ThreadLifecycleDemo.java
public class ThreadLifecycleDemo {

    public static void main(String[] args) throws InterruptedException {
        Object lock = new Object();

        Thread t = new Thread(() -> {
            synchronized (lock) {
                try {
                    lock.wait(5000);   // TIMED_WAITING state
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                }
            }
        });

        System.out.println(t.getState()); // NEW

        t.start();
        Thread.sleep(100);
        System.out.println(t.getState()); // TIMED_WAITING

        synchronized (lock) {
            lock.notifyAll();              // wake it up
        }
        Thread.sleep(100);
        System.out.println(t.getState()); // TERMINATED

        t.join();  // wait for thread to finish
    }
}

// Check state programmatically
Thread.State state = thread.getState();
switch (state) {
    case NEW          -> System.out.println("Not started yet");
    case RUNNABLE     -> System.out.println("Running or ready");
    case BLOCKED      -> System.out.println("Waiting for monitor lock");
    case WAITING      -> System.out.println("Waiting indefinitely");
    case TIMED_WAITING -> System.out.println("Waiting with timeout");
    case TERMINATED   -> System.out.println("Finished");
}

⚠️
RUNNABLE ≠ actually running! A RUNNABLE thread might be waiting in the OS run queue for a CPU time slice. The OS scheduler decides when it actually runs. Java has no "RUNNING" state separate from RUNNABLE.

    scope.fork(() -> fetchFromSecondary());
    scope.join();

Java Memory Model (JMM) & volatile
visibility · atomicity · happens-before
🧠
The Visibility Problem
why threads see stale values
Each CPU core has its own L1/L2 cache. When Thread-A writes a value, it may stay in CPU-1's cache. Thread-B on CPU-2 reads from its own cache — sees stale value! The JMM defines when writes by one thread are guaranteed to be visible to others.
Visibility problem — infinite loop!
public class VisibilityProblem {
    // ❌ BAD: Thread-B may NEVER see the update!
    static boolean running = true;

    public static void main(String[] args) throws InterruptedException {
        Thread t = new Thread(() -> {
            int i = 0;
            while (running) {  // may loop forever — cached CPU register!
                i++;
            }
            System.out.println("Stopped after " + i + " iterations");
        });
        t.start();
        Thread.sleep(1000);
        running = false;  // main thread writes — t may not see it!
        System.out.println("Set running=false");
    }
}

// ✅ FIX 1: volatile keyword
static volatile boolean running = true;
// Reads/writes always go to main memory — no caching

// ✅ FIX 2: synchronized
synchronized void setRunning(boolean val) { running = val; }
// Synchronized exit: flushes CPU cache to main memory

// ✅ FIX 3: AtomicBoolean
static AtomicBoolean running = new AtomicBoolean(true);
running.set(false);
if (running.get()) { ... }

volatile — Deep Dive
visibility YES · atomicity NO
volatile rules
public class VolatileDemo {
    private volatile int counter = 0;   // ✅ visible
    private volatile boolean flag = false;

    // ✅ volatile GUARANTEES:
    // 1. Every read sees the LATEST write (no CPU cache)
    // 2. Writes flushed to main memory immediately
    // 3. Happens-before: write X happens-before all reads seeing X

    // ❌ volatile DOES NOT guarantee:
    // Compound operations are NOT atomic!
    void increment() {
        counter++;  // ❌ NOT SAFE even with volatile!
        // counter++ is: READ counter → ADD 1 → WRITE counter
        // Two threads can interleave between READ and WRITE
    }

    // ✅ Use Case 1: simple flag checked by multiple threads
    volatile boolean shutdownRequested = false;
    void service() {
        while (!shutdownRequested) { doWork(); }
    }
    void shutdown() { shutdownRequested = true; }

    // ✅ Use Case 2: singleton double-checked locking (correct!)
    private static volatile Singleton instance;
    public static Singleton getInstance() {
        if (instance == null) {                   // 1st check (no lock)
            synchronized (Singleton.class) {
                if (instance == null) {           // 2nd check (with lock)
                    instance = new Singleton();   // volatile: no partial init visible
                }
            }
        }
        return instance;
    }
}

// volatile vs synchronized vs atomic:
// volatile     → visibility only (no mutual exclusion)
// synchronized → visibility + mutual exclusion (heavier)
// AtomicInteger → visibility + atomic CAS (lightweight for counters)
📻 volatile = live TV broadcast — everyone always sees the latest frame. But if two people try to change the channel at once, they still conflict!
📐
Happens-Before Relationship
JMM guarantee: if A happens-before B, B sees all of A's writes
RuleWhat guarantees happens-before
Program OrderEach action in thread happens-before later action in same thread
Monitor Locksynchronized exit happens-before the next synchronized entry on same monitor
volatile writeWrite to volatile field happens-before every read of that field
Thread startThread.start() call happens-before any action in the started thread
Thread joinAll actions in Thread T happen-before join(T) returns
TransitivityIf A happens-before B and B happens-before C, then A happens-before C
💡
happens-before doesn't mean "executes before in time" — it means the memory writes are guaranteed to be visible. It's about visibility, not ordering.
Happens-before example
int x = 0;
volatile boolean ready = false;

// Thread A
x = 42;          // write x
ready = true;    // volatile write — happens-before all reads of ready

// Thread B
if (ready) {     // volatile read
    print(x);    // guaranteed to see x = 42
    // Because: write(x) → write(ready) → read(ready) → read(x)
    //          (program order + volatile happens-before)
}

synchronized — Intrinsic Locks (Monitors)
mutual exclusion built into every Java object
🔐
synchronized — 4 Forms
All synchronized forms
public class Counter {
    private int count = 0;
    private final Object lockA = new Object();
    private final Object lockB = new Object();

    // ── Form 1: synchronized INSTANCE method ──────────────
    // Lock: this (the current object instance)
    public synchronized void increment() {
        count++;  // only one thread at a time
    }

    // ── Form 2: synchronized STATIC method ───────────────
    // Lock: Counter.class (class-level lock — ALL instances blocked)
    public static synchronized void staticMethod() {
        // ...
    }

    // ── Form 3: synchronized BLOCK — best practice ───────
    // Lock: explicit object — more control, smaller critical section
    public void incrementBlock() {
        // do non-critical work here (no lock held)
        System.out.println("Pre-lock work");

        synchronized (this) {       // acquire lock on "this"
            count++;                // only critical section locked
        }                           // lock released here

        // do non-critical work here too
    }

    // ── Form 4: separate locks for different resources ────
    private int resourceA = 0;
    private int resourceB = 0;

    public void updateA() {
        synchronized (lockA) { resourceA++; }  // doesn't block updateB!
    }
    public void updateB() {
        synchronized (lockB) { resourceB++; }  // different lock
    }
    // Threads updating A and B can run concurrently

    // ── wait / notify — must be inside synchronized ──────
    private final List<String> queue = new ArrayList<>();

    public synchronized void produce(String item) {
        queue.add(item);
        notifyAll();  // wake up waiting consumers
    }

    public synchronized String consume() throws InterruptedException {
        while (queue.isEmpty()) {  // ALWAYS use while, not if!
            wait();                // releases lock while waiting
        }
        return queue.remove(0);
    }
}
⚙️
How synchronized Works Internally
biased → lightweight → heavyweight lock
Every Java object has a monitor (mutex). JVM optimizes lock acquisition based on contention level.
1
Biased Lock (no contention)
First thread acquires lock — JVM biases object toward that thread. Next acquisition by same thread = nearly zero cost (just a field check). No CAS operation needed.
2
Lightweight Lock (low contention)
Second thread appears. Bias revoked. Both threads use CAS (Compare-And-Swap) to acquire lock via object's mark word. Lock spinning — tries CAS in a loop before parking. Avoids OS call.
3
Heavyweight Lock (high contention)
After spinning fails, thread calls OS. Blocked on mutex — full OS context switch. Most expensive. Thread enters BLOCKED state. Released thread may not immediately run.
⚠️
synchronized limitations: Cannot try-lock (what if lock held?), no timeout, no interruption, no fairness, cannot have multiple read threads. Use ReentrantLock for advanced needs.

Real Scenario: Thread-Safe Bank Account
BankAccount.java — synchronized correctly
public class BankAccount {
    private final String id;
    private volatile BigDecimal balance;  // visible immediately

    // ✅ synchronized ensures atomicity of read-check-update
    public synchronized boolean withdraw(BigDecimal amount) {
        if (balance.compareTo(amount) < 0) {
            throw new InsufficientFundsException("Balance: " + balance);
        }
        balance = balance.subtract(amount);  // atomic: no race condition
        return true;
    }

    public synchronized void deposit(BigDecimal amount) {
        balance = balance.add(amount);
    }

    // ✅ Transfer between accounts — careful with lock ordering!
    // ALWAYS lock in consistent order to prevent deadlock
    public static void transfer(BankAccount from, BankAccount to, BigDecimal amount) {
        // Order by ID ensures consistent lock acquisition order
        BankAccount first  = from.getId().compareTo(to.getId()) < 0 ? from : to;
        BankAccount second = first == from ? to : from;

        synchronized (first) {
            synchronized (second) {
                from.withdraw(amount);
                to.deposit(amount);
            }
        }
    }
}

ReentrantLock — synchronized on steroids
ReentrantLock complete example
public class AdvancedCounter {
    private int count = 0;
    // fair=true: threads get lock in FIFO order (prevents starvation)
    // fair=false: default, better throughput but possible starvation
    private final ReentrantLock lock = new ReentrantLock(true);
    private final Condition notEmpty = lock.newCondition();

    // ── Basic lock/unlock ──────────────────────────────────
    public void increment() {
        lock.lock();
        try {
            count++;                  // critical section
        } finally {
            lock.unlock();            // ALWAYS in finally — never forget!
        }
    }

    // ── tryLock: don't wait, return immediately ────────────
    public boolean tryIncrement() {
        if (lock.tryLock()) {         // returns false if lock held
            try {
                count++;
                return true;
            } finally {
                lock.unlock();
            }
        }
        return false;  // couldn't get lock, do something else
    }

    // ── tryLock with timeout ───────────────────────────────
    public boolean tryIncrementWithTimeout() throws InterruptedException {
        if (lock.tryLock(2, TimeUnit.SECONDS)) {  // wait max 2 seconds
            try {
                count++;
                return true;
            } finally {
                lock.unlock();
            }
        }
        return false;  // lock not acquired in 2 seconds
    }

    // ── lockInterruptibly — can be interrupted while waiting ─
    public void interruptibleIncrement() throws InterruptedException {
        lock.lockInterruptibly();  // throws if thread.interrupt() called
        try {
            count++;
        } finally {
            lock.unlock();
        }
    }

    // ── Condition variables (like wait/notify but per-condition) ─
    private final Queue<String> buffer = new LinkedList<>();
    private final Condition notFull  = lock.newCondition();
    private final Condition notEmptyC = lock.newCondition();
    private final int MAX = 10;

    public void put(String item) throws InterruptedException {
        lock.lock();
        try {
            while (buffer.size() == MAX) notFull.await();  // wait: buffer full
            buffer.add(item);
            notEmptyC.signal();                            // wake one consumer
        } finally { lock.unlock(); }
    }

    public String take() throws InterruptedException {
        lock.lock();
        try {
            while (buffer.isEmpty()) notEmptyC.await();   // wait: buffer empty
            String item = buffer.poll();
            notFull.signal();                             // wake one producer
            return item;
        } finally { lock.unlock(); }
    }

    // ── Reentrancy: same thread can lock multiple times ───
    public void outer() {
        lock.lock();
        try {
            inner();  // same thread locks again — no deadlock!
        } finally { lock.unlock(); }
    }
    public void inner() {
        lock.lock();  // hold count = 2, same thread
        try {
            count++;
        } finally { lock.unlock(); }  // hold count = 1
    }
    // lock.getHoldCount() tells current thread's hold count
}

ReadWriteLock — Multiple Readers, One Writer
Read lock: multiple threads can hold simultaneously (reads don't conflict). Write lock: exclusive — no other reads or writes. Perfect for caches and lookup tables that are read often but written rarely.
ReadWriteLock cache
public class RWCache<K, V> {
    private final Map<K, V> cache = new HashMap<>();
    private final ReadWriteLock rwLock = new ReentrantReadWriteLock();
    private final Lock readLock  = rwLock.readLock();
    private final Lock writeLock = rwLock.writeLock();

    public V get(K key) {
        readLock.lock();    // ← many threads can read simultaneously
        try {
            return cache.get(key);
        } finally { readLock.unlock(); }
    }

    public void put(K key, V value) {
        writeLock.lock();   // ← exclusive: blocks all readers + writers
        try {
            cache.put(key, value);
        } finally { writeLock.unlock(); }
    }
    // 100 reads happening → get() allows all concurrently
    // 1 write → put() blocks until all reads done, then exclusive
}
🔖
StampedLock — Optimistic Reading
Faster than ReadWriteLock. Optimistic read: try reading without lock. Validate stamp afterwards. If another thread wrote during read, retry with real read lock.
StampedLock optimistic read
private final StampedLock sl = new StampedLock();
private double x, y;  // point coordinates

public double distanceFromOrigin() {
    long stamp = sl.tryOptimisticRead(); // no lock! stamp = version
    double currentX = x;
    double currentY = y;

    if (!sl.validate(stamp)) {  // was there a write between read and now?
        stamp = sl.readLock();  // yes! get a real read lock
        try {
            currentX = x;
            currentY = y;
        } finally { sl.unlockRead(stamp); }
    }
    return Math.sqrt(currentX * currentX + currentY * currentY);
}

public void move(double deltaX, double deltaY) {
    long stamp = sl.writeLock();
    try { x += deltaX; y += deltaY; }
    finally { sl.unlockWrite(stamp); }
}

Atomic Classes — Lock-Free Operations
CAS — Compare-And-Swap hardware instruction
⚛️
How CAS Works
compare-and-swap — hardware-level atomicity
CAS is a single atomic CPU instruction: "if memory location has expected value, replace it with new value, otherwise fail". No locks, no OS calls. Used by AtomicInteger, AtomicLong, AtomicReference etc.
CAS concept + AtomicInteger
// CAS pseudocode (what CPU does atomically):
// boolean CAS(location, expected, newValue) {
//     if (*location == expected) {
//         *location = newValue;
//         return true;
//     }
//     return false;
// }

// AtomicInteger uses CAS internally:
AtomicInteger counter = new AtomicInteger(0);

// Atomic increment (uses CAS in a loop)
counter.incrementAndGet();   // ++counter, returns new value
counter.getAndIncrement();   // counter++, returns old value
counter.addAndGet(5);        // counter += 5, returns new
counter.getAndAdd(5);        // counter += 5, returns old

// compareAndSet — CAS directly
int current = counter.get();         // read
boolean success = counter.compareAndSet(current, current + 1);
// true if counter was still "current" when we tried to set
// false if another thread changed it — retry!

// Custom CAS-based operation (general pattern)
int result = counter.updateAndGet(current -> current * 2);  // atomic double
int prev    = counter.getAndUpdate(current -> Math.max(0, current - 1)); // floor 0

// accumulateAndGet — great for reducing
counter.accumulateAndGet(10, Math::max);  // atomic max

// AtomicReference for objects
AtomicReference<String> ref = new AtomicReference<>("initial");
ref.compareAndSet("initial", "updated");  // atomic if still "initial"

arr.incrementAndGet(3);                     // arr[3]++
arr.compareAndSet(3, expected, newValue);

// ── LongAdder — better than AtomicLong for high contention ─
LongAdder adder = new LongAdder();
adder.increment();          // uses cell striping — no contention
adder.add(5);
long total = adder.sum();   // sum all cells

LongAccumulator acc = new LongAccumulator(Long::max, Long.MIN_VALUE);
acc.accumulate(42);  // keeps running max
acc.get();           // get current max

// ── VarHandle (Java 9+) — most powerful ───────────
// Direct field access with atomic semantics
private volatile int value;
private static final VarHandle VALUE;
static {
    VALUE = MethodHandles.lookup()
        .findVarHandle(MyClass.class, "value", int.class);
}
VALUE.compareAndSet(this, expected, newVal);  // CAS on field

Race Conditions, Deadlocks & Livelocks
problems every concurrent programmer faces
🏁
Race Condition
outcome depends on thread timing
Two threads access shared data concurrently, at least one writes. Result depends on execution order — undefined, unpredictable. Classic: check-then-act, read-modify-write.
🏁 Two people editing same Google Doc simultaneously — last save wins, data lost
☠️
Deadlock
circular wait — all threads stuck forever
Thread A holds Lock-1, waits for Lock-2. Thread B holds Lock-2, waits for Lock-1. Neither can proceed. Requires 4 conditions: mutual exclusion, hold-and-wait, no preemption, circular wait.
☠️ Two cars at narrow bridge — each waiting for other to reverse. Nobody moves.
🔄
Livelock
threads active but make no progress
Both threads keep responding to each other's actions but neither makes progress. Like trying to pass someone in a hallway — both step same direction repeatedly.
🔄 Two polite people in hallway — both step aside same direction, keep blocking each other forever
☠️
Deadlock — Code + Detection + Fix
Deadlock example + fixes
// ❌ DEADLOCK — Thread A and B acquire locks in opposite order!
Object lockA = new Object();
Object lockB = new Object();

Thread threadA = new Thread(() -> {
    synchronized (lockA) {
        System.out.println("Thread A: got lockA");
        Thread.sleep(100);           // gives B time to grab lockB
        synchronized (lockB) {       // ← STUCK: B holds lockB
            System.out.println("Thread A: got lockB");
        }
    }
});

Thread threadB = new Thread(() -> {
    synchronized (lockB) {           // B gets lockB first
        System.out.println("Thread B: got lockB");
        Thread.sleep(100);
        synchronized (lockA) {       // ← STUCK: A holds lockA
            System.out.println("Thread B: got lockA");
        }
    }
});

// ✅ FIX 1: Lock ordering — always acquire locks in same order
// Use object hash or ID to determine order
void transfer(Account from, Account to) {
    Account first  = from.getId() < to.getId() ? from : to;
    Account second = first == from ? to : from;
    synchronized (first) {
        synchronized (second) { /* safe */ }
    }
}

// ✅ FIX 2: tryLock with timeout — back off and retry
boolean tryTransfer(Account from, Account to) throws InterruptedException {
    while (true) {
        if (from.lock.tryLock(100, MILLISECONDS)) {
            try {
                if (to.lock.tryLock(100, MILLISECONDS)) {
                    try {
                        // do transfer
                        return true;
                    } finally { to.lock.unlock(); }
                }
            } finally { from.lock.unlock(); }
        }
        Thread.sleep(new Random().nextInt(50)); // random backoff
    }
}

// ✅ FIX 3: Detect with jstack / jcmd
// jstack <pid> → look for "Found 1 deadlock"
// jcmd <pid> Thread.print
// ThreadMXBean:
ThreadMXBean tmxb = ManagementFactory.getThreadMXBean();
long[] deadlocked = tmxb.findDeadlockedThreads();
if (deadlocked != null) {
    ThreadInfo[] infos = tmxb.getThreadInfo(deadlocked, true, true);
    // log and alert!
}

Race Condition — Check-Then-Act & Read-Modify-Write
Race conditions + fixes
// ❌ RACE: Check-Then-Act
// Thread A: if (!map.containsKey(k)) → true
// Thread B: if (!map.containsKey(k)) → true (same!)
// Both insert! Duplicate key inserted.
if (!map.containsKey(key)) {
    map.put(key, value);  // ❌ race between check and put
}
// ✅ FIX: atomic operation
map.putIfAbsent(key, value);
map.computeIfAbsent(key, k -> expensiveCompute(k));

// ❌ RACE: Read-Modify-Write
// Thread A: reads count=5
// Thread B: reads count=5
// Both write count=6 → lost one increment!

// ✅ FIX options:
synchronized(this) { count++; }   // option 1
atomicCounter.incrementAndGet();  // option 2 (better)
LongAdder adder = new LongAdder(); adder.increment(); // option 3 (best under contention)

// ❌ RACE: Lazy initialization (without volatile)

    instance = new Singleton();   // ❌ partial construction visible!
}
// ✅ FIX: Initialization-on-demand holder (best pattern)

    private static class Holder {

    }
    public static Singleton getInstance() { return Holder.INSTANCE; }
}

ThreadPoolExecutor — Internals
the engine behind all Executors
Custom ThreadPoolExecutor
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    5,                              // corePoolSize: always alive
    20,                             // maximumPoolSize: max when busy
    60, TimeUnit.SECONDS,           // keepAliveTime: idle extra threads die after this
    new ArrayBlockingQueue<>(100),  // workQueue: buffer for waiting tasks
    new ThreadFactory() {           // custom thread naming
        AtomicInteger n = new AtomicInteger(1);
        public Thread newThread(Runnable r) {
            Thread t = new Thread(r, "worker-" + n.getAndIncrement());
            t.setDaemon(false);
            return t;
        }
    },
    new ThreadPoolExecutor.CallerRunsPolicy() // rejection policy
);

// When task submitted:
// 1. If active threads < corePoolSize → create new thread
// 2. Else if queue not full → add to queue
// 3. Else if active threads < maxPoolSize → create new thread
// 4. Else → apply rejection policy!

// Rejection Policies:
// AbortPolicy (default)  → throws RejectedExecutionException
// CallerRunsPolicy       → calling thread executes the task (backpressure)
// DiscardPolicy          → silently drops the task
// DiscardOldestPolicy    → drops oldest queued task, retries

// Monitor pool health
executor.getActiveCount();          // currently executing
executor.getQueue().size();         // waiting in queue
executor.getCompletedTaskCount();   // finished
executor.getTaskCount();            // submitted total

// Graceful shutdown
executor.shutdown();                // no new tasks, finish existing
executor.awaitTermination(30, TimeUnit.SECONDS);
// OR
executor.shutdownNow();             // interrupt running tasks, return queued
THREAD POOL FLOW
1
Task Submitted
executor.submit(task) or execute(runnable)
2
Core Thread Available?
If active < corePoolSize → create new thread, run immediately
3
Queue Not Full?
All core threads busy → add to work queue, wait for thread to pick up
4
Max Threads Not Reached?
Queue full → create extra thread (beyond core, up to max)
5
Reject!
At max threads + queue full → apply RejectionPolicy

scheduled.scheduleWithFixedDelay(() -> processQueue(),

    Executors.newVirtualThreadPerTaskExecutor();
// Submit 1 million tasks — no problem!

@Service
public class CheckoutService {
    private final ExecutorService ioExecutor =
        Executors.newVirtualThreadPerTaskExecutor();

    public CheckoutResult checkout(String userId, Cart cart) {
        // Run all 4 steps in PARALLEL
        CompletableFuture<User> userFuture =
            CompletableFuture.supplyAsync(
                () -> userService.findById(userId), ioExecutor);

        CompletableFuture<Boolean> inventoryFuture =
            CompletableFuture.supplyAsync(
                () -> inventoryService.checkAvailability(cart), ioExecutor);

        CompletableFuture<BigDecimal> priceFuture =
            CompletableFuture.supplyAsync(
                () -> pricingService.calculateTotal(cart, userId), ioExecutor);

        CompletableFuture<PaymentMethod> paymentFuture =
            CompletableFuture.supplyAsync(
                () -> paymentService.getDefaultMethod(userId), ioExecutor);

        // Wait for all, then create order
        return CompletableFuture
            .allOf(userFuture, inventoryFuture, priceFuture, paymentFuture)
            .thenApply(v -> {
                User user             = userFuture.join();
                Boolean inStock       = inventoryFuture.join();
                BigDecimal total      = priceFuture.join();
                PaymentMethod payment = paymentFuture.join();

                if (!inStock) throw new OutOfStockException();
                return new Order(user, cart, total, payment);
            })
            .thenCompose(order -> paymentService.processAsync(order))
            .thenApply(payment -> {
                inventoryService.reserve(cart);
                emailService.sendConfirmation(payment);
                return new CheckoutResult(payment);
            })
            .exceptionally(ex -> {
                log.error("Checkout failed for user {}", userId, ex);
                return CheckoutResult.failed(ex.getMessage());
            })
            .orTimeout(10, TimeUnit.SECONDS)
            .join();  // block here (in controller — last mile)
    }
}

Synchronizers — Coordination Primitives
CountDownLatch · CyclicBarrier · Semaphore · Phaser
// ── CountDownLatch ────────────────────────────────────────
// One-time gate: await() blocks until count reaches 0
CountDownLatch startGate = new CountDownLatch(1);  // trigger
CountDownLatch endGate   = new CountDownLatch(10); // wait for 10

for (int i = 0; i < 10; i++) {
    new Thread(() -> {
        startGate.await();     // all 10 threads wait here
        doWork();
        endGate.countDown();   // signal completion
    }).start();
}
startGate.countDown();   // fire! all 10 start simultaneously
endGate.await();         // wait for all to finish
// Use: test harness, service ready check, parallel initialization

// ── CyclicBarrier — reusable latch ───────────────────────
// All threads wait at barrier, then all proceed together
// Cyclic: resets after all threads pass, can be used again
CyclicBarrier barrier = new CyclicBarrier(3, () ->
    System.out.println("All arrived! Continuing..."));  // optional action

Thread t1 = new Thread(() -> { doPhase1(); barrier.await(); doPhase2(); });
Thread t2 = new Thread(() -> { doPhase1(); barrier.await(); doPhase2(); });
Thread t3 = new Thread(() -> { doPhase1(); barrier.await(); doPhase2(); });
// All 3 do phase1, then all proceed to phase2 together

// ── Semaphore — limit concurrent access ──────────────────
// Controls max number of threads accessing a resource
Semaphore dbSemaphore = new Semaphore(10); // max 10 concurrent DB connections

public Data queryDb() throws InterruptedException {
    dbSemaphore.acquire();  // wait for permit (blocks if 10 already using)
    try {
        return db.query();
    } finally {
        dbSemaphore.release();  // ALWAYS release in finally!
    }
}

// ── Exchanger — swap between exactly 2 threads ───────────
Exchanger<DataBuffer> exchanger = new Exchanger<>();
// Thread A: produces, Thread B: consumes — swap buffers!
Thread producer = new Thread(() -> {
    DataBuffer full = fillBuffer();
    DataBuffer empty = exchanger.exchange(full);  // swap!
    // now has empty buffer to fill again
});

Phaser phaser = new Phaser(1);  // register main thread
for (int i = 0; i < 5; i++) {
    phaser.register();           // register worker
    new Thread(() -> {
        doWork(phaser.getPhase());
        phaser.arriveAndAwaitAdvance(); // barrier each phase
    }).start();
}
phaser.arriveAndDeregister();  // main deregisters

Virtual Threads — Project Loom Deep Dive
Java 21 — millions of threads, zero OS cost
Virtual Thread Architecture — Carrier Thread Model
HOW VIRTUAL THREADS WORK
1
Virtual Thread Created
JVM creates virtual thread. Very cheap — just a heap object (~few KB). No OS call. No kernel thread created yet.
2
Scheduled on Carrier Thread
JVM scheduler mounts virtual thread on a ForkJoinPool worker (carrier thread). Virtual thread's stack loaded on carrier's stack. Now it runs!
3
Blocking I/O Called
Virtual thread calls Thread.sleep() or blocking socket read. JVM intercepts — instead of blocking OS thread, it unmounts virtual thread from carrier.
4
Carrier Thread Freed
Virtual thread's stack saved to heap. Carrier thread is FREE to run another virtual thread! Thousands of virtual threads blocked on I/O, but only N carrier threads (N = CPU cores).
5
I/O Completes — Remount
OS signals I/O complete. JVM scheduler finds available carrier thread. Virtual thread remounted. Continues exactly where it left off.
VIRTUAL vs PLATFORM THREAD MEMORY
🖥️ Platform Thread (1 per OS thread)
Stack: ~1MB (fixed) — OS managed
OS Thread object: ~50KB kernel memory
Total: ~1-2MB per thread
10,000 threads × 1MB = 10 GB RAM 😱
🪶 Virtual Thread (JVM managed)
Stack: ~few KB (grows dynamically on heap)
Object overhead: ~200 bytes
Total: ~1-10 KB per thread
1,000,000 threads × 5KB = 5 GB RAM ✅

Virtual Thread — Pitfalls
Avoid these patterns with virtual threads
// ⚠️ PITFALL 1: Pinning — synchronized blocks carrier thread!
// In Java 21, synchronized PINS virtual thread to carrier thread
// while blocked inside synchronized block

synchronized (obj) {
    Thread.sleep(1000);  // ❌ pins carrier thread for 1 second!
    db.query();          // ❌ carrier blocked during I/O!
}

// ✅ FIX: Replace synchronized with ReentrantLock
ReentrantLock lock = new ReentrantLock();
lock.lock();
try {
    Thread.sleep(1000);  // ✅ virtual thread parks, carrier freed
    db.query();
} finally { lock.unlock(); }

// Note: Java 24 fixes synchronized pinning!

// ⚠️ PITFALL 2: CPU-bound tasks — virtual threads don't help
// Virtual threads help with I/O blocking, not CPU work
// CPU-bound: use Platform threads with Fixed thread pool (N=cores)
// I/O-bound: use Virtual threads

// ⚠️ PITFALL 3: Thread pools for virtual threads are wrong!
ExecutorService pool = Executors.newFixedThreadPool(100); // ❌
// Don't pool virtual threads — they're cheap, create per task!
ExecutorService vtp = Executors.newVirtualThreadPerTaskExecutor(); // ✅

// ⚠️ PITFALL 4: Heavy ThreadLocal use
// ThreadLocal on virtual threads works but wastes memory
// Each VT gets its own ThreadLocal copy — with millions of VTs
// this can cause significant GC pressure
// ✅ Use ScopedValue for immutable request context

// ⚠️ PITFALL 5: Semaphore or bounded resources
// Virtual threads block on semaphore.acquire() — fine!
// But: don't forget these are now CHEAP so you might overwhelm DB
Semaphore dbConnLimit = new Semaphore(50);  // still need to limit DB connections!
dbConnLimit.acquire();
try { return db.query(); } finally { dbConnLimit.release(); }

// ThreadLocal: each thread gets its OWN independent copy of the variable
public class RequestContext {

    // ── Basic ThreadLocal ─────────────────────────────────
    private static final ThreadLocal<String> requestId =
        new ThreadLocal<>();

    // ── With initial value ────────────────────────────────

    // SimpleDateFormat is NOT thread-safe — each thread gets its own instance!

    // ── Request-scoped context (web applications) ─────────
    private static final ThreadLocal<UserContext> userContext =
        new ThreadLocal<>();

    // Called by web filter BEFORE every request
    public static void set(String reqId, String userId, String role) {
        userContext.set(new UserContext(reqId, userId, role));
    }

    public static UserContext get() {
        return userContext.get();
    }

    // ⚠️ CRITICAL: ALWAYS remove after request! Memory leak otherwise!

    // so ThreadLocal values in them also live forever = memory leak!
    public static void clear() {
        userContext.remove();
    }

            RequestContext.set(generateId(), getUser(), getRole());
            chain.doFilter(req, res);
        } finally {
            RequestContext.clear();  // ✅ ALWAYS clear in finally!
        }
    }
}

ScopedValue (Java 21) — Better ThreadLocal
Immutable, structured, no memory leaks. The modern replacement for ThreadLocal in concurrent code. Values are bound for the duration of a scope, automatically cleaned up.
ScopedValue vs ThreadLocal
// ── ScopedValue (Java 21) ─────────────────────────────────
public class Server {
    static final ScopedValue<Request> CURRENT_REQUEST =
        ScopedValue.newInstance();
    static final ScopedValue<User> CURRENT_USER =
        ScopedValue.newInstance();

    // Bind values for the scope of this method + all it calls
    void handle(Request request, User user) {
        ScopedValue
            .where(CURRENT_REQUEST, request)
            .where(CURRENT_USER, user)
            .run(() -> {
                processRequest();   // can read CURRENT_REQUEST
                validateUser();     // can read CURRENT_USER
                // both automatically cleaned up when run() returns
            });
    }

    void processRequest() {
        Request req = CURRENT_REQUEST.get();  // read scoped value
        log.info("Processing: {}", req.getPath());
    }
}

// ── ThreadLocal vs ScopedValue ────────────────────────────
// ThreadLocal:   mutable, set/get anywhere, memory leak risk, works in thread pools
// ScopedValue:   immutable, bound in scope, auto-cleanup, better with VirtualThreads
✅
Use ScopedValue for new code with Virtual Threads. Use ThreadLocal only for mutable per-thread state or when you need thread pool propagation with TransmittableThreadLocal.

public class TokenBucketRateLimiter {
    private final long capacity;          // max tokens
    private final long refillRate;        // tokens per second
    private final AtomicLong tokens;      // current tokens (atomic!)
    private volatile long lastRefillTime; // when last refilled

    public TokenBucketRateLimiter(long capacity, long refillPerSecond) {
        this.capacity = capacity;
        this.refillRate = refillPerSecond;
        this.tokens = new AtomicLong(capacity);
        this.lastRefillTime = System.nanoTime();
    }

    public boolean tryAcquire() {
        refill();
        // CAS loop: atomically decrement tokens
        while (true) {
            long current = tokens.get();
            if (current <= 0) return false;  // no tokens — rejected
            if (tokens.compareAndSet(current, current - 1)) {
                return true;  // acquired!
            }
            // CAS failed (another thread changed it) — retry
        }
    }

    private void refill() {
        long now = System.nanoTime();
        long elapsed = now - lastRefillTime;
        long tokensToAdd = (elapsed * refillRate) / 1_000_000_000L;
        if (tokensToAdd > 0) {
            tokens.updateAndGet(t -> Math.min(capacity, t + tokensToAdd));
            lastRefillTime = now;
        }
    }
}

Thread-safe cache with expiry
public class ExpiringCache<K, V> {
    private record Entry<V>(V value, long expiryNanos) {}

    private final ConcurrentHashMap<K, Entry<V>> map = new ConcurrentHashMap<>();
    private final long ttlNanos;

    public ExpiringCache(Duration ttl) {
        this.ttlNanos = ttl.toNanos();
    }

    public void put(K key, V value) {
        map.put(key, new Entry<>(value, System.nanoTime() + ttlNanos));
    }

    public Optional<V> get(K key) {
        Entry<V> entry = map.get(key);
        if (entry == null) return Optional.empty();
        if (System.nanoTime() > entry.expiryNanos()) {
            map.remove(key);              // lazy eviction
            return Optional.empty();
        }
        return Optional.of(entry.value());
    }

    // compute: atomically get or compute if absent
    public V getOrCompute(K key, Supplier<V> loader) {
        return map.computeIfAbsent(key, k ->
            new Entry<>(loader.get(), System.nanoTime() + ttlNanos)
        ).value();
    }
}

public class LogPipeline {
    // Bounded queue provides backpressure — producers block when full
    private final BlockingQueue<LogEvent> queue =
        new ArrayBlockingQueue<>(10_000);
    private volatile boolean running = true;
    private final ScheduledExecutorService scheduler =
        Executors.newSingleThreadScheduledExecutor();

    // PRODUCERS — many threads call this
    public void log(LogLevel level, String message) {
        LogEvent event = new LogEvent(level, message,
            Thread.currentThread().getName(), Instant.now());
        try {
            // offer with timeout — prevents unbounded wait
            if (!queue.offer(event, 500, TimeUnit.MILLISECONDS)) {
                // Queue full and timed out — drop or count dropped
                droppedCount.incrementAndGet();
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }

    // CONSUMER — single thread, batches writes for efficiency
    public void startConsumer() {
        Thread.ofVirtual().name("log-consumer").start(() -> {
            List<LogEvent> batch = new ArrayList<>(100);
            while (running || !queue.isEmpty()) {
                try {
                    // Poll with timeout — don't spin on empty queue
                    LogEvent event = queue.poll(100, TimeUnit.MILLISECONDS);
                    if (event != null) {
                        batch.add(event);
                        // Drain more without waiting
                        queue.drainTo(batch, 99);
                        flushBatch(batch);
                        batch.clear();
                    }
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                    break;
                }
            }
        });
    }

    private void flushBatch(List<LogEvent> batch) {
        // Bulk insert to DB/Elasticsearch — much faster than one-by-one
        logRepository.saveAll(batch);
    }

    public void shutdown() {
        running = false;
        // Give consumer time to drain remaining events
        scheduler.shutdown();
    }
}

// Usage: high-throughput logging with virtual threads
LogPipeline pipeline = new LogPipeline();
pipeline.startConsumer();

// 1000 concurrent requests all logging simultaneously
try (ExecutorService exec = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 1000; i++) {
        final int req = i;
        exec.submit(() -> {
            pipeline.log(INFO, "Processing request " + req);
            processRequest(req);
            pipeline.log(INFO, "Completed request " + req);
        });
    }
}

⚡ Thread Essentials
Thread.start()Creates OS thread and calls run(). Calling run() directly is NOT multithreading!
Thread.sleep(ms)Pauses current thread. Doesn't release lock. Throws InterruptedException.
thread.join()Wait for thread to finish. join(n) waits max n milliseconds.
thread.interrupt()Sets interrupt flag. Thread must check isInterrupted() or catch InterruptedException.
Thread.currentThread()Get reference to currently executing thread. isVirtual() → virtual thread check.
synchronized(obj)Acquire obj's monitor lock. Only one thread at a time. Releases on exit/exception.
obj.wait()Must be in synchronized block. Releases lock. Thread waits for notify().
obj.notifyAll()Wakes all waiting threads. Prefer notifyAll() over notify() — safer.
volatile fieldVisibility guarantee. Every read sees latest write. NOT atomic for compound ops.
AtomicIntegerThread-safe counter. incrementAndGet(), compareAndSet(). CAS-based, no locks.
🔒 Locks & Synchronizers
ReentrantLocktryLock(), timeout, interruptible, fair mode. ALWAYS unlock() in finally block!
ReadWriteLockMultiple concurrent readers OR exclusive writer. Good for read-heavy caches.
StampedLockOptimistic read (no lock, validate stamp), fastest for read-heavy workloads.
CountDownLatchawait() blocks until countDown() called N times. One-shot gate, not reusable.
CyclicBarrierAll threads wait at barrier, then all proceed. Reusable unlike CountDownLatch.
Semaphore(N)Limit concurrent access to resource. acquire()/release(). N = max threads.
LongAdderBetter than AtomicLong under high contention. Cell striping. Use for counters.
ConcurrentHashMapThread-safe map. computeIfAbsent(), merge() are atomic. Better than synchronizedMap.
BlockingQueueThread-safe producer-consumer queue. put() blocks when full, take() when empty.
ThreadLocalPer-thread variable. ALWAYS remove() in finally to prevent memory leaks!
❓ Interview Q&A — Part 1
sleep() vs wait()? sleep() — Thread method, doesn't release lock, only current thread, no notify needed. wait() — Object method, must be synchronized, RELEASES lock, woken by notify().
volatile vs synchronized? volatile: visibility only, no mutual exclusion, compound ops NOT safe. synchronized: visibility + atomicity + mutual exclusion. Use volatile for simple flags, synchronized for compound operations.
How to stop a thread? thread.interrupt() + thread checks isInterrupted() cooperatively. Never use Thread.stop() — deprecated, corrupts state. For executor: shutdownNow() sends interrupts.
What is deadlock? How to prevent? Circular lock dependency — A waits for B, B waits for A. Prevent: consistent lock ordering, tryLock() with timeout + backoff, avoid nested locks, use higher-level abstractions.
Why use thread pools? Thread creation is expensive (~1ms, ~1MB). Pools reuse threads. Control max threads (prevent OOM). Provide work queue for backpressure. Monitor and manage lifecycle.
❓ Interview Q&A — Part 2
Virtual threads vs Platform threads? Platform: 1:1 with OS thread, ~1MB stack, ~10K max. Virtual: JVM-managed, ~KB stack, millions possible, auto-park on I/O, best for I/O-bound work.
What is thread pinning? In Java 21, synchronized blocks prevent virtual thread from unmounting from carrier thread. Wastes a carrier thread while blocked. Fix: use ReentrantLock instead. Fixed in Java 24.
ConcurrentHashMap vs synchronizedMap? ConcurrentHashMap: segment locking, multiple threads write concurrently, CAS for reads. synchronizedMap: wraps HashMap, locks entire map for every read AND write — much slower.
What is happens-before? JMM guarantee: if A happens-before B, all writes by A are visible to B. Guaranteed by: synchronized exit/entry, volatile write/read, Thread.start(), Thread.join().
CompletableFuture vs Future? Future: get() blocks, no chaining, no exception handling. CompletableFuture: non-blocking callbacks (thenApply, thenCompose), combine (allOf, anyOf), error handling (exceptionally), composable pipelines.
