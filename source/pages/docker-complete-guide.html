Core Concepts — Image vs Container vs Registry
the holy trinity
📸
Docker Image
Blueprint / Template
A read-only template with everything needed to run your app — OS base, runtime, libraries, code, configs. Built from a Dockerfile. Immutable once built. Has layers (more on this below). Can be versioned with tags.
Read-only
Versioned with tags
📸 Like a class in OOP — defines the blueprint, doesn't run itself
📦
Docker Container
Running instance
A running instance of an Image. Has its own isolated filesystem, network, and processes. You can start, stop, restart, delete it. Multiple containers can run from the same image simultaneously. Has a writable layer on top.
Running process
Isolated
📦 Like an object (instance) created from a class — actually running in memory
🏪
Docker Registry
Image store
A storage and distribution system for Docker Images. Docker Hub is the default public registry (hub.docker.com). You can also run a private registry (ECR on AWS, GCR on GCP, Harbor self-hosted). Push/pull images like git push/pull.
Docker Hub (public)
ECR / GCR (private)
🏪 Like GitHub — but for Docker Images instead of code

Docker Architecture — How It All Fits Together
client–daemon–registry
Docker Architecture
👤 CLIENT LAYER — You talk to Docker via these
💻 docker CLI
The docker command you type in terminal. Sends requests to Docker Daemon via REST API. e.g., docker run nginx
🖥️ Docker Desktop
GUI app for Windows/Mac. Includes Docker Daemon + CLI + Kubernetes. Great for local development.
🔧 Docker Compose CLI
Runs multi-container apps defined in docker-compose.yml. Talks to daemon to start all services at once.
↕ REST API (Unix socket / TCP)
⚙️ DOCKER DAEMON (dockerd) — The engine, runs on the host OS
📋 Image Manager
Pulls images from registry, stores locally, manages layers. Handles docker pull, docker build.
📦 Container Manager
Creates, starts, stops, deletes containers. Manages lifecycle using containerd underneath.
🔀 Network Manager
Creates virtual networks (bridge, host, overlay). Enables container-to-container communication.
💾 Volume Manager
Manages persistent storage volumes. Mounts host paths or named volumes into containers.
↕ gRPC API
🏗️ CONTAINER RUNTIME LAYER — Actually creates containers
containerd
Industry-standard container runtime. Manages the complete lifecycle: image transfer, container execution, storage, networking. Used by Docker AND Kubernetes.
runc
Low-level OCI runtime. Actually creates the container — sets up namespaces, cgroups, and runs the process. Called by containerd per container start.
Linux OS Primitives
Namespaces (isolate PID, network, filesystem) + cgroups (limit CPU/RAM) = the magic that makes containers work without VMs.
↕ network
🏪 REGISTRY — Remote image storage
Docker Hub
Default public registry. Free tier available. Stores official images: nginx, postgres, redis, node, python...
Private Registries
AWS ECR, GCP Artifact Registry, GitHub Container Registry, Azure ACR, Harbor (self-hosted). For company proprietary images.
OCI Standard
Open Container Initiative defines the image format. Any OCI-compliant image works across Docker, Kubernetes, containerd.

Docker Image Layers — How Images Are Built
union filesystem + copy-on-write
🥞
Images = Stack of Layers
each instruction = one layer
Every Dockerfile instruction (FROM, RUN, COPY...) creates a new read-only layer. Layers are cached and shared between images. If two images share the same base layer, it's stored ONCE on disk. The container adds a thin writable layer on top.
🥞 Like a pancake stack — each layer adds to the one below. To change the bottom, you rebuild from there up.
⚡
Layer Caching
why build order matters
Docker caches each layer. If a layer hasn't changed, Docker reuses the cached version — massive speedup. But once a layer changes, ALL layers below it must rebuild. Best practice: copy package.json BEFORE copying source code — dependencies don't change as often!
# BAD — any code change rebuilds npm install! COPY . . RUN npm install # GOOD — dependencies cached separately COPY package.json package-lock.json ./ RUN npm install # cached unless package.json changes COPY . . # only this layer rebuilds on code change
Layer What it contains Mode
📝 Writable Layer
Container's runtime changes (new files, logs). Dies when container stops unless using a volume. Added by docker run, not docker build.
Read/Write
📂 COPY src code
Your application source files added to the image. Changes on every code push — rebuilt each time.
Read-only
📦 RUN npm install
node_modules installed. Large but cached as long as package.json doesn't change. Big savings!
Read-only
📋 COPY package.json
Dependency manifest. Triggers reinstall only when this changes. Smart ordering saves build time.
Read-only
⚙️ RUN apt-get install
System packages installed in the container (curl, git, etc). Usually cached after first build.
Read-only
🌊 FROM node:18-alpine
Base image — Alpine Linux + Node.js runtime. Shared across all images using this base. Pulled once from registry.
Base Layer

Dockerfile — Writing Your First Build Recipe
every instruction explained
📄 Dockerfile — Node.js App Example
# Start from official Node.js 18 Alpine image
FROM node:18-alpine

# Who maintains this image
LABEL maintainer="team@company.com"

# Set working directory inside container
WORKDIR /app

# Copy dependency files first (for cache)
COPY package*.json ./

# Install dependencies (cached layer)
RUN npm ci --only=production

# Copy application source code
COPY src/ ./src/

# Set environment variable
ENV NODE_ENV=production
ENV PORT=3000

# Expose port (documentation only)
EXPOSE 3000

# Create non-root user for security
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser

# Healthcheck — K8s/Docker uses this
HEALTHCHECK CMD wget -q localhost:3000/health || exit 1

# Run the app
CMD ["node", "src/index.js"]
📄 Multi-Stage Build — Go App (smaller images!)
# ─── Stage 1: BUILD ──────────────────────
FROM golang:1.21 AS builder

WORKDIR /build
COPY go.mod go.sum ./
RUN go mod download
COPY . .

# Compile to static binary
RUN CGO_ENABLED=0 go build -o app .


# ─── Stage 2: RUNTIME ────────────────────
# Only the compiled binary — no Go SDK!
FROM scratch

# Copy ONLY the binary from builder stage
COPY --from=builder /build/app /app

EXPOSE 8080
CMD ["/app"]


# Result:
# Builder image:  ~800MB (includes Go SDK)
# Final image:    ~5MB   (binary only!)
# 160x smaller!
FROM
Sets the base image. Every Dockerfile starts here. Use slim/alpine variants for smaller images. FROM node:18-alpine
RUN
Executes shell commands during build and commits the result as a new layer. Chain with && to reduce layers.
COPY / ADD
Copies files from host into image. COPY is preferred. ADD can unpack tarballs and fetch URLs (use carefully).
WORKDIR
Sets the working directory for subsequent instructions. Creates it if it doesn't exist. Like cd /app that sticks.
ENV
Sets environment variables available at build-time AND runtime. Override at docker run -e KEY=val.
ARG
Build-time variables only. Pass with docker build --build-arg VAR=val. NOT available at runtime (unlike ENV).
EXPOSE
Documents which port the container listens on. Does NOT actually publish the port — use -p in docker run for that.
CMD
Default command to run. Only one per Dockerfile. Can be overridden by docker run ... mycommand. Use JSON array format.
ENTRYPOINT
Like CMD but harder to override. Combine: ENTRYPOINT sets the executable, CMD provides default args. docker run img args passes args to entrypoint.
USER
Sets the user for subsequent RUN/CMD/ENTRYPOINT. Always run as non-root in production for security! Default is root (bad!).
HEALTHCHECK
Command Docker runs to check container health. Sets status to healthy/unhealthy. Used by Docker Compose and Kubernetes.
VOLUME
Declares mount point for external storage. Data in this path persists beyond container lifecycle when mounted.

Docker Networking — How Containers Communicate
bridge · host · none · overlay
🌉
Bridge Network
default mode
Default network for containers. Docker creates a virtual switch (docker0). Containers on the same bridge can talk to each other via IP. Use custom bridge for DNS-based container names!
# Create custom bridge network docker network create mynet # Run containers on same network docker run --network mynet --name db postgres docker run --network mynet --name api myapi # api can reach db by name "db"!
🌉 Virtual LAN — containers on same bridge are like PCs on the same WiFi
🔗
Host Network
no isolation
Container shares the host machine's network stack directly. No port mapping needed — container uses host ports directly. Best performance but zero network isolation. Only works on Linux.
docker run --network host nginx # nginx listens on host port 80 directly # No -p needed — but no isolation either!
🔗 Like plugging directly into the wall socket — maximum speed, no protection
🕳️
None Network
fully isolated
No networking at all — container has only a loopback interface. Completely isolated from the network. Use for batch processing containers that don't need network access. Maximum security.
docker run --network none data-processor # No inbound or outbound connections possible # Use for security-critical batch jobs
🕳️ Like a computer with no network cable — runs alone
🌐
Port Mapping
-p host:container
Publish container ports to the host. Format: host_port:container_port. Required for bridge networks to be accessible from outside. Multiple ports can be mapped.
# host:8080 → container:3000 docker run -p 8080:3000 myapp # Multiple ports docker run -p 80:80 -p 443:443 nginx # Random host port docker run -p 3000 myapp
🏢 Like a hotel front desk — external call to room 101 goes to guest 101 inside
🗺️
Overlay Network
multi-host (Swarm/K8s)
Connects containers across multiple Docker hosts. Used in Docker Swarm and Kubernetes. Creates a virtual network that spans machines. Containers communicate as if on the same host even across data centers.
# Only in Swarm mode docker network create -d overlay swarm-net # Containers on different hosts can reach each other!
🗺️ Like a VPN connecting offices across cities — seamless communication
🔍
DNS Resolution
container name = hostname
On custom bridge networks, Docker provides automatic DNS. Containers can reach each other by container name instead of IP. IPs change on restart — names don't. Essential for production setups.
# In your app code, connect to: "mongodb://db:27017" # not an IP! "redis://cache:6379" # container name "http://api:3000" # works in same network
📞 Like calling a colleague by name instead of memorising their extension

Docker Volumes — Persistent Storage
data that survives container death
💀
Problem: Containers are Ephemeral
When a container stops or is deleted, all data written inside it is GONE. A database container that stores data inside itself loses ALL data on restart. This is by design — containers should be stateless and replaceable.
💀 Like writing on a whiteboard that gets erased when you leave the room
💾
Solution: Mount External Storage
Mount storage from outside the container into it. Data lives outside the container lifecycle — survives stop, restart, delete. Three types: Named Volumes (managed by Docker), Bind Mounts (host path), tmpfs (RAM only).
💾 Like plugging in an external hard drive — data survives even if you throw away the computer
📀
Named Volumes
Docker-managed (BEST for prod)
Docker creates and manages the storage location. Independent of host directory structure. Portable, easy to back up, works on all platforms. Best choice for databases and persistent app data.
# Create a named volume docker volume create pgdata # Mount into container docker run -v pgdata:/var/lib/postgresql postgres # Inspect where Docker stores it docker volume inspect pgdata
📀 Like a managed storage account — you name it, Docker handles the rest
📁
Bind Mounts
host directory (BEST for dev)
Mount a specific directory from the HOST into the container. Changes on host reflect instantly in container and vice versa. Perfect for development — edit code on host, container picks up changes live!
# Mount current directory into /app docker run -v $(pwd):/app node:18 # Edit files locally → live in container! # Perfect for hot-reload development # Mount specific config file docker run -v ./nginx.conf:/etc/nginx/nginx.conf nginx
📁 Like Dropbox folder — both sides see the same files in real time
⚡
tmpfs Mount
RAM-only (fast, temporary)
Stores data in the host machine's RAM only. Extremely fast — no disk I/O. But data is lost when container stops. Great for temporary sensitive data (tokens, keys) that should never touch disk.
docker run --tmpfs /tmp:rw,size=64m myapp # /tmp inside container is in RAM # Lost on container stop — never on disk # Good for: sessions, temp tokens, cache
⚡ Like RAM vs SSD — blazing fast but gone when power off

Docker Compose — Multi-Container Apps
define your stack in one YAML file
📄 docker-compose.yml — Full Stack App
# docker-compose.yml
version: '3.9'

services:

  api:                       # Node.js backend
    build: ./backend          # build from Dockerfile
    ports:
      - "3000:3000"
    environment:
      - DB_URL=postgres://db:5432/myapp
      - REDIS_URL=redis://cache:6379
    depends_on:
      db:
        condition: service_healthy
    volumes:
      - ./backend:/app           # hot reload in dev
    networks:
      - app-network
    restart: unless-stopped

  frontend:                  # React app
    build: ./frontend
    ports:
      - "80:80"
    depends_on:
      - api
    networks:
      - app-network

  db:                        # PostgreSQL
    image: postgres:15
    environment:
      - POSTGRES_DB=myapp
      - POSTGRES_PASSWORD=secret
    volumes:
      - pgdata:/var/lib/postgresql
    healthcheck:
      test: ["CMD","pg_isready","-U","postgres"]
      interval: 10s
      retries: 5
    networks:
      - app-network

  cache:                     # Redis
    image: redis:7-alpine
    networks:
      - app-network

volumes:
  pgdata:                    # named volume

networks:
  app-network:
    driver: bridge
🎼
Why Docker Compose?
one command to rule them all
Instead of running 10 docker run commands with flags you always forget, define your entire stack in one YAML file. Start everything with a single command. Compose handles networking, volumes, and startup order automatically.
# Start entire stack docker compose up -d # View all logs together docker compose logs -f # Stop everything docker compose down # Rebuild and restart docker compose up --build -d # Scale a specific service docker compose up --scale api=3
🎼 Like a conductor score — one document describes how all instruments play together
⚙️
Key Compose Concepts
services — each container (api, db, cache)
depends_on — startup order + health conditions
networks — all services on same network by default
volumes — persistent data declaration
environment — env vars (use .env file for secrets)
restart: unless-stopped — auto-restart on crash
healthcheck — wait for service to be truly ready

🎯 Interview Q&A
Image vs Container? Image = read-only blueprint (like a class). Container = running instance of that image (like an object). Many containers from one image.
VM vs Container? VMs virtualize entire OS (heavy, slow, GBs). Containers share host OS kernel, only package app + libs (light, fast, MBs). Docker containers start in seconds vs minutes.
CMD vs ENTRYPOINT? CMD = default command, easily overridden. ENTRYPOINT = always runs, args appended. Use both: ENTRYPOINT for executable, CMD for default args.
COPY vs ADD? COPY just copies files. ADD can unpack tar files and fetch URLs. Best practice: always use COPY unless you specifically need ADD's extra features.
Volume vs Bind Mount? Volumes managed by Docker, portable, good for production. Bind mounts tie to host path, great for dev (live code reload). Use volumes in prod, bind mounts in dev.
How to reduce image size? Use alpine base, multi-stage builds, combine RUN commands, use .dockerignore, don't install dev dependencies, clean cache after apt-get.
What is layer caching? Each Dockerfile instruction = cached layer. Unchanged layers reuse cache = faster builds. Order matters — put frequently changing layers last.
Container security tips? Run as non-root user, use minimal base images, scan for vulnerabilities (Trivy, Snyk), use read-only filesystem where possible, never store secrets in images.
