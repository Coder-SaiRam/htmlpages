Section 1
SELECT & WHERE
Execution order · NULL handling · CASE · COALESCE · NULLIF · String & Date functions
What this section covers: SELECT is where you specify what columns to return, but it's one of the last clauses to execute. Understanding execution order prevents a whole class of bugs (like using aliases in WHERE). NULL is not a value — it's the absence of a value, and every comparison with NULL returns UNKNOWN, not TRUE or FALSE.
⚡ SQL Execution Order
SQL looks like it runs top-to-bottom (SELECT → FROM → WHERE), but internally the database executes clauses in a completely different order. This is why you can't use a SELECT alias in WHERE — WHERE runs at step 2, but SELECT aliases are created at step 5. Memorise this order to debug 90% of SQL errors.
1 FROM/JOIN→ 2 WHERE→ 3 GROUP BY→ 4 HAVING→ 5 SELECT→ 6 DISTINCT→ 7 ORDER BY→ 8 LIMIT/OFFSET
⚠️
Alias in WHERE → Error

You can't use SELECT salary*12 AS ctc FROM emp WHERE ctc > 1000000 — ctc doesn't exist at step 2. Fix: repeat the expression, or wrap in a CTE. You CAN use aliases in ORDER BY (runs at step 7).

SELECT picks which columns to show and can compute new values inline (salary*12, ROUND, etc.). WHERE filters rows before any grouping. LIMIT caps the result. Think of SELECT as "what do I want to see?" and WHERE as "which rows qualify?"

CASE is SQL's if-else. It evaluates conditions top-to-bottom and returns the first matching THEN value. ELSE is optional — without it, unmatched rows get NULL. You can use CASE anywhere in SELECT, WHERE, ORDER BY, or inside aggregate functions.

SELECT name, salary,
    CASE
        WHEN salary >= 100000 THEN 'L5 — Director'
        WHEN salary >=  85000 THEN 'L4 — Senior'
        WHEN salary >=  70000 THEN 'L3 — Mid'
        ELSE                        'L2 — Junior'
    END band
FROM employees ORDER BY salary DESC;

Section 2
JOINs
INNER · LEFT · RIGHT · FULL OUTER · CROSS · SELF — with exact output tables
What are JOINs? JOINs combine rows from two tables based on a related column (usually a foreign key). The type of JOIN determines what happens when there's no match — INNER silently drops unmatched rows, LEFT keeps all left-table rows regardless of match, FULL OUTER keeps everything. The two edge cases to always test: Ivy (NULL dept_id) and HR dept (no employees).
INNER JOIN — Only rows that match in both tables
INNER JOIN is the most common type. A row appears in the result only if the join condition is satisfied on BOTH sides. Any row that doesn't find a match is silently dropped. This is why Ivy disappears (her dept_id is NULL — no match in departments) and HR dept doesn't appear (no employees point to dept_id=40).
inner_join.sql
SQL
SELECT e.name, d.dept_name, e.salary
FROM   employees   e
JOIN   departments d ON e.dept_id = d.dept_id
ORDER BY e.salary DESC;

Section 3
GROUP BY & Aggregates
COUNT · SUM · AVG · MIN · MAX · HAVING · ROLLUP · FILTER
What GROUP BY does: It collapses multiple rows that share the same value into a single row, and lets you compute aggregates (COUNT, SUM, AVG, etc.) per group. A crucial rule: every column in SELECT must either be inside an aggregate function OR appear in GROUP BY. Use HAVING (not WHERE) to filter on aggregate results — because WHERE runs before grouping.
Aggregate Functions — COUNT, SUM, AVG, MIN, MAX
Aggregate functions compress many rows into one computed value. COUNT(*) counts all rows including NULLs; COUNT(col) counts only non-NULL values. AVG, SUM, MIN, MAX all ignore NULLs automatically — Ivy's NULL dept_id means she won't appear in dept-grouped results.
group_by.sql
SQL
SELECT
    d.dept_name,
    COUNT(e.emp_id)          headcount,
    SUM(e.salary)            total_payroll,
    ROUND(AVG(e.salary), 0) avg_salary,
    MIN(e.salary)            min_sal,
    MAX(e.salary)            max_sal
FROM   employees   e
JOIN   departments d ON e.dept_id = d.dept_id
GROUP BY d.dept_id, d.dept_name
ORDER BY avg_salary DESC;

HAVING — Filter Groups After Aggregation
HAVING is like WHERE but for groups. WHERE filters individual rows before grouping; HAVING filters the resulting groups. You must use HAVING when your condition involves an aggregate like COUNT(*) > 1 or SUM(salary) > 200000.
having.sql
SQL
-- Only departments with avg salary above 80000
SELECT d.dept_name,
    COUNT(e.emp_id)          headcount,
    ROUND(AVG(e.salary),0) avg_salary
FROM employees e
JOIN departments d ON e.dept_id = d.dept_id
GROUP BY d.dept_id, d.dept_name
HAVING   AVG(e.salary) > 80000
ORDER BY avg_salary DESC;

ROLLUP — Auto Subtotals & Grand Total
GROUP BY ROLLUP generates subtotals and a grand total automatically. It produces N+1 result rows: one per group value, plus a summary row where the group column is NULL (meaning "all groups combined"). Use COALESCE to label the NULL row as "ALL DEPTS" or "TOTAL".
rollup.sql
SQL

    COUNT(e.emp_id)                     headcount,
    SUM(e.salary)                       total_payroll
FROM employees e
JOIN departments d ON e.dept_id = d.dept_id
GROUP BY ROLLUP(d.dept_name);

FILTER — Conditional Aggregates
The FILTER clause lets you apply a condition inside an aggregate, computing different aggregates in the same GROUP BY query. It's cleaner and faster than the old CASE-inside-SUM trick: SUM(CASE WHEN condition THEN 1 ELSE 0 END).

Section 4
Subqueries & CTEs
Scalar subquery · EXISTS · NOT EXISTS · CTE · Recursive CTE · Gap Detection
What are CTEs? A CTE (Common Table Expression) is a named temporary result set defined with WITH name AS (...). It makes complex queries readable by breaking them into named steps you can reference like a table. Think of CTEs as giving a name to a subquery so you can reuse it. Recursive CTEs can traverse hierarchical data like org trees — impossible with regular JOINs.
Scalar Subquery — Inline Single-Value Computation
A scalar subquery returns exactly one value (one row, one column). It runs once per row of the outer query and can be used anywhere an expression is valid — in SELECT, WHERE, or HAVING. If it returns more than one row, you get an error.
scalar_subquery.sql
SQL
SELECT name, salary,
    (SELECT ROUND(AVG(salary),0) FROM employees) company_avg,
    salary - (SELECT ROUND(AVG(salary),0) FROM employees) diff
FROM employees ORDER BY diff DESC;

CTE — Employee vs Department Average
CTEs shine when a derived result (like "dept average salary") needs to be joined back to the original table. Without a CTE, you'd need a messy nested subquery. The WITH clause names the subquery once, and you reference it cleanly below — making the logic easy to read and debug.
cte.sql
SQL
WITH dept_avg AS (
    SELECT dept_id, ROUND(AVG(salary),0) avg_sal
    FROM   employees WHERE dept_id IS NOT NULL
    GROUP BY dept_id
)
SELECT
    e.name, d.dept_name, e.salary,
    da.avg_sal                        dept_avg,
    ROUND(e.salary - da.avg_sal, 0) vs_dept
FROM      employees  e
JOIN      departments d  ON e.dept_id  = d.dept_id
JOIN      dept_avg   da ON e.dept_id  = da.dept_id
ORDER BY vs_dept DESC;

Recursive CTE — Org Hierarchy Tree
Recursive CTEs have two parts: an anchor (starting rows — here, top-level managers with no manager themselves) and a recursive part that joins back to the CTE to get one more level. Always add a depth limit (WHERE depth < 10) to prevent infinite loops from bad data. This is the standard way to query tree/hierarchical data in SQL.
recursive_cte.sql
SQL
WITH RECURSIVE org AS (
    -- Anchor: top-level (no manager)
    SELECT emp_id, name, manager_id, 0 depth
    FROM   employees
    WHERE  manager_id IS NULL AND dept_id IS NOT NULL

    UNION ALL

    -- Recursive: reports of current level
    SELECT e.emp_id, e.name, e.manager_id, o.depth + 1
    FROM   employees e
    JOIN   org       o ON e.manager_id = o.emp_id
    WHERE  o.depth < 10    -- ⚠️ loop guard!
)
SELECT REPEAT('  ', depth) || name org_chart, depth
FROM  org ORDER BY emp_id;

Section 5
Window Functions
ROW_NUMBER · RANK · DENSE_RANK · LAG · LEAD · Running Totals · Moving Averages · Frame Clauses
What makes window functions special? Unlike GROUP BY which collapses rows, window functions add a computed column to every row without removing any rows. They "look through a window" at nearby rows — the current row, previous rows, next rows — while keeping the full result set intact. The OVER() clause defines the window: which rows to look at, and in what order.
GROUP BY vs OVER() — The Core Difference
GROUP BY compresses 9 employee rows into 3 department rows — you lose individual employee data. OVER() adds department-level context to every individual row without collapsing anything. This is why window functions are so powerful for comparisons like "how does this employee compare to their department?"

RANK vs DENSE_RANK vs ROW_NUMBER
All three functions assign a position number to each row within a partition. The difference is in how they handle ties. For Top-N filtering, always use DENSE_RANK — RANK skips numbers after ties which causes you to miss rows when filtering with WHERE rnk <= 2.
ranking.sql
SQL
SELECT name, dept_id, salary,
    ROW_NUMBER() OVER(PARTITION BY dept_id ORDER BY salary DESC) row_num,
    RANK()       OVER(PARTITION BY dept_id ORDER BY salary DESC) rnk,
    DENSE_RANK() OVER(PARTITION BY dept_id ORDER BY salary DESC) dense_rnk
FROM   employees WHERE dept_id IS NOT NULL
ORDER BY dept_id, salary DESC;

Top-N Per Group (Classic Pattern)
The most common window function interview question. The pattern: 1) Use DENSE_RANK() OVER(PARTITION BY group ORDER BY metric DESC) inside a CTE. 2) Filter the CTE with WHERE dense_rnk <= N. You cannot filter window function results directly in WHERE — window functions run at step 5, WHERE at step 2. Always wrap in a CTE first.
top_n.sql
SQL
WITH ranked AS (
    SELECT e.name, d.dept_name, e.salary,
        DENSE_RANK() OVER(
            PARTITION BY e.dept_id
            ORDER BY e.salary DESC
        ) rnk
    FROM  employees  e
    JOIN  departments d ON e.dept_id = d.dept_id
)
SELECT name, dept_name, salary, rnk
FROM   ranked WHERE rnk <= 2
ORDER BY dept_name, rnk;

LAG, Running Total & Moving Average
LAG(col) looks at the previous row's value — perfect for comparing current vs previous period. SUM() OVER (ORDER BY ...) creates a running total. AVG() OVER (ROWS BETWEEN 2 PRECEDING AND CURRENT ROW) creates a 3-period moving average. These are the backbone of time-series analysis in SQL.
lag_running.sql
SQL
SELECT name, joined, salary,
    LAG(salary) OVER(ORDER BY joined)  prev_hire_sal,
    SUM(salary) OVER(ORDER BY joined
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)  running_total,
    ROUND(AVG(salary) OVER(ORDER BY joined
        ROWS BETWEEN 2 PRECEDING AND CURRENT ROW), 0)  moving_avg_3
FROM employees ORDER BY joined;

Indexes — Backbone of Performance
B-Tree · Composite · Partial · Covering · Expression · GIN · When NOT to index
What is an index? An index is a separate data structure (usually a B-Tree) that the database maintains alongside a table — like the index at the back of a book. Without an index, the DB reads every single row (sequential scan = O(n)). With a B-Tree index, it uses binary search to find rows in O(log n). The trade-off: indexes speed up reads but slow down writes (every INSERT/UPDATE/DELETE must update the index too).
Sequential Scan vs Index Scan — Visual
Without an index, finding emp_id=7 in 9 rows requires reading all 9 rows. With a B-Tree index, you do binary search: split in half repeatedly until you find it. At 1 million rows, sequential scan = 1,000,000 reads; B-Tree = ~20 reads. This difference is why indexing foreign key columns is critical.

Row 5: emp_id=5 → not 7, continue...

Row 9: scan done (9 reads total)

Index Types — B-Tree, Composite, Partial, Covering, Expression, GIN
PostgreSQL offers many index types for different use cases. B-Tree handles most cases (equality, range, ordering). Composite indexes speed up multi-column WHERE clauses but require the leftmost column to be in the query. Partial indexes only index a subset of rows (e.g., active records). Covering indexes store extra columns so the DB never has to read the table row at all.
indexes.sql
SQL
-- BASIC B-TREE
CREATE INDEX idx_dept    ON employees(dept_id);
CREATE INDEX idx_salary  ON employees(salary);
CREATE UNIQUE INDEX idx_email ON employees(email);

-- COMPOSITE: leftmost prefix rule!
CREATE INDEX idx_dept_sal ON employees(dept_id, salary);
-- ✅ WHERE dept_id=10 AND salary>70000  ← uses both cols
-- ✅ WHERE dept_id=10                    ← uses leftmost
-- ❌ WHERE salary>70000                 ← skips dept_id → seq scan!

-- PARTIAL: only index relevant rows (smaller, faster)
CREATE INDEX idx_high_sal ON employees(salary)
    WHERE salary > 80000;   -- only 4 of 9 rows indexed

-- COVERING: no table row fetch needed!
CREATE INDEX idx_covering ON employees(dept_id)
    INCLUDE (name, salary);   -- Postgres 11+ INCLUDE

-- EXPRESSION: for function-based searches
CREATE INDEX idx_lower_name ON employees(LOWER(name));
-- Used by: WHERE LOWER(name) = 'alice'  (case-insensitive search)

-- ALWAYS use CONCURRENTLY in production (non-blocking!)
CREATE INDEX CONCURRENTLY idx_safe ON employees(salary);

Transactions & ACID
Atomicity · Consistency · Isolation · Durability · Isolation Levels · Locking · Deadlocks
Why transactions? A transaction groups multiple SQL statements into one all-or-nothing unit. Imagine deducting ₹10,000 from account A and adding it to account B — if the second statement crashes, the money disappears into the void without transactions. ACID properties guarantee correctness even during system failures or when thousands of users run queries simultaneously.
ACID Properties
ACID is not just a set of rules — it's a contract the database makes with your application. Atomicity ensures partial writes never persist. Consistency enforces all constraints (CHECK, FK, UNIQUE) before and after. Isolation prevents transactions from seeing each other's in-progress changes. Durability guarantees commits survive crashes via write-ahead logging (WAL).
A
Atomicity
All or nothing. Any step fails → ALL rolled back. No partial state.
💸 Debit AND credit, or neither
C
Consistency
DB goes from one valid state to another. All constraints hold before and after.
⚖️ Balance never goes negative
I
Isolation
Concurrent transactions behave as if sequential. Controlled by isolation levels.
👤 Each user sees consistent data
D
Durability
Committed data permanently written to disk via WAL. Survives power loss.
✅ Your booking is permanent
BEGIN, COMMIT, ROLLBACK, SAVEPOINT
BEGIN starts a transaction. All subsequent statements are part of the transaction until COMMIT (saves everything) or ROLLBACK (discards everything). SAVEPOINTs let you partially rollback — like an undo checkpoint inside the transaction. RETURNING after DML statements returns the modified rows without an extra SELECT.
transactions.sql
SQL
-- Budget transfer between departments
BEGIN;
    UPDATE departments SET budget = budget - 50000 WHERE dept_id = 10;
    UPDATE departments SET budget = budget + 50000 WHERE dept_id = 30;
    -- If second UPDATE fails → both rolled back
COMMIT;

-- SAVEPOINT: partial rollback inside a transaction
BEGIN;
    INSERT INTO orders(cust_id, total_amt, status) VALUES(1, 79999, 'pending');
    SAVEPOINT order_saved;
    INSERT INTO order_items(order_id, prod_id, qty) VALUES(LASTVAL(), 101, 1);
    -- If item insert fails:
    ROLLBACK TO order_saved;   -- undo item, keep order
COMMIT;

-- UPSERT: atomic insert-or-update
INSERT INTO employees(emp_id, name, dept_id, salary, joined)
VALUES(2, 'Bob', 10, 80000, '2020-07-15')
ON CONFLICT(emp_id) DO UPDATE SET
    salary = EXCLUDED.salary
RETURNING emp_id, name, salary;

💀
Deadlock Prevention Rule

Always acquire locks in the SAME ORDER across all transactions. If T1 locks Engineering then Finance, T2 must also lock Engineering then Finance — never reversed. PostgreSQL detects deadlocks and kills one transaction, causing errors and retries.

Table Partitioning — Range, List, Hash
Partitioning splits a logically single table into physical child tables. The key benefit: when you query with a filter on the partition key (e.g., order_date in 2024), the DB only scans the 2024 partition — "partition pruning." Warning: if you apply a function to the partition key (EXTRACT(YEAR FROM order_date)), pruning fails and you scan everything.
partitioning.sql
PostgreSQL
-- Parent table — declares partition strategy
CREATE TABLE orders_part (
    order_id   BIGSERIAL, cust_id INT NOT NULL,
    order_date DATE NOT NULL, total_amt NUMERIC(12,2)
) PARTITION BY RANGE(order_date);

-- Child partitions — one per year
CREATE TABLE orders_2024 PARTITION OF orders_part
    FOR VALUES FROM ('2024-01-01') TO ('2025-01-01');
CREATE TABLE orders_2025 PARTITION OF orders_part
    FOR VALUES FROM ('2025-01-01') TO ('2026-01-01');

-- ✅ Partition pruning works — only scans orders_2024
SELECT * FROM orders_part
WHERE  order_date >= '2024-01-01';

-- ❌ Pruning FAILS! Function on partition key
-- WHERE EXTRACT(YEAR FROM order_date) = 2024 → full scan!
Materialized Views — Cache Expensive Aggregations
A regular VIEW is just a saved query — it reruns every time you SELECT from it. A Materialized View stores the result on disk like a table. It must be REFRESHed to pick up new data. Use CONCURRENTLY to refresh without blocking reads (requires a unique index). Ideal for daily/hourly reports over millions of rows.
👁️ Regular VIEW
• No storage — just saved SQL
• Always fresh — runs on SELECT
• Slow for complex aggregations
• Cannot be indexed directly
✅ Materialized View
• Stores result on disk like a table
• Must REFRESH to update data
• Fast reads — already computed
• CAN be indexed for faster queries

Section 9
SQL Anti-Patterns
Common mistakes that cause bugs, silent data loss, or performance disasters — with fixes
Why anti-patterns matter: Most SQL performance disasters and subtle data bugs come from a small set of well-known mistakes. These patterns often "work" in development with small data, then explode in production at scale. This section shows the before/after for each — so you recognise them on sight.
= NULL & NOT IN with NULLs
NULL is not a value — it's "unknown." Any comparison with NULL returns UNKNOWN (not TRUE, not FALSE), so the WHERE clause never matches. This silently returns 0 rows — no error, no warning. Similarly, NOT IN (list with NULLs) silently returns 0 rows because of NULL propagation in three-valued logic.
❌ Broken — always 0 rows
-- dept_id = NULL is UNKNOWN, never TRUE
SELECT name FROM employees
WHERE  dept_id = NULL;   -- 0 rows always!

-- If manager_ids list contains NULL:
-- NOT IN returns 0 rows silently
SELECT name FROM employees
WHERE  emp_id NOT IN (1, 4, NULL);  -- 0 rows!
✅ Fixed
-- IS NULL is the only correct way
SELECT name FROM employees
WHERE  dept_id IS NULL;   -- Returns Ivy ✅

-- Use NOT EXISTS — safe with NULLs
SELECT name FROM employees e
WHERE NOT EXISTS (
    SELECT 1 FROM employees m
    WHERE  m.emp_id IN (1, 4)
      AND  m.emp_id = e.emp_id
);
Function on Indexed Column & OFFSET Pagination
Wrapping an indexed column inside a function breaks the index — the DB can't use the index because it can't predict what YEAR(date) returns for each row. For pagination, OFFSET scans and discards all previous pages on every request — page 500 requires reading 10,000+ rows just to throw them away. Keyset pagination jumps directly to the cursor position.
❌ Index Ignored & O(n) Pagination
-- Index on order_date is IGNORED
SELECT * FROM orders
WHERE EXTRACT(YEAR FROM order_date) = 2024;
-- Plan: Seq Scan (reads all rows!)

-- Page 500 → scans & discards 10,000 rows
SELECT * FROM orders
ORDER BY order_id
LIMIT 20 OFFSET 10000;
✅ Index Used & O(1) Keyset
-- Range comparison — index scan!
SELECT * FROM orders
WHERE  order_date >= '2024-01-01'
  AND  order_date <  '2025-01-01';

-- Keyset: always fast regardless of depth
SELECT * FROM orders
WHERE  order_id > 10000    -- cursor from last page
ORDER BY order_id LIMIT 20;

Your best performance debugging tool: EXPLAIN shows you how the database plans to execute a query — which indexes it uses, how it joins tables, how many rows it estimates. Month-Over-Month Revenue with % Change
A staple analytics query: calculate total revenue per month, then use LAG() to compare with the previous month. NULLIF prevents division-by-zero on the first month. This query uses a CTE to compute monthly totals first, then the outer query adds the window function comparison — you can't do both in one step.
mom_revenue.sql
PostgreSQL
WITH monthly AS (
    SELECT
        DATE_TRUNC('month', order_date) month,
        SUM(total_amt)                  revenue,
        COUNT(DISTINCT cust_id)           buyers
    FROM orders WHERE status = 'delivered'
    GROUP BY 1
)
SELECT
    TO_CHAR(month, 'Mon YYYY')       month,
    revenue,
    LAG(revenue) OVER(ORDER BY month) prev_month,
    ROUND((revenue - LAG(revenue) OVER(ORDER BY month))
        / NULLIF(LAG(revenue) OVER(ORDER BY month), 0)
        * 100, 1) pct_change
FROM  monthly ORDER BY month;

Job Queue with FOR UPDATE SKIP LOCKED
FOR UPDATE SKIP LOCKED is the correct pattern for a multi-worker job queue in PostgreSQL. Multiple workers run the same query simultaneously — each one atomically claims a job that no other worker has locked yet. Without SKIP LOCKED, workers would block on each other; with it, each worker instantly moves to the next available row.
job_queue.sql
PostgreSQL
CREATE TABLE job_queue (
    job_id     BIGSERIAL PRIMARY KEY,
    payload    JSONB, status VARCHAR(20) DEFAULT 'PENDING',
    run_at     TIMESTAMPTZ DEFAULT NOW()
);
CREATE INDEX idx_pending ON job_queue(status, run_at)
    WHERE status = 'PENDING';   -- partial index — only pending jobs

-- Each worker atomically claims one job
WITH claimed AS (
    SELECT job_id FROM job_queue
    WHERE  status = 'PENDING' AND run_at <= NOW()
    ORDER BY run_at LIMIT 1
    FOR UPDATE SKIP LOCKED   -- magic: skip rows locked by other workers
)
UPDATE job_queue
SET    status = 'PROCESSING'
WHERE  job_id = (SELECT job_id FROM claimed)
RETURNING *;

❓ Interview Q&A

What happens when you use = NULL in a WHERE clause?
It always returns zero rows. NULL comparisons return UNKNOWN, not TRUE or FALSE — not even NULL = NULL is TRUE. Always use IS NULL or IS NOT NULL. Similarly, NOT IN (1, 2, NULL) returns zero rows because NULL propagation in three-valued logic means the "NOT IN NULL" part is always UNKNOWN.
Why can't you use a SELECT alias in WHERE?
Because WHERE (step 2) runs before SELECT (step 5). The alias doesn't exist yet when WHERE evaluates. Fix: wrap in a CTE or subquery, or repeat the expression. You CAN use SELECT aliases in ORDER BY because that runs after SELECT (step 7).
INNER JOIN vs LEFT JOIN — when to use which?
INNER JOIN: when you only want rows with a match in both tables (unmatched rows silently disappear). LEFT JOIN: when you need ALL rows from the left table regardless of whether there's a match — unmatched right-side columns are NULL. The anti-join pattern (LEFT JOIN ... WHERE right.id IS NULL) finds rows in A with NO match in B.
ROW_NUMBER vs RANK vs DENSE_RANK — which for Top-N?
Always use DENSE_RANK for Top-N filtering. ROW_NUMBER: always unique (1,2,3,4) — arbitrary tiebreaker. RANK: ties same number, skips next (1,1,3) — WHERE rnk <= 2 misses the 3rd row when there's a tie at rank 2. DENSE_RANK: ties same number, no gap (1,1,2) — WHERE dense_rnk <= 2 always gets all rows at the top-N positions.
What is a covering index and when does it help most?
A covering index holds all columns a query needs so the DB never reads actual table rows (Index Only Scan). Create with INCLUDE: CREATE INDEX ON employees(dept_id) INCLUDE (name, salary). Most beneficial for read-heavy queries that always select the same few columns alongside a high-cardinality lookup column.
What is ACID and why does it matter?
Atomicity: all-or-nothing. Consistency: constraints always hold. Isolation: concurrent transactions don't interfere. Durability: committed data survives crashes. Without ACID, two users reading stock=1 simultaneously could both buy it, leaving stock=-1. ACID prevents this entire class of data corruption.
What is the N+1 query problem?
N+1 happens when code fetches N rows, then runs 1 additional query per row in a loop. Example: fetch 100 customers (1 query), then for each fetch their orders (100 more) = 101 total. The fix: a single JOIN or a WHERE IN subquery to get all data at once. Symptoms: response time proportional to result size, many near-identical queries in DB logs.
When should you NOT add an index?
Don't index: small tables (<10K rows — full scan faster), low-cardinality columns (boolean, status with 2-3 values — optimizer prefers full scan when >5-10% rows match), and heavily written columns (every INSERT/UPDATE pays index update cost). Always validate with EXPLAIN ANALYZE first — don't guess.
DELETE vs TRUNCATE vs DROP?
DELETE FROM t WHERE ...: removes rows, can be rolled back, fires triggers. TRUNCATE TABLE t: removes ALL rows instantly (DDL), faster, resets sequences, doesn't fire row-level triggers — but can be rolled back in PostgreSQL. DROP TABLE t: destroys entire table structure + data + all indexes, triggers, and constraints permanently.
What is the Leftmost Prefix Rule for composite indexes?
Given CREATE INDEX ON t(a, b, c): the index helps queries that filter on a alone, or a+b, or a+b+c. Queries filtering only on b or only on c cannot use this index (skips the leftmost column). Think of it like a phone book sorted by last name then first name — you can't look up by first name alone without scanning everything.
How do you safely add an index on a production table?
Use CREATE INDEX CONCURRENTLY idx_name ON tablename(col). Regular CREATE INDEX acquires a lock that blocks all writes until the index is built. CONCURRENTLY builds the index in the background without blocking — it takes longer but keeps the table available. Never drop indexes with plain DROP INDEX in production either — use CONCURRENTLY there too.
