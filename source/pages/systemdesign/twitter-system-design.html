Component 01

DNS & Load Balancer

The front door of the entire system — gets the request to the right datacenter.

🌐

DNS — Route 53 + GeoDNS

Resolves twitter.com → nearest datacenter IP
DNS is the internet's phone book. When you type twitter.com, your browser queries Route 53 which returns the IP of the nearest Twitter datacenter. Twitter uses GeoDNS — a user in Mumbai gets routed to the Asia-Pacific datacenter, a user in New York gets the US-East IP. This alone cuts latency by 100–200ms before a single byte of application code runs.
Twitter uses Anycast routing — multiple datacenters advertise the same IP prefix. BGP routing automatically selects the topologically closest datacenter. During a regional outage, BGP withdraws those routes and traffic automatically shifts to the next-best datacenter within seconds.

Benefits

  • Geographic load distribution
  • Automatic failover via BGP
  • Reduces network hops
  • DNS TTL controls failover speed
  • Zero application-layer changes

Limitations

  • DNS TTL causes stale caching (30s–5min)
  • Doesn't account for server load
  • Resolver location ≠ user location
  • Propagation delay during changes

Key Tradeoffs

  • Low TTL = faster failover but more DNS queries
  • High TTL = stale during incidents
  • Twitter uses 60s TTL as balance
⚖️

Load Balancer — Layer 4 & Layer 7

HAProxy for L4, Nginx/Envoy for L7 — distributes across thousands of app servers
Twitter runs a two-tier load balancing architecture. L4 load balancers (HAProxy) operate at the TCP level — they see IP packets and use consistent hashing to distribute connections. L7 load balancers (Nginx/Envoy) operate at HTTP — they can route based on URL path, headers, or method, enabling traffic splitting between services (e.g., /api/timeline → Timeline Service, /api/tweet → Tweet Service).

Benefits

  • Horizontal scale across N servers
  • Health checks auto-remove dead nodes
  • SSL termination at LB (offload servers)
  • Rate limiting at the edge
  • A/B testing via weighted routing

Drawbacks

  • LB itself can be a bottleneck
  • Need redundant LBs (active-passive)
  • Sticky sessions complicate stateless design
  • Extra network hop per request

Algorithms used

  • Round-robin for uniform servers
  • Least connections for variable load
  • Consistent hashing for cache locality
  • IP hash for session affinity
Twitter Reality

Twitter's load balancers handle ~30 billion requests/day. They run redundant LB pairs in active-passive mode — if the active LB dies, the passive takes over via VRRP (Virtual Router Redundancy Protocol) with sub-second failover. The LB layer also does DDoS mitigation — connection rate limits, SYN cookie protection, and anycast black-holing for attack traffic.

Component 02

API Gateway & App Servers

Authentication, rate limiting, and the microservices that own each domain.

🔒

API Gateway

Single entry point — auth, rate limiting, routing, observability
The API Gateway is the bouncer and router for all API traffic. Every request must pass through it. It validates OAuth 2.0 tokens (Twitter uses Bearer tokens), enforces rate limits per user/app (Twitter API v2: 500K tweets/month for basic apps), routes to the correct microservice, and emits distributed traces for observability.
OAuth 2.0 Bearer tokens Rate limiting: token bucket JWT validation Request tracing (OpenTelemetry) Circuit breaker

Benefits

  • Single auth enforcement point
  • API versioning (v1/v2) isolation
  • Request/response transformation
  • Circuit breaking prevents cascade failures
  • Centralized rate limiting

Drawbacks

  • Single point of failure if not HA
  • Adds ~1–5ms latency per request
  • Gateway becomes a complex monolith
  • Versioning can get messy at scale
Rate limiting uses a token bucket algorithm per (user_id, endpoint) stored in Redis. Each endpoint has a bucket with a max capacity and a refill rate. When a user exceeds their limit, the API returns HTTP 429 with a Retry-After header. Without this, a single viral tweet would let bots scrape millions of requests.

Component 03 — The Hard Problem

Fan-out Service

The most complex piece of Twitter's architecture — distributing one tweet to 300M timelines.

📢

Fan-out on Write vs Fan-out on Read

The fundamental architectural dilemma — Twitter does both, and here's why
When Katy Perry (100M followers) posts a tweet, Twitter must somehow make that tweet appear in 100 million people's home timelines. There are two pure approaches — and Twitter uses a hybrid of both.
Twitter's hybrid approach: For regular users (≤50K followers), tweet IDs are pushed into each follower's cached timeline on Cassandra immediately (fan-out on write). For celebrities (Katy Perry, Obama), their tweets are NOT pre-distributed — instead, when a follower opens their timeline, the system fetches the celebrity's recent tweets and merges them on the fly (fan-out on read). This avoids writing Katy Perry's tweet into 100M rows.

Fan-out on Write Pros

  • Timeline reads are O(1) — pre-built
  • No N queries at read time
  • Read latency is predictable
  • Works for majority of users

Fan-out on Write Cons

  • Massive write amplification
  • Lady Gaga tweet = 50M DB writes
  • Celebrity tweets stall the pipeline
  • Kafka lag during viral moments

Why Hybrid?

  • 95% of users: <50K followers → push
  • 5% celebrity accounts → pull
  • Keeps write amplification bounded
  • Read latency still fast for both paths
Twitter Engineering Blog — the Katy Perry Problem

Twitter's engineers call this the "thundering herd" problem. When Katy Perry's account went viral in 2013, their fan-out workers couldn't keep up — timelines were showing tweets 5 minutes late. The hybrid solution (celebrity threshold at ~50K followers) was their fix. The fan-out service reads the social graph from FlockDB, batches follower IDs into groups of 100, and writes tweet_id references into each follower's Cassandra timeline row — all asynchronously via Kafka workers.

Component 04

Timeline Service

How 200 billion timelines per day get built and served in under 200ms.

📜

Home Timeline Architecture

Cassandra timeline store + Redis cache + celebrity merge
A user's home timeline is essentially a sorted list of tweet_ids. It's stored in Cassandra as a column family: row key = user_id, columns = (timestamp, tweet_id). The most recent 800 tweet_ids per user are cached in Redis. When you open Twitter, the timeline service: (1) fetches the cached tweet IDs from Redis, (2) fetches any un-cached celebrity tweets from their profile, (3) merges and sorts everything, (4) hydrates the tweet_ids into full tweet objects by fetching from the Tweet service or Memcached.
1

Redis cache lookup

Check if user_id's timeline is cached. Returns up to 800 tweet_ids. Cache hit: ~0.5ms. If miss, fall back to Cassandra.

2

Celebrity tweet merge

For each celebrity the user follows (determined by FlockDB), fetch their last N tweets. Done in parallel with step 1.

3

Merge + sort by timestamp

Combine the pre-built timeline IDs with the freshly-fetched celebrity tweet IDs. Sort by timestamp descending. Top 20 are the first page.

4

Hydrate tweet IDs → full objects

Batch-fetch tweet content, user info, like/retweet counts from Memcached (or MySQL if cache miss). Typically 1 round-trip via pipelined Redis/Memcached.

5

Return ranked timeline

Apply ML ranking (engagement signals, recency, relationship strength) and return the final JSON payload. Total target: <200ms end-to-end.

Cassandra Timeline Store

  • Linear horizontal scaling
  • Tunable consistency (quorum reads)
  • Excellent for time-series data
  • Multi-datacenter replication built-in
  • High write throughput natively

Cassandra Limitations

  • No JOINs — data must be pre-modeled
  • Read performance degrades with tombstones
  • Complex data modeling required
  • Compaction pauses can cause latency spikes

Consistency choice

  • Write: quorum (majority of nodes)
  • Read: local quorum (fast)
  • Replication factor: 3 per DC
  • Eventual → timeline can be seconds stale
Twitter Timeline Engineering

Twitter calls their timeline storage "Manhattan" (a custom distributed key-value store built on top of MySQL at first, now purpose-built). Each timeline is stored as a list of up to 800 tweet IDs. When the list exceeds 800, old entries are trimmed. If a user scrolls to older tweets, the system does a real-time query to Cassandra. The Redis layer holds timelines for active users only — accounts inactive for 30+ days are evicted from cache, saving enormous memory.

Component 05

Database Layer

Multiple databases, each chosen for its specific access pattern — no one-size-fits-all.

🗄️

MySQL (Sharded) — Tweet & User Store

The source of truth for tweets, users, and relationships
Tweet and user data is stored in sharded MySQL. Twitter historically chose MySQL for its ACID guarantees (tweets must never be lost or duplicated), mature tooling, and their engineers' familiarity. A single MySQL instance can handle ~5,000 writes/sec — insufficient for 6,000 tweets/sec — so the data is sharded.
Sharding strategy: Tweet table sharded by tweet_id (a Snowflake ID). User table sharded by user_id. The shard key is embedded in the ID itself — the high bits of the Snowflake ID contain the datacenter ID and worker ID, so routing to the right shard requires zero external lookup.

MySQL Sharding Pros

  • ACID guarantees on tweet writes
  • Horizontal scale via sharding
  • Read replicas for read scale-out
  • Snowflake IDs enable shard-local writes
  • Battle-tested at web scale

MySQL Sharding Cons

  • Cross-shard joins are impossible
  • Re-sharding is painful operation
  • Hot shards if key distribution is uneven
  • Schema changes require careful migration
-- Snowflake tweet_id: 64 bits total -- [1 unused][41 timestamp ms][10 worker ID][12 sequence] shard_id = tweet_id >> (10 + 12) & 0x3FF -- top 10 bits = worker = shard -- Schema (simplified) CREATE TABLE tweets ( tweet_id BIGINT PRIMARY KEY, -- Snowflake ID user_id BIGINT NOT NULL, content TEXT, created_at TIMESTAMP DEFAULT NOW(), media_ids JSON, INDEX (user_id, created_at) -- for profile timeline ) ENGINE=InnoDB;
FlockDB — Social Graph Database

Twitter built FlockDB, a distributed graph database (open-sourced in 2010) for storing follower/following relationships. It's built on top of MySQL with custom sharding. The core schema is just: (source_id, destination_id, position). FlockDB is optimized for two operations: "give me all followers of user X" and "give me all users that X follows". These queries happen on every timeline build — billions of times per day.

Component 06

Caching Layer

Without caching, 2.3M read QPS would obliterate any database. Cache is what makes Twitter possible.

⚡

Twemcache + Redis — Multi-Tier Cache

Twitter's custom Memcached fork + Redis for complex data structures
Twitter uses Twemcache (their Memcached fork with better eviction policies) for object caching — tweet objects, user profiles, like counts. Redis is used for data structures that need more than simple get/set: sorted sets for timelines, sets for follower lists, atomic counters for like/retweet counts, and pub/sub for real-time notifications.
Cache LayerTechnologyWhat's storedEviction
L1 — Object cacheTwemcache (cluster)Full tweet JSON, user profilesLRU + custom hotness score
L2 — Timeline cacheRedis ClusterList of 800 tweet_ids per active userLRU + inactive user eviction (30d)
L3 — Counter cacheRedis (atomic INCR)Like counts, retweet counts, follower countsWrite-through to DB every N ops
L4 — Session cacheRedisAuth tokens, user sessionsTTL-based expiry
L5 — Trending cacheRedis Sorted SetsHashtag scores (ZINCRBY)Sliding window TTL

Cache Benefits

  • Absorbs 99%+ of read traffic
  • Sub-millisecond response times
  • Counters are atomic (no race conditions)
  • Redis cluster auto-shards data
  • Reduces DB cost dramatically

Cache Challenges

  • Cache invalidation on tweet delete
  • Counter drift (cache vs DB discrepancy)
  • Cold start problem (cache stampede)
  • Hot keys — celebrity user_ids overwhelm one shard

Hot Key Problem

  • Katy Perry's profile on one Redis node
  • Millions of reads/sec to that node
  • Fix: local in-process cache on app servers
  • Fix: read from any of N replica shards
The cache stampede: when a cache key expires, thousands of app servers simultaneously miss the cache and hammer the DB with the same query. Twitter's solution: probabilistic early expiry — slightly before the TTL expires, a random subset of servers regenerates the cache proactively, preventing the simultaneous expiry.
Twemcache — Twitter's Memcached fork

Twitter open-sourced Twemcache after modifying Memcached to support per-item TTLs with jitter (preventing mass simultaneous evictions), a DECR floor at 0 (counters can't go negative), and automatic slab rebalancing for more efficient memory use. Their Memcached cluster serves over 30 billion requests/day with a hit rate above 95%.

Component 07

Media Storage & CDN

100TB of images and videos per day — stored, transcoded, and served globally.

🎬

S3 + Video Pipeline + CloudFront CDN

Upload → transcode → store → serve from edge, globally
When you upload a photo or video: the client uploads directly to S3 via a pre-signed URL (bypassing app servers entirely — saves bandwidth). An S3 event trigger fires a Lambda/worker that kicks off the media processing pipeline: images are resized into multiple resolutions (thumb, small, medium, large, orig), videos are transcoded into multiple bitrates (360p, 480p, 720p, 1080p) using FFmpeg workers on Kubernetes. Processed media is stored back in S3 and served via CloudFront CDN from 450+ edge locations worldwide.
1

Client requests pre-signed S3 URL

App server generates a time-limited URL signed with AWS credentials. Client uploads directly to S3. App server never touches the bytes.

2

S3 event → Media Processing Queue

S3 ObjectCreated event fires to SQS/Kafka. Workers pick up the job and begin transcoding. Videos are processed in parallel by multiple workers.

3

Image resizing

Original image is resized to 4 standard sizes using ImageMagick/Sharp. Twitter also strips EXIF data (privacy) and runs AI content moderation.

4

Video transcoding (HLS/DASH)

FFmpeg produces multiple bitrate renditions. Master HLS playlist allows adaptive bitrate streaming — browser picks quality based on network speed.

5

Store → CDN → serve

Processed files stored in S3. CloudFront serves from nearest edge. Cache-Control: max-age=31536000 (1 year immutable, since URLs contain content hash).

CDN Benefits

  • Serves from 450+ PoPs globally
  • Reduces origin bandwidth by 99%
  • Absorbs DDoS at edge
  • Immutable URLs → infinite cache TTL
  • HTTP/2 multiplexing reduces RTT

CDN Tradeoffs

  • Cache invalidation is hard
  • Cold region misses go to origin
  • Cost increases with egress volume
  • Long-tail content rarely cached
Twitter Media Numbers

Twitter serves over 1.6 billion media items per day via CDN. Their video pipeline processes videos up to 512MB and 2:20 minutes. Using adaptive bitrate streaming (HLS), a user on a 3G connection automatically gets 360p while a fiber user gets 1080p — same URL, different quality segments served by the CDN based on bandwidth estimation.

Component 08

Search Service

Real-time indexing of 500M tweets per day with sub-second query response.

🔍

Elasticsearch + Real-Time Indexing

Tweets searchable within seconds of posting
Twitter's search is particularly hard because tweets must be searchable within seconds of being posted (real-time search). A tweet about a breaking earthquake should appear in search immediately. Traditional search engines batch-index — Twitter needed a streaming indexing pipeline.
Architecture: Tweets flow through Kafka to a fleet of Elasticsearch indexing workers that parse, tokenize, and write to the Elasticsearch cluster. Twitter built a custom inverted index on top of Lucene (the engine powering Elasticsearch) called Earlybird — designed for real-time append-only indexing and optimized for Twitter's unique query patterns (recency-ranked results, hashtag queries).

Elasticsearch Pros

  • Full-text search with relevance scoring
  • Horizontal sharding out of the box
  • Aggregations for trending topics
  • Near real-time indexing (<1s)
  • Fuzzy matching / typo tolerance

Elasticsearch Cons

  • Not ACID — indexing can lag
  • Memory intensive (inverted index in RAM)
  • Re-indexing at scale is expensive
  • Split-brain risk in ES cluster

Why Earlybird?

  • Standard Lucene: rebuild index for updates
  • Earlybird: append-only in-memory index
  • Tweets never update (immutable)
  • Perfect fit for Lucene's segment model
Twitter Search — Earlybird Architecture

Twitter's Earlybird (open-sourced as part of Apache Lucene work) keeps recent tweets in a RAM-resident inverted index. Older tweets are compacted into immutable on-disk segments. A search query fans out across all Earlybird shards in parallel, each returning its top-K results, which are then merged and re-ranked by a blending service that considers recency, engagement, user authority, and personal relevance signals.

Component 09

Notification Service

Delivering billions of push notifications per day — without spamming users or crashing devices.

🔔

Push, WebSocket & Email Delivery

FCM → Android · APNs → iOS · WebSocket → web · SES → email
Every like, retweet, reply, follow, and mention generates a notification. At Twitter's scale, this is a fan-out problem: Lady Gaga's tweet liked by 1M people = 1M notification events, each potentially triggering a push to the original poster. The notification service consumes events from Kafka, applies throttling and deduplication rules, and dispatches to FCM (Android), APNs (iOS), WebSocket (web), or email (SES).

Benefits

  • Async delivery — no user-facing latency
  • Deduplication prevents notification storms
  • Rate limiting per user (max 1 notif/min)
  • Retry logic for delivery failures

Challenges

  • Delivery receipts unreliable (FCM/APNs)
  • Millions of stale device tokens
  • Notification thundering herd on viral events
  • Per-user preference management at scale

Throttling tradeoff

  • Too many notifs → user turns them off
  • Too few → user misses important events
  • ML model predicts optimal notification timing
  • Batching: "You have 30 new likes"
Notification at Celebrity Scale

When a celebrity gets 100K likes on a tweet in 1 minute, naively sending 100K individual push notifications is catastrophic — it would crash their phone. Twitter's notification service rate-limits per recipient: if you receive more than N notifications of the same type in a time window, they're batched into a summary: "Barack Obama: 45K people liked your tweet". The Kafka consumer for notifications uses a sliding window counter in Redis to track notification frequency per user.

Component 10 — The Nervous System

Kafka Event Bus

Every async operation in Twitter flows through Kafka — the backbone of the entire system.

⚡

Apache Kafka — Distributed Event Streaming

Fan-out, analytics, notifications, search indexing — all via Kafka topics
Kafka is the central nervous system of Twitter's backend. Every significant action — tweet posted, user followed, tweet liked — publishes an event to a Kafka topic. Multiple consumer groups read from the same topics independently: the fan-out service, the search indexer, the notification service, the analytics pipeline, the trending service. No service knows about another service — they only know about Kafka topics.

Kafka Benefits

  • Producers decouple from consumers
  • Multiple independent consumer groups
  • 7-day message retention (replay ability)
  • Millions of messages/sec throughput
  • Built-in partitioning for parallelism
  • Consumer offset control (reprocess from any point)

Kafka Tradeoffs

  • Messages are eventually processed (async)
  • Ordering only within a partition
  • Consumer lag during spike events
  • Complex ops: ZooKeeper/KRaft management
  • At-least-once delivery → idempotent consumers required
Kafka at Twitter Scale

Twitter runs hundreds of Kafka topics across multiple clusters. The "tweets" topic alone handles over 6,000 messages/second baseline, spiking to 30,000+ during major events (New Year's Eve, World Cup goals, breaking news). Kafka's replication factor of 3 means each message is written to 3 brokers before acknowledgment — a broker can die and no data is lost. Consumer groups process messages in parallel: 50 fan-out workers each reading a different partition, giving 50× parallelism on a single topic.

In your interview: after designing the system, proactively say "the biggest tradeoff in this design is X". For Twitter, say: "I chose fan-out on write because the 10:1 read ratio means optimizing reads is worth the write amplification — but for celebrity accounts with millions of followers, I'd switch to fan-out on read to avoid Kafka consumer lag."
