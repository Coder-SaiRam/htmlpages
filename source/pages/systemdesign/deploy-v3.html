The CI/CD Pipeline

Every commit travels through a sequence of automated gates before it reaches users. Each gate catches a different class of bug — fail at any gate and the pipeline stops.

⚙

Each stage has a clear role. Lint and type-check run in 30 seconds and catch syntax errors, TypeScript mismatches, and undefined references before any test runs. Unit tests validate business logic in isolation — a 247-test suite with 87% coverage. Integration tests spin up an ephemeral Postgres container and verify that the services wire together correctly, catching SQL query errors and API contract mismatches. Security scanning checks npm packages against the CVE database and runs SAST analysis for injection patterns. Docker build verifies the application can be containerised — a stage that catches missing files, wrong base images, and missing environment variables. Smoke tests run after staging deploy, hitting 7 critical endpoints to confirm the service actually started successfully.

yaml — .github/workflows/ci.yml
name: CI/CD Pipeline on: pull_request: { branches: [main] } push: { branches: [main] } jobs: quality: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - run: npm ci - run: npm run lint && npm run typecheck - run: npm test -- --coverage --coverageThreshold='{"global":{"lines":80}}' - run: npm run test:integration build: needs: quality # Only runs if quality passes steps: - run: npm audit --audit-level=high - run: | docker build -t myapp:sha-${{ github.sha }} . docker push $ECR/myapp:sha-${{ github.sha }} deploy-staging: needs: build if: github.ref == 'refs/heads/main' steps: - run: kubectl set image deployment/api api=myapp:sha-${{ github.sha }} - run: npm run test:smoke -- --url https://staging.myapp.com deploy-production: needs: deploy-staging environment: production # Requires manual approval from release-managers team steps: - run: ./scripts/canary-deploy.sh --weight 10 --observe 300

When to use Blue-Green

  • Payment processing changes
  • Authentication system updates
  • Database schema migrations
  • When you need instant rollback
  • Any change where 1% user exposure is too much

When to use Canary

  • New API endpoints at scale
  • Algorithm changes (ranking, search)
  • Performance-sensitive code
  • New features with uncertain load impact
  • When you want real traffic validation

When Rolling is fine

  • Frontend/UI changes
  • Non-critical backend services
  • Internal tools
  • Low-risk config changes
  • Any stateless service where v1+v2 can coexist

Avoid Recreate in prod

  • Forces real downtime window
  • Users see error pages during deploy
  • No gradual validation
  • Full rollback requires another deploy
  • Acceptable only for dev environments

The key insight is that green receives zero live traffic during the entire deployment and validation phase. You can run smoke tests, integration checks, and even load tests against green using an internal URL before any user sees it. When you're satisfied, the load balancer switch is a single command that takes effect in milliseconds — and blue remains warm and ready for an instant rollback for the next 24 hours.

bash — blue-green switch (AWS ALB)
# 1. Deploy v2.15 to Green (zero traffic) kubectl set image deployment/api-green api=myapp:sha-b9e3 -n production kubectl rollout status deployment/api-green --timeout=5m # 2. Validate against internal green URL ./smoke-test.sh --url http://green-internal.myapp.com # 3. Gradually shift traffic for w in 10 25 50 100; do aws elbv2 modify-rule --rule-arn $RULE --actions "[ {\"TargetGroupArn\":\"$BLUE_TG\",\"Weight\":$((100-w))}, {\"TargetGroupArn\":\"$GREEN_TG\",\"Weight\":$w}]" sleep 120 ./check-error-rate.sh --threshold 0.005 || exit 1 done # Rollback in 5 seconds if needed: aws elbv2 modify-rule --actions '[{"TargetGroupArn":"$BLUE_TG","Weight":100}]'

Canary releases work because you can observe real production behaviour on a small population before committing the full fleet. The typical progression — 1% → 5% → 10% → 25% → 50% → 100% — with 5-minute observation windows between each step. Tools like Argo Rollouts automate this: they query Prometheus metrics at each step and abort automatically if thresholds are exceeded. The automated rollback is the key feature — you do not need an engineer watching dashboards at 3 AM.

// argo rollouts — automated canary Define your canary strategy in YAML with metric thresholds. Argo Rollouts queries Prometheus every 60 seconds during each step. If errorRate > 0.01 triggers 3 consecutive times, it automatically aborts, routes all traffic back to the stable version, and fires a PagerDuty alert. No human needs to be watching.

Feature Flags

Decouple deployment from release. Ship code to production with the feature off, then roll it out to users independently — no new deployment required.

🚩

Feature Flag Rollout — 100 Users

Drag the slider — watch users receive the new feature based on a deterministic hash of their user ID
Rollout: 0%
● new feature   ● old experience    0 / 100 enabled

The core insight is that deployment and release become two separate events. Code is deployed on Tuesday with the flag OFF. The product team enables it for 5% of users on Thursday with no deployment, no risk window, and no engineer involvement. If anything goes wrong, the flag is flipped OFF in 5 seconds — no rollback, no incident. The hash used for bucket assignment must be deterministic: hash(flagName + userId) % 100 so the same user always sees the same experience across sessions and devices.

typescript — feature flag service
async function isEnabled(flag: string, userId: string): Promise<boolean> { const cfg = await redis.hGetAll(`ff:${flag}`); if (!cfg?.enabled || cfg.enabled === 'false') return false; // Internal team always on (allow-list) if (cfg.allowList?.split(',').includes(userId)) return true; // Deterministic percentage bucket const bucket = murmurHash(`${flag}:${userId}`) % 100; return bucket < Number(cfg.rolloutPct ?? 0); } // Usage — wrap new code path, keep old path intact const showNewDashboard = await isEnabled('new-dashboard', user.id); return showNewDashboard ? renderNewDashboard() : renderOldDashboard();

Production is on fire at 3 AM. A hotfix bypasses the normal pipeline queue to reach production in minutes — but it must still pass tests.

bash — hotfix workflow
# 1. Branch from the PRODUCTION TAG — not from main # (main may have unreleased features you don't want to ship) git checkout -b hotfix/fix-payment-null v2.14.3 # 2. Make the targeted fix # Before: const amount = order.currency.amount; ← crashes if null # After: const amount = order.currency?.amount ?? 0; git commit -m "fix(payment): null guard on currency in processCharge" git tag v2.14.4 && git push origin hotfix/fix-payment-null --tags # 3. CI runs targeted tests only (2 min vs 8 min full suite) # 4. Deploy directly to production with abbreviated smoke test kubectl set image deployment/api api=myapp:sha-hotfix-v2.14.4 -n production # 5. CRITICAL: cherry-pick back to main — without this, next # regular deploy will re-introduce the bug git checkout main && git cherry-pick v2.14.4 && git push origin main
// the #1 hotfix mistake Forgetting the cherry-pick back to main. The on-call engineer fixes production at 4 AM, breathes a sigh of relief, and goes to sleep. The next morning a routine PR merges, CI runs, and the next deployment overwrites the hotfix because it was never committed to the main branch. The same bug returns. This has happened at nearly every engineering team at least once. Some teams automate it: a GitHub Actions workflow opens a cherry-pick PR automatically when a hotfix tag is pushed.

// expand-contract: the only safe DB migration pattern Rolling back app code is trivial. Rolling back a dropped column requires restoring from backup — measured in hours. The expand-contract pattern keeps you safe: Phase 1 — add the new column with a default, deploy. Old and new code both work. Phase 2 — deploy code that writes to both columns, backfill existing rows. Phase 3 — remove old column once no code reads it. Each phase is independently rollback-safe. Never drop a column in the same deploy that removes its code reference.

End-to-End: Dark Mode Feature

JIRA-456 goes from Sarah's first commit to 1.2 million users over 9 days. Every decision, every command, every metric.

🌙

Complete Feature Delivery Walkthrough

Trunk-based dev · GitHub Actions CI · Blue-Green deploy · LaunchDarkly flags

The team uses trunk-based development, Kubernetes on AWS, GitHub Actions, and LaunchDarkly for feature flags. The app has 1.2M daily active users and blue-green deployments via AWS ALB. Dark mode requires a UI component, a backend API, and a new database column.

D1

Branch, code, wrap in flag immediately

Sarah cuts a branch from main and wraps the DarkModeToggle component in a feature flag on the very first commit — not when the feature is "done." This enables merging incomplete work to main safely. The DB migration is Phase 1 only: ADD COLUMN dark_mode BOOL DEFAULT FALSE (non-locking in MySQL 8).

D1

PR opens — CI runs in 6m 44s

Lint (28s), type-check (45s), unit tests 247 tests at 87% coverage (2m14s), integration tests (4m02s), security scan clean. Two engineers review — one change request on the migration script, Sarah fixes, both approve, PR merges to main.

D1

Auto-deploy to Dev — flag stays OFF

CI builds myapp:sha-a4f2c1d, pushes to ECR, rolls out to dev namespace, runs smoke tests. All pass. Sarah enables the flag for herself only in LaunchDarkly dev (allow-list: sarah@myapp.com). The toggle appears for her. Zero other users see any change.

D2

QA testing — bug found and fixed

Release manager promotes to staging. QA enables the flag for the 10-person QA team. After 3 hours: all pass except one — revenue chart colours invisible in dark mode on Safari 17. Sarah fixes in a follow-up PR, deploys to staging, QA re-tests and approves.

D3

Production deployment — blue-green

Migration runs first (0.041s, zero downtime). New image deploys to green (20 pods). Smoke tests pass. Traffic shifts: 10% → observe 5 min (error 0.04% ✓) → 100%. Blue kept warm for 24h. The feature flag is still OFF — zero users see any change. Deployment and release are fully decoupled.

D3–5

Staged flag rollout — no deploys needed

Day 3 at 2 PM: 5% rollout → 60,000 users → 12 positive support messages, zero bugs. Day 4: 20% → stable overnight. Day 4 at 3 PM: 50% → slight settings page traffic uptick (users exploring), expected. Day 5: 100% → 1.2M users enabled, blog post and in-app notification published.

D14

Cleanup — flag removed, Blue decommissioned

The feature flag wrapper is deleted from the codebase — the component is now always rendered. PR opened, CI passes, deployed without incident. Blue environment decommissioned after 11 days with zero rollback events. JIRA-456 closed. Dark mode adoption: 31% of users enabled it within the first week.

// deploy on Tuesday, never Friday No production deploys within 48 hours of a weekend. If a Friday 4 PM deploy causes a problem, your on-call engineer is debugging over the weekend with limited team support. Tuesday–Thursday mid-morning is the optimal window: full team available, metrics monitored for a full business day, and time to rollback before end of day if needed. This is a cultural rule, not a technical one — but it saves more engineer weekends than any amount of testing.
