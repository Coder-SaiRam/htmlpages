Block Storage

Raw storage volumes attached to compute — the foundation of every database and operating system.

🧱

Block Storage — AWS EBS, SAN, NVMe

Fixed-size block devices presented as raw disks to the OS

Block storage presents storage as a sequence of fixed-size blocks (typically 512 bytes to 4KB). The OS formats the block device with a filesystem (ext4, XFS, NTFS) and manages file placement. The storage system has no concept of files — it just stores and retrieves raw blocks. This makes block storage the fastest possible storage layer, and the right choice whenever a database or OS needs direct disk control.

AWS EBS gp3 AWS EBS io2 Google Persistent Disk Azure Managed Disks NVMe SSD SAN (iSCSI/FC)
⚡

AWS EBS gp3

General purpose SSD. 3,000 IOPS baseline, up to 16,000 IOPS. 1GB–16TB. Detachable and re-attachable. Only one EC2 at a time (except Multi-Attach io2).

$0.08/GB-month
🔥

EBS io2 Block Express

Provisioned IOPS for mission-critical workloads. Up to 256,000 IOPS, 4,000 MB/s throughput. Used for Oracle RAC, SAP HANA, high-transaction MySQL.

$0.125/GB-month
💎

NVMe SSD (local)

Instance-attached NVMe. Lowest latency possible (<100μs). Ephemeral — data lost on stop/terminate. Used for temp storage, caches, databases with own replication.

Included with instance
🏢

SAN (iSCSI / Fibre Channel)

Enterprise shared storage fabric. Multiple hosts access the same LUNs. Used in data centres for Oracle RAC, VMware vSAN. High cost, high reliability.

On-premise

Block storage pros

  • Lowest latency of any storage type
  • Full POSIX filesystem control
  • Best for transactional databases
  • Fine-grained IOPS provisioning
  • Snapshots for backups

Block storage cons

  • Tied to one server (usually)
  • Not horizontally scalable by default
  • More expensive than object storage
  • Requires capacity planning upfront
  • No built-in geo-replication

Use block storage when

  • Running a relational database
  • Need <1ms disk latency
  • OS needs to format the volume
  • Running VMs (VM images on EBS)
  • High-frequency trading, OLTP
// interview key numbers EBS gp3 baseline: 3,000 IOPS / 125 MB/s — tunable to 16,000 IOPS / 1,000 MB/s independently of size (unlike gp2 where IOPS scaled with GB). NVMe local SSD: ~500K–2M IOPS. A PostgreSQL instance on io2 can handle ~30,000 TPS. A 1TB EBS gp3 volume costs ~$80/month.

File Storage

Shared file systems accessible by multiple servers simultaneously — the NFS model at cloud scale.

📁

File Storage — AWS EFS, NFS, GlusterFS

POSIX-compliant hierarchical file storage mounted by multiple compute nodes simultaneously

File storage exposes a shared filesystem over a network. Multiple servers mount the same filesystem simultaneously and see the same files. This is fundamentally different from block storage (which typically attaches to one server). File storage is the right choice when multiple compute nodes need to share files — shared configuration, model weights, user-uploaded content accessible by all app servers.

File storage pros

  • Multiple servers share same filesystem
  • Standard POSIX APIs (no code change)
  • Auto-scales capacity (EFS is elastic)
  • No capacity planning needed
  • Ideal for shared ML model weights

File storage cons

  • 3–10× more expensive than object storage
  • Higher latency than block storage
  • Not suited for small random I/O
  • Locking semantics are complex in clusters

File vs Object tradeoff

  • File: need POSIX APIs, mount as disk
  • Object: need HTTP, arbitrary scale
  • File: shared across servers easily
  • Object: versioning built in, cheaper
  • File: for ML training data during job
// real-world use at scale Netflix uses AWS EFS to store and serve their encoding job scripts across hundreds of Lambda functions simultaneously. When a new movie file is uploaded, a fleet of Lambda workers all mount the same EFS volume to access shared FFmpeg binaries and encoding profiles. Without shared file storage, each Lambda would need to download these assets independently — wasting seconds and bandwidth on every invocation.

Object Storage & Amazon S3

The most versatile storage system ever built — unlimited scale, 11 nines durability, and $0.023/GB.

🪣

Amazon S3 — In Complete Depth

S3 is not just a file store — it's a globally distributed, infinitely scalable object database with versioning, lifecycle policies, and event triggers

S3 stores data as objects — each object has a key (path-like string), a value (binary blob up to 5TB), and metadata (key-value pairs). Objects live in buckets. Unlike a filesystem, there is no true directory hierarchy — only key prefixes that look like paths. S3 is accessed entirely via HTTP REST API, making it accessible from anywhere with no special software.

📌

Versioning

Keep all versions of every object. Enables undelete, recovery from accidental overwrites. Each version has its own key + version ID. Cost: N× storage for N versions.

🔐

Pre-signed URLs

Generate time-limited URLs that grant temporary access to private objects. Upload or download directly from client browser — bypasses your servers entirely. Critical for media uploads.

⚡

S3 Event Notifications

Trigger Lambda, SQS, or SNS on object events (PUT, DELETE, etc.). Foundation of serverless data pipelines: upload image → trigger Lambda → resize → store back to S3.

🌍

S3 Cross-Region Replication

Automatically replicate objects to a bucket in another region. Used for disaster recovery, global distribution of assets, compliance (data residency requirements).

🔒

S3 Object Lock

WORM (Write Once Read Many) compliance. Objects cannot be deleted or overwritten for a fixed retention period. Required for SEC Rule 17a-4, FINRA, HIPAA compliance.

📡

S3 Select / Athena

Query S3 objects with SQL without loading them entirely. S3 Select filters within a single object (CSV/JSON/Parquet). Athena queries entire data lakes. Pay per query TB scanned.

1

Static website hosting

Host entire React/Next.js SPA from S3 + CloudFront. Zero server cost. Enable S3 static website hosting → configure index.html + error.html → point CloudFront distribution at S3 origin.

2

Data lake / analytics

Store petabytes of raw data in Parquet or ORC format in S3. Query with Athena ($5/TB scanned) or Spark (EMR). No database required — S3 IS the data lake. Cost: $23/TB-month vs $1000+/TB for a data warehouse.

3

Media pipeline (the YouTube pattern)

Client → pre-signed URL → direct upload to S3 → S3 Event → Lambda/SQS → transcoding workers → processed files stored in S3 → CloudFront serves to users. App servers never touch media bytes.

4

Backup & disaster recovery

Automated snapshots, DB backups, log archives stored in S3. Lifecycle policy transitions to Glacier after 30 days. Cross-region replication for DR. 11 nines durability means you've effectively never lost a backup.

5

Distributed artifact store

Docker image layers (ECR uses S3 under the hood), npm packages, Python wheels, Terraform state files, Kubernetes etcd backups — S3 is the universal artifact store for DevOps pipelines.

Best practice: enable S3 Block Public Access at the account level, use bucket policies that deny non-TLS requests, enable CloudTrail logging on all S3 data events, and use AWS Config to continuously audit for public buckets.

Object storage strengths

  • Virtually unlimited scale (no capacity planning)
  • 11 nines durability (3× replication across AZs)
  • Built-in versioning, lifecycle, replication
  • Cheapest persistent storage ($0.023/GB)
  • HTTP API accessible from anywhere
  • Serverless — no infrastructure to manage

Object storage limitations

  • High latency (100ms+ first byte vs <1ms disk)
  • No in-place updates (must overwrite object)
  • No POSIX filesystem semantics
  • Not suitable for frequently updated data
  • Eventual consistency (now strong in S3)
  • Egress costs add up at high bandwidth

Distributed File Systems

HDFS, Ceph, GlusterFS — the infrastructure layer that powers big data processing at exabyte scale.

🕸

HDFS, Ceph, GlusterFS — How They Work

Commodity hardware scaled to exabytes through replication and distributed metadata

Distributed file systems split large files into blocks (HDFS default: 128MB), replicate each block across multiple nodes (default: 3×), and maintain a metadata server (NameNode in HDFS) that tracks which nodes hold which blocks. Reading a 1TB file means reading 8,000 blocks in parallel from hundreds of DataNodes — achieving aggregate throughput impossible from a single disk.

🐘

HDFS (Hadoop)

The original distributed FS. 128MB blocks, 3× replication, central NameNode. Write-once-read-many. Deployed at Yahoo (200PB+), Facebook (hundreds of PB), LinkedIn. S3 is replacing it for new builds but existing deployments are massive.

Big data workloads
🦑

Ceph

Swiss-army knife: object + block + file storage in one system. No single point of failure (unlike HDFS NameNode). Used by CERN (200PB), OpenStack clouds. Ceph RBD powers OpenStack Cinder block storage.

Multi-protocol
🧊

GlusterFS

Peer-to-peer distributed FS. No central metadata server (like HDFS NameNode). Files distributed using DHT (Distributed Hash Table). Used for Kubernetes PersistentVolumes in on-premise clusters. Red Hat owns it.

Kubernetes friendly
☁️

Apache Ozone

HDFS successor designed for the object storage era. S3-compatible API, scales to billions of objects (HDFS struggles with 500M+ files). Designed for cloud-native workloads. Used at Tencent at petabyte scale.

HDFS replacement
// HDFS vs S3 — the modern choice The industry trend is "S3 as the data lake, Spark/Trino for compute" replacing on-premise HDFS. S3 separates storage from compute (Databricks, EMR, Athena can all query the same S3 data independently) while HDFS co-locates compute and storage. New architectures almost always choose S3. HDFS remains dominant only in existing large on-premise deployments.

In-Memory Storage

When disk is too slow — storing data entirely in RAM for microsecond access.

⚡

Redis, Memcached, VoltDB, SAP HANA

Not just a cache — in-memory stores are first-class data systems

In-memory storage keeps the entire dataset in RAM. Reads and writes happen in microseconds — no disk I/O, no filesystem overhead. The tradeoff is cost (RAM is ~100× more expensive per GB than SSD) and volatility (RAM loses data on power loss). Modern systems address this with persistence mechanisms (Redis AOF/RDB) and clustering (Redis Cluster, Hazelcast) for both durability and scale.

In-memory strengths

  • Sub-millisecond access (100μs typical)
  • Rich data structures in Redis
  • Atomic operations (INCR, MULTI/EXEC)
  • Pub/sub messaging built-in
  • Persistence options available

In-memory constraints

  • RAM is 50–100× more expensive than SSD
  • Data set limited by total RAM
  • Volatility risk without persistence config
  • Not a primary database for large datasets

Time-Series Storage

Metrics, IoT sensor data, financial ticks — data that is fundamentally ordered by time requires purpose-built storage.

📈

InfluxDB, TimescaleDB, Prometheus, Apache Druid

Why general-purpose databases fail at time-series — and what purpose-built TSDB does differently

Time-series data has a unique write pattern: always appending new data, rarely updating the past. General-purpose databases struggle because every insert must maintain B-tree index ordering. Time-series databases exploit the temporal ordering to achieve 10–100× better write throughput and 90%+ compression through delta encoding and gorilla compression.

📊

InfluxDB

Purpose-built TSDB. Line protocol ingestion. TSM (Time-Structured Merge Tree) storage engine. 10M+ writes/sec. Used for infrastructure metrics, IoT telemetry. InfluxDB Cloud (managed) or OSS.

Pure time-series
⏱

TimescaleDB

PostgreSQL extension with automatic time-based partitioning (hypertables). Full SQL. Best of both worlds — TSDB performance + PostgreSQL ecosystem. 10–100× better than vanilla Postgres for time data.

PostgreSQL-based
🔥

Prometheus

Monitoring-focused TSDB. Pull-based metrics collection. PromQL query language. 15-day default retention. Used with Grafana for infrastructure dashboards. Not designed for long-term storage (use Thanos/Cortex for that).

Monitoring/alerting
🐉

Apache Druid

Real-time OLAP. Sub-second queries on billions of rows of time-series. Used for user behaviour analytics (clicks, events). Powers Airbnb analytics, Netflix dashboards. Pre-aggregates at ingestion time.

Real-time analytics
// why TSDBs compress so well A CPU metric stored naively: {"time": 1703001234, "cpu": 73.2} = ~30 bytes. In InfluxDB with gorilla compression: the timestamp delta (say, 10 seconds from last) is stored as a varint (~2 bytes) and the float64 XOR delta (~3 bits for small changes) — achieving 10–15 bytes per data point on average. A week of per-second CPU metrics shrinks from 2.5GB to under 200MB.

Search & Analytics Storage

Elasticsearch, Solr, Meilisearch, OpenSearch — inverted indexes and the power of full-text search at scale.

🔍

Search Engines & Vector Databases

The inverted index is the most important data structure in modern information retrieval

A search engine stores an inverted index — mapping every word to every document containing it. "Alice" → [doc1, doc7, doc342]. A full-text query looks up the word in the index and gets document IDs instantly, rather than scanning every document. Elasticsearch stores this index in Lucene segments on disk, with recent data in RAM for near-real-time search.
