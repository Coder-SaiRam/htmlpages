FUNDAMENTALS
What is Recursion?
Understanding the core concept and components
🪆 REAL-WORLD ANALOGY: RUSSIAN NESTING DOLLS

Imagine opening a set of Russian nesting dolls (Matryoshka dolls). When you open one doll, you find another smaller doll inside. You keep opening dolls until you reach the tiniest doll that doesn't open anymore.

This is exactly how recursion works:

  • Opening a doll = Making a recursive function call
  • Finding a smaller doll = Getting a smaller subproblem
  • The tiniest doll = Base case (stops the recursion)
  • Closing dolls back up = Functions returning values back up the call stack

Just like you can't keep opening dolls forever, recursion must have a stopping point. And just like you close each doll after handling the smaller ones, each function call completes after its recursive calls return.

RECURSION DEFINITION
  • Recursion: A function that calls itself to solve a smaller version of the same problem
  • Base Case: The terminating condition that stops recursion
  • Recursive Case: The part where function calls itself with modified parameters
  • Call Stack: Memory structure that tracks function calls
📖 THEORY: WHY RECURSION EXISTS

The Fundamental Principle: Many problems have a natural self-similar structure. Solving the big problem requires solving smaller versions of the same problem.

Mathematical Foundation: Recursion is based on mathematical induction:

  • Base case = Prove the statement holds for n=0 or n=1
  • Recursive case = Prove if it holds for n-1, it holds for n
  • Conclusion = The statement holds for all n

When Problems Are Naturally Recursive:

  • Tree and graph traversals (each node has subtrees)
  • Divide and conquer algorithms (split into smaller pieces)
  • Combinatorial problems (make a choice, recurse on remaining)
  • Mathematical sequences (Fibonacci, factorial)
🧠 BEGINNER'S MENTAL MODEL

Step 1: Trust the Magic

When starting with recursion, the biggest mental hurdle is "trusting" that the recursive call will work. Don't try to trace every single call in your head - that's impossible for large inputs!

Step 2: The Leap of Faith

  • Assume the function works correctly for smaller inputs
  • Focus ONLY on: "If I had the solution for n-1, how would I build the solution for n?"
  • Don't mentally trace the entire recursion tree - trust the process

Step 3: Think in Layers

  • Layer 1: What's the simplest possible input? (Base case)
  • Layer 2: If I solve a smaller problem, can I use that solution?
  • Layer 3: How do I combine the smaller solution to get my answer?

// Classic Example: Factorial
public int factorial(int n) {
    // BASE CASE: Stop condition
    if (n == 0 || n == 1) {
        return 1;
    }

    // RECURSIVE CASE: Break down the problem
    return n * factorial(n - 1);
}

🔬 DEEP DIVE: CALL STACK MECHANICS

What Happens in Memory:

Every time a function is called, the system creates a stack frame containing:

  • Local variables: All variables declared in the function
  • Parameters: Values passed to the function
  • Return address: Where to go back after function completes
  • Return value space: Where to store the result

Stack Growth Direction: The stack grows "downward" (toward lower memory addresses) as new functions are called.

Memory Limit: Each thread has a limited stack size (typically 1-2 MB). Too many recursive calls = StackOverflowError!

💡 AHA MOMENTS TO LOOK FOR
  • Aha #1: The Stack Isn't Magic - It's just a data structure storing function calls. Each call creates a new frame with its own variables.
  • Aha #2: Functions Pause, They Don't Disappear - When factorial(4) calls factorial(3), factorial(4) doesn't disappear - it pauses and waits.
  • Aha #3: Unwinding Is Automatic - You don't need to explicitly "unwind" the stack. As functions return, the stack naturally pops frames.
  • Aha #4: Base Case Is The Anchor - Without it, functions keep pausing forever, waiting for answers that never come.
  • Aha #5: Space = Time - Recursive depth directly correlates to space complexity. n calls = O(n) space.
🏢 ANOTHER ANALOGY: ORGANIZATIONAL HIERARCHY

Think of a manager delegating work to employees:

  • CEO (initial call): "I need to know the total company headcount"
  • Division Head (recursive call): "I'll ask each department head and sum the results"
  • Department Manager (recursive call): "I'll ask each team lead and sum the results"
  • Team Lead (recursive call): "I'll count my team members directly"
  • Individual Employee (base case): "Count me as 1 person"

The CEO doesn't personally visit every employee. Each manager delegates to the level below, and answers bubble back up. This is recursion in organizational form!

KEY COMPONENTS OF RECURSION
  • Base Case: MUST have at least one to prevent infinite recursion
  • Progress Toward Base: Each recursive call must move closer to base case
  • Trust the Recursion: Assume the recursive call works for smaller inputs
  • Combine Results: Use the result from recursive call to build solution
⚠️ COMMON CONFUSION POINTS FOR BEGINNERS
  • Confusion #1: "How do I trace ALL the calls?" - DON'T! You can't hold 1000 function calls in your head. Focus on one level: base case + one recursive step.
  • Confusion #2: "Where does the return value go?" - It goes back to whoever called the function. Think of it like returning a borrowed book to the person who lent it to you.
  • Confusion #3: "Why doesn't my variable change?" - Each function call has its OWN copy of local variables. Changing n in factorial(3) doesn't affect n in factorial(4).
  • Confusion #4: "When should I use recursion vs iteration?" - Use recursion when the problem has natural substructure (trees, backtracking). Use iteration when you need better space efficiency.
  • Confusion #5: "What's the difference between direct and indirect recursion?" - Direct: func() calls func(). Indirect: func() calls helper() which calls func().

CORE CONCEPTS
The Three Rules of Recursion
Essential principles every recursive function must follow
🛗 ANALOGY: THE ELEVATOR

Think of recursion like taking an elevator down multiple floors, then coming back up:

  • Base Case = Ground Floor: The elevator must eventually reach ground floor (otherwise you'd go into the basement forever!)
  • Moving Toward Base = Going Down Floors: Each floor you descend (n-1, n-2) brings you closer to ground floor
  • Calling Itself = Pressing the Down Button: At each floor, you press the button to go down one more floor
  • Returning = Coming Back Up: Once you hit ground floor, the elevator returns back up, floor by floor

If the elevator never reached ground floor, it would keep going down forever (stack overflow!). The base case is your safety net.

Rule Description Example
1. Base Case Must have a stopping condition if (n == 0) return 1;
2. Move Toward Base Each call must progress toward the base case factorial(n-1) reduces n each time
3. Call Itself Function must call itself recursively return n * factorial(n-1);
📖 DEEP DIVE: WHY THESE THREE RULES?

Rule 1 - Base Case (The Safety Net):

  • Without it: Infinite recursion → Stack overflow → Program crash
  • Think of it as the "simplest case I can solve directly without recursion"
  • Common bases: empty array, zero, single element, null node
  • Multiple bases are okay! Example: fibonacci has TWO base cases (n=0 and n=1)

Rule 2 - Progress Toward Base (The Guarantee of Termination):

  • Each recursive call must get "closer" to the base case
  • For numbers: decrease (n-1) or divide (n/2)
  • For arrays: reduce size (arr[1:] or left half)
  • For strings: remove characters (s[1:] or s[:-1])
  • Violation example: calling func(n) instead of func(n-1) → infinite loop

Rule 3 - Call Itself (The Essence of Recursion):

  • This is what makes it recursive (vs just a regular function)
  • The function delegates work to a "smaller version of itself"
  • Can be direct (func calls func) or indirect (func calls helper, helper calls func)
ThreeRulesExamples.java
java
// EXAMPLE 1: Sum of array
public int sum(int[] arr, int index) {
    // RULE 1: Base case - reached end of array
    if (index >= arr.length) return 0;

    // RULE 2: Progress - index increases (moves toward arr.length)
    // RULE 3: Calls itself
    return arr[index] + sum(arr, index + 1);
}

// EXAMPLE 2: Fibonacci
public int fibonacci(int n) {
    // RULE 1: Base cases - TWO base cases here!
    if (n == 0) return 0;
    if (n == 1) return 1;

    // RULE 2: Progress - n decreases by 1 and 2
    // RULE 3: Calls itself TWICE (binary recursion)
    return fibonacci(n - 1) + fibonacci(n - 2);
}

// EXAMPLE 3: Reverse string
public String reverse(String s) {
    // RULE 1: Base case - string length 0 or 1
    if (s.length() <= 1) return s;

    // RULE 2: Progress - string gets shorter (remove first char)
    // RULE 3: Calls itself
    return reverse(s.substring(1)) + s.charAt(0);
    // Last char + reverse of rest
}

// EXAMPLE 4: Power function
public int power(int base, int exp) {
    // RULE 1: Base case - anything to power 0 is 1
    if (exp == 0) return 1;

    // RULE 2: Progress - exponent decreases
    // RULE 3: Calls itself
    return base * power(base, exp - 1);
    // base^exp = base * base^(exp-1)
}
COMMON PITFALLS TO AVOID
  • Missing Base Case: Causes StackOverflowError (infinite recursion)
  • Wrong Base Case: May never be reached or return incorrect result
  • Not Progressing: Calling with same parameter (e.g., func(n) instead of func(n-1))
  • Multiple Base Cases: Don't forget edge cases (e.g., negative numbers)
  • Excessive Recursion: Some problems are better solved iteratively
🎤 INTERVIEW STRATEGY: COMMUNICATING RECURSION
  • Step 1: State the Approach - "I'll solve this recursively by breaking it into smaller subproblems"
  • Step 2: Identify Base Case - "The base case is when [simplest scenario]. In that case, we return [value]"
  • Step 3: Explain Recursive Case - "For the recursive case, we [do something], then call ourselves with [smaller input]"
  • Step 4: Show Combine Logic - "We combine the result by [operation] with the current value"
  • Step 5: Trace Small Example - Walk through n=3 or n=4, not n=10
  • Step 6: Analyze Complexity - "Time: O(?) because [reason]. Space: O(?) for call stack"
  • Step 7: Discuss Alternatives - "We could also solve this iteratively, but recursion is cleaner for [reason]"
✅ EDGE CASES TO ALWAYS CHECK
  • Empty Input: null array, empty string, 0 value
  • Single Element: Often a base case (array with 1 item, n=1)
  • Negative Numbers: Does your base case handle n < 0?
  • Large Input: Will you hit stack overflow? (n > 10000)
  • Duplicate Elements: Does your algorithm handle duplicates correctly?
  • Special Values: 0, -1, Integer.MAX_VALUE, null
RecursionMistakes.java
java
// ❌ WRONG: Missing base case
public int badFactorial(int n) {
    return n * badFactorial(n - 1); // Infinite recursion!
}

// ❌ WRONG: Not progressing toward base case
public int badFactorial2(int n) {
    if (n == 0) return 1;
    return n * badFactorial2(n); // Same n! Infinite loop!
}

// ✅ CORRECT: Has base case and progresses
public int goodFactorial(int n) {
    if (n <= 1) return 1; // Base case (handles 0 and 1)
    return n * goodFactorial(n - 1); // Progresses toward base
}

📖 UNDERSTANDING RECURSION TYPES

Different recursion types aren't just academic classifications - they have real implications for performance, stack usage, and problem-solving approaches. Recognizing which type fits your problem helps you write more efficient code.

Key Insight: The number of recursive calls determines the time complexity growth rate. More calls = exponential growth potential.

1. Linear Recursion (Single Recursive Call)

🚂 ANALOGY: TRAIN CARS

Think of linear recursion like a train where each car is connected to exactly ONE car behind it. You walk from the front car to the last car (going down recursion), then walk back (returning values).

LinearRecursion.java
java
// Linear Recursion: One recursive call per execution
public int sum(int n) {
    if (n == 0) return 0;
    return n + sum(n - 1); // Single recursive call
}

// Time: O(n), Space: O(n) for call stack
🎯 WHEN TO RECOGNIZE LINEAR RECURSION
  • Problem involves sequential processing (one item at a time)
  • Only ONE direction to explore at each step
  • Keywords: "sum of", "count", "reverse", "traverse linked list"
  • Examples: Array traversal, linked list operations, factorial, GCD
🌍 REAL-WORLD USE CASES
  • File Path Traversal: Following a single path /home/user/documents/file.txt
  • Undo/Redo Operations: Going back one step at a time
  • Linked List Processing: Visiting each node sequentially

2. Binary Recursion (Two Recursive Calls)

🌳 ANALOGY: FAMILY TREE

Binary recursion is like exploring a family tree where each person has exactly two parents. To count your ancestors, you count your mother's ancestors + your father's ancestors + 2 (your parents).

BinaryRecursion.java
java
// Binary Recursion: Two recursive calls per execution
public int fibonacci(int n) {
    if (n <= 1) return n;
    return fibonacci(n - 1) + fibonacci(n - 2); // Two recursive calls
}

// Time: O(2^n) - Exponential!, Space: O(n) for deepest call
⚠️ PERFORMANCE WARNING
  • Binary recursion without memoization is EXPONENTIALLY SLOW
  • fibonacci(40) makes over 1 BILLION function calls!
  • ALWAYS consider memoization for binary recursion
  • Or convert to iterative approach for better performance

  • Binary Search: Split search space in half each time
  • Merge Sort: Split array into two halves, sort each, merge
  • Tree Algorithms: Process left subtree and right subtree
  • Decision Trees: Each decision has two branches (yes/no)

3. Multiple Recursion (Many Recursive Calls)

🛣️ ANALOGY: MAZE WITH MANY PATHS

Multiple recursion is like standing at an intersection with many roads. You need to explore EACH road to find all possible paths to the exit. This is the essence of backtracking.

MultipleRecursion.java
java
// Multiple Recursion: Many recursive calls (e.g., in loops)
public void printPermutations(String prefix, String remaining) {
    if (remaining.length() == 0) {
        System.out.println(prefix);
        return;
    }

    for (int i = 0; i < remaining.length(); i++) {
        String newPrefix = prefix + remaining.charAt(i);
        String newRemaining = remaining.substring(0, i) +
                             remaining.substring(i + 1);
        printPermutations(newPrefix, newRemaining); // Multiple calls in loop
    }
}

// Time: O(n! * n) - Factorial!, Space: O(n) for recursion depth
🎯 WHEN TO RECOGNIZE MULTIPLE RECURSION
  • Problem requires exploring ALL possible choices
  • Recursive calls happen inside a LOOP
  • Keywords: "all permutations", "all combinations", "all subsets", "find all solutions"
  • Examples: Permutations, N-Queens, Sudoku solver, word search
🌍 REAL-WORLD USE CASES
  • Puzzle Solvers: Sudoku, crossword, Rubik's cube
  • Game AI: Chess move generation, tic-tac-toe
  • Scheduling: Finding all valid meeting time combinations
  • Path Finding: Finding all paths from A to B in a graph

4. Tail Recursion (Optimizable)

🎯 ANALOGY: RELAY RACE

Tail recursion is like a relay race where the runner doesn't wait around after passing the baton. They're done! The current runner has ALL the information needed - no need to come back.

TailRecursion.java
java
// Tail Recursion: Recursive call is the last operation
public int factorialTail(int n, int accumulator) {
    if (n == 0) return accumulator;
    return factorialTail(n - 1, n * accumulator); // Last operation
}

// Can be optimized by compiler to iterative form (not in Java though)
// Time: O(n), Space: O(1) with tail call optimization
📖 WHY TAIL RECURSION IS SPECIAL

The Key Difference: In tail recursion, there's NO WORK left to do after the recursive call returns. The function can immediately pass the result up.

Compiler Optimization: Smart compilers can convert tail recursion to a loop, eliminating stack frames:

  • Instead of creating new stack frame → Reuse the current frame
  • Space complexity drops from O(n) to O(1)
  • Performance equivalent to iterative solution

Languages with Tail Call Optimization (TCO): Scheme, Haskell, Scala, Kotlin (sometimes)

Languages WITHOUT TCO: Java, Python, JavaScript (mostly)

TailVsNonTail.java
java
// NON-TAIL RECURSION (has work after recursive call)
public int factorial(int n) {
    if (n == 0) return 1;
    int result = factorial(n - 1); // Recursive call
    return n * result;              // WORK AFTER CALL! (multiplication)
}

// TAIL RECURSION (no work after recursive call)
public int factorialTail(int n, int acc) {
    if (n == 0) return acc;
    return factorialTail(n - 1, n * acc); // NO WORK AFTER! Just return result
}

// Call: factorialTail(5, 1)
// → factorialTail(4, 5)
// → factorialTail(3, 20)
// → factorialTail(2, 60)
// → factorialTail(1, 120)
// → factorialTail(0, 120)
// → return 120
🎯 WHEN TO USE TAIL RECURSION
  • You need recursion but want better space efficiency
  • Use an accumulator parameter to carry results forward
  • Common in functional programming languages
  • Useful when: calculating running totals, traversing large structures
📊 RECURSION TYPES COMPARISON
Type Calls Per Level Time Complexity Space Complexity Best For
Linear 1 O(n) O(n) Sequential processing, linked lists
Binary 2 O(2^n) without memo O(n) Divide & conquer, tree problems
Multiple n (variable) O(n!) or O(n^n) O(n) Backtracking, finding all solutions

CHOOSING THE RIGHT TYPE
  • Linear: Simple problems with sequential reduction (sum, reverse)
  • Binary: Problems with two subproblems (fibonacci, binary search tree)
  • Multiple: Problems with many choices (permutations, combinations)
  • Tail: When you want potential optimization (accumulators pattern)

🎨 REAL-WORLD ANALOGY: ASSEMBLY LINE PATTERNS

Imagine different types of assembly lines in a factory:

  • Process-Then-Pass (Process & Recurse): Worker stamps a part, then passes it down. Each worker processes before passing along. Like reading a book page-by-page from start to finish.
  • Pass-Then-Process (Recurse & Process): Pass the part all the way to the end, then each worker processes on the way back. Like building a tower of blocks - you place the bottom one last!
  • Split-and-Merge (Divide & Conquer): Split work into two lines, process both separately, then merge at the end. Like sorting cards by dividing the pile in half repeatedly.
  • Coordinator-Worker (Helper Method): A coordinator manages the workflow while workers do the actual processing. Like a manager delegating tasks to a team.

Understanding these patterns is like learning the basic moves in chess - master them, and you can solve most problems!

📖 THEORY: THE FOUR FOUNDATIONAL PATTERNS

Why Only Four Patterns? Every recursive problem boils down to a few key questions:

  • When do I process? Before recursion (going down) or after (coming back up)?
  • How many recursive calls? One (linear) or multiple (divide & conquer)?
  • What state do I track? None, or need additional parameters?

Pattern Selection Logic:

  • Process order matters? → Choose between Process-and-Recurse vs Recurse-and-Process
  • Can I split the problem? → Use Divide and Conquer
  • Need extra state? → Use Helper Method Pattern
🎯 PATTERN SELECTION MATRIX
Pattern When to Use Characteristics Examples
Process & Recurse Process current, then handle rest Forward processing, build result going down Print array, count nodes, prefix sum
Recurse & Process Need result from below first Backward processing, build result going up Reverse operations, factorial, postorder

🧠 MENTAL MODEL: THE RECURSION TIMELINE

Visualize recursion in two phases:

Phase 1: Going Down (The Journey Out)

  • Each function call moves "deeper" into the problem
  • Parameters get smaller (n-1, index+1, left/right halves)
  • Stack grows (functions waiting to complete)
  • "Process & Recurse" does work here

Phase 2: Coming Back Up (The Return Journey)

  • Base case reached, now returning values
  • Each function completes and returns to its caller
  • Stack shrinks (functions finishing)
  • "Recurse & Process" does work here

Key Insight: The recursion stack is like a bookmark in a book - it remembers where you were so you can come back and continue!

Pattern 1: Process and Recurse

WHEN TO USE
  • Need to perform an action at each step
  • Building result as you go down the recursion
  • Examples: Print elements, count nodes, calculate sum
ProcessAndRecurse.java
java
// Pattern: Process current, then recurse
public void printArray(int[] arr, int index) {
    // Base case
    if (index >= arr.length) return;

    // Process: Print current element
    System.out.print(arr[index] + " ");

    // Recurse: Move to next
    printArray(arr, index + 1);
}

// Count elements
public int countElements(int[] arr, int index) {
    if (index >= arr.length) return 0;
    return 1 + countElements(arr, index + 1);
}
💡 INTUITION: WHY PROCESS FIRST?

The Forward Pass Pattern: Use this when the order is "first to last" - you want to handle elements in the order you encounter them.

Think of it like: Reading a book page by page. You read page 1, then move to page 2, then page 3. Each page is "processed" (read) before moving forward.

Key characteristic: The work happens BEFORE the recursive call. The call stack builds up, but the important work is done on the way down.

  • Wrong base case: Using index == arr.length is fine, but index >= arr.length is safer (handles edge cases)

  • Modifying parameters: Don't do index++ - pass index+1 to keep function pure
  • Off-by-one errors: Trace with small inputs (n=1, n=2) to verify your indices

Pattern 2: Recurse and Process

WHEN TO USE
  • Need result from recursive call before processing
  • Building result on the way back up
  • Examples: Reverse operations, bottom-up calculations
RecurseAndProcess.java
java
// Pattern: Recurse first, then process
public void printReverse(int[] arr, int index) {
    // Base case
    if (index >= arr.length) return;

    // Recurse first: Go to end
    printReverse(arr, index + 1);

    // Process: Print on way back (reverse order)
    System.out.print(arr[index] + " ");
}

// Calculate factorial (need result from below)
public int factorial(int n) {
    if (n <= 1) return 1;
    int result = factorial(n - 1); // Recurse first
    return n * result; // Then process
}
💡 INTUITION: WHY RECURSE FIRST?

The Backward Pass Pattern: Use this when you need information from "ahead" or want to process in reverse order.

Think of it like: Stacking plates. You put plate 1 down, then plate 2 on top, then plate 3. When removing them, you must take plate 3 first, then 2, then 1 - the reverse of how you placed them!

Key characteristic: Recursion happens FIRST. Processing happens AFTER returning from recursion. The actual work happens as the call stack unwinds.

  • Cleanup/Teardown: Initialize in order, cleanup in reverse
  • Reverse Polish Notation: Process operators after operands

Pattern 3: Divide and Conquer

WHEN TO USE
  • Split problem into smaller subproblems
  • Solve each subproblem recursively
  • Combine results from subproblems
  • Examples: Merge sort, binary search, quick sort
DivideAndConquer.java
java
// Pattern: Divide, Conquer, Combine
public int binarySearch(int[] arr, int target, int left, int right) {
    // Base case
    if (left > right) return -1;

    // Divide: Split problem in half
    int mid = left + (right - left) / 2;

    if (arr[mid] == target) return mid;

    // Conquer: Recursively search one half
    if (arr[mid] > target) {
        return binarySearch(arr, target, left, mid - 1);
    } else {
        return binarySearch(arr, target, mid + 1, right);
    }
}

// Find maximum in array
public int findMax(int[] arr, int left, int right) {
    if (left == right) return arr[left];

    int mid = (left + right) / 2;
    int leftMax = findMax(arr, left, mid);
    int rightMax = findMax(arr, mid + 1, right);

    // Combine: Take maximum of both halves
    return Math.max(leftMax, rightMax);
}
💡 INTUITION: THE POWER OF SPLITTING

The Divide-and-Conquer Mindset: "This problem is too big. Let me split it in half and solve each half separately."

Think of it like: Looking for a word in a dictionary. Instead of checking every page, you open to the middle. Is your word before or after? Now you only have to search half the dictionary! Repeat this splitting until you find your word.

Key characteristic: Multiple recursive calls (usually 2), then combine their results. Often achieves O(log n) time complexity!

Three distinct phases:

  • Divide: Split the problem into smaller subproblems
  • Conquer: Solve each subproblem recursively
  • Combine: Merge the solutions of subproblems

⚠️ COMMON MISTAKES: PATTERN 3
  • Integer overflow: Use mid = left + (right-left)/2 not (left+right)/2
  • Infinite recursion: Ensure mid is between left and right, and at least one element is excluded in recursive calls
  • Incorrect base case: For single element, return that element. For empty, return appropriate default
  • Forgetting to combine: After getting both halves' results, you MUST combine them (max, merge, etc.)
  • Off-by-one in bounds: Be careful with mid-1 and mid+1 to avoid infinite loops

  • Searching Algorithms: Binary search in sorted arrays/trees
  • Sorting Algorithms: Merge sort, quick sort
  • Computational Geometry: Closest pair of points
  • Image Processing: Divide image into quadrants for parallel processing
  • MapReduce: Divide data, process in parallel, combine results
  • Fast Fourier Transform: Divide signal into frequency components
✨ OPTIMIZATION TIP: DIVIDE & CONQUER
  • Tail Recursion: Binary search can be tail-recursive (only one branch explored)
  • Iterative Alternative: Binary search is often better implemented iteratively
  • Parallelization: Both halves can be computed in parallel (great for multicore)
  • Cache Efficiency: Splitting improves cache locality compared to linear scan

💡 INTUITION: THE COORDINATOR PATTERN

The Wrapper Approach: "I need to track extra information, but I don't want users to worry about it."

Think of it like: A restaurant's front-of-house vs. kitchen. Customers (users) see a simple menu and ordering process (public method). The kitchen (helper method) handles all the complex tracking, timing, and coordination behind the scenes.

Key characteristic: Two methods - one public (clean API) and one private (does the real work with extra parameters).

Common scenarios for helper methods:

  • Index tracking: User provides array, helper tracks current position
  • Accumulator: Build result across recursive calls
  • Path tracking: Remember the path taken through tree/graph
  • Level tracking: Count depth or level in tree
  • Parent reference: Keep track of parent nodes

// Example 1: Path Tracking in Tree
public List<List<Integer>> findAllPaths(TreeNode root) {
    List<List<Integer>> paths = new ArrayList<>();
    findPathsHelper(root, new ArrayList<>(), paths);
    return paths;
}

private void findPathsHelper(TreeNode node, List<Integer> current,
                                 List<List<Integer>> paths) {
    if (node == null) return;
    current.add(node.val);

    if (node.left == null && node.right == null) {
        paths.add(new ArrayList<>(current)); // Leaf: save path
    }

    findPathsHelper(node.left, current, paths);
    findPathsHelper(node.right, current, paths);
    current.remove(current.size() - 1); // Backtrack!
}

// Example 2: Count with Condition
public int countEvens(int[] arr) {
    return countEvensHelper(arr, 0, 0);
}

private int countEvensHelper(int[] arr, int index, int count) {
    if (index >= arr.length) return count;
    int newCount = arr[index] % 2 == 0 ? count + 1 : count;
    return countEvensHelper(arr, index + 1, newCount);
}

  • Exposing helper method: Keep helper private - users shouldn't see implementation details
  • Too many parameters: If helper needs >4 parameters, consider creating a class to hold state

🌲 ANALOGY: WALKING THROUGH A MAZE

Imagine you're in a maze trying to find all exits. At each intersection, you:

  • Choose a path - Pick one direction to explore
  • Walk down it - See where it leads (recursive exploration)
  • Hit a dead end or find exit - Record the solution or fail
  • Walk back to the intersection - This is "backtracking"!
  • Try the next path - Explore other options

You systematically try every path, backtracking whenever you hit a dead end. This is exactly how backtracking algorithms work!

📖 THEORY: WHY BACKTRACKING EXISTS

The Core Problem: Some problems require exploring ALL possible solutions. There's no clever shortcut - you must try every combination.

Naive Approach Problem: If you generated all possibilities upfront (like using nested loops), you'd need exponentially many loops. For n=10, you'd need 10 nested loops!

Backtracking Solution: Use recursion to simulate dynamic depth of loops. The recursion depth adapts to the problem size.

Key Insight: Backtracking is DFS (Depth-First Search) on a decision tree. Each level = one decision. Each branch = one choice.

🧠 MENTAL MODEL: THE DECISION TREE

Every backtracking problem can be visualized as a tree:

  • Root = Starting state (empty solution)
  • Nodes = Partial solutions (choices made so far)
  • Edges = Decisions/choices available
  • Leaves = Complete solutions or dead ends
  • DFS Traversal = How backtracking explores the tree

Backtracking = "Go deep until you find a solution or dead end, then come back up and try the next branch"

📌 UNIVERSAL BACKTRACKING TEMPLATE
  • Choose - Make a decision and add it to your current path
  • Explore - Recursively explore further with this choice
  • Unchoose - Remove the choice (backtrack) to try other options
BacktrackTemplate.java
java
public class BacktrackTemplate {
    public void backtrack(State state, List<Result> results) {
        // BASE CASE: Found a valid solution
        if (isValidSolution(state)) {
            results.add(new Result(state));
            return;
        }

        // ITERATE through all possible choices
        for (Choice choice : getPossibleChoices(state)) {
            // 1. CHOOSE: Make a choice
            state.choose(choice);

            // 2. EXPLORE: Recurse with this choice
            backtrack(state, results);

            // 3. UNCHOOSE: Backtrack (undo choice)
            state.unchoose(choice);
        }
    }

    // Example: Generate all permutations
    public List<List<Integer>> permute(int[] nums) {
        List<List<Integer>> results = new ArrayList<>();
        backtrackPermute(nums, new ArrayList<>(), new boolean[nums.length], results);
        return results;
    }

    private void backtrackPermute(int[] nums, List<Integer> current,
                                   boolean[] used, List<List<Integer>> results) {
        // Base case: permutation complete
        if (current.size() == nums.length) {
            results.add(new ArrayList<>(current));
            return;
        }

        for (int i = 0; i < nums.length; i++) {
            if (used[i]) continue;  // Skip used elements

            current.add(nums[i]);        // CHOOSE
            used[i] = true;
            backtrackPermute(nums, current, used, results);  // EXPLORE
            current.remove(current.size() - 1);  // UNCHOOSE
            used[i] = false;
        }
    }
}

👔 ANALOGY: TRYING ON OUTFITS

Imagine you have 3 shirts [A, B, C] and want to try on each in different orders:

  • Try A first, then you have [B, C] left to try
  • Try B next, then you have [C] left → outfit: A-B-C
  • Take off B and C (backtrack to A)
  • Try C next, then B left → outfit: A-C-B
  • Take off all (backtrack to start), try B first...

This is permutations! Order matters (A-B-C is different from B-A-C), and you use ALL items exactly once.

📖 PERMUTATIONS DEEP DIVE

Definition: A permutation is an arrangement of all elements where order matters.

Count: n! (factorial) permutations for n distinct elements

  • 3 elements: 3! = 6 permutations
  • 4 elements: 4! = 24 permutations
  • 10 elements: 10! = 3,628,800 permutations!

Why n!? For first position: n choices. Second: n-1 choices. Third: n-2 choices. So: n × (n-1) × (n-2) × ... × 1 = n!

Permutations.java
java
// Permutations: Order matters, use all elements
public List<List<Integer>> permute(int[] nums) {
    List<List<Integer>> result = new ArrayList<>();
    backtrack(nums, new ArrayList<>(), result);
    return result;
}

private void backtrack(int[] nums, List<Integer> current, List<List<Integer>> result) {
    // BASE CASE: All elements used (permutation complete)
    if (current.size() == nums.length) {
        result.add(new ArrayList<>(current)); // MUST copy! Not reference
        return;
    }

    // TRY each element that hasn't been used yet
    for (int num : nums) {
        if (current.contains(num)) continue;  // Skip if already used

        current.add(num);                     // CHOOSE: Add to current permutation
        backtrack(nums, current, result);     // EXPLORE: Recurse with this choice
        current.remove(current.size() - 1);   // UNCHOOSE: Remove for next iteration
    }
}

// TIME: O(n! × n) - n! permutations, O(n) to build each
// SPACE: O(n) for recursion depth (excluding output)

Handling Duplicates

🔍 WHY DUPLICATES ARE TRICKY

With duplicates like [1,1,2], naive approach generates: [1,1,2], [1,2,1], [1,1,2] (duplicate!), [1,2,1] (duplicate!), ...

The Problem: We treat identical elements as different, generating the same permutation multiple times.

The Solution: Sort first, then skip element if it's same as previous AND previous wasn't used. This ensures we only use duplicates in order.

PermutationsWithDuplicates.java
java
// With Duplicates: [1,1,2] → [[1,1,2], [1,2,1], [2,1,1]]
public List<List<Integer>> permuteUnique(int[] nums) {
    List<List<Integer>> result = new ArrayList<>();
    Arrays.sort(nums);  // MUST SORT to group duplicates together!
    backtrackUnique(nums, new ArrayList<>(), new boolean[nums.length], result);
    return result;
}

private void backtrackUnique(int[] nums, List<Integer> current,
                             boolean[] used, List<List<Integer>> result) {
    if (current.size() == nums.length) {
        result.add(new ArrayList<>(current));
        return;
    }

    for (int i = 0; i < nums.length; i++) {
        if (used[i]) continue;  // Already used in current permutation

        // KEY INSIGHT: Skip duplicate if previous same number not used
        // This ensures we only use duplicates in sorted order
        if (i > 0 && nums[i] == nums[i-1] && !used[i-1]) continue;

        current.add(nums[i]);
        used[i] = true;
        backtrackUnique(nums, current, used, result);
        current.remove(current.size() - 1);
        used[i] = false;
    }
}

// Valid: [1(index 0), 1(index 1), 2]
// Skip:  [1(index 1), 1(index 0), 2] - violates ordering!
// This prevents duplicate permutations

  • For duplicates: sort first, skip if nums[i] == nums[i-1] && !used[i-1]
  • MUST copy list when adding: new ArrayList<>(current) not current
  • Time complexity is O(n! × n): n! permutations, O(n) to copy each

private void backtrackCombine(int n, int k, int start,
                              List<Integer> current, List<List<Integer>> result) {
    if (current.size() == k) {  // Found k elements
        result.add(new ArrayList<>(current));
        return;
    }

    for (int i = start; i <= n; i++) {
        current.add(i);
        backtrackCombine(n, k, i + 1, current, result);  // Start from i+1
        current.remove(current.size() - 1);
    }
}

public List<List<Integer>> combinationSum(int[] candidates, int target) {
    List<List<Integer>> result = new ArrayList<>();
    Arrays.sort(candidates);
    backtrackComboSum(candidates, target, 0, new ArrayList<>(), result);
    return result;

private void backtrackComboSum(int[] candidates, int target, int start,
                               List<Integer> current, List<List<Integer>> result) {
    if (target == 0) {
        result.add(new ArrayList<>(current));
        return;
    }
    if (target < 0) return;  // Exceeded target

    for (int i = start; i < candidates.length; i++) {
        current.add(candidates[i]);
        backtrackComboSum(candidates, target - candidates[i], i, current, result);  // i not i+1 (reuse)
        current.remove(current.size() - 1);
    }
}

public List<List<String>> solveNQueens(int n) {
    List<List<String>> result = new ArrayList<>();
    char[][] board = new char[n][n];
    for (char[] row : board) Arrays.fill(row, '.');
    backtrack(board, 0, result);
    return result;
}

private void backtrack(char[][] board, int row, List<List<String>> result) {
    if (row == board.length) {  // Placed all queens
        result.add(construct(board));
        return;
    }

    for (int col = 0; col < board.length; col++) {
        if (!isValid(board, row, col)) continue;  // Check constraints

        board[row][col] = 'Q';              // Choose
        backtrack(board, row + 1, result);  // Explore next row
        board[row][col] = '.';              // Unchoose
    }
}

private boolean isValid(char[][] board, int row, int col) {
    // Check column
    for (int i = 0; i < row; i++) {
        if (board[i][col] == 'Q') return false;
    }

    // Check upper-left diagonal
    for (int i = row - 1, j = col - 1; i >= 0 && j >= 0; i--, j--) {
        if (board[i][j] == 'Q') return false;
    }

    // Check upper-right diagonal
    for (int i = row - 1, j = col + 1; i >= 0 && j < board.length; i--, j++) {
        if (board[i][j] == 'Q') return false;
    }

    return true;
}

private List<String> construct(char[][] board) {
    List<String> result = new ArrayList<>();
    for (char[] row : board) {
        result.add(new String(row));
    }
    return result;
}
💡 OPTIMIZATION
  • Use sets to track occupied columns and diagonals: Set<Integer> cols, diag1, diag2
  • Diagonal formula: upper-left = row - col, upper-right = row + col
  • This reduces validation from O(n) to O(1)

📖 THEORY: STRING RECURSION PATTERNS

Why Strings Are Special in Recursion:

  • Immutable vs Mutable: Java strings are immutable, so use StringBuilder for efficiency
  • Character-level Decisions: Each character presents a choice (include/exclude, position, case)
  • Order Matters: Unlike sets, character order is significant
  • Exponential Growth: n characters can form 2ⁿ subsequences, n! permutations

Three Core String Recursion Patterns:

  • Build by Adding: Start empty, add chars (permutations, combinations)
  • Build by Removing: Start with full string, remove chars
  • Binary Choice: At each position, include or exclude (subsequences)

Visualizing String Recursion:

Think of the string as a sequence of decision points:

  • At each character: You have choices (use it here, skip it, swap it, change case, etc.)
  • Each path: Leads to a different result string

  • Pruning: Skip branches that violate constraints (duplicates, invalid patterns)

// Method 1: Swap-based (in-place)
private void backtrack(char[] chars, int index, List<String> result) {
    if (index == chars.length) {
        result.add(new String(chars));
        return;
    }

    for (int i = index; i < chars.length; i++) {
        swap(chars, index, i);              // Choose
        backtrack(chars, index + 1, result); // Explore
        swap(chars, index, i);              // Unchoose (backtrack)
    }
}

// Method 2: StringBuilder-based (clearer)
private void backtrackWithSB(String s, StringBuilder current,
                             boolean[] used, List<String> result) {
    if (current.length() == s.length()) {
        result.add(current.toString());
        return;
    }

    for (int i = 0; i < s.length(); i++) {
        if (used[i]) continue;

        current.append(s.charAt(i));
        used[i] = true;
        backtrackWithSB(s, current, used, result);
        current.deleteCharAt(current.length() - 1);
        used[i] = false;
    }
}

    backtrackUnique(chars, new StringBuilder(), new boolean[chars.length], result);
    return result;
}

private void backtrackUnique(char[] chars, StringBuilder current,
                             boolean[] used, List<String> result) {
    if (current.length() == chars.length) {
        result.add(current.toString());
        return;
    }

    for (int i = 0; i < chars.length; i++) {
        if (used[i]) continue;
        // Skip duplicates: if current == previous and previous not used
        if (i > 0 && chars[i] == chars[i-1] && !used[i-1]) continue;

        current.append(chars[i]);
        used[i] = true;
        backtrackUnique(chars, current, used, result);
        current.deleteCharAt(current.length() - 1);
        used[i] = false;
    }

💡 STRING MANIPULATION TIPS
  • Use StringBuilder for building strings (mutable, efficient)
  • Swap-based approach is O(1) space but modifies input
  • Always sort first when handling duplicates
⚠️ COMMON MISTAKES: STRING PERMUTATIONS
  • String Immutability: Don't use String concatenation in loops - use StringBuilder!
  • Forgetting to Backtrack: After recursive call, must undo changes (remove char, swap back)
  • Not Handling Duplicates: "AAB" gives duplicate results without sorting + skip logic
  • Wrong Skip Condition: Must check !used[i-1] not used[i-1] for duplicates
  • StringBuilder Reuse: Each recursive path should not interfere with others
💡 INTUITION: SWAP VS BUILD APPROACH

Swap Approach: Fix positions left-to-right. At index i, try swapping with every position j >= i. More space-efficient (O(1) extra space).

Build Approach: Gradually build the string by appending available characters. More intuitive, easier to handle duplicates. Uses O(n) extra space for tracking.

When to use which?

  • Use Swap: When string has no duplicates, want minimal space
  • Use Build: When handling duplicates, need clearer logic, interviewer preference

A palindrome is like looking in a mirror - the word reads the same forwards and backwards!

Palindrome Partitioning is like cutting a sentence into words where each word is a mirror word:

  • "RACECAR" - Already a perfect mirror (one piece)
  • "AABAA" - Can split as "AA-B-AA" or "A-ABA-A" or even "AA-BAA" - multiple ways!

Each split point is a decision: "Should I cut here or keep going?"

💡 INTUITION: PALINDROME PARTITION LOGIC

The Two-Step Dance:

  • Step 1: Try all cut positions - For each starting position, try cutting at every possible end
  • Step 2: Validate the piece - Is substring[start...end] a palindrome?
  • Step 3: Recurse on remainder - If valid, partition the rest of the string

Why this works: We explore every possible way to partition, checking each piece for palindrome property.

// Partition string into all palindromic substrings
// "aab" → [["a","a","b"], ["aa","b"]]
public List<List<String>> partition(String s) {
    List<List<String>> result = new ArrayList<>();
    backtrack(s, 0, new ArrayList<>(), result);
    return result;
}

private void backtrack(String s, int start, List<String> current,
                      List<List<String>> result) {
    if (start == s.length()) {
        result.add(new ArrayList<>(current));
        return;
    }

    for (int end = start + 1; end <= s.length(); end++) {
        String substring = s.substring(start, end);
        if (isPalindrome(substring)) {
            current.add(substring);              // Choose
            backtrack(s, end, current, result);  // Explore
            current.remove(current.size() - 1);  // Unchoose
        }
    }
}

private boolean isPalindrome(String s) {
    int left = 0, right = s.length() - 1;
    while (left < right) {
        if (s.charAt(left++) != s.charAt(right--)) {
            return false;
        }
    }
    return true;
}

// OPTIMIZATION: Pre-compute palindrome table with DP
private boolean[][] computePalindromeTable(String s) {
    int n = s.length();
    boolean[][] dp = new boolean[n][n];

    // Single characters are palindromes
    for (int i = 0; i < n; i++) {
        dp[i][i] = true;
    }

    // Check for length 2
    for (int i = 0; i < n - 1; i++) {
        if (s.charAt(i) == s.charAt(i + 1)) {
            dp[i][i + 1] = true;
        }
    }

    // Check for lengths 3+
    for (int len = 3; len <= n; len++) {
        for (int i = 0; i <= n - len; i++) {
            int j = i + len - 1;
            if (s.charAt(i) == s.charAt(j) && dp[i + 1][j - 1]) {
                dp[i][j] = true;
            }
        }
    }

    return dp;
}
🎯 PALINDROME PATTERNS
  • Two Pointer: Expand from center or shrink from edges
  • DP Table: Pre-compute O(n²) for O(1) lookup
  • Recursion: isPalindrome(s) = (s[0] == s[n-1]) && isPalindrome(s[1..n-2])
⚠️ COMMON MISTAKES: PALINDROMES
  • Repeated Computation: Checking isPalindrome repeatedly - use DP table for O(1) lookups!
  • Off-by-one in substring: substring(start, end) is exclusive of end, not inclusive
  • Empty partitions: Ensure you don't create empty substrings
  • Not copying list: When adding to result, must use new ArrayList<>(current)
✨ OPTIMIZATION: PALINDROME DP TABLE
  • Time Trade-off: O(n²) preprocessing to build DP table, then O(1) per check
  • Worth it when: Multiple palindrome checks on same string (like in partitioning)
  • DP Formula: dp[i][j] = (s[i] == s[j]) && dp[i+1][j-1]
  • Base cases: Single char = true, Two chars = s[i] == s[i+1]

📚 ANALOGY: HIGHLIGHTING TEXT

Imagine highlighting parts of a sentence:

  • Substrings: Highlight consecutive words/letters (like underlining a phrase continuously)
  • Subsequences: Highlight any selection of words keeping order (like picking keywords)

Example: "THE CAT"

  • Substrings: "T", "TH", "THE", "HE", "H", "E", " ", " C", "C", "CA", "CAT", "A", "AT", "T"
  • Subsequences: Can skip middle parts - "TC" (skip everything between), "TEAT" (skip "H" and "C"), etc.
📖 THEORY: SUBSTRING VS SUBSEQUENCE

Fundamental Difference:

  • Substring: MUST be contiguous. Like a continuous highlight.
  • Subsequence: Can skip characters but MUST maintain order.

Counting:

  • Substrings: O(n²) total for string of length n. (n choose 2 ways to pick start and end)
  • Subsequences: O(2ⁿ) total. (Each char: include or exclude = 2 choices, n chars = 2ⁿ)

Generation Approach:

  • Substrings: Two nested loops (all start positions × all end positions)
  • Subsequences: Recursion with binary choice (include current char or skip it)

// All substrings: Contiguous sequences
// "abc" → ["a", "ab", "abc", "b", "bc", "c"]
public List<String> allSubstrings(String s) {
    List<String> result = new ArrayList<>();
    for (int i = 0; i < s.length(); i++) {
        for (int j = i + 1; j <= s.length(); j++) {
            result.add(s.substring(i, j));
        }
    }
    return result;  // O(n²) substrings, O(n³) time with substring creation
}

// All subsequences: Pick or skip each character (order preserved)
// "abc" → ["", "a", "b", "c", "ab", "ac", "bc", "abc"]
public List<String> allSubsequences(String s) {
    List<String> result = new ArrayList<>();
    backtrack(s, 0, new StringBuilder(), result);
    return result;
}

private void backtrack(String s, int index, StringBuilder current,
                      List<String> result) {
    if (index == s.length()) {
        result.add(current.toString());
        return;
    }

    // Choice 1: Include current character
    current.append(s.charAt(index));
    backtrack(s, index + 1, current, result);
    current.deleteCharAt(current.length() - 1);

    // Choice 2: Exclude current character
    backtrack(s, index + 1, current, result);
}

// Letter Case Permutations: "a1b2" → ["a1b2","a1B2","A1b2","A1B2"]
public List<String> letterCasePermutation(String s) {
    List<String> result = new ArrayList<>();
    backtrackCase(s.toCharArray(), 0, result);
    return result;
}

private void backtrackCase(char[] chars, int index, List<String> result) {
    if (index == chars.length) {
        result.add(new String(chars));
        return;
    }

    backtrackCase(chars, index + 1, result);  // Keep current case

    if (Character.isLetter(chars[index])) {
        chars[index] ^= (1 << 5);  // Toggle case (XOR trick)
        backtrackCase(chars, index + 1, result);
        chars[index] ^= (1 << 5);  // Restore
    }
}
💡 SUBSTRING VS SUBSEQUENCE
  • Substring: Contiguous, O(n²) count, use nested loops
  • Subsequence: Can skip chars, O(2ⁿ) count, use recursion
  • XOR trick for case toggle: c ^= (1 << 5) or c ^= 32
⚠️ COMMON MISTAKES: SUBSEQUENCES
  • Confusing with Substrings: Subsequences can SKIP characters, substrings CANNOT
  • Order Violation: Must maintain original order - "CA" is valid from "CAT" but "AC" is not
  • Exponential Explosion: Be aware 2ⁿ can be HUGE (2²⁰ = 1 million possibilities!)
  • Case Toggle Bug: XOR only works for letters, check with Character.isLetter()
💡 INTUITION: THE BINARY CHOICE

For Subsequences: At each position, you have exactly 2 choices:

  • Include the character - Add it to current sequence
  • Exclude the character - Skip it, move to next

This creates a binary decision tree with 2ⁿ leaves (all possible subsequences).

Think of it like: A checklist where you tick or skip each item:

  • For "ABC": [✓✓✓]=ABC, [✓✓✗]=AB, [✓✗✓]=AC, [✓✗✗]=A, [✗✓✓]=BC, [✗✓✗]=B, [✗✗✓]=C, [✗✗✗]=""

  • Divide-and-Conquer Friendly: Can split at any point

Three Main Array Recursion Patterns:

  • Element Selection: For each element, choose to include or exclude (subsets)
  • Position Assignment: For each element, decide which group/partition it belongs to
  • Sequential Processing: Process elements in order, making decisions based on cumulative state

Complexity Insights:

  • Subsets: 2ⁿ possibilities (each element in or out)
  • Partitions: Exponential, depends on K (number of groups)
  • With Constraints: Often can prune significantly (target sum, equal partitions)

public List<List<Integer>> subsets(int[] nums) {
    List<List<Integer>> result = new ArrayList<>();
    backtrack(nums, 0, new ArrayList<>(), result);
    return result;
}

private void backtrack(int[] nums, int start, List<Integer> current,
                      List<List<Integer>> result) {
    result.add(new ArrayList<>(current));  // Add at every level

    for (int i = start; i < nums.length; i++) {
        current.add(nums[i]);
        backtrack(nums, i + 1, current, result);
        current.remove(current.size() - 1);
    }
}

// Partition array into k equal sum subsets
// [4,3,2,3,5,2,1], k=4 → true (can partition into 4 subsets of sum 5)
public boolean canPartitionKSubsets(int[] nums, int k) {
    int sum = 0;
    for (int num : nums) sum += num;
    if (sum % k != 0) return false;  // Can't divide evenly

    int target = sum / k;
    Arrays.sort(nums);
    reverse(nums);  // Start with largest (pruning optimization)

    return backtrack(nums, new int[k], 0, target);
}

private boolean backtrack(int[] nums, int[] buckets, int index, int target) {
    if (index == nums.length) {
        // Check if all buckets equal target
        for (int bucket : buckets) {
            if (bucket != target) return false;
        }
        return true;
    }

    for (int i = 0; i < buckets.length; i++) {
        // Pruning: skip if adding exceeds target
        if (buckets[i] + nums[index] > target) continue;

        buckets[i] += nums[index];  // Choose
        if (backtrack(nums, buckets, index + 1, target)) return true;  // Explore
        buckets[i] -= nums[index];  // Unchoose

        // Pruning: if bucket was empty and failed, skip other empty buckets
        if (buckets[i] == 0) break;
    }

    return false;
}

// Partition Equal Subset Sum: [1,5,11,5] → true ([1,5,5] and [11])
public boolean canPartition(int[] nums) {
    int sum = 0;
    for (int num : nums) sum += num;
    if (sum % 2 != 0) return false;  // Odd sum can't be split equally

    return backtrack(nums, 0, 0, sum / 2);
}

private boolean backtrack(int[] nums, int index, int currentSum, int target) {
    if (currentSum == target) return true;
    if (index == nums.length || currentSum > target) return false;

    // Include current number
    if (backtrack(nums, index + 1, currentSum + nums[index], target)) {
        return true;
    }

    // Exclude current number
    return backtrack(nums, index + 1, currentSum, target);
}
⚠️ PERFORMANCE PITFALLS
  • Sorting and starting with largest elements prunes search space
  • Skip empty buckets after failure (all empty buckets are equivalent)
  • Use DP memoization for overlapping subproblems
💡 INTUITION: SUBSET VS PARTITION

Key Difference:

  • Subsets: Can pick any elements, don't need to use all → 2ⁿ possibilities
  • Partitions: MUST use all elements, dividing into groups → fewer valid arrangements

Partition Insight: Once you decide one group, the rest is determined (like cutting a pizza - one cut determines both pieces!)

Optimization Key: For K-partition, sort largest-first. Why? Large elements have fewer placement options, so try them first to fail fast!

private void backtrack(String s, int start, List<String> parts,
                      List<String> result) {
    if (parts.size() == 4) {
        if (start == s.length()) {  // Used all characters
            result.add(String.join(".", parts));
        }
        return;
    }

    // Try 1, 2, or 3 digits for current part
    for (int len = 1; len <= 3 && start + len <= s.length(); len++) {
        String part = s.substring(start, start + len);

        // Validate: no leading zeros (except "0"), value <= 255
        if ((part.startsWith("0") && part.length() > 1) ||
            Integer.parseInt(part) > 255) {
            continue;
        }

        parts.add(part);
        backtrack(s, start + len, parts, result);
        parts.remove(parts.size() - 1);
    }
}

        availableDigits--;
    }

    return result;
}

📖 THEORY: WHY TREES ARE NATURALLY RECURSIVE

Trees Are Defined Recursively:

  • A tree is a node with two subtrees (left and right)
  • Each subtree is also a tree (recursive definition!)
  • Base case: null node (empty tree)

Key Insight: Any tree problem can be solved by:

  • 1. Solve for current node
  • 2. Recursively solve for left subtree
  • 3. Recursively solve for right subtree
  • 4. Combine results

Time Complexity: Visit each node once = O(n) where n = number of nodes

Space Complexity: O(h) where h = height (call stack depth). Balanced tree: O(log n), Skewed: O(n)

Traversal Order Use Cases Example

Inorder Left → Root → Right BST sorted output, expression parsing Print sorted BST
Postorder Left → Right → Root Delete tree, postfix expressions, dependency resolution Calculate folder sizes
Level-order Level by level (BFS) Shortest path, level-based operations Organization hierarchy

🧠 MENTAL MODEL: THE TRUST PRINCIPLE

The Most Important Concept for Tree Recursion:

Don't think about the whole tree! Only think about:

  • Current node: What do I do with this node?
  • Trust recursion: Assume my recursive calls will correctly handle subtrees
  • Combine results: How do I use the answers from left and right?

Example: Tree Height

  • Don't trace the entire tree mentally!
  • Just think: "My height = 1 + max(left height, right height)"
  • Trust that recursive calls will return correct left/right heights

// PREORDER: Root → Left → Right
public void preorder(TreeNode root) {
    if (root == null) return;
    process(root);              // Process root first
    preorder(root.left);        // Then left subtree
    preorder(root.right);       // Then right subtree
}

// INORDER: Left → Root → Right (gives sorted order for BST)
public void inorder(TreeNode root) {
    if (root == null) return;
    inorder(root.left);         // Left subtree first
    process(root);              // Then process root
    inorder(root.right);        // Then right subtree
}

// POSTORDER: Left → Right → Root
public void postorder(TreeNode root) {
    if (root == null) return;
    postorder(root.left);       // Left subtree first
    postorder(root.right);      // Then right subtree
    process(root);              // Process root last
}

// Max Depth: Height of tree
public int maxDepth(TreeNode root) {
    if (root == null) return 0;
    int leftDepth = maxDepth(root.left);
    int rightDepth = maxDepth(root.right);
    return 1 + Math.max(leftDepth, rightDepth);
}

// Is Balanced: Left and right subtrees differ by at most 1
public boolean isBalanced(TreeNode root) {
    return checkHeight(root) != -1;
}

private int checkHeight(TreeNode node) {
    if (node == null) return 0;

    int leftHeight = checkHeight(node.left);
    if (leftHeight == -1) return -1;  // Left subtree unbalanced

    int rightHeight = checkHeight(node.right);
    if (rightHeight == -1) return -1;  // Right subtree unbalanced

    if (Math.abs(leftHeight - rightHeight) > 1) return -1;  // Current node unbalanced

    return 1 + Math.max(leftHeight, rightHeight);
}
🎯 TRAVERSAL CHEAT SHEET
  • Preorder: Root first - good for copying tree, prefix expressions
  • Inorder: Left→Root→Right - BST gives sorted order
  • Postorder: Children first - good for deletion, postfix expressions
⚠️ COMMON MISTAKES: TREE RECURSION
  • Null Pointer Exception: Always check if (node == null) at start!
  • Wrong Traversal: Inorder only gives sorted output for BST, not general binary tree
  • Forgetting to Return: Must return value from recursive calls
  • Not Combining Results: For height/sum problems, must combine left+right results
  • Global Variables: Avoid when possible - pass state through parameters or return values
💡 INTUITION: WHEN TO USE WHICH TRAVERSAL

Ask yourself: "When do I need to process the current node?"

  • Need current node value FIRST? → Preorder (parent before children)
  • Need children's values FIRST? → Postorder (children before parent)
  • Need to visit in sorted order (BST)? → Inorder
  • Need to process level by level? → Level-order (BFS)

Practical Test: For tree height, you need children's heights before computing current height → Postorder!

🗺️ ANALOGY: TREASURE HUNTING

Path problems are like treasure hunting in a maze:

  • Path Sum I: "Is there a path to treasure with exact value X?" → Boolean search
  • Path Sum II: "Find ALL paths to treasure with value X" → Collect all solutions
  • Path Sum III: "Count paths anywhere in forest" → Can start/end anywhere
  • Max Path Sum: "Find most valuable path in forest" → Optimization problem

Each decision point: Should I go left? Right? Both? Track your path as you explore!

💡 INTUITION: PATH TRACKING

The Path Pattern:

  • Going Down: Add current node to path, subtract from remaining sum
  • At Leaf: Check if sum matches target
  • Coming Back: Remove node from path (backtrack!)

Why Backtrack? The path list is shared across recursive calls. Must undo changes so it's correct for sibling branches!

Key Insight: Path Sum I/II need root-to-leaf. Path Sum III allows any path → need to try starting from every node!

// Path Sum I: Does root-to-leaf path exist with given sum?
public boolean hasPathSum(TreeNode root, int targetSum) {
    if (root == null) return false;
    if (root.left == null && root.right == null) {  // Leaf node
        return targetSum == root.val;
    }
    int remaining = targetSum - root.val;
    return hasPathSum(root.left, remaining) || hasPathSum(root.right, remaining);
}

// Path Sum II: Find all root-to-leaf paths with given sum
public List<List<Integer>> pathSum(TreeNode root, int targetSum) {
    List<List<Integer>> result = new ArrayList<>();
    backtrack(root, targetSum, new ArrayList<>(), result);
    return result;
}

private void backtrack(TreeNode node, int remaining, List<Integer> path,
                      List<List<Integer>> result) {
    if (node == null) return;

    path.add(node.val);  // Choose

    if (node.left == null && node.right == null && remaining == node.val) {
        result.add(new ArrayList<>(path));  // Found valid path
    }

    backtrack(node.left, remaining - node.val, path, result);   // Explore
    backtrack(node.right, remaining - node.val, path, result);

    path.remove(path.size() - 1);  // Unchoose
}

// Path Sum III: Count all paths (not just root-to-leaf) with given sum
public int pathSumIII(TreeNode root, int targetSum) {
    if (root == null) return 0;

    // Paths starting from current node
    int pathsFromRoot = countPaths(root, targetSum);

    // Paths in left and right subtrees
    int pathsInLeft = pathSumIII(root.left, targetSum);
    int pathsInRight = pathSumIII(root.right, targetSum);

    return pathsFromRoot + pathsInLeft + pathsInRight;
}

private int countPaths(TreeNode node, long remaining) {
    if (node == null) return 0;

    int count = (remaining == node.val) ? 1 : 0;
    count += countPaths(node.left, remaining - node.val);
    count += countPaths(node.right, remaining - node.val);

    return count;
}

// Binary Tree Maximum Path Sum (can start/end anywhere)
private int maxSum = Integer.MIN_VALUE;

public int maxPathSum(TreeNode root) {
    maxPathSumHelper(root);
    return maxSum;
}

private int maxPathSumHelper(TreeNode node) {
    if (node == null) return 0;

    // Get max path sum from left and right (ignore negative)
    int left = Math.max(0, maxPathSumHelper(node.left));
    int right = Math.max(0, maxPathSumHelper(node.right));

    // Update global max (path through current node)
    maxSum = Math.max(maxSum, left + right + node.val);

    // Return max single path for parent
    return node.val + Math.max(left, right);
}
💡 PATH SUM PATTERNS
  • Root-to-Leaf: Check for leaf (left==null && right==null)
  • Any Path: Try starting from each node, use prefix sum or recursion
  • Max Path: Update global variable, return single path to parent
⚠️ COMMON MISTAKES: PATH PROBLEMS
  • Forgetting Backtrack: Must remove node from path when returning!
  • Not Checking Leaf: Path must end at leaf (both children null)
  • Integer Overflow: Use long for Path Sum III with large values
  • Negative Values: Max path may exclude negative subtrees (use Math.max(0, ...))
  • Global Variable Misuse: Reset between test cases or use instance variables

🧩 ANALOGY: REBUILDING FROM BLUEPRINTS

Imagine you have two blueprints of the same building:

  • Preorder (Front View): Shows building entrance first, then wings
  • Inorder (Floor Plan): Shows left rooms, center, then right rooms
  • Postorder (Foundation Up): Shows basement first, ending with roof

The Reconstruction Process:

  • Preorder tells you WHERE to start (the root/entrance)
  • Inorder tells you WHAT'S LEFT and WHAT'S RIGHT of that point
  • Recurse on left and right portions separately

You need BOTH blueprints because one shows structure order, the other shows spatial arrangement!

📖 THEORY: WHY YOU NEED TWO TRAVERSALS

Key Insight: One traversal alone is ambiguous!

  • Preorder alone: Can't tell which nodes are left vs right children
  • Inorder alone: Can't tell which node is the root

The Magic Combination:

  • Preorder: First element is always root
  • Inorder: Root position divides left subtree (before root) from right subtree (after root)
  • Postorder: Last element is always root

Algorithm:

  • 1. Find root (first in preorder / last in postorder)
  • 2. Locate root in inorder → determines left/right split
  • 3. Calculate size of left subtree
  • 4. Recursively build left subtree from corresponding ranges
  • 5. Recursively build right subtree from remaining ranges

// Build tree from Preorder and Inorder
// Preorder: [3,9,20,15,7], Inorder: [9,3,15,20,7]
public TreeNode buildTree(int[] preorder, int[] inorder) {
    Map<Integer, Integer> inorderMap = new HashMap<>();
    for (int i = 0; i < inorder.length; i++) {
        inorderMap.put(inorder[i], i);  // Value → Index mapping
    }
    return build(preorder, 0, preorder.length - 1, inorder, 0, inorder.length - 1, inorderMap);
}

private TreeNode build(int[] preorder, int preStart, int preEnd,
                      int[] inorder, int inStart, int inEnd,
                      Map<Integer, Integer> inorderMap) {
    if (preStart > preEnd || inStart > inEnd) return null;

    int rootVal = preorder[preStart];  // First element in preorder is root
    TreeNode root = new TreeNode(rootVal);

    int rootIndex = inorderMap.get(rootVal);  // Find root in inorder
    int leftSize = rootIndex - inStart;  // Size of left subtree

    // Build left subtree
    root.left = build(preorder, preStart + 1, preStart + leftSize,
                     inorder, inStart, rootIndex - 1, inorderMap);

    // Build right subtree
    root.right = build(preorder, preStart + leftSize + 1, preEnd,
                      inorder, rootIndex + 1, inEnd, inorderMap);

    return root;
}

private TreeNode build2(int[] postorder, int postStart, int postEnd,
                       int[] inorder, int inStart, int inEnd,
                       Map<Integer, Integer> inorderMap) {
    if (postStart > postEnd || inStart > inEnd) return null;

    int rootVal = postorder[postEnd];  // Last element in postorder is root
    TreeNode root = new TreeNode(rootVal);

    int rootIndex = inorderMap.get(rootVal);
    int leftSize = rootIndex - inStart;

    root.left = build2(postorder, postStart, postStart + leftSize - 1,
                      inorder, inStart, rootIndex - 1, inorderMap);

    root.right = build2(postorder, postStart + leftSize, postEnd - 1,
                       inorder, rootIndex + 1, inEnd, inorderMap);

    return root;
}
🎯 TREE CONSTRUCTION RULES
  • Preorder + Inorder: First of preorder = root, split inorder at root
  • Postorder + Inorder: Last of postorder = root, split inorder at root
  • Preorder + Postorder: Only possible for full binary trees
  • Use HashMap for O(1) root lookup in inorder array
⚠️ COMMON MISTAKES: TREE CONSTRUCTION
  • Index Calculation Errors: Off-by-one in subtree range calculations is extremely common!
  • Not Using HashMap: Linear search for root in inorder is O(n²) total complexity
  • Duplicate Values: Algorithm assumes all values are unique
  • Wrong Range: Make sure left/right subtree ranges don't overlap or skip elements
💡 INTUITION: THE INDEX DANCE

The Tricky Part: Mapping between preorder/postorder indices and inorder indices.

Key Variables:

  • rootIndex in inorder: Splits into left/right
  • leftSize: Number of nodes in left subtree = rootIndex - inStart
  • Left subtree in preorder: [preStart+1 ... preStart+leftSize]
  • Right subtree in preorder: [preStart+leftSize+1 ... preEnd]

Mental Model: Think of sliding windows on both arrays that move in sync!

What is Memoization?
Cache results to avoid redundant calculations
📖 ANALOGY: THE RECIPE BOOK

Imagine you're learning to cook. Two approaches:

  • Without Memoization: Every time you want to make spaghetti, you call your grandmother, ask for the recipe, write it down, throw it away after cooking, then call her again next time. Wasteful!
  • With Memoization: First time you call grandma and get the recipe, you write it in your recipe book. Next time you want spaghetti, you just look it up in the book! Much faster!

Memoization = keeping a "recipe book" of solutions you've already computed. Look up first, compute only if not found.

📖 THEORY: WHY MEMOIZATION IS REVOLUTIONARY

The Problem: Many recursive algorithms recompute the SAME values over and over. fibonacci(5) calculates fibonacci(3) twice!

The Impact: Without memoization, fibonacci(40) takes ~2 billion calls. WITH memoization: just 40 calls! That's a 50 million times speedup!

Requirements for Memoization:

  • Overlapping Subproblems: Same calculation appears multiple times in recursion tree
  • Pure Functions: Same input ALWAYS gives same output (no randomness, no side effects)
  • Optimal Substructure: Optimal solution built from optimal sub-solutions

Memoization vs Dynamic Programming:

  • Memoization: Top-down approach (recursion + caching)
  • DP: Bottom-up approach (iterative, build table)
  • Both solve same problems, memoization often easier to write!
🧠 MENTAL MODEL: THE CACHE TABLE

Visualize memoization as a table/map where:

  • Keys = Function parameters (the "question")
  • Values = Computed results (the "answer")
  • Algorithm: Always check table first. If answer exists, return it. If not, compute it, store it, then return it.

It's like having a cheat sheet during an exam - but you wrote the cheat sheet yourself as you solved problems!

📌 MEMOIZATION = RECURSION + CACHING
  • Store results of expensive function calls
  • Return cached result when same inputs occur again
  • Converts O(2ⁿ) exponential time to O(n) or O(n²) polynomial time

// House Robber: Max money without robbing adjacent houses
public int rob(int[] nums) {
    int[] memo = new int[nums.length];
    Arrays.fill(memo, -1);
    return robFrom(nums, 0, memo);
}

private int robFrom(int[] nums, int index, int[] memo) {
    if (index >= nums.length) return 0;
    if (memo[index] != -1) return memo[index];

    // Choice 1: Rob this house, skip next
    int robCurrent = nums[index] + robFrom(nums, index + 2, memo);

    // Choice 2: Skip this house
    int skipCurrent = robFrom(nums, index + 1, memo);

    memo[index] = Math.max(robCurrent, skipCurrent);
    return memo[index];
}

// Decode Ways: "226" → ["BBF", "BZ", "VF"] = 3 ways
public int numDecodings(String s) {
    int[] memo = new int[s.length()];
    Arrays.fill(memo, -1);
    return decode(s, 0, memo);
}

private int decode(String s, int index, int[] memo) {
    if (index == s.length()) return 1;  // Found valid decoding
    if (s.charAt(index) == '0') return 0;  // Invalid
    if (memo[index] != -1) return memo[index];

    int ways = decode(s, index + 1, memo);  // Take 1 digit

    if (index + 1 < s.length()) {
        int twoDigit = Integer.parseInt(s.substring(index, index + 2));
        if (twoDigit <= 26) {
            ways += decode(s, index + 2, memo);  // Take 2 digits
        }
    }

    memo[index] = ways;
    return ways;
}

// Longest Common Subsequence: "abcde" and "ace" → 3 ("ace")
public int longestCommonSubsequence(String text1, String text2) {
    int[][] memo = new int[text1.length()][text2.length()];
    for (int[] row : memo) Arrays.fill(row, -1);
    return lcs(text1, text2, 0, 0, memo);
}

private int lcs(String s1, String s2, int i, int j, int[][] memo) {
    if (i == s1.length() || j == s2.length()) return 0;
    if (memo[i][j] != -1) return memo[i][j];

    if (s1.charAt(i) == s2.charAt(j)) {
        // Characters match: include and move both pointers
        memo[i][j] = 1 + lcs(s1, s2, i + 1, j + 1, memo);
    } else {
        // Don't match: try skipping from either string
        int skip1 = lcs(s1, s2, i + 1, j, memo);
        int skip2 = lcs(s1, s2, i, j + 1, memo);
        memo[i][j] = Math.max(skip1, skip2);
    }

    return memo[i][j];
}

// Edit Distance: Min operations to convert word1 → word2
public int minDistance(String word1, String word2) {
    int[][] memo = new int[word1.length()][word2.length()];
    for (int[] row : memo) Arrays.fill(row, -1);
    return editDist(word1, word2, 0, 0, memo);
}

private int editDist(String s1, String s2, int i, int j, int[][] memo) {
    if (i == s1.length()) return s2.length() - j;  // Insert remaining
    if (j == s2.length()) return s1.length() - i;  // Delete remaining
    if (memo[i][j] != -1) return memo[i][j];

    if (s1.charAt(i) == s2.charAt(j)) {
        memo[i][j] = editDist(s1, s2, i + 1, j + 1, memo);  // No operation
    } else {
        int insert = editDist(s1, s2, i, j + 1, memo);
        int delete = editDist(s1, s2, i + 1, j, memo);
        int replace = editDist(s1, s2, i + 1, j + 1, memo);
        memo[i][j] = 1 + Math.min(insert, Math.min(delete, replace));
    }

    return memo[i][j];
}

// Unique Paths: Grid m×n, can only move right or down
public int uniquePaths(int m, int n) {
    int[][] memo = new int[m][n];
    return paths(0, 0, m, n, memo);
}

private int paths(int row, int col, int m, int n, int[][] memo) {
    if (row == m - 1 && col == n - 1) return 1;  // Reached destination
    if (row >= m || col >= n) return 0;  // Out of bounds
    if (memo[row][col] != 0) return memo[row][col];

    memo[row][col] = paths(row + 1, col, m, n, memo) +  // Move down
                     paths(row, col + 1, m, n, memo);    // Move right

    return memo[row][col];
}

The 4-Step Optimization Evolution
Complete journey from recursion to space-optimized DP
🏗️ ANALOGY: BUILDING A HOUSE

Imagine four ways to build houses in a neighborhood:

  • Step 1 - Pure Recursion: Build each house from scratch, calculating everything yourself. Every time someone asks "how to build a kitchen?", you figure it out from first principles. SLOW but thorough understanding.
  • Step 2 - Memoization: First time you build a kitchen, you write down the steps in a notebook. Next time someone asks, you look it up. Much faster! But still building on-demand.
  • Step 3 - Tabulation (DP): Instead of waiting for requests, you create a complete blueprint catalog systematically: "Here's how to build foundation, walls, roof, kitchen..." Everything pre-calculated in order.
  • Step 4 - Space Optimization: You realize you don't need ALL blueprints at once. Keep only the last 2 pages you need. Like a rolling window - much less storage!
📖 THEORY: WHY WE OPTIMIZE STEP-BY-STEP

The Evolution Makes Sense:

  • Start with Recursion: Matches the problem definition naturally. Easy to write and understand. Shows your thinking clearly.
  • Add Memoization: Minimal code change (add a cache). Fixes exponential blowup. Still recursive (keeps intuition).
  • Convert to Tabulation: Eliminates recursion overhead. More predictable performance. Better for analysis.
  • Optimize Space: Final polish. Reduces memory usage. Critical for large inputs.

Key Insight: Each step builds on the previous! You're not learning 4 different approaches - you're learning ONE approach that evolves.

    if (n <= 1) return n;

    // RECURSIVE CASE: F(n) = F(n-1) + F(n-2)
    return fib(n - 1) + fib(n - 2);
}

// CLIMBING STAIRS: Pure Recursion
// "How many ways to climb n stairs taking 1 or 2 steps?"
// Time: O(2ⁿ), Space: O(n)
public int climbStairs(int n) {
    // BASE CASES
    if (n <= 2) return n; // 1 stair: 1 way, 2 stairs: 2 ways

    // RECURSIVE LOGIC:
    // To reach stair n, you either:
    //   - Take 1 step from stair (n-1): climbStairs(n-1) ways
    //   - Take 2 steps from stair (n-2): climbStairs(n-2) ways
    return climbStairs(n - 1) + climbStairs(n - 2);
}

  • Problem size is small (combinatorial enumeration with early stopping)

  • You see the SAME subproblem appearing multiple times in the recursion tree
  • Time complexity is exponential (2ⁿ, n!) but problem should be polynomial
  • You're calculating "how many ways" or "optimal value" → Usually needs DP!

STEP 2
Memoization - Top-Down DP with Caching
Add a cache to remember computed results
📓 ANALOGY: THE ANSWER KEY

Imagine doing homework. Two approaches:

  • Without Memoization: Each time you encounter "What's 7×8?", you count on fingers or calculate from scratch.
  • With Memoization: First time you calculate 7×8=56, you write it in your notebook. Next time you see 7×8, you just look it up instantly!

Memoization = Keeping an answer key for subproblems you've already solved.

📖 THEORY: FROM EXPONENTIAL TO POLYNOMIAL

The Magic of Memoization:

  • Core Idea: Before computing, check if answer is cached. If yes, return it. If no, compute and cache.
  • Requirement: Overlapping subproblems (same calculation appears multiple times)
  • Impact: Transforms O(2ⁿ) to O(n) or O(n²) - MASSIVE speedup!
  • Trade-off: Use extra memory to save time

Why It's Top-Down:

  • Still uses recursion (top-down approach)
  • Calculates only what's needed (lazy evaluation)
  • Natural extension of recursive thinking

// FIBONACCI: Memoization (Top-Down DP)
// Time: O(n) - Each subproblem calculated once
// Space: O(n) - Cache array + O(n) recursion stack = O(n)
public int fib(int n) {
    int[] memo = new int[n + 1];
    Arrays.fill(memo, -1); // -1 means "not calculated yet"
    return fibMemo(n, memo);
}

private int fibMemo(int n, int[] memo) {
    // BASE CASES
    if (n <= 1) return n;

    // CHECK CACHE: If already calculated, return cached result
    if (memo[n] != -1) {
        return memo[n]; // ✓ CACHE HIT - instant return!
    }

    // CALCULATE: Not in cache, compute it
    memo[n] = fibMemo(n - 1, memo) + fibMemo(n - 2, memo);

    // RETURN: Answer now cached for future use
    return memo[n];
}

// CLIMBING STAIRS: Memoization
// Time: O(n), Space: O(n)
public int climbStairs(int n) {
    int[] memo = new int[n + 1];
    return climbMemo(n, memo);
}

private int climbMemo(int n, int[] memo) {
    if (n <= 2) return n;
    if (memo[n] > 0) return memo[n]; // Cache check

    memo[n] = climbMemo(n - 1, memo) + climbMemo(n - 2, memo);
    return memo[n];
}

// ALTERNATIVE: Using HashMap (better for sparse inputs)
Map<Integer, Integer> cache = new HashMap<>();

public int fibHashMap(int n) {
    if (n <= 1) return n;
    if (cache.containsKey(n)) return cache.get(n);

    int result = fibHashMap(n - 1) + fibHashMap(n - 2);
    cache.put(n, result);
    return result;
}

STEP 3
Tabulation - Bottom-Up DP
Build the table iteratively from base cases upward
📊 ANALOGY: FILLING A SPREADSHEET

Imagine filling out a spreadsheet:

  • Memoization (Top-Down): Someone asks "What's in cell E5?" You check if it's filled. If not, you figure out its formula, which might need C3 and D4. You recursively fill dependencies as needed.
  • Tabulation (Bottom-Up): You systematically fill the spreadsheet row by row, left to right. By the time you reach E5, all its dependencies (C3, D4) are already filled. No recursion needed!

Tabulation = Filling the table in the right order so all dependencies are already computed.

📖 THEORY: WHY TABULATION IS OFTEN BETTER

Advantages of Bottom-Up:

  • No Recursion Overhead: Iterative = no call stack, no stack overflow risk
  • Predictable Performance: Always computes all subproblems, easy to analyze
  • Better Cache Locality: Sequential array access is cache-friendly
  • Easier Space Optimization: Can clearly see which old values aren't needed

When Memoization is Better:

  • Not All Subproblems Needed: Sparse problem space (e.g., LCS with very different strings)
  • Recursion is Natural: Tree problems, complex state space
  • Unclear Order: Hard to determine iterative fill order

The Conversion Process:

  • 1. Identify States: What parameters change? Those become table dimensions.
  • 2. Determine Base Cases: Initialize table with base values.
  • 3. Find Fill Order: What order ensures dependencies are ready?
  • 4. Write Transition: Convert recursive relation to iterative update.

// FIBONACCI: Tabulation (Bottom-Up DP)
// Time: O(n) - Single loop through all values
// Space: O(n) - DP table
public int fib(int n) {
    // EDGE CASE
    if (n <= 1) return n;

    // CREATE TABLE: dp[i] = fibonacci(i)
    int[] dp = new int[n + 1];

    // BASE CASES: Initialize known values
    dp[0] = 0;
    dp[1] = 1;

    // FILL TABLE: Build from bottom up
    for (int i = 2; i <= n; i++) {
        // RECURRENCE: F(i) = F(i-1) + F(i-2)
        dp[i] = dp[i - 1] + dp[i - 2];
    }

    // ANSWER: Final value in table
    return dp[n];
}

// CLIMBING STAIRS: Tabulation
// Time: O(n), Space: O(n)
public int climbStairs(int n) {
    if (n <= 2) return n;

    int[] dp = new int[n + 1];
    dp[1] = 1; // 1 way to reach stair 1
    dp[2] = 2; // 2 ways to reach stair 2

    for (int i = 3; i <= n; i++) {
        // Ways to reach stair i = ways from (i-1) + ways from (i-2)
        dp[i] = dp[i - 1] + dp[i - 2];
    }

    return dp[n];
}

// MIN COST CLIMBING STAIRS: More complex example
// cost[i] = cost to step on stair i
// Can start from step 0 or 1, can climb 1 or 2 steps
public int minCostClimbingStairs(int[] cost) {
    int n = cost.length;
    int[] dp = new int[n + 1]; // dp[i] = min cost to reach stair i

    // BASE: Can start at stair 0 or 1 for free
    dp[0] = 0;
    dp[1] = 0;

    // BUILD TABLE
    for (int i = 2; i <= n; i++) {
        // To reach stair i, either:
        //   - Come from (i-1) and pay cost[i-1]
        //   - Come from (i-2) and pay cost[i-2]
        int fromPrev1 = dp[i - 1] + cost[i - 1];
        int fromPrev2 = dp[i - 2] + cost[i - 2];
        dp[i] = Math.min(fromPrev1, fromPrev2);
    }

    return dp[n]; // Minimum cost to reach top (past last stair)
}

💡 INTUITION: DEPENDENCY DIRECTION

Key Insight for Tabulation:

  • Identify Dependencies: dp[i] depends on which previous values?
  • Fill in Order: Ensure dependencies are computed before they're needed
  • 1D Problems: Usually left-to-right (small to large)
  • 2D Problems: Usually top-to-bottom, left-to-right (or by diagonals)

Example: Fibonacci

  • dp[i] depends on dp[i-1] and dp[i-2]
  • So fill left to right: 0 → 1 → 2 → 3 → ... → n
  • When computing dp[i], dp[i-1] and dp[i-2] are already ready!
🎯 MEMOIZATION → TABULATION CONVERSION GUIDE
  1. Identify Parameters: Memoization has func(n, memo) → Table needs size n+1
  2. Create Table: int[] dp = new int[n+1]
  3. Initialize Base Cases: What were the if (n <= 1) return n? Those become dp[0]=0, dp[1]=1
  4. Determine Loop Range: Start after base cases, go to n
  5. Convert Recursive Call to Table Lookup: func(n-1) becomes dp[i-1]
  6. Return Final Value: return dp[n]
⚠️ COMMON TABULATION MISTAKES
  • Wrong Loop Range: Starting at 0 when base cases are 0 and 1 (should start at 2)
  • Off-by-One Errors: Array size n vs n+1, accessing dp[i-1] when i=0
  • Wrong Fill Order: Computing dp[i] before its dependencies are ready
  • Forgetting Base Cases: Not initializing dp[0], dp[1], etc.
  • Wrong Return: Returning dp[n-1] instead of dp[n]

STEP 4
Space Optimization - Reducing Memory Usage
Keep only what you need, discard the rest
🎬 ANALOGY: SLIDING WINDOW VIEW

Imagine watching a movie on a train:

  • Tabulation Approach: You record the ENTIRE journey on video from start to finish. At the end, you watch the last frame. But you're carrying around hours of footage you'll never watch again!
  • Space Optimized: You just look out the window. You see the current view and maybe remember the last few frames. That's all you need! Why carry the whole movie?

Space optimization = Recognizing you don't need all historical data, just the recent relevant values.

📖 THEORY: IDENTIFYING OPTIMIZATION OPPORTUNITIES

When Can We Optimize Space?

  • Limited Dependencies: dp[i] only depends on a few recent values (like dp[i-1], dp[i-2])
  • Sequential Access: We fill the table in order and never look back far
  • No Path Reconstruction: We only need the final answer, not the path/choices taken

Common Patterns:

  • 1D → O(1): If dp[i] depends on last k values, keep only those k variables
  • 2D → 1D: If dp[i][j] depends only on current row and previous row, keep 2 rows
  • 2D → O(1): If dp[i][j] depends only on a few cells, track just those

When We CAN'T Optimize:

  • Need to reconstruct the solution path (e.g., print actual subsequence, not just length)
  • Random access needed (dependencies aren't localized)
  • Already using minimum space (genuinely need all values)

// FIBONACCI: Space Optimized
// Time: O(n) - Same as tabulation
// Space: O(1) - Only 2 variables! (vs O(n) array)
public int fib(int n) {
    if (n <= 1) return n;

    // INSIGHT: dp[i] = dp[i-1] + dp[i-2]
    // We only need the LAST 2 values, not the entire array!
    int prev2 = 0; // This was dp[0]
    int prev1 = 1; // This was dp[1]

    for (int i = 2; i <= n; i++) {
        // CALCULATE: current = prev1 + prev2 (like dp[i] = dp[i-1] + dp[i-2])
        int current = prev1 + prev2;

        // SLIDE WINDOW: Move forward (discard oldest, shift values)
        prev2 = prev1; // What was prev1 is now prev2
        prev1 = current; // Current becomes prev1
    }

    return prev1; // Last calculated value
}

// CLIMBING STAIRS: Space Optimized
// Time: O(n), Space: O(1)
public int climbStairs(int n) {
    if (n <= 2) return n;

    int prev2 = 1; // Ways to reach stair 1
    int prev1 = 2; // Ways to reach stair 2

    for (int i = 3; i <= n; i++) {
        int current = prev1 + prev2;
        prev2 = prev1;
        prev1 = current;
    }

    return prev1;
}

// MIN COST CLIMBING STAIRS: Space Optimized
// Time: O(n), Space: O(1)
public int minCostClimbingStairs(int[] cost) {
    int n = cost.length;
    int prev2 = 0; // Cost to reach stair 0
    int prev1 = 0; // Cost to reach stair 1

    for (int i = 2; i <= n; i++) {
        int current = Math.min(prev1 + cost[i - 1], prev2 + cost[i - 2]);
        prev2 = prev1;
        prev1 = current;
    }

    return prev1;
}

// 2D → 1D OPTIMIZATION EXAMPLE: Unique Paths
// Grid m×n, dp[i][j] = ways to reach cell (i,j)
// ORIGINAL: O(m×n) space for full 2D table
// OPTIMIZED: O(n) space - only keep current row!
public int uniquePaths(int m, int n) {
    // INSIGHT: dp[i][j] = dp[i-1][j] + dp[i][j-1]
    // Only need PREVIOUS ROW and CURRENT ROW - not all rows!

    int[] dp = new int[n];
    Arrays.fill(dp, 1); // First row: all 1s (only one way to reach)

    for (int i = 1; i < m; i++) { // For each row
        for (int j = 1; j < n; j++) { // For each column
            // dp[j] currently holds dp[i-1][j] (value from previous row)
            // dp[j-1] holds dp[i][j-1] (value from same row, previous column)
            dp[j] = dp[j] + dp[j - 1];
        }
    }

    return dp[n - 1]; // Bottom-right cell
}

Think of it like a train window:

  • As you move forward, old scenery disappears from view
  • You only see the current and recent past (the window view)
  • You don't need to remember what you saw 100 miles ago!

Applying to DP:

  • Identify the "window size": How many previous values does dp[i] depend on?
  • Keep only that many variables: For Fibonacci, window = 2 (prev1, prev2)
  • Slide the window forward: After calculating new value, shift: oldest gone, new becomes recent
🎯 TABULATION → SPACE OPTIMIZATION CHECKLIST
  1. Analyze Dependencies: What values does dp[i] actually use? (e.g., dp[i-1], dp[i-2])
  2. Count Variables Needed: If depends on last k values, need k variables
  3. Rename Variables: prev1 (most recent), prev2 (second recent), etc.
  4. Initialize: Set prev variables to base case values
  5. In Loop: Calculate current using prev values
  6. Slide Window: Shift variables (prev2=prev1, prev1=current)
  7. Return: Return the most recent variable (usually prev1 or current)
🎯 CAN YOU SPACE OPTIMIZE? DECISION MATRIX

dp[i][j] uses dp[i-1][j], dp[i][j-1] ✅ YES → O(n) Keep 1 row instead of full 2D table

⚠️ COMMON SPACE OPTIMIZATION MISTAKES
  • Wrong Slide Order: Must update prev2 BEFORE prev1! Order matters!
  • Losing Path Information: Optimizing away the table means you can't reconstruct the actual solution sequence
  • Overwriting Too Soon: In 2D→1D, be careful not to overwrite values still needed in current iteration
  • Wrong Return Value: Make sure to return the right variable after loop ends
  • Micro-optimization Trap: Don't optimize if O(n) space is fine for problem constraints

Only optimize if asked:

  • Don't Do It Unprompted: Tabulation with O(n) space is usually fine. Space optimization adds complexity.

LeetCode #70: Climbing Stairs

You are climbing a staircase with n steps. Each time you can climb 1 or 2 steps. In how many distinct ways can you climb to the top?

// ========================================
// STEP 1: PURE RECURSION
// Time: O(2ⁿ), Space: O(n)
// ========================================
public int climbStairs_Step1(int n) {
    // BASE CASES
    if (n <= 2) return n;

    // RECURRENCE: ways(n) = ways(n-1) + ways(n-2)
    // Explanation: To reach step n, either:
    //   - Take 1 step from (n-1): ways(n-1) possibilities
    //   - Take 2 steps from (n-2): ways(n-2) possibilities
    return climbStairs_Step1(n - 1) + climbStairs_Step1(n - 2);
}

// ========================================
// STEP 2: MEMOIZATION (Top-Down DP)
// Time: O(n), Space: O(n)
// ========================================
public int climbStairs_Step2(int n) {
    int[] memo = new int[n + 1];
    return climbMemo(n, memo);
}

private int climbMemo(int n, int[] memo) {
    // BASE CASES
    if (n <= 2) return n;

    // CHECK CACHE: Already computed?
    if (memo[n] > 0) return memo[n];

    // COMPUTE & CACHE
    memo[n] = climbMemo(n - 1, memo) + climbMemo(n - 2, memo);
    return memo[n];
}

// ========================================
// STEP 3: TABULATION (Bottom-Up DP)
// Time: O(n), Space: O(n)
// ========================================
public int climbStairs_Step3(int n) {
    // EDGE CASE
    if (n <= 2) return n;

    // CREATE TABLE: dp[i] = ways to reach step i
    int[] dp = new int[n + 1];

    // INITIALIZE BASE CASES
    dp[1] = 1; // One way to reach step 1
    dp[2] = 2; // Two ways to reach step 2

    // FILL TABLE from bottom up
    for (int i = 3; i <= n; i++) {
        dp[i] = dp[i - 1] + dp[i - 2];
    }

    return dp[n];
}

// STEP 4: SPACE OPTIMIZED

public int climbStairs_Step4(int n) {
    // EDGE CASE
    if (n <= 2) return n;

    // SLIDING WINDOW: Only track last 2 values
    int prev2 = 1; // Ways to reach step 1
    int prev1 = 2; // Ways to reach step 2

    // ITERATE and slide window forward
    for (int i = 3; i <= n; i++) {
        int current = prev1 + prev2;

        // SLIDE: Move window forward
        prev2 = prev1; // Old prev1 becomes prev2
        prev1 = current; // Current becomes prev1
    }

    return prev1;
}

📖 PROBLEM STATEMENT

LeetCode #1143: Longest Common Subsequence

Given two strings text1 and text2, return the length of their longest common subsequence (LCS). A subsequence can skip characters but must maintain order.

Examples:

  • text1="abcde", text2="ace" → 3 (LCS is "ace")
  • text1="abc", text2="abc" → 3 (LCS is "abc")
  • text1="abc", text2="def" → 0 (no common subsequence)

Key Insight: At each position (i, j), we have two choices:

  • If text1[i] == text2[j]: Match! LCS = 1 + LCS(i+1, j+1)
  • If text1[i] != text2[j]: Skip one character → max(LCS(i+1, j), LCS(i, j+1))

// ========================================
// STEP 1: PURE RECURSION
// Time: O(2^(m+n)) - Exponential!
// Space: O(m+n) - Recursion depth
// ========================================
public int longestCommonSubsequence_Step1(String text1, String text2) {
    return lcsRecursive(text1, text2, 0, 0);
}

private int lcsRecursive(String s1, String s2, int i, int j) {
    // BASE CASES: Reached end of either string
    if (i == s1.length() || j == s2.length()) {
        return 0; // No more characters to match
    }

    // CASE 1: Characters match!
    if (s1.charAt(i) == s2.charAt(j)) {
        // Include this character, move both pointers forward
        return 1 + lcsRecursive(s1, s2, i + 1, j + 1);
    }

    // CASE 2: Characters don't match
    // Try skipping character from s1 OR skipping from s2, take maximum
    int skipS1 = lcsRecursive(s1, s2, i + 1, j);
    int skipS2 = lcsRecursive(s1, s2, i, j + 1);
    return Math.max(skipS1, skipS2);
}

// ========================================
// STEP 2: MEMOIZATION (Top-Down DP)
// Time: O(m×n) - Each state calculated once
// Space: O(m×n) - 2D cache + O(m+n) recursion stack
// ========================================
public int longestCommonSubsequence_Step2(String text1, String text2) {
    int[][] memo = new int[text1.length()][text2.length()];
    // Initialize with -1 (meaning "not calculated")
    for (int[] row : memo) {
        Arrays.fill(row, -1);
    }
    return lcsMemo(text1, text2, 0, 0, memo);
}

private int lcsMemo(String s1, String s2, int i, int j, int[][] memo) {
    // BASE CASES
    if (i == s1.length() || j == s2.length()) {
        return 0;
    }

    // CHECK CACHE: Already computed?
    if (memo[i][j] != -1) {
        return memo[i][j]; // ✓ CACHE HIT!
    }

    // COMPUTE
    if (s1.charAt(i) == s2.charAt(j)) {
        memo[i][j] = 1 + lcsMemo(s1, s2, i + 1, j + 1, memo);
    } else {
        int skipS1 = lcsMemo(s1, s2, i + 1, j, memo);
        int skipS2 = lcsMemo(s1, s2, i, j + 1, memo);
        memo[i][j] = Math.max(skipS1, skipS2);
    }

    return memo[i][j];
}

// ========================================
// STEP 3: TABULATION (Bottom-Up DP)
// Time: O(m×n) - Fill entire table
// Space: O(m×n) - 2D DP table
// ========================================
public int longestCommonSubsequence_Step3(String text1, String text2) {
    int m = text1.length();
    int n = text2.length();

    // CREATE TABLE: dp[i][j] = LCS length for text1[i..] and text2[j..]
    // We add +1 to dimensions for base case (empty string)
    int[][] dp = new int[m + 1][n + 1];

    // BASE CASES: dp[m][*] = 0 and dp[*][n] = 0 (already initialized to 0)

    // FILL TABLE: Build from bottom-right to top-left
    // Why? Because dp[i][j] depends on dp[i+1][j+1], dp[i+1][j], dp[i][j+1]
    for (int i = m - 1; i >= 0; i--) {
        for (int j = n - 1; j >= 0; j--) {
            if (text1.charAt(i) == text2.charAt(j)) {
                // Match: include this character
                dp[i][j] = 1 + dp[i + 1][j + 1];
            } else {
                // No match: take max of skipping from either string
                dp[i][j] = Math.max(dp[i + 1][j], dp[i][j + 1]);
            }
        }
    }

    // ANSWER: LCS starting from beginning of both strings
    return dp[0][0];
}

// ========================================
// STEP 4: SPACE OPTIMIZED (1D Array)
// Time: O(m×n) - Same computation
// Space: O(n) - Only 2 rows instead of full 2D table!
// ========================================
public int longestCommonSubsequence_Step4(String text1, String text2) {
    int m = text1.length();
    int n = text2.length();

    // INSIGHT: dp[i][j] only depends on current row and next row
    // We can use just 2 arrays: current and next
    int[] next = new int[n + 1]; // Next row (initially all 0s for base case)
    int[] current = new int[n + 1]; // Current row we're computing

    // FILL TABLE: Bottom to top
    for (int i = m - 1; i >= 0; i--) {
        for (int j = n - 1; j >= 0; j--) {
            if (text1.charAt(i) == text2.charAt(j)) {
                // next[j+1] is like dp[i+1][j+1]
                current[j] = 1 + next[j + 1];
            } else {
                // next[j] is like dp[i+1][j]
                // current[j+1] is like dp[i][j+1]
                current[j] = Math.max(next[j], current[j + 1]);
            }
        }

        // SLIDE WINDOW: Current row becomes next row for next iteration
        int[] temp = next;
        next = current;
        current = temp; // Reuse old 'next' array as 'current'
    }

    // ANSWER: After all iterations, 'next' holds the final row
    return next[0];
}

💡 INTUITION: 2D → 1D SPACE OPTIMIZATION

Key Observation for 2D DP:

  • Look at what dp[i][j] depends on
  • In LCS: dp[i][j] depends on dp[i+1][j+1], dp[i+1][j], dp[i][j+1]
  • All dependencies are from current row or next row (row i+1)
  • We don't need rows i+2, i+3, ... i+100!
  • Solution: Keep only 2 rows at a time, sliding them forward

General Pattern:

  • If dp[i][j] depends only on row i and row i+1 → Use 2 arrays
  • If dp[i][j] depends only on row i and row i-1 → Use 2 arrays
  • If dp[i][j] depends on diagonal (i-1,j-1) and same row → Use 1 array + 1 variable!
⚠️ COMMON MISTAKES: 2D SPACE OPTIMIZATION
  • Wrong Swap Order: Must swap arrays AFTER inner loop completes, not during
  • Overwriting Too Soon: Be careful when using single array - might overwrite values still needed
  • Index Confusion: Remember next[j] is like dp[i+1][j], current[j+1] is like dp[i][j+1]
  • Path Reconstruction: Can't reconstruct actual LCS with space optimization - only length!
  • Wrong Iteration Direction: LCS goes bottom-up. Some problems go top-down. Check dependencies!
🎯 2D → 1D TRANSFORMATION CHECKLIST
  1. Analyze Dependencies: What rows does dp[i][j] access? (current, previous, next?)
  2. Count Rows Needed: If depends on 2 rows → keep 2 arrays
  3. Choose Array Names: prev and current (top-down) OR current and next (bottom-up)
  4. Initialize First: Set up base case in first array
  5. Fill Pattern: Process in correct order (row by row)
  6. Map Indices: dp[i-1][j] → prev[j], dp[i][j] → current[j]
  7. Swap After Row: After inner loop, current becomes prev for next iteration
  8. Return Correct Array: Return from the "current" array after all iterations

📖 PROBLEM STATEMENT

0/1 Knapsack Problem

Given n items, each with a weight weights[i] and value values[i], and a knapsack with capacity W, find the maximum total value you can carry. Each item can be taken once (0/1 choice).

Example:

  • Capacity = 5, weights = [1, 2, 3], values = [60, 100, 120]
  • Best choice: Take item 1 (weight=2, value=100) + item 2 (weight=3, value=120) = 220
  • Cannot take all three (total weight = 6 > 5)

Key Insight: For each item at index i and remaining capacity w:

  • Option 1: Skip item → value = knapsack(i+1, w)
  • Option 2: Take item (if fits) → value = values[i] + knapsack(i+1, w - weights[i])
  • Choose maximum of both options
🎒 ANALOGY: PACKING FOR VACATION

Imagine packing a suitcase with limited space:

  • Each item has importance (value) and takes space (weight)
  • For each item, you decide: "Should I pack this or not?"
  • Goal: Maximize total importance of packed items without exceeding suitcase capacity
  • Constraint: Can't take partial items (either pack it or leave it - that's the "0/1")

This is exactly the knapsack problem - a fundamental optimization challenge!

// ========================================
// STEP 1: PURE RECURSION
// Time: O(2ⁿ) - Each item: take or skip = 2 choices
// Space: O(n) - Recursion depth
// ========================================
public int knapsack_Step1(int[] weights, int[] values, int capacity) {
    return knapsackRecursive(weights, values, 0, capacity);
}

private int knapsackRecursive(int[] w, int[] v, int i, int cap) {
    // BASE CASE: No items left or no capacity
    if (i == w.length || cap == 0) {
        return 0; // No value can be added
    }

    // OPTION 1: Skip current item
    int skipItem = knapsackRecursive(w, v, i + 1, cap);

    // OPTION 2: Take current item (if it fits)
    int takeItem = 0;
    if (w[i] <= cap) {
        // Take this item: add its value + solve for remaining items and capacity
        takeItem = v[i] + knapsackRecursive(w, v, i + 1, cap - w[i]);
    }

    // RETURN: Maximum value from both options
    return Math.max(skipItem, takeItem);
}

// ========================================
// STEP 2: MEMOIZATION (Top-Down DP)
// Time: O(n×W) - Each state (i, capacity) calculated once
// Space: O(n×W) - 2D cache + O(n) recursion
// ========================================
public int knapsack_Step2(int[] weights, int[] values, int capacity) {
    int n = weights.length;
    // memo[i][cap] = max value starting from item i with capacity cap
    int[][] memo = new int[n][capacity + 1];

    // Initialize with -1 (means "not computed")
    for (int[] row : memo) {
        Arrays.fill(row, -1);
    }

    return knapsackMemo(weights, values, 0, capacity, memo);
}

private int knapsackMemo(int[] w, int[] v, int i, int cap, int[][] memo) {
    // BASE CASE
    if (i == w.length || cap == 0) {
        return 0;
    }

    // CHECK CACHE
    if (memo[i][cap] != -1) {
        return memo[i][cap]; // ✓ CACHE HIT!
    }

    // COMPUTE
    int skipItem = knapsackMemo(w, v, i + 1, cap, memo);

    int takeItem = 0;
    if (w[i] <= cap) {
        takeItem = v[i] + knapsackMemo(w, v, i + 1, cap - w[i], memo);
    }

    // CACHE AND RETURN
    memo[i][cap] = Math.max(skipItem, takeItem);
    return memo[i][cap];
}

// ========================================
// STEP 3: TABULATION (Bottom-Up DP)
// Time: O(n×W) - Fill entire table
// Space: O(n×W) - 2D DP table
// ========================================
public int knapsack_Step3(int[] weights, int[] values, int capacity) {
    int n = weights.length;

    // CREATE TABLE: dp[i][w] = max value using items 0..i-1 with capacity w
    // We use (n+1) rows to include base case (0 items)
    int[][] dp = new int[n + 1][capacity + 1];

    // BASE CASE: dp[0][*] = 0 (no items) and dp[*][0] = 0 (no capacity)
    // Already initialized to 0 by default

    // FILL TABLE: Process each item
    for (int i = 1; i <= n; i++) {
        for (int w = 0; w <= capacity; w++) {
            // Option 1: Don't take item i-1 (0-indexed array)
            dp[i][w] = dp[i - 1][w];

            // Option 2: Take item i-1 (if it fits)
            if (weights[i - 1] <= w) {
                int takeValue = values[i - 1] + dp[i - 1][w - weights[i - 1]];
                dp[i][w] = Math.max(dp[i][w], takeValue);
            }
        }
    }

    // ANSWER: Max value with all items and full capacity
    return dp[n][capacity];
}

// ========================================
// STEP 4: SPACE OPTIMIZED (Single Array!)
// Time: O(n×W) - Same computation
// Space: O(W) - Only 1 row! (From O(n×W) to O(W))
// ========================================
public int knapsack_Step4(int[] weights, int[] values, int capacity) {
    int n = weights.length;

    // INSIGHT: dp[i][w] only depends on dp[i-1][w] and dp[i-1][w-weight[i]]
    // Both from PREVIOUS ROW! We only need 1 array if we iterate BACKWARD!
    int[] dp = new int[capacity + 1];

    // BASE CASE: dp[0] = 0 (no capacity), already initialized

    // FILL TABLE: Process each item
    for (int i = 0; i < n; i++) {
        // CRITICAL: Iterate BACKWARD to avoid overwriting values we still need!
        // If we go forward, dp[w-weight] gets updated before we use it
        for (int w = capacity; w >= weights[i]; w--) {
            // Try taking item i
            int takeValue = values[i] + dp[w - weights[i]];
            dp[w] = Math.max(dp[w], takeValue);
            // dp[w] was "previous row value" (skip item)
            // Now dp[w] is "current row value" (best of skip or take)
        }
    }

    // ANSWER: Maximum value at full capacity
    return dp[capacity];
}

💡 INTUITION: WHY BACKWARD ITERATION WORKS

The Key Insight:

  • In 2D: dp[i][w] depends on dp[i-1][w] (skip) and dp[i-1][w-weight] (take)
  • Both dependencies are from the PREVIOUS ROW (row i-1)
  • When we collapse to 1D, dp[w] initially holds the "previous row" value

Forward iteration problem:

  • If we go left-to-right, we update dp[2] first
  • Then when we compute dp[5], we read dp[5-3=2]
  • But dp[2] now has the CURRENT row value, not previous row!
  • We've used the same item twice! ❌

Backward iteration solution:

  • Go right-to-left: w = capacity down to weight[i]
  • When we compute dp[5], we read dp[2]
  • dp[2] hasn't been updated yet, still has previous row value!
  • Perfect! Each item used at most once ✓

Mental Model: Backward iteration ensures we "consume" old values before overwriting them.

🎯 KNAPSACK-SPECIFIC OPTIMIZATION PATTERN

Knapsack is Special:

  • Unlike LCS, knapsack can use single array instead of 2 arrays
  • Requirement: Iterate capacity BACKWARD
  • Why different from LCS? LCS needs both dp[i][j+1] (same row) and dp[i+1][j] (next row)
  • Knapsack: Only needs previous row values, never needs same-row values we already updated

General Rule:

  • If dp[i][w] depends ONLY on row i-1 → Use 1 array with backward iteration
  • If dp[i][w] depends on row i-1 AND same row i → Use 2 arrays
⚠️ COMMON KNAPSACK MISTAKES
  • Forward Iteration in Step 4: CRITICAL ERROR! Will use items multiple times. Always go backward!
  • Off-by-One in Tabulation: dp[i][w] represents items 0..i-1, so use weights[i-1]
  • Not Checking if Item Fits: Must check weights[i] <= w before taking
  • Wrong Base Case: Need dp[0][*] = 0 (no items) AND dp[*][0] = 0 (no capacity)
  • Forgetting to Initialize Memo: Use -1 or create wrapper with initialization

FUNCTION backtrack(state, results):
    // BASE CASE: Found valid solution
    IF is_valid_solution(state):
        results.add(state.copy())
        RETURN

    // TRY all possible choices
    FOR EACH choice IN get_choices(state):
        // 1. CHOOSE
        state.add(choice)

        // 2. EXPLORE
        backtrack(state, results)

        // 3. UNCHOOSE (backtrack)
        state.remove(choice)

TIME: O(b^d) where b=branching, d=depth  |  SPACE: O(d)

📌 Permutations Template

permutations.txt
pseudocode
FUNCTION permute(nums, current, used, results):
    // BASE: All elements used
    IF current.size() == nums.length:
        results.add(current.copy())
        RETURN

    FOR i = 0 TO nums.length - 1:
        IF used[i]:
            CONTINUE  // Skip if already in current permutation

        current.add(nums[i])
        used[i] = TRUE
        permute(nums, current, used, results)
        current.remove_last()
        used[i] = FALSE

⚠️ For duplicates: Sort first, skip if nums[i]==nums[i-1] && !used[i-1]
TIME: O(n! × n)  |  SPACE: O(n)

📌 Combinations/Subsets Template

combinations.txt
pseudocode
FUNCTION subsets(nums, start, current, results):
    results.add(current.copy())  // ✨ Add at EVERY level

    FOR i = start TO nums.length - 1:
        current.add(nums[i])
        subsets(nums, i + 1, current, results)  // i+1: No reuse
        current.remove_last()

VARIATIONS:
- Combinations (k elements): Add only when current.size() == k
- Combination Sum (reuse allowed): Pass i instead of i+1
- Target sum: Track remaining, prune if exceeded

TIME: O(2^n)  |  SPACE: O(n)

📌 Tree DFS Template

tree-dfs.txt
pseudocode
FUNCTION dfs(node):
    IF node IS NULL:
        RETURN base_value

    // Process current node (optional)
    process(node.val)

    // Recurse on children
    left_result = dfs(node.left)
    right_result = dfs(node.right)

    // Combine results
    RETURN combine(node.val, left_result, right_result)

EXAMPLES:
- Max Depth: RETURN 1 + max(left_result, right_result)
- Sum: RETURN node.val + left_result + right_result
- Exists: RETURN left_result || right_result

TIME: O(n)  |  SPACE: O(h) where h=height

📌 Path Sum Template

path-sum.txt
pseudocode
FUNCTION hasPathSum(node, target):
    IF node IS NULL:
        RETURN FALSE

    // Leaf node: Check if sum matches
    IF node.left IS NULL AND node.right IS NULL:
        RETURN target == node.val

    remaining = target - node.val
    RETURN hasPathSum(node.left, remaining) OR
           hasPathSum(node.right, remaining)

💡 FOR COLLECTING ALL PATHS:
   - Use backtracking with path list
   - Add node before recursion
   - Remove after recursion (backtrack)

TIME: O(n)  |  SPACE: O(h)

📌 Tree Construction Template

FUNCTION buildTree(preorder, inorder):
    // Create map for O(1) inorder lookup
    inorder_map = create_index_map(inorder)

    FUNCTION build(pre_start, pre_end, in_start, in_end):
        IF pre_start > pre_end:
            RETURN NULL

        root_val = preorder[pre_start]  // First in preorder
        root_index = inorder_map[root_val]
        left_size = root_index - in_start

        root = NEW TreeNode(root_val)
        root.left = build(pre_start+1, pre_start+left_size,
                         in_start, root_index-1)
        root.right = build(pre_start+left_size+1, pre_end,
                          root_index+1, in_end)
        RETURN root

    RETURN build(0, len-1, 0, len-1)

⚠️ Postorder: root = postorder[post_end] (last element)
TIME: O(n)  |  SPACE: O(n)

📌 1D Memoization Template

1d-memo.txt
pseudocode
FUNCTION solve(n):
    memo = ARRAY of size n+1, fill with -1

    FUNCTION helper(i):
        // BASE CASE
        IF i meets base condition:
            RETURN base_value

        // CHECK CACHE
        IF memo[i] != -1:
            RETURN memo[i]

        // COMPUTE & CACHE
        memo[i] = compute_from_subproblems(helper(i-1), helper(i-2), ...)
        RETURN memo[i]

    RETURN helper(n)

EXAMPLES: Fibonacci, Climbing Stairs, House Robber
TIME: O(n) - Each state computed once  |  SPACE: O(n)

📌 2D Memoization Template

2d-memo.txt
pseudocode
FUNCTION solve(s1, s2):
    memo = 2D ARRAY [len1][len2], fill with -1

    FUNCTION helper(i, j):
        // BASE CASES
        IF i OR j out of bounds:
            RETURN base_value

        // CHECK CACHE
        IF memo[i][j] != -1:
            RETURN memo[i][j]

        // COMPUTE based on choices
        IF s1[i] == s2[j]:
            memo[i][j] = 1 + helper(i+1, j+1)  // Match
        ELSE:
            memo[i][j] = MAX(helper(i+1, j), helper(i, j+1))  // Skip one

        RETURN memo[i][j]

    RETURN helper(0, 0)

EXAMPLES: LCS, Edit Distance, Unique Paths
TIME: O(m×n)  |  SPACE: O(m×n)

Recursion Best Practices
Pro tips for writing clean recursive code
✨ DESIGN PRINCIPLES
  • Trust the recursion: Assume recursive calls work correctly, focus on base case and combination logic
  • Identify subproblems: Ask "How can I make this input smaller?"
  • Define clear base cases: Handle all edge cases explicitly
  • State modification: Pass by value for simplicity, or use backtracking for efficiency
⚠️ COMMON PITFALLS
  • Missing base case: Leads to infinite recursion and stack overflow
  • Wrong base case logic: Off-by-one errors, returning wrong value
  • Not reducing input: Passing same input → infinite recursion
  • Forgetting to backtrack: In permutations/combinations, must undo choices
  • Deep copy vs shallow copy: Use new ArrayList<>(current) when adding to results
  • Exponential without memo: O(2ⁿ) becomes O(n) with caching
💡 OPTIMIZATION TECHNIQUES
  • Memoization: Cache results to avoid recomputation
  • Pruning: Skip branches that can't lead to solution (e.g., sum exceeded)
  • Sorting: Sort input to group duplicates and enable pruning
  • Early termination: Return immediately when solution found
  • Space optimization: Use iterative approach if call stack too deep
🎯 PATTERN RECOGNITION
  • "All possible" → Backtracking (permutations, combinations, subsets)
  • "Optimal" → Memoization/DP (max, min, count)
  • "Tree/Graph traversal" → DFS recursion
  • "Divide input" → Divide and conquer (merge sort, binary search)
  • "Process children first" → Postorder recursion
  • "Overlapping subproblems" → Add memoization
⏱️ TIME COMPLEXITY PATTERNS
  • Linear recursion: T(n) = T(n-1) + O(1) → O(n)
  • Binary recursion: T(n) = 2T(n-1) + O(1) → O(2ⁿ)
  • Divide & conquer: T(n) = 2T(n/2) + O(n) → O(n log n)
  • With memoization: O(2ⁿ) → O(n) or O(n²)
  • Tree recursion: O(n) where n = nodes

Tips for fixing recursive bugs
✨ DEBUGGING STRATEGIES
  • Add Print Statements: Print parameters at function entry/exit
  • Trace Small Inputs: Manually trace n=2 or n=3 on paper
  • Check Base Case: Verify all edge cases handled (empty, null, 0, 1)
  • Verify Reduction: Ensure input gets smaller each call
  • Use Debugger: Step through with small input to see call stack
  • Draw Call Tree: Visualize recursion tree for small input
🔍 COMMON BUG PATTERNS
  • Stack Overflow: Missing/wrong base case, or input not reducing
  • Wrong Answer: Base case returns wrong value, or combination logic flawed
  • Infinite Loop: Not reducing input, or base case never reached
  • Missing Results: Forgot to collect/add result, or wrong pruning
  • Duplicates: Not handling duplicates properly (need sort + skip)
  • Wrong Order: Using wrong traversal (pre/in/post order)

  • ✅ Return values from recursive calls are used?

  • ✅ Backtracking done after exploration?
  • ✅ Memoization added if overlapping subproblems?

Recursion vs Iteration
When to use which approach
⚖️ RECURSION VS ITERATION COMPARISON
Aspect Recursion Iteration
Code Clarity ✅ Often more elegant and intuitive ⚠️ Can be complex for tree/graph problems
Space ⚠️ O(depth) call stack ✅ O(1) usually
Performance ⚠️ Function call overhead ✅ Slightly faster
Stack Overflow Risk ⚠️ Yes, for deep recursion ✅ No
Tree/Graph Problems ✅ Natural fit ⚠️ Need explicit stack/queue
Backtracking ✅ Very natural ❌ Very complex
✅ USE RECURSION WHEN
  • Problem has natural recursive structure (trees, graphs)
  • Backtracking needed (permutations, combinations)
  • Divide and conquer approach fits well
  • Code clarity more important than performance
  • Input size is moderate (won't cause stack overflow)
✅ USE ITERATION WHEN
  • Input size very large (risk of stack overflow)
  • Simple linear processing (loop works fine)
  • Performance critical (avoid function call overhead)
  • Tail recursion (easily converted to iteration)
  • Space is very constrained
💡 TAIL RECURSION → ITERATION
  • Tail Recursion: Recursive call is the last operation
  • Example: return recurse(n-1) with no operation after
  • Conversion: Replace with while loop, update variables instead of recursing
  • Benefit: Some compilers optimize tail recursion automatically
