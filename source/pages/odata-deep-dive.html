What is OData?
📖
The Plain-English Explanation
Before the protocol — the problem it solves and why it exists.

Imagine you have a database of products and customers. You want to expose this data over HTTP so apps, Excel, Power BI, and mobile apps can all consume it. You could build a REST API — but then you have to write logic for filtering, sorting, pagination, field selection, and relationship traversal for every endpoint.

OData (Open Data Protocol) is a standardised way to expose data over HTTP so that the client decides what to query, how to filter, sort, page, and which related data to include — without the server needing custom endpoints for each combination.

Think of it as “SQL over HTTP”. You query a URL almost like writing a SQL SELECT, and the server returns JSON (or XML). The protocol defines the grammar of those queries so every OData server speaks the same language.

Anatomy of an OData URL
Every OData URL has exactly the same structure. Learn this once, read any OData URL forever.
URL Anatomy
Base URL
The root of your OData service.
https://api.example.com/odata/v4/
Resource Path
Entity set name, key, or navigation property.
Products  or  Products(42)  or  Products(42)/Supplier
?
Start of query string
$select
Comma-separated list of properties to return. Like SQL SELECT columns.
$filter
Boolean expression to filter rows. Like SQL WHERE clause.
$orderby
Sort expression. Name asc, Price desc
$top / $skip
Pagination: return top N, skip first N. Always use $orderby with $skip.
$expand
Include related entities inline (eliminates extra round-trips).
$count
$count=true adds @odata.count to response with total matching rows.
$search
Free-text search across searchable properties. Server decides what “searchable” means.
$format
Override Accept header: $format=json or $format=xml
Full Example URL
https://api.example.com/odata/v4/Products?$select=Id,Name,Price&$filter=Price gt 100 and Category eq 'Electronics'&$orderby=Price desc&$top=10&$skip=20&$count=true&$expand=Supplier

$metadata & the Entity Data Model
The self-describing contract of your OData service — what it is, how to read it
📄
What is $metadata?
Hit /$metadata on any OData service to get a machine-readable description of all entities, their types, relationships, and capabilities.

Every OData service exposes a special endpoint: GET /odata/v4/$metadata. It returns an XML document (CSDL — Common Schema Definition Language) that describes:

  • All Entity Types (like tables) with their properties and data types
  • All Navigation Properties (foreign key relationships)
  • All Entity Sets (the URL-addressable collections)
  • All Functions and Actions exposed by the service
  • All Annotations (UI hints, capability declarations, validation rules)
metadata_request.httpHTTP
GET /odata/v4/$metadata HTTP/1.1
Host: api.example.com
Accept: application/xml
200 OKCSDL XML — machine-readable service contract
<?xml version="1.0" encoding="UTF-8"?>
<edmx:Edmx Version="4.0" xmlns:edmx="http://docs.oasis-open.org/odata/ns/edmx">
  <edmx:DataServices>
    <Schema Namespace="MyService" xmlns="http://docs.oasis-open.org/odata/ns/edm">

      <!-- Entity Type = a table definition -->
      <EntityType Name="Product">
        <Key><PropertyRef Name="Id"/></Key>
        <Property Name="Id"       Type="Edm.Int32"    Nullable="false"/>
        <Property Name="Name"     Type="Edm.String"   Nullable="false"/>
        <Property Name="Price"    Type="Edm.Decimal"  Scale="2"/>
        <Property Name="Category" Type="Edm.String"/>
        <Property Name="InStock"  Type="Edm.Boolean"/>
        <Property Name="CreatedAt" Type="Edm.DateTimeOffset"/>
        <!-- Navigation Property = foreign key relationship -->

          Type="MyService.Supplier" Nullable="false"/>
        <NavigationProperty Name="OrderItems"
          Type="Collection(MyService.OrderItem)"/>

</edmx:Edmx>

System Query Options
Every $option explained with real URLs and responses
🔬
$select — choose your columns
Like SQL SELECT. Reduces payload size. Essential for performance.
select.urlURL
# Return only Id, Name, Price — not the whole entity
GET /odata/v4/Products?$select=Id,Name,Price

# Select from expanded navigation property
GET /odata/v4/Products?$expand=Supplier($select=Name,Country)

# Wildcard (v4.01) — all declared properties
GET /odata/v4/Products?$select=*
200 OKOnly selected fields returned
{
  "@odata.context": ".../$metadata#Products(Id,Name,Price)",
  "value": [
    { "Id": 101, "Name": "iPhone 15",  "Price": 79999 },
    { "Id": 102, "Name": "MacBook Air", "Price": 114999 }
  ]
}
📄
$top, $skip, $count — pagination
Server-side pagination. Always combine $skip with $orderby for stable results.
pagination.urlURL
# Page 1: first 5 products
GET /odata/v4/Products?$top=5&$orderby=Id asc

# Page 2: skip first 5, get next 5
GET /odata/v4/Products?$top=5&$skip=5&$orderby=Id asc

# Include total count in response (separate DB query)
GET /odata/v4/Products?$top=5&$count=true&$orderby=Price desc

# Count only — just the number, no data
GET /odata/v4/Products/$count
200 OK$count=true adds @odata.count
{
  "@odata.count": 47,   // total matching, ignores $top/$skip
  "value": [ /* 5 products */ ],
  "@odata.nextLink": "...?$top=5&$skip=5&$orderby=Price desc&$count=true"
}
⚠️
Server-driven paging: Servers can ignore your $top and return fewer results with a @odata.nextLink. Always follow nextLink for complete results — never assume page size.
🔄
$orderby — sorting
Multi-column sort, ascending/descending, sort on navigation property.
orderby.urlURL
# Single column ascending (default)
GET /odata/v4/Products?$orderby=Name

# Descending
GET /odata/v4/Products?$orderby=Price desc

# Multi-column sort
GET /odata/v4/Products?$orderby=Category asc,Price desc

# Sort by navigation property (v4)
GET /odata/v4/Products?$orderby=Supplier/Country asc,Name asc

# Null handling (v4.01)
GET /odata/v4/Products?$orderby=Price desc nulls last
🔎
$search — free-text search
Broad full-text search across all searchable properties. Server defines what “searchable” means.
search.urlURL
# Simple term search
GET /odata/v4/Products?$search=iPhone

# Phrase search (quoted)
GET /odata/v4/Products?$search="MacBook Air"

# Boolean search expressions
GET /odata/v4/Products?$search=iPhone OR MacBook
GET /odata/v4/Products?$search=Apple NOT refurbished

# Combine with $filter
GET /odata/v4/Products?$search=iPhone&$filter=Price lt 80000
💡
$search vs $filter: $search is semantic / relevance-based. $filter is exact/logical matching. Use $search when the user types a search box term. Use $filter for programmatic filtering with exact values.

String Functions
Pattern matching, substring, case operations.
string_filters.urlURL
# Starts with
GET /odata/v4/Products?$filter=startswith(Name, 'Mac')

# Ends with
GET /odata/v4/Products?$filter=endswith(Name, 'Pro')

# Contains (case-sensitive on most servers)
GET /odata/v4/Products?$filter=contains(Name, 'Book')

# Case-insensitive: tolower() trick
GET /odata/v4/Products?$filter=tolower(Name) eq 'iphone 15'

# String length
GET /odata/v4/Products?$filter=length(Name) gt 10

# Substring match (indexOf)
GET /odata/v4/Products?$filter=indexof(Description, 'wireless') gt -1

# Concatenation
GET /odata/v4/Products?$filter=concat(Category, Name) eq 'ElectronicsiPhone 15'

Lambda Operators — any() and all()
Filter collections based on related entity conditions. Most powerful $filter feature.
💡
Lambda operators let you filter an entity based on whether ANY or ALL of its related collection entities match a condition. Essential for filtering orders by their line items, products by their tags, etc.
lambda.urlURL
# any(): orders that have AT LEAST ONE item with price > 10000
GET /odata/v4/Orders?$filter=Items/any(i: i/UnitPrice gt 10000)

# all(): orders where ALL items are in stock
GET /odata/v4/Orders?$filter=Items/all(i: i/Product/InStock eq true)

# any() with nested navigation
GET /odata/v4/Customers?$filter=Orders/any(o: o/Items/any(i: i/Product/Category eq 'Electronics'))

# any() with no predicate = collection is non-empty
GET /odata/v4/Customers?$filter=Orders/any()

# Negate: customers with NO orders
GET /odata/v4/Customers?$filter=not Orders/any()
200 OKCustomers who placed at least one Electronics order

$expand & Navigation Properties
Fetch related data in one request — eliminates N+1 round trips
🔗
What are Navigation Properties?
Typed relationships between entities, defined in $metadata, traversable in URLs.
💡
Navigation Properties are the OData equivalent of SQL JOINs. Defined in $metadata, they let you traverse from one entity to related entities via URL paths or $expand. They replace multiple HTTP round-trips with a single request.
navprop.urlURL
# Traverse navigation property as URL path
GET /odata/v4/Products(101)/Supplier         # single entity nav
GET /odata/v4/Products(101)/OrderItems       # collection nav
GET /odata/v4/Products(101)/Supplier/Country  # property of nav

# Basic $expand — include supplier inline
GET /odata/v4/Products?$expand=Supplier

# Multiple $expand
GET /odata/v4/Products?$expand=Supplier,OrderItems

# Nested $expand (Product > Order > Customer)
GET /odata/v4/Products?$expand=OrderItems($expand=Order($expand=Customer))
200 OKProduct with Supplier expanded inline — no second request needed
{
  "@odata.context": ".../$metadata#Products",
  "value": [
    {
      "Id": 101,
      "Name": "iPhone 15",
      "Price": 79999,
      "Supplier": {         // expanded navigation property
        "Id": 5,
        "Name": "Apple Inc.",
        "Country": "USA"
      }
    }
  ]
}
📦
$expand with nested query options
Filter, select, order, and limit expanded collections.
expand_nested.urlURL
# $expand with $select on the expanded entity
GET /odata/v4/Customers?$expand=Orders($select=Id,OrderDate,TotalAmt)

# $expand with $filter on the expanded collection
GET /odata/v4/Customers?$expand=Orders($filter=Status eq 'delivered')

# $expand with $top on collection (last 3 orders per customer)
GET /odata/v4/Customers?$expand=Orders($orderby=OrderDate desc;$top=3)

# $expand with $count
GET /odata/v4/Customers?$expand=Orders($count=true)

# Full nested options: select + filter + orderby + top
GET /odata/v4/Customers
  ?$select=Id,Name
  &$expand=Orders(
    $select=Id,TotalAmt,Status;
    $filter=Status eq 'delivered';
    $orderby=TotalAmt desc;
    $top=2
  )
# Note: semicolon (;) separates options inside $expand(), not &
200 OKEach customer with their top 2 delivered orders
{
  "value": [
    {
      "Id": 1,
      "Name": "Aditya Kumar",
      "Orders": [
        { "Id": 1002, "TotalAmt": 114999, "Status": "delivered" },
        { "Id": 1001, "TotalAmt": 79999,  "Status": "delivered" }
      ]
    }
  ]
}
⚠️
Performance warning: Deeply nested $expand with large collections causes N+1 problems server-side. Many OData servers limit expand depth (MaxExpansionDepth). Always add $select inside $expand to limit payload. Avoid expanding large collections without $top.

CRUD Operations
Create, Read, Update, Delete — HTTP verbs, headers, ETag concurrency
✏️
CREATE — POST
POST to an entity set creates a new entity. Server generates the key.
create.httpHTTP

}

🔄
UPDATE — PATCH vs PUT
PATCH = partial update (recommended). PUT = full replacement (dangerous — omitted fields become null).
✅ PATCH — partial update
PATCH /odata/v4/Products(109) HTTP/1.1
Content-Type: application/json
If-Match: W/"MjAyNC0wNC0wMQ=="

{
  "Price": 69999,
  "InStock": false
}
// Only Price and InStock updated.
// All other fields unchanged.
❌ PUT — full replacement (risky)
PUT /odata/v4/Products(109) HTTP/1.1
Content-Type: application/json

{
  "Name": "Samsung Galaxy S24",
  "Price": 69999
  // Category, InStock, Supplier not sent
  // → they become null/default!
}
// Returns 204 No Content
🔒
If-Match / ETag (Optimistic Concurrency): Send the ETag you received from GET in the If-Match header of PATCH/DELETE. If another client changed the entity since you read it, the server returns 412 Precondition Failed instead of overwriting their changes. Always use ETags for concurrent edits.
412 Precondition FailedAnother user modified this entity — your ETag is stale
{
  "error": {
    "code":    "PreconditionFailed",
    "message": "The ETag value in the If-Match header does not match the current ETag."
  }
}
🗑️
DELETE — Remove an entity
Simple DELETE on the keyed URL. Use If-Match for safe concurrent deletes.
delete.httpHTTP
DELETE /odata/v4/Products(109) HTTP/1.1
If-Match: W/"MjAyNC0wNC0wMQ=="   // safe concurrent delete

# Cascading delete via nav property
DELETE /odata/v4/Orders(1001)/Items(3) HTTP/1.1
204 No ContentEntity deleted — no body returned
// 204 No Content — success, empty body

Functions & Actions
Custom operations beyond basic CRUD
🧩
Functions vs Actions — the difference
Functions are GET (side-effect free). Actions are POST (may modify data).
FunctionAction
HTTP methodGETPOST
Side effectsNone (safe, idempotent)May modify data
ParametersIn URLIn request body (JSON)
Can be composedYes (in $filter, $orderby)No
ExampleCalculate discount priceProcess payment, send email
🔧
Bound vs Unbound Functions/Actions
Bound = operates on a specific entity/set. Unbound = standalone operation.
functions.urlURL
# Unbound function (standalone, like a static method)
GET /odata/v4/GetTopProducts(count=5)

# Bound function on entity type (like instance method)
GET /odata/v4/Products(101)/GetRelatedProducts()

# Bound function on entity set (operates on the whole collection)
GET /odata/v4/Products/GetProductsByCategory(category='Electronics')

# Function composable with query options
GET /odata/v4/Products/GetMostExpensive()?$select=Name,Price&$top=3

# Unbound action (POST, parameters in body)
POST /odata/v4/SendConfirmationEmail
Content-Type: application/json

$batch — Batch Requests
Multiple operations in one HTTP round-trip. Critical for mobile and SAP Fiori.
📦
Why Batch?
Mobile networks are slow. Each HTTP request has latency overhead. $batch bundles multiple requests into one.
💡
SAP Fiori apps heavily use $batch. When a user saves a complex form with master-detail data, SAP generates a $batch with dozens of operations. This reduces roundtrips from N to 1, dramatically improving perceived performance on mobile networks.

OData Injection Prevention
$filter expressions are user-supplied. Never build them with string concatenation.
❌ Vulnerable — string concatenation
// JavaScript — NEVER do this
const userInput = "x' or 1 eq 1 or Name eq '";
const url = `/odata/v4/Products`
  + `?$filter=Name eq '${userInput}'`;

// Results in:
// $filter=Name eq 'x' or 1 eq 1 or Name eq ''
// Returns ALL products — data leak!
✅ Safe — proper encoding
// Encode single quotes by doubling them
function odataString(val) {
  return "'"
    + val.replace(/'/g, "''")
    + "'";
}

const safe = odataString(userInput);
// Input: x' or 1 eq 1
// Output: 'x'' or 1 eq 1'

// Use OData client libraries:
// SAP UI5: ODataUtils
// .NET: Microsoft.OData.Client
// JavaScript: odata-query npm package

❓ Interview & Real-World Q&A

What is OData and why would you use it over plain REST?
OData is an OASIS-standard protocol for querying and manipulating data over HTTP. Unlike plain REST where every endpoint needs custom filter/sort/pagination logic, OData standardises these operations via URL query options ($filter, $orderby, $top, $skip, $expand). Clients can query what they need without new endpoints. Best for enterprise/BI scenarios, SAP, and data-driven apps.
What is $metadata and why does it matter?
GET /$metadata returns a CSDL XML document describing all entity types, their properties, data types, navigation properties, functions, and actions. It's the service's self-describing contract. Tools like Power BI, SAP UI5, and Microsoft OData clients auto-generate type-safe code from $metadata. It eliminates manual API documentation drift.
What's the difference between $filter and $search?
$filter is exact/logical matching with full boolean expressions: Price gt 100. $search is semantic full-text search across searchable properties — like a search box. Use $filter for programmatic queries with exact values, $search for user-typed search terms. They can be combined.
How do ETags work in OData?
ETags are concurrency tokens. When you GET an entity, you receive @odata.etag. When you PATCH or DELETE, send that value in If-Match: W/"abc". If another client changed the entity since your read, the server returns 412 Precondition Failed instead of silently overwriting. Use If-Match: * to skip the check.
Why does SAP OData v2 require a CSRF token?
SAP NetWeaver Gateway requires a Cross-Site Request Forgery token for all write operations (POST, PATCH, DELETE). First GET the service root with X-CSRF-Token: Fetch to receive a token. Then include that token as X-CSRF-Token: {token} on all write requests. SAP UI5 ODataModel handles this automatically.
What is the any() lambda operator and when do you use it?
any() filters entities based on whether at least one item in a related collection matches a condition. Example: Orders/any(o: o/TotalAmt gt 10000) returns customers who placed at least one order over ₹10,000. Without lambdas, this would require a subquery or client-side filtering.
What is $batch and when should you use it?
$batch bundles multiple HTTP requests into a single multipart HTTP call. Use it when: loading a master-detail page (fetch order + line items + customer in one call), saving a complex form with multiple entity changes, or in mobile apps where latency is high. Operations in a changeset boundary are atomic — all or nothing.
PATCH vs PUT in OData — which should you use?
Always PATCH for updates. PATCH is partial — only the properties you send are updated. PUT replaces the entire entity — omitted properties become null/default, which can silently corrupt data. PUT is only useful when you explicitly want to clear fields by omitting them.
